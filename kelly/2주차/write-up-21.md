## ❓질문

```text
HTTP 1.1 이후로, GET에도 Body에 데이터를 실을 수 있게 되었습니다. 그럼에도 불구하고 왜 아직도 이런 방식을 지양하는 것일까요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

HTTP 최신 스펙인 RFC 9110에 따르면, GET 메서드에 Body를 포함하는 것이 기술적으로 불가능하거나 금지된 것은 아닙니다. 그럼에도 불구하고 실무에서 이 방식을 강력히 지양하는 이유는 '전 세계 웹 인프라 장비들과의 심각한 호환성(Compatibility) 문제'와 **'GET의 본질인 캐싱 메커니즘의 훼손'** 때문입니다.

크게 세 가지 이유로 정리할 수 있습니다.

첫째, **중간 프록시 및 클라이언트 장비의 무조건적인 바디 유실(Drop) 위험** 때문입니다.
클라이언트와 백엔드 서버 사이에는 수많은 공유기, 방화벽, 프록시 서버, CDN, 그리고 로드밸런서가 존재합니다. 이 중 수많은 오래된 레거시 장비들이나 특정 클라이언트 라이브러리들은 여전히 **'GET 요청에는 Body가 없다'는 오래된 관습을 바탕으로 하드코딩**되어 있습니다. 따라서 이 장비들을 거치는 동안 GET의 Message Body가 강제로 삭제(Drop)되거나, 서버가 Body를 읽기 전에 요청을 에러로 판단하고 차단해 버리는 예측 불가능한 장애가 실무에서 빈번히 발생합니다.

둘째, **웹 캐시(Cache) 아키텍처의 한계** 때문입니다.
웹 프록시 서버나 CDN(Cloudflare 등)이 GET의 응답을 캐싱할 때, 기준이 되는 캐시 키(Cache Key)는 오직 'URL 주소'와 '쿼리 스트링'뿐입니다. 만약 GET에 Body를 실어 보내면, Body 안의 데이터가 달라져도 URL 주소가 같다면 캐시 서버는 이를 구별하지 못하고 **이전 유저의 바디 요청에 대한 캐싱된 응답을 잘못 내려주는 치명적인 정합성 오류**가 발생할 수 있습니다.

셋째, **메서드의 시맨틱(의미론) 오염**입니다.
GET은 본질적으로 서버의 상태를 변경하지 않고 리소스를 가져오는 '안전하고 명확한 조회'의 의미를 가집니다. 여기에 굳이 무거운 Body를 싣는 것은 HTTP 프로토콜이 수십 년간 유지해 온 단순성과 직관성을 깨뜨려 팀 협업 및 API 가독성을 떨어뜨립니다.

따라서 실무에서는 검색 조건이 너무 많아 쿼리 스트링 길이 제한(보통 브라우저 기준 2KB~8KB)을 초과하는 대용량 조회가 필요할 경우, GET에 Body를 넣는 위험한 시도 대신 **차라리 POST 메서드를 사용하되 응답 결과에 캐시 제어 헤더를 정교하게 세팅하는 방식을 선택**하여 인프라적 안정성을 확보합니다.

### 꼬리 질문

- 🤗 스포 방지!

  표준 기술의 한계와 실무 호환성, 그리고 대용량 조회 시 POST 우회 전략까지 완벽하게 답변했기 때문에, 면접관은 실제 Spring Boot 환경이나 유명 오픈소스(Elasticsearch 등)의 사례를 들며 한 단계 더 깊은 토론으로 유도할 것입니다.

    - [ ]  방금 대용량 데이터 조회 시 GET 대신 POST를 쓰는 대안을 말씀해 주셨는데, 실제로 전 세계적인 대용량 검색 엔진 오픈소스인 **Elasticsearch**의 API 명세서를 보면 데이터를 조회(Search)할 때 GET과 POST를 둘 다 열어두고 Body를 적극적으로 사용합니다. Elasticsearch가 이런 과감한 설정을 채택할 수 있었던 배경이나 이유는 무엇일까요?
    - [ ]  Spring Boot 환경에서 쿼리 파라미터가 20~30개 이상으로 너무 많아질 때, 이를 컨트롤러 메서드 파라미터에서 하나하나 `@RequestParam`으로 받으면 코드가 매우 지저분해집니다. GET 메서드 환경에서 이 대량의 파라미터들을 우아하게 객체로 바인딩하기 위해 자바/스프링에서 제공하는 메커니즘은 무엇인가요? (예: 쿼리 스트링을 DTO로 바로 받는 `@ModelAttribute` 데이터 바인딩 원리)
    - [ ]  만약 POST 메서드를 조회 목적으로 사용하게 된다면, HTTP 상태 코드는 일반적인 생성 성공인 **201 Created**를 내려주어야 할까요, 아니면 조회 성공인 **200 OK**를 내려주어야 할까요? REST API 관점에서 어울리는 상태 코드와 그 이유를 설명해 주세요.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

이 질문은 "최신 기술 표준(RFC 스펙)이 허용하더라도, 현실 세계의 웹 생태계와 레거시 인프라 장비들이 가진 하드웨어적/소프트웨어적 파편화(Compatibility) 문제를 인지하고 있는가?"를 검증하기 위한 것입니다.

엔지니어는 스펙 문서만 보고 개발하는 게 아니라, 내 코드가 거쳐 갈 전 세계의 다양한 네트워크 장비들과의 **호환성 및 트레이드오프**를 최우선으로 고려해야 함을 알고 있는지 보려는 의도입니다.

### 답변 분석

- **이론(스펙)과 현실(인프라)의 완벽한 분리**: "표준 문서상으로는 되지만, 현실 인프라 장비들이 못 버틴다"라는 명확한 근거를 대어 기술적 깊이를 증명했습니다.
- **캐시 아키텍처의 구체적 메커니즘(Cache Key) 제시**: 단순히 "캐싱에 안 좋다"가 아니라 "캐시 서버는 URL과 쿼리 스트링만으로 캐시 키를 조합하기 때문에 Body가 바뀌어도 오작동한다"는 내부 메커니즘을 정확히 짚었습니다. 면접관이 무릎을 탁 칠 만한 포인트입니다.
- **리스크 발생 시의 현실적인 대안(Best Practice) 제시**: 쿼리 파라미터의 길이 제한이라는 실제 문제 상황을 가정하고, 이때 GET 대신 **POST를 쓰며 캐시 헤더를 조율한다**는 실무 우회 기법까지 깔끔하게 제시했습니다.

### 핵심 키워드

- 중간 레거시 인프라(프록시, 로드밸런서)의 바디 유실(Drop) 리스크
- 캐시 키(Cache Key) 매핑 오류로 인한 데이터 정합성 파괴
- URL 길이 제한 초과 시 POST 우회 설계 전략
