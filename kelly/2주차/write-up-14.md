## ❓질문

```text
200 (ok) 와 201 (created) 의 차이에 대해 설명해 주세요.
```

---
## 😎 답변 및 꼬리 질문

### 답변

**200 OK**와 **201 Created**의 가장 본질적인 차이는 '서버 측에 새로운 리소스가 실제로 생성되었는가'와 '생성된 리소스의 접근 경로를 함께 제공하는가'에 있습니다.

먼저 **200 OK**는 클라이언트의 요청이 성공적으로 처리되었음을 알리는 가장 범용적인 응답 코드입니다. 주로 데이터를 조회하는 `GET` 요청이나, 기존 데이터를 수정하는 `PUT`, `PATCH` 요청의 결과로 사용되며, 응답 본문(Body)에 클라이언트가 요청한 데이터나 처리 결과 메시지가 포함됩니다.

반면 **201 Created**는 클라이언트의 요청으로 인해 **서버에 새로운 리소스가 성공적으로 생성되었음**을 나타내는 특수 목적의 응답 코드입니다. 주로 회원가입, 게시글 작성 같은 `POST` 요청의 결과로 사용됩니다.

특히 201 Created 응답을 내릴 때는 HTTP 규격상 **응답 헤더의 `Location` 필드에 방금 생성된 따끈따끈한 리소스의 식별자나 절대 경로 URI를 반드시 포함하여 반환해야 합니다.** 이를 통해 클라이언트는 body를 파싱하지 않고도 헤더만 보고 새 리소스의 위치를 파악하여 즉시 상세 페이지로 리다이렉트하는 등의 후속 작업을 우아하게 처리할 수 있습니다.

### 꼬리 질문

- 🤗 스포 방지!

  REST API 아키텍처와 성공 응답 규격을 명확히 이해하고 있기 때문에, 면접관은 2xx 클래스의 또 다른 특수 코드나 비동기 처리 상황에서의 응답 설계로 깊게 들어올 것입니다.

    - [ ]  클라이언트가 회원탈퇴를 하거나 게시글을 삭제하는 **`DELETE` 요청**을 보냈고 서버에서 성공적으로 삭제 처리가 끝났다면, 이때는 **200 OK**, **201 Created**, 아니면 또 다른 어떤 응답 코드를 내려주는 것이 가장 RESTful한 설계일까요? (예: 204 No Content 유도)
    - [ ]  만약 대용량 동영상 인코딩 요청이나 대량의 이메일 발송 요청처럼, `POST` 요청을 받았으나 서버 내부에서 백그라운드로 오래 실행되어야 해서 '요청 접수는 성공했지만 아직 리소스 생성이 완료되지는 않은 상태'일 때는 201 대신 어떤 응답 코드를 사용하는 것이 적절할까요? (예: 202 Accepted)
    - [ ]  Spring Boot로 REST 컨트롤러를 개발할 때, 단순 String이나 DTO를 리턴하면 기본적으로 `200 OK`로 나갑니다. 방금 말씀하신 `201 Created` 코드와 함께 `Location` 헤더를 담아서 클라이언트에게 깔끔하게 응답을 내려주려면 Spring에서 제공하는 어떤 객체를 활용하여 코드를 작성해야 하나요? (예: `ResponseEntity.created(URI).body(...)` 사용법)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

모든 성공 요청에 습관적으로 `200 OK`만 반환하는 브론즈/실버 개발자와, HTTP 스펙을 명확히 이해하고 비즈니스 행위에 맞는 정확한 시맨틱(Semantic, 의미론적) 코드를 부여하는 골드 개발자를 구분하기 위한 질문입니다.

면접관이 듣고 싶어 하는 핵심은 "201 Created는 단순히 성공했다는 의미를 넘어 서버에 '새로운 리소스가 물리적으로 생성되었음'을 보장하며, 생성된 리소스의 위치(Location 헤더)까지 클라이언트에게 친절하게 가이드해 주는 규격"이라는 점을 명확히 아는가입니다.

### 답변 분석

- **핵심 기준 제시**: "리소스 생성 여부"와 "접근 경로 제공 여부"라는 두 가지 명확한 기준 축으로 두 코드의 차이를 선명하게 갈랐습니다.
- **HTTP 메서드와의 유기적 매핑**: 200 OK는 `GET/PUT/PATCH`, 201 Created는 `POST`와 주로 엮인다는 실무적 매핑 관계를 명시했습니다.
- **숨겨진 스펙(`Location` 헤더)의 언급**: 실버 답변은 "201은 생성되었을 때 씁니다"로 끝납니다. 골드 답변은 "201 Created는 규격상 `Location` 헤더에 생성된 리소스의 URI를 실어 보내야 한다"는 핵심 스펙을 정확히 짚었습니다. 면접관은 이 대목에서 지원자가 HTTP 표준 규격을 정말 제대로 이해하고 API를 설계하는 사람이라고 확신하게 됩니다.

### 핵심 키워드

- 범용적 성공(200) vs 실제 리소스 생성 완료(201)
- 주로 매핑되는 HTTP 메서드의 차이 (`GET/PUT` vs `POST`)
- 201 응답 시 `Location` 헤더를 통한 새 리소스 URI 제공 의무
