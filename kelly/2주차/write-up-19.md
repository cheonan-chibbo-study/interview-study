## ❓질문

```text
GET과 POST의 차이는 무엇인가요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

**GET**과 **POST**의 가장 근본적인 차이는 '요청의 목적(시맨틱)'과 **'멱등성 여부'**, 그리고 '데이터 전송 방식'에 있습니다.

첫째, **설계 목적과 데이터 전송 방식**이 다릅니다.
**GET**은 서버로부터 리소스를 조회(Read)하기 위한 메서드입니다. 요청 데이터는 URL의 쿼리 스트링(Query String)을 통해 전달되므로 데이터 크기에 제한이 있고 브라우저 히스토리에 그대로 노출됩니다.
반면 **POST**는 서버의 리소스를 생성(Create)하거나 컨트롤러 계층의 비즈니스 로직을 실행하기 위한 메서드입니다. 데이터는 HTTP 요청 본문(Message Body)에 담겨 전송되므로, 대용량 데이터나 미디어 파일을 보낼 수 있고 겉으로 노출되지 않습니다.

둘째, 안전성과 멱등성(Idempotency)의 차이입니다.
**GET은 안전하고 멱등한 메서드**입니다. 호출해도 서버의 상태를 변경하지 않으며, 백 번을 호출해도 항상 같은 조회 결과를 반환합니다. 이 특성 덕분에 브라우저나 CDN 등 웹 인프라에서 **적극적으로 응답 결과를 캐싱(Caching)하여 성능을 최적화**할 수 있습니다.
반면 **POST는 안전하지도 멱등하지도 않습니다.** 호출할 때마다 서버에 새로운 데이터가 계속 추가되므로 서버 상태가 가변합니다. 따라서 POST 요청은 **원칙적으로 캐싱이 불가능**하며, 네트워크 지연으로 인한 중복 요청이 발생하지 않도록 실무에서 별도의 방어 로직(멱등성 키, 중복 클릭 방지 등)을 설계해야 합니다.

마지막으로 흔히 'POST가 Body에 데이터를 담기 때문에 보안상 안전하다'고 오해하지만, HTTP 프로토콜을 평문으로 전송하면 Body 데이터도 패킷 스니핑을 통해 쉽게 탈취될 수 있습니다. 따라서 **진정한 의미의 보안은 GET과 POST의 차이가 아니라, HTTPS 암호화 채널을 적용했는지 여부로 판단**해야 합니다.

### 꼬리 질문

- 🤗 스포 방지!

  기본적인 차이점과 캐싱, 보안, 멱등성까지 모두 다루었기 때문에 면접관은 한 단계 더 파고들어 프로토콜의 하부 메커니즘이나 실제 브라우저의 전송 스펙 차이로 꼬리를 물 것입니다.

    - [ ]  방금 GET은 쿼리 스트링으로, POST는 Body로 보낸다고 하셨는데, 현대 HTTP 스펙(RFC 규격)상 **GET 메서드 요청에 Body를 실어서 보내는 것**이 기술적으로 가능한가요? 만약 보낸다면 서버는 이를 어떻게 처리해야 할까요?
    - [ ]  POST 요청은 비멱등하기 때문에, 웹 브라우저에서 어떤 폼(Form)을 POST로 제출한 뒤 '새로고침(F5)'을 누르면 브라우저가 경고창을 띄우며 재전송 여부를 묻습니다. 이 현상을 방지하기 위해 백엔드에서 사용하는 설계 패턴이 무엇인지 아시나요? (예: PRG - Post/Redirect/Get 패턴 유도)
    - [ ]  일부 대형 프레임워크나 네트워크 라이브러리 중에는 GET은 패킷을 1번에 보내고, POST는 헤더를 먼저 보낸 뒤 TCP ACK를 받고 바디를 보내는 **'2번의 패킷 전송(2-way handshake)'** 구조로 동작한다는 이야기가 있습니다. 이 구조가 실제 성능이나 대역폭에 미치는 영향에 대해 들어보신 적이 있나요?

## 🤔 질문 의도 및 답변 분석

### 질문 의도

"GET은 조회고 POST는 생성이다", "GET은 URL에 데이터가 보이고 POST는 Body에 숨겨져서 안전하다" 수준의 답변은 브론즈와 실버 답변입니다. 특히 'POST가 보안상 안전하다'는 식의 단순 암기는 면접관들이 가장 기피하는 오개념입니다.

면접관이 진짜 기대하는 골드 답변은 **두 메서드의 설계 목적(시맨틱)의 차이를 명확히 짚고, 전송 메커니즘의 차이(쿼리 파라미터 vs Body)와 웹 캐시/인프라 활용 관점, 그리고 앞서 배운 안전성과 멱등성의 개념까지 유기적으로 엮어서 설명하는 것** 입니다.

### 답변 분석

- **다각도의 기준을 통한 대조**: 목적(시맨틱), 전송 방식(URL vs Body), 멱등성, 캐싱 여부 등 여러 축을 기반으로 명쾌하게 비교하여 지식의 스펙트럼이 넓음을 증명합니다.
- **핵심 특성(캐싱)과의 연결**: GET이 멱등하기 때문에 "웹 인프라와 브라우저가 캐싱을 적극적으로 활용할 수 있다"는 백엔드 성능 최적화 관점을 자연스럽게 녹여냈습니다.
- **치명적인 오개념(보안)의 직접적인 교정**: 면접관들이 유도 심문으로 가장 많이 던지는 "POST가 GET보다 보안에 좋죠?"라는 함정을 미리 파악하고, "Body에 담는다고 안전한 게 아니라 HTTPS 암호화가 본질이다"라며 정면으로 짚어주었습니다. 면접관 입장에서 "이 지원자는 제대로 알고 있구나"라며 100점을 줄 수밖에 없는 강력한 고득점 포인트입니다.

### 핵심 키워드

- 조회 및 멱등/캐싱 가능(GET) vs 생성 및 비멱등/캐싱 불가(POST)
- 쿼리 스트링 전송 vs 메시지 바디(Message Body) 전송
- Body 전송은 보안의 본질이 아님 (HTTPS 암호화의 중요성)
