## 😎 답변 및 꼬리 질문

### 답변

HTTP 메서드는 클라이언트가 서버에게 '이 리소스를 가지고 어떤 행위(동사)를 하고 싶은지'를 지시하는 표준 규격입니다. URI가 목적지인 '명사'라면, HTTP 메서드는 행동을 나타내는 '동사' 역할을 합니다. 주요 메서드는 크게 5가지가 있습니다.

- **GET**: 리소스를 조회할 때 사용하며, 본문(Body) 대신 쿼리 파라미터를 통해 데이터를 전달합니다.
- **POST**: 새 리소스를 생성하거나 컨트롤러 계층의 비즈니스 로직을 실행할 때 사용하며, 주로 Body에 데이터를 담아 보냅니다.
- **PUT**: 리소스를 전체 교체(Update)할 때 사용하며, 만약 대상 리소스가 없으면 새로 생성합니다.
- **PATCH**: 리소스의 **일부만 수정**할 때 사용합니다.
- **DELETE**: 특정 리소스를 삭제할 때 사용합니다.

이 메서드들을 설계할 때 백엔드 엔지니어로서 반드시 고려해야 하는 핵심 속성이 두 가지 있습니다.

첫째는 안전성(Safe)으로, 호출해도 서버의 리소스를 변경하지 않는 특성입니다. **GET**이 대표적입니다.
둘째는 멱등성(Idempotent)으로, 동일한 요청을 한 번 보내나 백 번 보내나 **서버의 리소스 상태가 똑같이 유지되는 특성**입니다. **GET, PUT, DELETE는 멱등성을 보장**합니다. 예를 들어 PUT은 같은 데이터로 여러 번 덮어써도 최종 상태가 같고, DELETE 역시 한 번 삭제하나 여러 번 삭제하나 대상이 없는 상태는 동일하기 때문입니다.

반면 **POST는 호출할 때마다 새로운 리소스가 계속 생성되므로 멱등하지 않습니다.** 실무에서는 결제나 주문 같은 non-idempotent한 API를 설계할 때, 중복 요청으로 인한 리스크를 방지하기 위해 '멱등성 키(Idempotency Key)' 등을 도입하여 데이터 정합성을 방지하곤 합니다.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

모든 개발자가 GET, POST, PUT, DELETE가 각각 어떤 역할을 하는지는 알고 있습니다.

브론즈와 실버는 단순 각 메서드의 기능 정의만 나열하고 끝냅니다.

면접관이 진짜 검증하고 싶은 것은 "REST API 설계 원칙에 맞게 각 메서드의 시맨틱(Semantic, 의미론적) 한 차이를 정확히 알고 쓰는가"와, 특히 면접에서 단골로 파고드는 핵심 속성인

- **안전성(Safe)**
- **아이뎀포턴트(Idempotent, 등등성/멱등성)**
- 캐시 가능성(Cacheable)

개념을 명확히 구분하여 설명할 수 있는가입니다.

### 답변 분석

- **명사와 동사의 직관적 비유**: URI(명사)와 HTTP 메서드(동사)의 관계를 명확히 짚어 RESTful API의 기본 철학을 이해하고 있음을 보여줍니다.
- **PUT과 PATCH의 디테일한 차이**: 많은 개발자가 뭉뚱그리는 'PUT(전체 교체)'과 'PATCH(일부 수정)'의 차이를 선명하게 갈랐습니다.
- **안전성과 멱등성(Idempotency)의 정교한 매핑**: 단순히 메서드 종류만 나열하지 않고, **'안전성'과 '멱등성'의 개념을 정확한 예시(PUT, DELETE가 왜 멱등한지)와 함께 서술**했습니다.
- **실무적 결제/주문 이슈 연결**: POST의 무서운 특성(비멱등성)을 짚고, 이를 실무에서 해결하기 위한 **'멱등성 키'** 개념을 슬쩍 언급하여 "이 사람은 대규모 트래픽이나 결제 도메인에서 발생할 수 있는 중복 요청 장애 대책을 알고 있구나"라는 강한 인상을 심어줍니다.

### 핵심 키워드

- URI(명사)와 HTTP 메서드(동사)의 시맨틱 매핑
- PUT(전체 교체) vs PATCH(일부 수정)의 차이
- 안전성(Safe) 및 멱등성(Idempotent)의 수학적/실무적 의미
