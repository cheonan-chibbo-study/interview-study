## ❓질문

```text
필요하다면 저희가 직접 응답코드를 정의해서 사용할 수 있을까요? 예를 들어 285번 처럼요.
```

---
## 😎 답변 및 꼬리 질문

### 답변

기술적으로는 서버가 응답 헤더에 285번이라는 숫자를 임의로 실어 보낼 수 있고, 프론트엔드와 약속만 된다면 작동은 가능합니다. 하지만 **실무 아키텍처 관점에서는 HTTP 표준 규격을 준수해야 하므로 285번 같은 임의의 커스텀 응답 코드를 정의해서 사용하는 것은 지양해야 합니다.**

이유는 웹 생태계가 **표준 HTTP 응답 코드(RFC 규격)를 기반으로 빌드업**되어 있기 때문입니다.

첫째, **중간 인프라 장비의 오작동을 유발**합니다. 클라이언트와 백엔드 서버 사이에는 Nginx 같은 웹 서버, AWS ALB 같은 로드밸런서, 그리고 CDN(Cloudflare 등)이나 프록시 서버 등 수많은 인프라 장비들이 존재합니다. 이 장비들은 285번이라는 표준에 없는 코드를 만나면 에러로 판단해 요청을 차단하거나, 로그 분석 및 모니터링 대시보드에서 정상적인 트래픽 통계를 내지 못하는 리스크가 발생합니다.

둘째, **브라우저의 기본 동작 메커니즘과 충돌**합니다. 브라우저는 2xx는 성공, 3xx는 리다이렉션이라는 큰 틀의 규격에 맞춰 내부 엔진이 동작합니다. 표준에 없는 코드는 브라우저가 다음 동작을 예측할 수 없게 만듭니다.

따라서 실무에서는 임의의 HTTP 상태 코드를 만드는 대신, **상태 코드는 표준 규격(예: 400 Bad Request)을 그대로 준수하여 내리되, 상세한 비즈니스 에러나 세부 상태는 응답 본문(JSON Body) 안에 우리만의 커스텀 코드(예: `code: "CHICKEN-285"`)를 정의하여 전달하는 방식**이 가장 안전하고 올바른 설계라고 생각합니다.

### 꼬리 질문

- 🤗 스포 방지!

  표준 규격과 JSON 바디를 통한 커스텀 에러 설계를 언급했기 때문에, 면접관은 실제 API 규격을 설계할 때의 구체적인 구현체나 표준 스펙에 대한 질문으로 고도화할 것입니다.

    - [ ]  방금 JSON 바디에 커스텀 에러 코드를 담는 대안을 말씀해 주셨는데, 최근 전 세계적으로 REST API의 에러 응답 포맷을 통일하기 위해 제정된 표준 스펙인 **RFC 7807 (Problem Details for HTTP APIs)** 규격에 대해 들어보셨거나 사용해 보신 적이 있나요?
    - [ ]  만약 프론트엔드 개발자가 'HTTP 상태 코드를 매번 확인하기 귀찮으니, 모든 응답(성공이든 실패든)의 HTTP 상태 코드는 전부 **200 OK**로 통일하고, 실제 성공/실패 여부는 JSON 바디의 `isSuccess: false`를 보고 판단하게 해달라'고 요청한다면 백엔드 엔지니어로서 어떻게 대답하시겠습니까?
    - [ ]  표준 HTTP 응답 코드 중에는 아직 정식 표준은 아니지만 특정 기업이나 프레임워크에서 커스텀하게 쓰다가 유명해진 코드들도 있습니다. 혹시 아시는 게 있나요? (예: Cloudflare의 52x 에러 시리즈, 혹은 Spring에서 쓰이는 특수 코드 등)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

기술적으로는 백엔드 개발자가 응답 코드를 285번이든 999번이든 마음대로 설정해서 보낼 수 있습니다.

하지만 면접관이 진짜 보고 싶은 것은 "기술적으로 가능한 것과 아키텍처적으로 올바른 것을 구분할 수 있는가?"입니다.

HTTP 표준 규격을 무시하고 임의의 코드를 만들었을 때, 브라우저나 로드밸런서(Nginx, AWS ALB 등) 같은 외부 인프라 장비들이 이 코드를 인식하지 못해 발생하는 사이드 이펙트(Side Effect)를 인지하고 있는지 검증하려는 것입니다.

### 답변 분석

- **기술적 가능성과 아키텍처적 올바름의 분리**: "작동은 하지만 쓰면 안 된다"라는 명확한 선을 그어, 넓은 시야를 가진 엔지니어임을 증명했습니다.
- **인프라 관점(네트워크 레이어)의 사이드 이펙트 제시**: 단순히 "협업에 안 좋다" 수준을 넘어, **'Nginx, 로드밸런서(ALB), CDN, 프록시'** 등 실제 서버 환경에 존재하는 인프라 장비들이 표준 외 코드를 만났을 때 생기는 모니터링/라우팅 문제를 정확히 짚었습니다.
- **명확한 대안(우아한 설계) 제시**: 무조건 "안 됩니다"로 끝내지 않고, "HTTP 상태 코드는 표준을 유지하되, 세부 상태는 JSON 바디에 담는다"라는 실무에서 실제로 사용하는 대안(Best Practice)을 제시해 완벽한 골드 답변을 완성했습니다.

### 핵심 키워드

- HTTP 표준 규격(RFC 스펙) 준수
- 중간 인프라 장비(Nginx, 로드밸런서, CDN)의 오작동 리스크
- HTTP 상태 코드는 표준으로, 세부 상태는 JSON 바디로 분리
