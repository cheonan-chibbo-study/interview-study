```text
POST와 PUT, PATCH의 차이는 무엇인가요?
```

## 😎 답변 및 꼬리 질문

### 답변

**POST, PUT, PATCH**는 모두 서버의 데이터를 변경하는 메서드이지만, '행위의 목적(시맨틱)'과 **'자원의 변경 방식'**, 그리고 '멱등성 여부'에서 명확한 차이가 있습니다.

첫째, **POST**는 새로운 리소스를 생성(Create)할 때 사용하며, 세 메서드 중 유일하게 **멱등성을 보장하지 않습니다.** 클라이언트가 동일한 POST 요청을 여러 번 보내면 서버에는 매번 새로운 리소스가 중복 생성됩니다. 또한, URI 설계 시 생성될 리소스의 ID를 클라이언트가 알지 못하므로, `/api/chickens`처럼 부모 컬렉션의 URI를 목적지로 삼습니다.

둘째, **PUT**은 리소스를 전체 교체(Replace)할 때 사용하며, **멱등성을 보장합니다.** 요청 시 기존 리소스를 완전히 덮어쓰기 때문에, 동일한 데이터로 수백 번을 호출해도 최종 서버의 리소스 상태는 항상 같습니다. 만약 해당 URI에 대상 리소스가 존재하지 않으면, 새로운 리소스를 생성하는 `Create` 역할을 겸하기도 합니다. 전체 교체 방식이기 때문에, 수정할 필드뿐만 아니라 **기존 리소스의 모든 필드 데이터를 누락 없이 전체 다 보내야 하는 특징**이 있습니다. 만약 특정 필드를 누락하고 보내면, 서버 측 데이터에서 해당 필드가 `null`이나 디폴트 값으로 초기화되어 데이터가 유실되는 장애가 생길 수 있습니다.

셋째, **PATCH**는 리소스의 일부만 수정(Partial Update)할 때 사용하며, **설계 방식에 따라 멱등할 수도 있고 아닐 수도 있습니다.**
필요한 필드만 쏙 골라서 변경하기 때문에 대역폭을 아낄 수 있습니다. 예를 들어 유저의 이름만 바꾼다면 `{"name": "서은"}`과 같이 변경할 데이터만 전송합니다. 값의 대입 형식(`age = 20`)으로 설계된 PATCH는 백 번을 수행해도 결과가 같아 멱등하지만, 기존 값에 의존하는 가변 연산(`age = age + 1`) 형태로 설계된 PATCH는 호출할 때마다 상태가 변하므로 비멱등(Non-idempotent)하게 동작할 수 있습니다.

요약하자면, 새로운 자원 생성은 **POST**, 기존 자원의 완전한 덮어쓰기는 **PUT**, 자원의 특정 일부분만 효율적으로 수정할 때는 **PATCH**를 사용하는 것이 올바른 RESTful API 설계 규칙입니다.

### 꼬리 질문

- 🤗 스포 방지!

  자원의 변경 방식과 멱등성, 그리고 필드 유실 이슈까지 완벽하게 짚었기 때문에 면접관은 실제 백엔드 프레임워크(Spring Boot)와 JPA 환경에서 이 메서드들이 어떻게 매핑되고 작동하는지 기술 스택 관점으로 파고들 확률이 높습니다.

    - [ ]  방금 PUT은 전체 교체라 모든 필드를 다 보내야 하고 안 그러면 `null`이 될 수 있다고 하셨는데, Spring Data JPA 환경에서 **`Repository.save()`** 메서드를 호출할 때 엔티티의 일부 필드가 누락된 DTO를 넘겨주면 내부적으로 어떤 쿼리가 나가며 어떤 문제가 생길까요?
    - [ ]  현대 웹 서비스의 API들을 보면, 리소스를 수정하는 행위(Update)임에도 불구하고 PUT이나 PATCH를 쓰지 않고 그냥 **POST 메서드 하나로 통일해서 처리하는 경우**를 종종 볼 수 있습니다. RESTful한 설정을 포기하면서까지 POST로 수정 API를 만드는 실무적인 장점이나 이유는 무엇이 있을까요?
    - [ ]  PATCH 메서드를 완벽하게 표준 스펙대로 구현하기 위해 RFC 6902에서는 'JSON Patch'라는 표준 규격(`[{"op": "replace", "path": "/name", "value": "서은"}]`)을 제안하고 있습니다. 실무에서 이 표준 형식을 그대로 쓰지 않고 보통 일반 JSON 객체로 PATCH를 처리하는 이유와, 이때 자바 진영에서 누락된 필드와 `null`로 명시된 필드를 구분하기 위해 사용하는 기술(예: `Optional` 처리 혹은 `JsonMergePatch`)에 대해 아시는 대로 설명해 주세요.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

"POST는 생성, PUT과 PATCH는 수정이다" 수준의 단순 암기는 브론즈 답변입니다.

면접관이 진짜 보고 싶은 골드 포인트는 두 가지입니다.

1. **PUT과 PATCH의 리소스 수정 방식의 근본적인 차이**: '전체 교체(PUT)'와 '일부 수정(PATCH)'의 메커니즘을 정확히 이해하고 있는가.
2. **멱등성(Idempotency)의 차이**: 자원 변경 메서드들 사이에서 어떤 메서드가 멱등하고 어떤 메서드가 멱등하지 않은지, 그로 인해 실무에서 어떤 예외 처리가 필요한지 알고 있는가.

### 답변 분석

- **명확한 축을 기반으로 한 비교**: 생성(POST), 전체 교체(PUT), 일부 수정(PATCH)이라는 목적 구분을 첫 문장에서 명확히 가르고 들어갔습니다.
- **PUT의 실무적 위험성(Side Effect) 짚기**: 실버 답변은 단순히 "PUT은 전체 수정입니다"로 끝납니다. 골드 답변은 "일부 필드를 누락하고 보내면 `null`로 덮어써져 데이터가 유실되는 위험이 있다"는 실무적인 트레이드오프와 주의점을 정확히 설명했습니다. 이 부분에서 면접관은 실제 API 설계 트러블슈팅 경험이 있는 개발자라고 판단하게 됩니다.
- **PATCH의 조건부 멱등성 재강조**: PATCH가 대입식일 때와 연산식일 때 멱등성이 어떻게 변하는지 디테일을 놓치지 않고 설명하여 스펙의 깊이를 보여주었습니다.

### 핵심 키워드

- 새로운 자원 생성(POST) vs 전체 교체(PUT) vs 일부 수정(PATCH)
- PUT 호출 시 필드 누락에 따른 데이터 유실 위험성 (`null` 덮어쓰기)
- 멱등(PUT)과 비멱등(POST), 그리고 조건부 멱등(PATCH)의 구분
