## 😎 답변 및 꼬리 질문

### 답변

**Connectionless**가 네트워크 연결 관점에서의 특성이라면, **Stateless**는 애플리케이션 데이터 관점에서의 특성입니다.

먼저 Connectionless(비연결성)는 클라이언트가 요청을 보내고 서버가 응답을 마치면 **그 즉시 TCP/IP 연결을 끊는 특성**입니다. 연결을 계속 유지하지 않기 때문에 서버의 커넥션 자원을 획기적으로 아낄 수 있어, 수많은 클라이언트의 요청을 효율적으로 처리할 수 있습니다.

반면 Stateless(무상태)는 서버가 클라이언트의 **이전 요청 상태나 비즈니스 문맥을 저장하지 않는 특성**입니다. 즉, 방금 전 연결에서 어떤 데이터를 주고받았는지 서버는 기억하지 않으므로, 모든 요청은 독립적이고 완전한 정보를 담고 있어야 합니다.

이 두 특성이 결합하여 HTTP는 서버가 최소한의 커넥션만 유지하면서(Connectionless), 동시에 상태 관리를 위한 메모리 부담도 지지 않게 되어(Stateless), 전 세계적인 대규모 트래픽을 감당할 수 있는 압도적인 수평 확장성(Scale-out)을 얻게 되었습니다.

### 꼬리 질문

  두 개념의 장점과 한계를 모두 언급했기 때문에, 면접관은 실제 이 기술이 발전하면서 생긴 변형이나 성능 최적화 기법에 대해 파고들 것입니다.

    - [ ]  Connectionless가 자원을 아껴준다는 장점이 있지만, 매 요청마다 TCP의 3-Way Handshake를 새로 수행해야 하므로 네트워크 오버헤드가 크다는 단점도 있습니다. 현대 HTTP에서는 이 문제를 어떻게 해결하고 있나요? (예: HTTP Persistent Connection, Keep-Alive 특징)
    - [ ]  Connectionless의 단점을 보완하기 위해 Keep-Alive를 사용해 연결을 유지하게 된다면, 그것은 HTTP의 Connectionless 특성이 깨진 것이라고 보아야 할까요? 아니면 여전히 유지되고 있는 것일까요?
    - [ ]  서버가 Stateless하기 때문에 클라이언트는 매 요청마다 인증 정보(예: JWT 토큰이나 세션 ID)를 HTTP 헤더에 실어 보내야 합니다. 이로 인해 발생하는 보안적 위협(예: 토큰 탈취, XSS, CSRF)에는 어떤 것들이 있으며 어떻게 방어할 수 있을까요?

## 🤔 질문 의도 및 답변 분석

### 질문 의도

많은 지원자가 Stateless(무상태)와 Connectionless(비연결성)를 "둘 다 서버가 기억을 안 하고 연결을 끊는 것 아닌가?"라며 뭉뚱그려 답변합니다. 면접관이 이 질문을 통해 확인하고 싶은 것은 **두 개념이 네트워크의 서로 다른 레이어와 관점(애플리케이션 계층의 '데이터' 관점 vs 전송 계층의 '연결' 관점)에서 정의된다는 점을 정확히 인지하고 있는가**입니다. 그리고 이 두 특성이 결합했을 때 대규모 웹 생태계에서 어떤 시너지를 내는지 이해하고 있는지 검증하려 합니다.

### 답변 분석

- **명확한 기준을 통한 개념 분리**: "Connectionless는 **연결(Network)** 관점, Stateless는 **데이터/상태(Application)** 관점"이라는 명확한 기준선을 먼저 제시했습니다. 이 한 문장만으로도 면접관은 지원자가 두 개념을 혼동하지 않고 정확히 구조적으로 이해하고 있음을 인지합니다.
- **인과관계와 목적 서술**: 각 특성이 아끼는 자원이 무엇인지(Connectionless는 커넥션 자원, Stateless는 서버 메모리 자원)를 구체적으로 짚었습니다.
- **두 개념의 시너지(결론) 제시**: 단순히 두 단어의 사전적 정의를 나열하는 데 그치지 않고, 이 두 특성이 합쳐져서 결국 '대규모 트래픽 처리를 위한 수평 확장성'이라는 웹의 궁극적인 아키텍처적 목적을 어떻게 달성하는지 한 호흡으로 묶어냈습니다.

### 핵심 키워드

- 연결 관점(Connectionless) vs 데이터 관점(Stateless)
- 커넥션 자원 절약 vs 서버 메모리 절약
- 두 특성의 시너지를 통한 수평 확장성(Scale-out)
