## ❓질문

```text
Connectionless의 논리대로면 성능이 되게 좋지 않을 것으로 보이는데, 해결 방법이 있을까요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

지적하신 대로 초기 Connectionless 방식은 매번 TCP 핸드셰이크를 수행해야 하므로 상당한 네트워크 오버헤드가 발생했습니다. 이를 해결하기 위해 HTTP는 구조적 장점은 유지하되, **전송 계층에서의 커넥션 재사용 기술**을 도입하여 성능을 극대화했습니다.

대표적인 해결 방법은 HTTP 지속 연결(Persistent Connection)입니다. HTTP/1.1부터 기본 스펙으로 채택된 이 방식은 헤더에 `Connection: keep-alive`를 설정하여, 한 번 맺은 TCP 커넥션을 일정 시간 동안 유지하며 여러 개의 요청과 응답을 주고받을 수 있도록 만들었습니다.

나아가 **HTTP/2에서는 멀티플렉싱(Multiplexing)** 기술을 도입하여 단 하나의 TCP 커넥션 위에서 여러 개의 요청과 응답을 순서에 상관없이 동시에(병렬로) 주고받을 수 있게 개선했고, 최근 **HTTP/3에서는 TCP 대신 UDP 기반의 QUIC 프로토콜**을 채택하여 핸드셰이크 단계 자체를 단 1-RTT(Round Trip Time) 이하로 줄임으로써 비연결성 구조에서 오는 성능 저하 문제를 완전히 해결하고 있습니다.

### 꼬리 질문

- 🤗 스포 방지!

  네트워크 성능 최적화 기술들을 깊게 다루었기 때문에, 면접관은 각 버전별 핵심 기술의 상세 메커니즘이나 백엔드 애플리케이션 레벨에서의 다른 자원 재사용 기술로 꼬리를 물 가능성이 높습니다.

    - [ ]  HTTP/1.1의 Keep-Alive 방식이 커넥션을 재사용하게 해주지만, 응답이 올 때까지 다음 요청이 대기해야 하는 'HOLB(Head of Line Blocking)' 문제가 여전히 존재합니다. HTTP/2의 멀티플렉싱은 이 HOLB 문제를 구체적으로 어떻게 해결하나요?
    - [ ]  네트워크 단에서 커넥션을 재사용하는 Keep-Alive 기술처럼, 백엔드 애플리케이션 레벨(예: Java/Spring Boot)에서 데이터베이스 연결 비용을 줄이기 위해 이와 유사하게 자원을 재사용하는 기술이 무엇인지 아시나요? (예: DB Connection Pool)
    - [ ]  HTTP/3에서 사용되는 QUIC 프로토콜은 패킷 손실이 발생했을 때 HTTP/2와 비교하여 어떤 장점을 가지나요?

## 🤔 질문 의도 및 답변 분석

### 질문 의도

Connectionless의 치명적인 단점은 매 요청마다 **TCP 3-Way Handshake**와 **4-Way Handshake**를 반복해야 하므로 네트워크 지연(Latency)이 누적된다는 점입니다.

면접관이 이 질문을 통해 확인하고 싶은 것은 "이론적인 비연결성의 한계를 극복하기 위해, HTTP 프로토콜이 버전별로 어떤 성능 최적화 메커니즘(지속 연결, 파이프라이닝, 멀티플렉싱 등)을 도입하며 진화해 왔는가?"를 실무적인 네트워크 지식과 엮어 설명할 수 있는지 확인하는 것입니다.

### 답변 분석

- **질문자의 통찰에 대한 공감 및 문제 구체화**: 질문자가 제시한 '성념 손해'의 원인이 네트워크 단에서의 'TCP 핸드셰이크 반복 비용'임을 명확히 짚고 넘어갔습니다.
- **시간 흐름(버전별)에 따른 완벽한 해결책 나열**:
    - 단순히 "Keep-Alive를 씁니다"로 끝나면 실버 답변입니다.
    - 골드 답변은 HTTP/1.1(Keep-Alive) -> HTTP/2(Multiplexing) -> HTTP/3(UDP/QUIC)의 발전 과정을 일목요연하게 정리하여, 프로토콜 단에서 이 성능 문제를 해결하기 위해 엔지니어들이 어떻게 고도화해 왔는지를 깊이 있게 보여줍니다.
- **개념의 정합성 유지**: 네트워크 연결은 효율적으로 재사용(Keep-Alive)하되, HTTP 요청/응답 단위의 독립성(Stateless, Connectionless의 철학)은 훼손하지 않는 논리를 유지했습니다.

### 핵심 키워드

- HTTP 지속 연결 (Keep-Alive)
- HTTP/2 멀티플렉싱 (Multiplexing)
- HTTP/3 UDP 기반 QUIC 프로토콜
