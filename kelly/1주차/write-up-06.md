## ❓질문

```text
TCP의 keep-alive와 HTTP의 keep-alive의 차이는 무엇인가요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

TCP의 Keep-Alive와 HTTP의 Keep-Alive는 **작동하는 네트워크 계층과 존재 목적에서 명확한 차이**가 있습니다.

먼저 HTTP의 Keep-Alive는 7계층(애플리케이션 계층)에서 동작하며, 목적은 '성능 최적화를 위한 연결 재사용'입니다. 클라이언트와 서버 간에 한 번 맺은 TCP 커넥션을 끊지 않고 유지하여, 연속된 HTTP 요청 시 핸드셰이크 오버헤드를 줄이기 위해 사용됩니다.

반면 TCP의 Keep-Alive는 4계층(전송 계층)에서 동작하며, 목적은 '연결의 유효성 검증 및 유지'입니다. 두 엔드포인트 사이에 한동안 데이터 왕래가 없을 때, '상대방이 아직 살아있는지, 혹은 중간의 방화벽이 커넥션을 강제로 끊지 않았는지' 확인하기 위해 주기적으로 빈 패킷(Probe)을 보내어 **커넥션의 생존 여부를 확인**하는 메커니즘입니다.

요약하자면, HTTP의 Keep-Alive는 **불필요한 핸드셰이크를 줄여 성능을 높이기 위한 기술**이고, TCP의 Keep-Alive는 **연결이 유효한지 감시하고 유실된 커넥션을 정리하기 위한 기술**입니다.

### 꼬리 질문

- 🤗 스포 방지!

  두 Keep-Alive의 차이를 완벽히 파악했다면, 면접관은 이 기술들이 실무 환경(OS 설정, 대규모 트래픽 톰캣 서버 등)에서 어떻게 부작용을 일으키거나 조율되는지 실제 운영 관점의 질문을 던질 것입니다.

    - [ ]  HTTP Keep-Alive를 무한정 길게 잡으면 성능 상 매우 좋을 것 같은데, 실무에서 타이머(`Keep-Alive: timeout=5`)를 두고 일정 시간 뒤에 커넥션을 끊어버리는 이유는 무엇일까요?
    - [ ]  TCP Keep-Alive의 기본 OS 주기(OS 커널 스펙)는 보통 2시간 정도로 매우 길게 잡혀 있습니다. 만약 실무에서 이 주기를 그대로 방치하면, 장애가 발생했을 때 서버 자원(소켓) 관점에서 어떤 문제가 생길 수 있을까요?
    - [ ]  클라이언트가 HTTP 요청을 보내고 물리적인 선이 끊어지거나 갑자기 전원이 꺼지는 'Dying Gasp' 상황이 발생했을 때, 서버는 HTTP와 TCP 중 어떤 Keep-Alive 메커니즘을 통해 이 죽은 연결을 감지하고 정리하게 될까요?

## 🤔 질문 의도 및 답변 분석

### 질문 의도

이 질문은 지원자가 "네트워크 계층 구조(OSI 7 Layer)를 명확히 이해하고, 각 계층이 담당하는 책임과 역할을 구분할 수 있는가?"를 검증하기 위한 것입니다.

이름은 같은 'Keep-Alive'이지만 담당하는 계층, 작동 방식, 그리고 존재하는 목적이 완전히 다릅니다. 이를 구조적으로 분리하여 설명할 수 있다면 네트워크 동작 원리를 완벽하게 꿰뚫고 있다는 강력한 증거가 됩니다.

### 답변 분석

- **계층(Layer)과 목적(Purpose)을 기준으로 한 이분법적 정리**: "HTTP는 7계층/연결 재사용", "TCP는 4계층/생존 확인"이라는 대칭 구조를 명확히 세워 답변의 가독성과 설득력을 극대화했습니다.
- **현상에 대한 구체적 메커니즘 서술**: TCP Keep-Alive를 설명할 때 '데이터 왕래가 없을 때 주기적으로 빈 패킷을 보내 감시한다'는 구체적인 동작 원리를 포함하여 지식의 깊이를 보여줍니다.
- **깔끔한 한 줄 요약**: 면접관이 장황한 설명 속에서 핵심을 놓치지 않도록, 마지막에 두 기술의 본질(성능 최적화 vs 생존 감시)을 한 번 더 임팩트 있게 정리해 주었습니다.

### 핵심 키워드

- 7계층 연결 재사용 (HTTP) vs 4계층 생존 여부 확인 (TCP)
- 핸드셰이크 오버헤드 감소
- 주기적 생존 확인 패킷 (TCP Probe)
