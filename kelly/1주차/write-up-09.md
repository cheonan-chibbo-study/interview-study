## 😎 답변 및 꼬리 질문

### 답변

HTTP/3의 가장 핵심적인 특징은 지난 수십 년간 사용해 온 **TCP 프로토콜을 버리고, UDP 기반으로 개발된 QUIC(Quick UDP Internet Connections) 프로토콜을 전송 계층에 채택했다는 점**입니다. 이 변화를 통해 HTTP/2의 한계를 극복하고 성능을 극대화했습니다.

주요 특징은 크게 세 가지입니다.

첫째, **전송 계층(TCP 레벨)의 HOL Blocking 문제를 완벽히 해결**했습니다. HTTP/2는 하나의 TCP 커넥션을 공유하기 때문에, 단 하나의 패킷만 유실되어도 재전송이 완료될 때까지 전송 계층에서 병목이 생겼습니다. 반면 HTTP/3는 UDP 기반의 QUIC을 사용하여 각 스트림을 완전히 독립적인 패킷으로 처리합니다. 따라서 특정 패킷이 유실되더라도 다른 스트림의 데이터 전송에는 아무런 영향을 주지 않습니다.

둘째, **초기 연결 지연 시간(Latency)의 획기적인 단축**입니다. TCP를 사용할 때는 연결을 맺기 위한 TCP 핸드셰이크와 보안을 위한 TLS 핸드셰이크가 각각 따로 일어나 여러 번의 왕복 시간(RTT)이 필요했습니다. 하지만 QUIC은 **연결 수립과 TLS 암호화 핸드셰이크를 하나의 과정으로 결합**하여, 첫 연결 시 단 1-RTT만 소요되며, 이미 연결했던 서버에 재연결할 때는 **0-RTT**로 요청을 보낼 수 있습니다.

셋째, **모바일 환경에서의 '연결성 이동(Connection Migration)' 보장**입니다. TCP 기반에서는 와이파이(Wi-Fi)를 쓰다가 셀룰러(LTE/5G) 망으로 전환되면 IP 주소가 바뀌므로 기존 TCP 커넥션이 끊어지고 처음부터 다시 핸드셰이크를 해야 했습니다. 반면 HTTP/3는 IP 주소가 아닌 고유한 '커넥션 ID(Connection ID)'로 연결을 식별하기 때문에, 네트워크 환경이 바뀌어도 끊김 없이 부드럽게 통신을 이어갈 수 있는 실무적 강점이 있습니다.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

HTTP/3는 단순히 "이전 버전보다 조금 더 빨라졌다" 수준이 아니라, 웹 탄생 이래 수십 년간 신뢰성의 표준이었던 **TCP를 버리고 UDP를 선택한 거대한 패러다임의 전환**입니다.

면접관이 확인하고 싶은 것은 "HTTP/2가 해결하지 못했던 TCP 레벨의 고질적인 한계(전송 계층 HOLB)를 HTTP/3가 어떻게 UDP(QUIC)를 통해 아키텍처적으로 완전히 해결했는지, 그리고 모바일 중심의 현대 네트워크 환경에서 어떤 실무적 이점을 가져다주는가"를 명확히 이해하고 있는지 검증하는 것입니다.

### 답변 분석

- **핵심 패러다임 전환 선언**: 답변의 시작에서 "TCP를 버리고 UDP 기반의 QUIC을 채택했다"는 본질을 명확히 선언하여 기본 개념을 완벽히 쥐고 흔들고 있음을 보여줍니다.
- **이전 버전(HTTP/2)의 한계와 인과관계 연결**: 왜 UDP로 가야만 했는지에 대한 당위성을 "HTTP/2에 남아있던 전송 계층 레벨의 HOLB"를 통해 논리적으로 풀어냈습니다.
- **실무 및 현대 네트워크 트렌드 반영**: 핸드셰이크 오버헤드를 RTT 개념으로 구체화하고, 현대 모바일 환경에서 사용자가 겪는 가장 흔한 문제인 '망 전환 시 연결 끊김 현상'을 커넥션 ID 기반의 Connection Migration 개념으로 명쾌하게 짚어내어 실무 감각을 어필했습니다.

### 핵심 키워드

- UDP 기반의 QUIC 프로토콜 채택
- 1-RTT / 0-RTT 핸드셰이크 (연결 지연 감소)
- 커넥션 ID 기반의 연결성 이동 (Connection Migration)
