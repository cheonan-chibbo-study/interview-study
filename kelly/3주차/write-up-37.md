```text
TCP와 UDP 중 어느 프로토콜이 Checksum을 수행할까요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

결론부터 말씀드리면 **TCP와 UDP 모두 헤더에 체크섬(Checksum) 필드를 가지고 있으며, 데이터 무결성 검증을 위한 체크섬 연산을 수행합니다.** 다만 두 프로토콜 간에는 '필수 여부'와 '오류 처리 방식'에서 명확한 차이가 존재합니다.

첫째, TCP는 체크섬 수행이 절대적인 필수(Mandatory)입니다.
신뢰성을 생명으로 하는 프로토콜이기 때문에 세그먼트 전송 과정에서 단 1비트의 오염도 허용하지 않습니다. 따라서 송·수신 측 모두 무조건 체크섬을 계산해야 하며, 만약 수신 측에서 체크섬 오류를 감지하면 해당 패킷을 즉시 버린(Drop) 후, TCP의 핵심 메커니즘인 재전송 요구(ARQ)를 통해 송신 측에 깨끗한 패킷을 다시 보내달라고 요청합니다.

둘째, UDP는 전통적인 환경(IPv4)에서 체크섬 수행이 선택(Optional)입니다.
실시간성과 극단적인 속도를 중요시하는 UDP의 특성상, 체크섬을 연산하는 시간조차 아끼고 싶다면 송신 측에서 체크섬 필드를 `0`으로 채워 보냄으로써 검증을 완전히 생략할 수 있습니다.

만약 UDP에서 체크섬을 활성화하여 사용하다가 수신 측에서 오류(비트 오염)를 감지하면, TCP처럼 '다시 보내달라'는 재전송 요청을 결코 하지 않습니다. **오염된 패킷이 상위 애플리케이션 레이어로 올라가 시스템을 망치는 것을 방지하기 위해, 그저 그 자리에서 조용히 패킷을 파기(Drop)하고 끝냅니다.** 참고로 최신 인프라 표준인 IPv6 환경에 이르러서는 UDP에서도 체크섬 수행이 필수(Mandatory)로 스펙이 변경되었습니다. IPv6에서는 하부 레이어(IP 계층)의 헤더 체크섬이 아예 삭제되었기 때문에, 최소한의 패킷 오염을 방지하기 위해 UDP 레이어에서의 체크섬을 강제하도록 아키텍처가 진화했기 때문입니다.

### 꼬리 질문

- 🤗 스포 방지!

  체크섬의 필수성 여부와 IPv6 인프라적 제약까지 완벽하게 파고들었기 때문에, 면접관은 UDP의 이러한 '기능 없음'을 소프트웨어적으로 어떻게 보완하는지 실무 레이어 질문으로 확장할 것입니다.

    - [ ]  방금 UDP는 체크섬 에러가 나면 재전송 없이 조용히 패킷을 드롭(Drop)한다고 하셨습니다. 그렇다면 우리가 앞서 공부했던 **UDP 기반의 HTTP/3(QUIC)** 환경에서는 패킷이 유실되거나 드롭되었을 때, 웹 통신의 신뢰성을 지키기 위해 이 유실된 패킷을 누가, 어떻게 재전송하게 만드나요? (예: 전송 계층이 아닌 애플리케이션/QUIC 소스 코드 레이어에서의 재전송 메커니즘 구현 유도)
    - [ ]  UDP 체크섬 연산을 할 때, 전송 계층(L4)의 헤더와 데이터뿐만 아니라 네트워크 계층(L3)의 IP 주소 정보 일부를 가져와서 함께 연산하는 **'의사 헤더(Pseudo Header)'** 개념이 들어갑니다. 전송 계층 프로토콜이 왜 굳이 하위 레이어인 IP 헤더의 정보까지 끌어와서 체크섬을 구하는 것일까요? (예: 잘못된 호스트로 배달되는 라우팅 오류까지 이중 검증하기 위함)
    - [ ]  실무에서 실시간 온라인 게임(예: FPS 게임)의 네트워크 엔진을 설계할 때, 캐릭터의 이동 좌표 데이터는 UDP 체크섬을 켜서 오염 시 드롭하는 게 맞을까요, 아니면 체크섬을 끄고 오염된 채로라도 빠르게 받는 게 맞을까요? 엔지니어로서의 트레이드오프 판단을 설명해 주세요.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

많은 지원자가 "UDP는 신뢰성이 없으니까 체크섬 같은 검증 연산도 안 할 것이다"라고 잘못 추측하여 함정에 빠집니다.

면접관이 확인하고 싶은 것은 "UDP가 비록 최소한의 기능만 가진 프로토콜이지만, 최소한의 '오염된 패킷 필터링'을 위해 체크섬 스펙을 열어두었다는 점을 정확히 아는가"와 "TCP에서는 체크섬이 필수(Mandatory)인 반면, UDP(특히 IPv4 환경)에서는 선택(Optional)적으로 꺼둘 수 있다는 프로토콜 규격의 차이"를 완벽히 인지하고 있는지 검증하는 것입니다.

### 답변 분석

- **함정 정면 돌파 및 팩트 체크**: "둘 다 수행한다"고 명쾌하게 두괄식 답변을 던져 면접관의 함정 질문을 완벽히 방어했습니다.
- **프로토콜별 '필수(Mandatory) vs 선택(Optional)' 대조**: TCP는 무조건 필수이지만, UDP(IPv4)는 필드를 0으로 밀어버리면 생략 가능하다는 **프로토콜 바이너리 레벨의 디테일**을 정확히 짚었습니다.
- **오류 대처 메커니즘의 차이 재강조**: 오류 발생 시 TCP의 '재전송(신뢰성 보장)'과 UDP의 '조용한 파기(비신뢰성 유지)'를 명확히 대조하여 두 프로토콜의 철학을 일관되게 관통했습니다.
- **최신 인프라 트렌드(IPv6 스펙) 제시**: 앞선 질문의 빌드업을 활용해, **IPv6 환경에서는 UDP 체크섬도 필수가 되었다는 최신 네트워크 규격 명세**를 덧붙여 전공 지식의 깊이가 독보적임을 증명했습니다.

### 핵심 키워드

- TCP, UDP 모두 체크섬 필드 내장 (둘 다 수행 가능)
- TCP = 무조건 필수 / UDP(IPv4) = 0으로 채워 생략 가능(Optional)
- IPv6 환경에서는 UDP 체크섬도 필수(Mandatory)로 규격 격상
