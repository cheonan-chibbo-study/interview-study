## 😎 답변 및 꼬리 질문

### 답변

CORS(Cross-Site Resource Sharing)는 'SOP 정책으로 인해 차단된 교차 출처 간의 리소스 공유를 안전하게 허용하기 위해, 브라우저와 서버가 HTTP 헤더를 통해 상호 협약하는 웹 표준 메커니즘'입니다.

CORS의 본질은 에러가 아니라 'SOP라는 기본 방어벽을 안전하게 우회할 수 있도록 허용하는 화이트리스트 정책'입니다. 브라우저는 다른 출처로 비동기 요청을 보낼 때 안전성을 검증하기 위해 크게 3가지 방식으로 동작합니다.

첫째, 가장 대표적인 **예비 요청 (Preflight Request)** 방식입니다.
브라우저는 실제 본 요청을 보내기 전, HTTP **`OPTIONS` 메서드**를 사용해 타겟 서버로 '예비 요청'을 먼저 날립니다. 이때 헤더에 자신의 출처를 알리는 `Origin`과 본 요청에서 사용할 `Access-Control-Request-Method` 등을 실어 보냅니다. 서버가 이 요청을 받고 '너의 출처와 메서드를 허용한다'는 응답인 **`Access-Control-Allow-Origin`** 헤더를 돌려주면, 브라우저는 그제야 안전함을 인지하고 진짜 본 요청(`GET`, `POST` 등)을 전송합니다.

둘째, **단순 요청 (Simple Request)** 방식입니다.
예비 요청을 생략하고 곧바로 본 요청을 보낸 뒤, 서버의 응답 헤더를 보고 브라우저가 사후 판단하는 방식입니다. 다만 이 방식은 조건이 매우 까다롭습니다. HTTP 메서드가 `GET`, `HEAD`, `POST` 중 하나여야 하고, 헤더 역시 `Accept`, `Content-Type` 등 전통적인 기본 헤더만 허용되며, 결정적으로 `Content-Type`은 `application/json`이 아닌 `application/x-www-form-urlencoded`나 `multipart/form-data` 등 클래식한 포맷만 허용됩니다. 따라서 **현대적인 JSON 기반 REST API 환경에서는 대부분 Preflight 방식으로 동작**하게 됩니다.

셋째, **인증된 요청 (Credentialed Request)** 방식입니다.
쿠키나 Authorization Bearer 토큰 등 인증 정보를 요청 헤더에 싣고 다른 출처의 API를 호출할 때 사용됩니다. 이때 브라우저는 보안 강화를 위해 서버 측에 더욱 엄격한 조건을 요구합니다. 백엔드 서버는 반드시 응답 헤더에 `Access-Control-Allow-Credentials: true`를 명시해야 하며, 이때 허용 출처를 뜻하는 **`Access-Control-Allow-Origin` 헤더의 값에 와일드카드(`*`)를 절대 사용할 수 없고 명확한 단 하나의 출처 도메인을 주소로 특정**해야만 브라우저 차단벽이 해제됩니다.

결론적으로 CORS 에러를 해결한다는 것은, 무작위로 설정을 끄는 것이 아니라 **백엔드 서버가 브라우저에게 '우리 API는 저 프론트엔드의 출처(Origin)를 신뢰하니 데이터를 넘겨주어도 좋다'고 HTTP 응답 헤더를 정교하게 세팅해 주는 과정**을 의미합니다.

  CORS의 내부 동작 스펙과 세 가지 유형을 완벽하게 관통했기 때문에, 면접관은 실제 스프링 환경에서의 구현체나 인프라(프록시)를 활용한 우회 전략으로 실전 능력을 시험할 것입니다.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

많은 주니어 개발자가 CORS를 "프론트엔드와 백엔드 포트가 다르면 발생하는 에러" 정도로 단순하게 대답하고, 해결책도 그저 외부 라이브러리나 스프링 어노테이션(@CrossOrigin)을 복사 붙여넣기 하는 수준에 그칩니다.

면접관이 확인하고 싶은 골드 포인트는 "CORS가 사실 에러가 아니라 '브라우저와 서버가 협력하여 안전하게 리소스를 공유하기 위한 표준 프로토콜 규격'임을 인지하고 있는가"와 "단순 요청(Simple Request), 예비 요청(Preflight), 인증된 요청(Credentialed)이라는 3가지 핵심 동작 흐름을 정확히 이해하고 제어할 수 있는가"입니다.

### 답변 분석

- **CORS의 철학적 정의 수정**: "CORS는 에러다"라는 흔한 편견을 깨고, "SOP를 극복하기 위해 브라우저와 서버가 협약하는 표준 프로토콜이자 화이트리스트 제도"라는 완벽한 아키텍처적 정의를 내렸습니다.
- **3대 동작 메커니즘의 완벽한 분화**: Preflight(`OPTIONS` 메서드 활용), Simple Request(엄격한 조건과 JSON 환경에서의 한계), Credentialed Request(쿠키 연동 및 와일드카드 금지 규칙)라는 세 가지 동작 스펙을 완벽하게 쪼개어 설명했습니다.
- **실무적 주의점(와일드카드 제약) 명시**: 인증 정보 연동 시 백엔드가 흔히 실수하는 **`withCredentials`와 `Access-Control-Allow-Origin: *` 충돌 장애 현상**을 정확하게 언급하여 실무 설계 역량을 증명했습니다.

### 핵심 키워드

- OPTIONS 메서드를 활용한 예비 요청 (Preflight)
- 현대 JSON API 환경에서 Simple Request가 작동하기 어려운 이유
- 인증 정보 포함 시 `Allow-Credentials` 활성화 및 와일드카드() 사용 금지 규칙
