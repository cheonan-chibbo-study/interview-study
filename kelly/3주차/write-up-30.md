## 😎 답변 및 꼬리 질문

### 답변

**XSS**와 **CSRF**의 가장 본질적인 차이는 '공격의 대상(타겟)'과 '공격자가 유저의 인증 토큰이나 쿠키를 직접 쥐고 흔들 수 있는가'에 있습니다.

첫째, XSS(Cross-Site Scripting)는 **'사용자(클라이언트)'를 타겟으로 하는 공격**입니다.
웹사이트에 악성 자바스크립트 코드를 삽입하여, 이를 방문한 다른 유저의 브라우저에서 코드가 강제로 실행되게 만듭니다. 이 과정에서 공격자는 유저의 세션 ID 쿠키나 JWT 토큰을 직접 탈취(Read)하여 빼내 오거나, 화면을 위조하는 등 클라이언트 측에서 악의적인 행위를 직접 제어할 수 있습니다.

둘째, CSRF(Cross-Site Request Forgery)는 **'서버'를 타겟으로 하는 공격**입니다.
공격자는 사용자의 인증 정보나 토큰을 **직접 빼내 오거나 읽을 수 없습니다.** 대신, 사용자가 이미 특정 사이트(예: 은행)에 로그인되어 있어 **브라우저에 인증 쿠키가 남아있는 상태**를 악용합니다. 공격자가 만든 악성 낚시 사이트에 유저가 접속하면, 유저의 브라우저가 유저도 모르게 은행 서버로 'A계좌로 1억 원 송금'과 같은 무단 요청(Request)을 강제로 보내게 만듭니다. 은행 서버는 브라우저가 자동으로 실어 보낸 인증 쿠키를 보고 유저가 직접 보낸 정상 요청으로 오인하여 승인해 버립니다.

요약하자면, XSS는 악성 스크립트를 실행해 유저의 정보를 '탈취(Read)'하는 것이 목적이고, CSRF는 유저의 권한을 도용해 유저 몰래 서버에 특정 명령을 '수행(Write)'하게 만드는 것이 목적입니다.

따라서 백엔드에서의 방어 대책도 다릅니다. XSS는 입력값의 특수문자를 치환하는 **HTML 에스케이핑 필터**나 쿠키의 **`HttpOnly` 속성**으로 방어하지만, CSRF는 난수 토큰을 검증하는 **CSRF Token 메커니즘**이나 쿠키의 **`SameSite` 속성**을 사용해 다른 도메인에서 출발한 무단 요청을 차단하는 방식으로 방어해야 합니다.

### 꼬리 질문

- 🤗 스포 방지!

  XSS와 CSRF의 구조적 메커니즘 차이를 완벽히 분리해 냈기 때문에, 면접관은 CSRF의 방어 대책인 `SameSite` 속성의 디테일이나 최신 아키텍처 환경에서의 CSRF 유효성으로 파고들 것입니다.

    - [ ]  방금 CSRF 방어를 위해 쿠키의 **`SameSite` 속성**을 언급해 주셨습니다. `SameSite` 속성에 부여할 수 있는 3가지 옵션(`Strict`, `Lax`, `None`)이 각각 크로스 사이트 요청을 어떻게 통제하는지 설명해 주세요. (최신 크롬 브라우저의 디폴트 정책 포함)
    - [ ]  Spring Security 프레임워크를 쓰면 기본적으로 **CSRF 방어 필터(`CsrfFilter`)가 활성화**되어 있습니다. 그런데 실무에서 완전히 무상태(Stateless)인 **JWT 토큰 인증 방식**을 채택하거나 **REST API 서버**를 빌드할 때는 많은 백엔드 엔지니어들이 `.csrf().disable()` 코드를 통해 CSRF 방어를 꺼버리곤 합니다. REST API와 JWT 환경에서는 왜 CSRF 공격으로부터 상대적으로 안전하다고 판단하는 것일까요? (예: 쿠키 자동 전송 메커니즘의 배제)
    - [ ]  만약 우리 웹 서비스에 **XSS 취약점**이 뚫려서 해커가 마음대로 자바스크립트를 실행할 수 있는 상태가 되었다면, 우리가 서버에 구축해 둔 **CSRF Token 방어벽**은 무력화될까요, 아니면 여전히 안전할까요? (예: XSS가 뚫리면 CSRF 방어도 동시 붕괴되는 인과관계 유도)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

많은 개발자가 이름의 유사성(Cross-Site...) 때문에 두 개념을 뭉뚱그려 암기하곤 합니다.

면접관이 확인하고 싶은 골드 포인트는 "XSS는 '사용자 브라우저에 악성 코드를 실행'하는 클라이언트 타겟 공격인 반면, CSRF는 '사용자의 인증된 권한을 도용하여 서버에 무단 요청을 전송'하는 서버 타겟 공격"이라는 본질적인 메커니즘의 차이를 정확히 구별하여 설명할 수 있는가입니다.

### 답변 분석

- **가장 선명한 대조 기준 제시**: "XSS는 탈취(Read/클라이언트 타겟)", "CSRF는 무단 수행(Write/서버 타겟)"이라는 핵심 요약을 명확히 선언하여 기본기를 단단하게 보여줍니다.
- **공격자의 리소스 접근 권한 차이 설명**: CSRF에서 공격자는 유저의 쿠키 내용을 결코 읽을 수 없고, '자동 탑재 기능'만 이용한다는 미묘하고도 중요한 기술적 한계를 정확히 짚었습니다. (실버 답변은 이를 구별하지 못함)
- **방어 대책의 정교한 연결**: 두 취약점의 메커니즘이 다른 만큼, 백엔드 엔지니어로서 꺼내 들어야 하는 방어 솔루션(`HttpOnly` vs `SameSite`, `CSRF Token`)을 1:1로 깔끔하게 매핑하여 실무 아키텍처 역량을 증명했습니다.

### 핵심 키워드

- 유저 스크립트 실행 및 정보 탈취(XSS) vs 유저 권한 도용 및 무단 명령 수행(CSRF)
- 쿠키 값을 직접 가로챔(XSS) vs 쿠키 값은 읽지 못하고 브라우저 자동 전송만 악용(CSRF)
- HTML 에스케이핑(XSS 방어) vs CSRF Token & SameSite 쿠키(CSRF 방어)
