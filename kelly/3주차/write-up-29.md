## 😎 답변 및 꼬리 질문

### 답변

XSS(Cross-Site Scripting)는 공격자가 취약한 웹사이트에 **악성 자바스크립트 스크립트를 삽입하여, 이 웹사이트를 방문하는 '다른 무고한 사용자의 브라우저'에서 해당 스크립트가 강제로 실행되도록 만드는 클라이언트 사이드 공격 기법**입니다.

이 공격을 통해 해커는 피해자의 세션 ID 쿠키를 탈취하여 계정을 가로채거나, 사용자의 페이지를 변조하고, 악성 유포지로 리다이렉트시킬 수 있습니다. XSS는 크게 3가지 유형으로 분류됩니다.

첫째, **Stored(저장형) XSS**입니다.
가장 위험한 형태로, 공격자가 게시판 댓글이나 프로필 영역에 악성 스크립트를 작성하여 **서버의 데이터베이스에 영구적으로 저장**시키는 방식입니다. 이 게시글을 조회하는 모든 유저의 브라우저에서 스크립트가 실행되므로 피해 규모가 가장 큽니다.

둘째, **Reflected(반사형) XSS**입니다.
서버에 스크립트를 저장하지 않고, **URL의 쿼리 파라미터 등에 악성 스크립트를 포함시켜 서버에 요청하면, 서버가 이를 그대로 응답 화면에 '반사'하여 출력**할 때 발생합니다. 공격자는 악성 스크립트가 포함된 링크를 피싱 이메일 등으로 유저에게 클릭하도록 유도하여 공격을 수행합니다.

셋째, **DOM-based(DOM 기반) XSS**입니다.
서버와 상관없이, 브라우저 측에서 자바스크립트가 URL의 해시값 코드 등을 파싱하여 **DOM(Document Object Model) 구조를 동적으로 안전하지 않게 조작할 때 브라우저 내부에서 발생하는 취약점**입니다.

백엔드 엔지니어로서 XSS를 방어하기 위한 핵심 대책은 다음과 같습니다.

1. **HTML 에스케이핑(Escaping) 필터 적용**: 사용자가 입력한 문자열 중 `<, >, &, ", '` 같은 특수문자를 문자가 아닌 HTML 엔티티(예: `&lt;, &gt;`)로 치환하여 브라우저가 이를 코드가 아닌 단순 텍스트로 인식하게 만듭니다. Spring Boot 환경에서는 네이버의 **Lucida 네이버 XSS Filter** 같은 오픈소스를 서블릿 필터 레이어에 등록하여 모든 요청의 텍스트를 전처리합니다.
2. **안전한 HTML 렌더링 라이브러리 사용**: 만약 마크다운 에디터처럼 의도적으로 HTML 태그를 허용해야 한다면, 무조건 허용하는 것이 아니라 **네이버 HTML Sanifier**나 **OWASP Java HTML Sanitizer**를 사용해 `script`, `onerror` 같은 위험한 태그와 속성만 쏙 골라내는 화이트리스트 기반 새니타이징(Sanitizing)을 수행해야 합니다.
3. **쿠키 보안 및 CSP 설정**: 앞서 세션 로그인에서 언급한 쿠키의 **`HttpOnly` 속성**을 활성화하여 자바스크립트가 `document.cookie`로 세션 ID를 탈취하는 것을 원천 차단하고, HTTP 응답 헤더에 CSP(Content Security Policy)를 설정하여 브라우저가 지정된 안전한 도메인의 스크립트만 실행하도록 강제합니다.

### 꼬리 질문

- 🤗 스포 방지!

  CSRF랑 XSS는 어떤 차이가 있나요?

  XSS는 프론트엔드에서만 막을 수 있나요?

  XSS의 원리와 유형, 그리고 백엔드/인프라적 방어 수단까지 완벽하게 스케치했기 때문에, 면접관은 또 다른 대표적인 웹 취약점인 CSRF와의 비교나 구체적인 프레임워크의 동작 원리로 깊게 들어올 것입니다.

    - [ ]  XSS와 이름이 비슷해서 자주 비교되는 웹 취약점으로 CSRF(Cross-Site Request Forgery, 사이트 간 요청 위조)가 있습니다. 이 둘의 공격 목적과 메커니즘의 결정적인 차이는 무엇인가요? (예: 유저의 권한 탈취/스크립트 실행 vs 유저의 권한을 이용한 무단 요청 전송)
    - [ ]  최근 현대 프론트엔드 프레임워크인 **React**나 **Vue.js** 등은 기본적으로 데이터 바인딩을 할 때 텍스트를 자동으로 에스케이핑(Escaping)해 주어 XSS에 상대적으로 안전하다고 합니다. 그럼에도 불구하고 프론트엔드 단에서 의도적으로 이 방어벽을 깨부수고 XSS 취약점을 만들어내는 위험한 함수나 속성에는 어떤 것들이 있을까요? (예: React의 `dangerouslySetInnerHTML`, Vue의 `v-html`)
    - [ ]  사용자 입력을 받을 때 **'검증(Validation)'** 단계에서 스크립트 문자열(`<script>`)이 들어오면 에러를 뱉으며 요청을 차단하는 방식(Blacklist)과, 일단 받고 저장/출력할 때 '치환(Escaping)'하는 방식 중 XSS 방어 관점에서 어떤 쪽이 더 안전하고 올바른 아키텍처일까요? (예: 우회 공격 스펙을 고려한 Escaping의 당위성)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

"악성 스크립트를 삽입해서 공격하는 것이다"라는 단순 정의는 브론즈 답변입니다.
면접관이 진짜 확인하고 싶은 골드 포인트는 세 가지입니다.

1. **공격의 본질**: XSS는 서버를 직접 해킹하는 것이 아니라, **서버를 매개체로 삼아 '무고한 다른 사용자의 브라우저(클라이언트)'를 공격하는 기법**임을 인지하고 있는가.
2. **XSS의 3가지 유형**: Stored, Reflected, DOM-based XSS의 메커니즘 차이를 구분할 수 있는가.
3. **실무적 방어 대책**: 백엔드 개발자로서 입력값 검증과 에스케이핑(Escaping), 그리고 보안 헤더를 통해 이를 어떻게 원천 차단하는지 알고 있는가.

### 답변 분석

- **공격 대상의 명확한 정의**: XSS가 데이터베이스를 파괴하는 공격이 아니라 **'타인의 브라우저'를 오염시키는 공격**이라는 본질을 완벽히 선언했습니다.
- **유형별 뼈대 파악**: Stored, Reflected, DOM-based의 트리거 원인(DB 저장 여부, URL 반사 여부, 브라우저 자바스크립트 자체 처리 여부)을 명쾌하게 나누어 비교했습니다.
- **실무 기술 셋(오픈소스 명칭) 제시**: 단순히 "필터를 씁니다"가 아니라, 자바 진영과 현업에서 표준적으로 사용하는 **`Naver XSS Filter`, `OWASP Sanitizer`** 같은 구체적인 구현체 명칭을 던져 실무 역량을 입증했습니다.
- **다중 방어벽(Defense in Depth) 아키텍처 제시**: 백엔드의 코드 레벨 방어(Escaping, Sanitizing)에 그치지 않고, 브라우저가 제공하는 인프라 레벨의 방어벽인 `HttpOnly 쿠키`와 `CSP(콘텐츠 보안 정책) 헤더`까지 아울러 설명함으로써 빈틈없는 완벽한 보안 의식을 보여주었습니다.

### 핵심 키워드

- 타인의 브라우저를 타겟팅하는 클라이언트 사이드 스크립트 주입 공격
- 3대 유형 (Stored, Reflected, DOM-based)
- 다중 방어 (HTML 에스케이핑 필터, HttpOnly 쿠키, CSP 헤더)
