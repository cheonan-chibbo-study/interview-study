## ❓질문

```text
XSS는 프론트엔드에서만 막을 수 있나요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

아닙니다. XSS는 결코 프론트엔드만의 영역이 아니며, **백엔드와 프론트엔드가 각자의 레이어에서 상호 협력하여 '다중 방어벽(Defense in Depth)'을 구축해야만 완벽히 막을 수 있습니다.**

물론 현대 프론트엔드 프레임워크(React, Vue 등)는 데이터를 출력할 때 자동으로 에스케이핑(Escaping)을 해주어 1차적인 방어벽 역할을 합니다. 하지만 **보안의 대전제는 '클라이언트가 보내는 모든 입력값은 변조될 수 있으므로 절대 신뢰해서는 안 된다'는 점**입니다.

공격자는 프론트엔드 화면을 거치지 않고 Postman이나 curl 명령어로 백엔드 API에 직접 악성 스크립트가 담긴 `POST` 요청을 찌를 수 있습니다. 만약 백엔드가 이를 검증 없이 그대로 데이터베이스에 저장(Stored XSS)해 버린다면, 우리 서비스의 웹 아키텍처 전체가 오염되는 결과를 초래합니다.

따라서 백엔드 엔지니어는 다음과 같은 핵심적인 역할을 수행해야 합니다.

첫째, **데이터 인입 단계에서의 글로벌 필터링**입니다.
서블릿 필터(Servlet Filter)나 인터셉터 레이어에서 악성 특수문자를 HTML 엔티티로 변환하는 글로벌 XSS 필터(예: 네이버 Lucy XSS Filter)를 적용하여, 오염된 데이터가 데이터베이스에 아예 발을 붙이지 못하도록 '원천 차단'해야 합니다.

둘째, 비즈니스 목적에 따른 새니타이징(Sanitizing)입니다.
마크다운 에디터처럼 의도적으로 HTML 태그 입력을 허용해야 하는 비즈니스 요구사항이 있다면, 백엔드 단에서 `OWASP Java HTML Sanitizer` 같은 화이트리스트 기반 라이브러리를 통해 `script`나 `onerror` 같은 위험한 태그와 속성만 정교하게 걸러내어 저장해야 합니다.

셋째, **인프라 및 브라우저 보안 헤더 지원**입니다.
쿠키에 `HttpOnly` 옵션을 부여하여 자바스크립트가 세션 ID를 읽지 못하게 원천 차단하고, HTTP 응답 헤더에 `Content-Security-Policy(CSP)`를 설정해 주는 등의 **최종적인 보안 가이드라인을 제공하는 주체 역시 백엔드 서버**입니다.

결론적으로, 화면을 안전하게 그리는 것은 프론트엔드의 몫이지만, **데이터의 무결성을 지키고 시스템 전체의 보안 아키텍처를 견고히 하는 것은 백엔드의 절대적인 책임**입니다.

### 꼬리 질문

- 🤗 스포 방지!

  프론트엔드와 백엔드의 역할 분담 및 상호 방어 논리를 완벽히 펼쳤기 때문에, 면접관은 실제 구현 상의 한계나 트레이드오프를 꼬리 질문으로 던질 것입니다.

    - [ ]  방금 백엔드 인입 단계에서 모든 요청을 에스케이핑(Escaping)하는 글로벌 필터를 언급하셨습니다. 그런데 만약 사용자가 보내는 데이터가 JSON 포맷(`application/json`)일 때, Spring Boot의 기본 JSON 컨버터인 **Jackson 라이브러리** 환경에서는 일반적인 서블릿 필터 방식의 XSS 처리가 먹히지 않는 기술적 한계가 있습니다. 이 문제는 실무에서 어떻게 해결하나요? (예: `ObjectMapper`에 커스텀 `CharacterEscapes` 등록 유도)
    - [ ]  만약 프론트엔드 개발자가 '백엔드에서 이미 DB에 넣을 때 `<`를 `&lt;`로 다 치환(Escaping)해서 저장해 버리면, 우리 프론트엔드가 브라우저가 아닌 모바일 앱(iOS/Android) 환경이나 엑셀 다운로드 기능을 구현할 때 화면에 `&lt;script&gt;`라는 가공 가공되지 않은 텍스트가 그대로 노출되는 사이드 이펙트가 생긴다'며 반발한다면 백엔드 엔지니어로서 어떻게 조율하시겠습니까? (저장 시점 vs 출력 시점의 에스케이핑 트레이드오프)
    - [ ]  XSS 방어의 최종 진화형이라고 불리는 **CSP(Content Security Policy, 콘텐츠 보안 정책)** 헤더를 백엔드 서버에서 설정해 준다고 하셨는데, 이 CSP 헤더가 정확히 브라우저에게 어떤 제약 조건(예: 인라인 스크립트 실행 금지, 허용된 도메인 스크립트만 로드 등)을 강제하여 XSS를 방어하는지 설명해 주세요.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

XSS는 최종적으로 브라우저(클라이언트)에서 실행되기 때문에, 프론트엔드가 화면에 데이터를 렌더링할 때 신경 써야 하는 취약점인 것은 맞습니다.

하지만 면접관이 진짜 확인하고 싶은 것은 "백엔드가 '화면 렌더링은 내 관할이 아니다'라며 책임을 회피하지 않고, 데이터의 원천 저장소인 데이터베이스(DB)를 안전하게 보호하기 위해 백엔드 레이어에서 수행해야 하는 '상호 방어(Defense in Depth)'의 중요성을 알고 있는가"입니다.

### 답변 분석

- **보안의 대전제 선언**: "클라이언트의 입력은 절대 신뢰하지 않는다"는 백엔드 보안의 가장 핵심적인 철학을 명시하여 면접관에게 신뢰감을 주었습니다.
- **API 직접 공격 시나리오 제시**: 프론트엔드 화면을 우회하여 API에 직접 악성 데이터를 찌르는 상황(Postman, curl)을 구체적으로 언급함으로써, 왜 백엔드 검증이 필수적인지 인과관계를 완벽하게 입증했습니다.
- **백엔드의 3대 구체적 방어선 명시**: 단순히 "백엔드도 막아야 합니다"로 끝내지 않고, 글로벌 필터(입력단), 새니타이저(저장단), HttpOnly/CSP(인프라/출력단)라는 명확한 구현 기술과 아키텍처 구조를 조리 있게 설명했습니다.

### 핵심 키워드

- 다중 방어 (Defense in Depth) 및 클라이언트 불신
- 프론트엔드를 우회하는 API 직접 공격 방어 (데이터 무결성)
- 글로벌 필터(Escaping), 새니타이저(Sanitizing), 보안 헤더(CSP)의 백엔드 주도권
