## ❓질문

```text
그렇다면, 스레드는 PCB를 갖고 있을까요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

**아닙니다. 스레드는 독립적인 PCB를 소유하지 않습니다.** PCB(Process Control Block)는 운영체제로부터 메모리 공간, 파일 디스크립터 같은 **'자원을 할당받는 단위'인 프로세스 고유의 관리 구조체**이기 때문입니다.

대신, 하나의 프로세스 내부에서 독립적인 '실행 흐름(Context)'을 유지해야 하는 각 스레드들은 자신만의 고유한 메타데이터 구조체인 TCB(Thread Control Block, 스레드 제어 블록)를 할당받아 관리됩니다.

### 1. PCB와 TCB의 포함 관계

하나의 프로세스 내에 멀티 스레드가 생성되면, 커널 공간에 있는 해당 프로세스의 **PCB는 내부에 생성된 여러 TCB들을 가리키는 포인터(리스트 형태)를 링크**하여 관리합니다.

- **PCB가 들고 있는 것**: 페이지 테이블(가상 메모리 영토), 파일 디스크립터 목록, 유저 ID/권한 등 모든 형제 스레드가 공통으로 '공유하는 자원'의 메타데이터가 담깁니다.
- **TCB가 들고 있는 것**: 개별 작업자(스레드)가 독립적인 CPU 실행 궤적을 유지하기 위해 필요한 최소한의 '실행 컨텍스트'가 담깁니다.

### 2. TCB(Thread Control Block) 내부의 핵심 정보

스레드가 교체되는 '스레드 컨텍스트 스위칭' 시 데이터 파괴를 막기 위해 TCB는 다음과 같은 필드를 독립적으로 유지합니다.

- **Thread ID (TID)**: 프로세스 내부에서 스레드를 구별하는 고유 번호입니다.
- **스레드 상태 (Thread State)**: 스레드가 현재 실행(`Running`), 준비(`Ready`), 대기(`Blocked`) 중인지 기록합니다.
- **PC (Program Counter)**: 이 스레드가 다음번에 실행해야 할 **독립적인 코드 라인의 메모리 주소**입니다. (형제 스레드들이 똑같은 Code 영역을 공유하더라도, 각자 실행 중인 라인은 다르기 때문에 PC는 무조건 TCB에 독립적으로 존재해야 합니다.)
- **레지스터 스냅샷**: 이 스레드가 CPU 연산 장치 위에서 직전까지 계산하던 날날의 레지스터 값(중간 계산 결과)을 백업하는 공간입니다.
- **스택 포인터 (Stack Pointer)**: 이 스레드만의 독립된 **유저 스택 공간의 주소**를 가리킵니다.

### # 리눅스(Linux) 커널의 반전 스펙

참고로 우리가 실무에서 사용하는 리눅스 커널의 경우, 아키텍처 비대화를 막기 위해 프로세스와 스레드를 엄격히 구분하지 않고 둘 다 `task_struct`라는 단일 구조체(LWP - Light Weight Process)로 통합 관리합니다.

다만, 메모리와 자원을 공유하지 않는 `task_struct`들의 묶음을 **프로세스**, `CLONE_VM` 등의 플래그를 켜서 메모리와 파일 시스템을 공유하도록 링킹된 `task_struct`들을 **스레드**로 취급하는 방식으로 대칭성을 유지합니다.

요약하자면, 스레드는 시스템 자원 주소록인 PCB를 직접 가지지 않고 **프로세스의 PCB를 형제들과 공동 공유**하며, 오직 자신의 독립 실행 흐름을 보장하기 위한 **개별 명세서인 TCB를 나침반 삼아 작동**합니다.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 이 질문을 통해 확인하고 싶은 골드 포인트는 "스레드가 프로세스의 자원을 공유하면서도 어떻게 독립적인 실행 흐름(Context)을 유지하는지 그 물리적 관리 장치(TCB)를 이해하고 있는가"입니다. 스레드가 PCB의 하부 오너쉽 구조로 엮여 들어가는 형태를 아는지가 핵심입니다.

### 답변 분석

- **이중 구조(PCB-TCB)의 명확한 분리**: 프로세스 단위의 자원(PCB)과 스레드 단위의 실행 흐름(TCB)이 커널 공간에서 **어떻게 포인터 구조로 포함**되는지 아키텍처 맵을 완벽하게 증명했습니다.
- **공유와 독립 필드의 완벽한 당위성 해설**: 특히 "동일한 코드를 공유하더라도 실행 라인이 다르므로 PC 레지스터와 스택 포인터는 무조건 TCB에 고유하게 존재해야 한다"는 컴퓨터 구조적 인과관계를 찔러 신뢰도를 극대화했습니다.
- **리눅스 커널 실무 스펙(`task_struct` / LWP) 터치**: 책에 나오는 이론에만 갇혀있지 않고, 실제 현업 인프라인 리눅스가 **LWP(경량 프로세스)** 개념을 빌려 두 블록을 어떻게 우아하게 통합 구현했는지까지 언급하여 면접관을 완전히 압도할 수 있습니다.

### 핵심 키워드

- 스레드는 자원 관리용 PCB 대신 실행 흐름 관리용 TCB(Thread Control Block)를 소유
- TCB 내부의 독립 필수 필드: 스레드 고유 ID, `PC(Program Counter)`, 레지스터 백업 영역, 스택 포인터
- 리눅스 커널에서의 구현 스펙: `task_struct` (LWP) 단일 구조체를 통한 자원 공유 매핑
