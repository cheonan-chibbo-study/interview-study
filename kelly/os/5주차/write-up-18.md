## ❓질문

```text
자식 프로세스가 상태를 알리지 않고 죽거나, 부모 프로세스가 먼저 죽게 되면 어떻게 처리하나요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

"자식 프로세스와 부모 프로세스의 생명 주기가 일치하지 않아 어느 한쪽이 먼저 종료되는 경우, 리눅스 커널은 시스템 자원 누수를 막기 위해 **좀비(Zombie) 프로세스**와 **고아(Orphan) 프로세스**라는 두 가지 메커니즘으로 이 상황을 처리합니다.

### 1. 자식 프로세스가 상태를 알리지 않고 먼저 죽은 경우 (좀비 프로세스)

자식 프로세스가 작업을 마치고 `exit()`을 호출하면 메모리, 파일 디스크립터 같은 물리 자원은 즉시 OS에 반환됩니다. 하지만 자식이 왜 죽었는지(종료 코드)를 부모에게 보고해야 하므로, 커널 내부의 PID와 최소한의 메타데이터(`task_struct`)는 지워지지 않고 커널에 남게 되는데, 이 상태를 '좀비 프로세스'라고 합니다.

- **정상 처리 메커니즘**: 부모 프로세스가 **`wait()` 또는 `waitpid()` 시스템 콜**을 호출하여 자식의 종료 상태(Exit Status)를 읽어가는 순간, 커널은 대기 중이던 좀비 프로세스의 `task_struct`와 PID를 완전히 소멸(Reap)시킵니다.
- **상태를 무시할 경우의 문제점**: 만약 부모 프로세스가 바쁘거나 버그가 있어서 `wait()`을 영원히 호출하지 않는다면 좀비 프로세스가 커널에 계속 쌓이게 됩니다. 메모리는 늘어나지 않지만, 리눅스 시스템 전체의 **최대 PID 개수는 한정되어 있으므로 'PID 고갈 장애'가 발생하여 시스템 내에서 더 이상 새로운 프로세스나 소켓을 생성할 수 없는 리스크**가 발생합니다.

### 2. 부모 프로세스가 자식보다 먼저 죽어버린 경우 (고아 프로세스)

부모 프로세스가 비정상 종료되거나 먼저 작업을 끝내고 사라졌는데, 자식 프로세스는 여전히 백그라운드 등에서 살아 움직이는 상황입니다. 이 자식들을 '고아 프로세스'라고 부릅니다.

- **커널의 구제 메커니즘 (입양)**: 리눅스 커널은 고아 프로세스가 된 자식들을 그대로 방치하지 않습니다. 부모가 종료되는 즉시, 커널은 이 고아 프로세스들의 부모 PID(PPID)를 리눅스 최상위 대부 프로세스인 `init` (현대 리눅스에서는 PID 1번인 `systemd`) 프로세스로 강제 변경(Re-parenting)합니다.
- **최종 종료 처리**: `systemd` 프로세스는 백그라운드에서 주기적으로 `wait()` 시스템 콜을 호출하는 루프를 돌고 있습니다. 따라서 고아 프로세스가 나중에 자신의 작업을 다 마치고 죽어서 좀비 상태가 되더라도, **새로운 양부모가 된 `systemd`가 즉시 자식의 종료 상태를 수거하고 PCB 자원을 깨끗하게 청소**해 줍니다.

요약하자면, 자식이 먼저 죽으면 부모가 **`wait()`을 해줄 때까지 좀비 상태로 대기**하며, 부모가 먼저 죽으면 **PID 1번인 `systemd`가 고아를 입양**하여 추후 좀비 상태가 되더라도 안전하게 청소되도록 아키텍처적으로 방어합니다."

### 꼬리 질문

- 🤗 스포 방지!

  자식과 부모의 비정상 종료 시 커널 레벨의 방어선(wait, systemd)을 완벽히 정렬해 냈기 때문에, 면접관은 "그렇다면 실무 자바/스프링 환경이나 리눅스 터미널에서 개발자가 이 좀비 프로세스를 어떻게 모니터링하고 프로그래밍적으로 해결할 것인가"라는 실전 엔지니어링 역량으로 꼬리를 물 확률이 매우 높습니다.

    - [ ]  리눅스 환경에서 `top`이나 `ps` 명령어를 쳤을 때, 특정 프로세스가 현재 '좀비(Zombie)' 상태에 빠져 있는지는 어떤 상태 플래그 문자를 보고 식별할 수 있나요? 또한, 좀비 프로세스는 이미 죽은 유령이라 `kill -9 [좀비PID]` 명령어를 내려도 절대 죽지 않는데, 이 좀비를 터미널에서 강제로 소멸시키려면 어떻게 조치해야 하나요?
    - [ ]  자바 가상 머신(JVM) 기반의 스프링 부트 애플리케이션 내부에서 `ProcessBuilder` 라이브러리를 사용해 리눅스의 외부 쉘 스크립트나 C언어 프로그램을 자식 프로세스로 실행시키는 로직을 구현했습니다. 이때 자바 코드 단에서 방금 설명하신 '좀비 프로세스 누수'를 막기 위해 반드시 체인으로 호출해 주어야 하는 자바 내장 API 메서드는 무엇인가요?
    - [ ]  **런타임에 부모가 죽어도 종료되지 않고, 시스템이 켜져 있는 동안 백그라운드에 완전히 상주하면서 로그를 수집하거나(Fluentd) 클라이언트의 요청을 대기하는 웹 서버(Nginx) 같은 프로세스들을 무엇이라 부르나요? 이 프로세스들은 고아 프로세스의 특성을 어떻게 의도적으로 활용하여 설계되는지 설명해 주세요.**

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 확인하고 싶은 골드 포인트는 두 가지입니다.

1. **좀비(Zombie) 프로세스 처리와 자원 고갈 방어**: 자식이 죽어도 부모가 리포트를 받기 전까지 커널 메타데이터(`task_struct`, PID)가 남는 이유와, 이를 **`wait()` 시스템 콜**로 해제하는 원리를 아는가.
2. **고아(Orphan) 프로세스 구제 메커니즘**: 부모가 먼저 죽어 버려 길을 잃은 자식 프로세스를 리눅스의 대부(代父) 프로세스인 `init` (또는 `systemd`)가 어떻게 입양하여 고립을 막는지 이해하고 있는가.

### 답변 분석

- **현상과 하부 자원의 정확한 매핑**: 좀비 프로세스가 메모리를 먹는 것이 아니라 **PID와 커널 메타데이터(`task_struct`)를 점유하여 PID 고갈을 유발한다**는 로우레벨 팩트를 정확히 짚었습니다.
- **핵심 커널 장치의 명시**: 좀비를 해결하는 **`wait()` 시스템 콜**과 고아를 구제하는 PID 1번 `systemd` (Re-parenting)의 실무 구조를 완벽하게 연결해 서술했습니다.

### 핵심 키워드

- 자식 선종료 시 `wait()` 미호출로 인해 PCB/PID가 남는 좀비 프로세스 (PID 고갈 유발)
- 부모 선종료 시 최상위 프로세스로 부모가 재매핑되는 고아 프로세스 (Re-parenting)
- PID 1번 `systemd` 프로세스의 자동 자원 회수 루틴
