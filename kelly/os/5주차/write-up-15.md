## 😎 답변 및 꼬리 질문

### 답변

PCB(Process Control Block, 프로세스 제어 블록)는 운영체제 커널이 프로세스를 스케줄링하고 관리하기 위해, 프로세스의 생성과 동시에 커널 메모리 영역에 할당하는 고유의 메타데이터 구조체(Task Descriptor)입니다.

CPU 코어는 한 번에 단 하나의 프로세스 명령만 실행할 수 있으므로, 수많은 프로세스를 번갈아 구동하는 멀티태스킹을 구현하려면 **과거 실행 상태를 정확히 '기억'하고 '복원'할 장치**가 필요합니다. PCB가 바로 그 기억 장치 역할을 수행합니다.

PCB 내부에 저장되는 대표적인 핵심 정보들은 다음과 같습니다.

1. **프로세스 식별자 (PID)**: 시스템 전체에서 각 프로세스를 유일하게 구별하기 위한 고유 고정 번호입니다.
2. **프로세스 상태 (Process State)**: 현재 프로세스가 생성(`New`), 준비(`Ready`), 실행(`Running`), 대기(`Waiting`), 종료(`Terminated`) 중 어떤 상태인지를 기록하여 스케줄러의 타겟이 되도록 합니다.
3. **프로그램 카운터 (Program Counter, PC)**: 이 프로세스가 다음번에 실행해야 할 **실행 코드 라인의 메모리 주소**를 가리킵니다. CPU를 뺏겼다가 다시 얻었을 때 어디서부터 코드를 재개해야 하는지 알려주는 이정표입니다.
4. **CPU 레지스터 스냅샷 (CPU Registers)**: 누산기(`Accumulator`), 인덱스 레지스터, 스택 포인터 등 CPU 내부의 하드웨어 레지스터들이 직전까지 들고 있던 **날날의 연산 데이터 값들을 백업**해 두는 공간입니다.
5. **CPU 스케줄링 정보**: 프로세스의 우선순위(Priority), 스케줄링 큐를 가리키는 포인터 등 스케줄러 알고리즘에 필요한 데이터입니다.
6. **메모리 관리 정보**: 프로세스의 영토(Code, Data, Heap, Stack)를 매핑해 둔 가상 메모리의 **페이지 테이블(Page Table) 주소** 또는 세그먼트 테이블 정보가 담겨 있어 메모리 침범을 방지합니다.
7. **입출력 상태 정보**: 프로세스가 현재 열어두고 사용 중인 **파일 디스크립터(File Descriptor) 목록**이나 할당받은 I/O 장치들의 리스트가 관리됩니다.

PCB가 가장 빛을 발하는 순간은 **컨텍스트 스위칭(Context Switching)** 시점입니다.
타이머 인터럽트 등으로 인해 프로세스 A에서 프로세스 B로 CPU 제어권이 넘어갈 때, CPU는 프로세스 A의 현재 `PC` 주소와 `레지스터` 값들을 **프로세스 A의 PCB**에 남김없이 저장(Save)합니다. 이어서 새롭게 구동할 **프로세스 B의 PCB**를 열어 과거에 저장해 두었던 `PC`와 `레지스터` 값들을 CPU 하드웨어 칩셋 위로 그대로 복원(Restore)하여 실행 흐름을 이어갑니다.

보안 관점에서도 매우 엄격하게 다루어집니다. 만약 PCB가 일반 유저 메모리 영역에 있다면 악의적인 프로그램이 다른 프로세스의 우선순위를 낮추거나 파일 디스크립터를 훔쳐볼 수 있으므로, **PCB는 오직 운영체제 특권 권한으로만 접근할 수 있는 '커널 공간(Kernel Space)'에 철저히 격리되어 관리**됩니다.

### 꼬리 질문

- 🤗 스포 방지!

  그렇다면, 스레드는 PCB를 갖고 있을까요?

  PCB의 본질과 구조를 완벽하게 정렬해 내셨기 때문에, 면접관은 PCB의 파생 구조체인 스레드 제어 블록(TCB)과의 관계나, 리눅스 커널의 실제 구현체 명칭으로 깊이를 시험하려 할 것입니다.

    - [ ]  방금 프로세스마다 고유한 PCB가 할당된다고 하셨습니다. 그렇다면 하나의 프로세스 내부에서 구동되는 **멀티 스레드(Multi-thread)** 환경인 경우, 커널은 각 스레드들을 구별하고 실행 흐름을 백업하기 위해 어떤 제어 블록 구조체를 추가로 생성하여 PCB와 연동하나요? (예: **TCB - Thread Control Block / 스레드 제어 블록**의 존재와 PCB와의 포인터 연결 구조 유도)
    - [ ]  우리가 주로 사용하는 **리눅스(Linux) 운영체제 커널의 실제 C언어 소스 코드**를 보면, 'PCB'라는 이름의 구조체는 명시되어 있지 않습니다. 리눅스 커널에서 이 PCB 역할을 완벽하게 대행하는 핵심 구조체의 명칭은 무엇인가요? (예: **`task_struct` 구조체** 및 `thread_info` 유도)
    - [ ]  컨텍스트 스위칭이 발생하여 PCB의 내용이 교체될 때, CPU 내부의 하드웨어 주소 변환 캐시 장치인 TLB (Translation Lookaside Buffer)는 기존 프로세스의 가상 메모리 주소 파편을 들고 있으면 위험하므로 통째로 플러시(Flush, 초기화)되거나 무효화됩니다. 이로 인해 발생하는 성능적 트레드오프(Overhead)에 대해 설명해 주세요.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 확인하고 싶은 골드 포인트는 세 가지입니다.

1. **PCB의 저장 위치와 보안성**: PCB가 왜 유저 영역이 아닌 보호된 커널 메모리 영역(Kernel Space)에 할당되는지 그 당위성을 아는가.
2. **핵심 명세 정보**: PCB 내부의 수많은 필드 중 컨텍스트 스위칭과 스케줄링에 직결되는 핵심 필드(`PC`, `Register`, `PID`, `State`)를 정확히 짚어낼 수 있는가.
3. **컨텍스트 스위칭과의 연계**: CPU 주도권이 넘어갈 때 PCB가 어떤 역할을 수행하는지 물리적 메커니즘을 이해하고 있는가.

### 답변 분석

- **존재의 물리적 위치(Kernel Space) 명시**: PCB가 메모리의 어디에 위치하는지 명확히 짚고, 이를 보안 관점(메타데이터 변조 방지)과 엮어 서술함으로써 구조적 시야를 증명했습니다.
- **필드 정보의 기능적 열거**: 의미 없는 단순 나열이 아니라, `PC`는 실행 재개 위치, `레지스터`는 연산 데이터 백업, `페이지 테이블`은 영토 보호 등 **각 필드가 운영체제 내에서 수행하는 기여도**를 명확하게 매핑했습니다.
- **컨텍스트 스위칭 시퀀스의 연결**: 앞서 마스터했던 인터럽트/스위칭 메커니즘과 PCB의 구조를 엮어, "A의 PCB에 Save, B의 PCB에서 Restore"라는 명쾌한 물리적 이동 경로를 시각적으로 잘 풀어냈습니다.

### 핵심 키워드

- 커널 공간(Kernel Space)에 생성되는 프로세스 관리용 메타데이터 구조체
- 컨텍스트 백업 및 복원의 기준점이 되는 `PC(Program Counter)`와 `레지스터` 필드
- 가상 메모리 주소(페이지 테이블) 및 파일 디스크립터 목록 관리
