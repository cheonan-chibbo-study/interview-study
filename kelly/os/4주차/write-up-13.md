## 😎 답변 및 꼬리 질문

### 답변

프로세스(Process)는 컴퓨터의 디스크(스토리지)에 바이너리 형태로 저장되어 있던 정적인 **'프로그램(Program)'이 메모리에 적재(Load)되어, 가상 메모리 공간과 자원을 할당받고 CPU에 의해 실제로 실행 중인 동적인 상태**를 의미합니다. 즉, 운영체제 관점에서 '태스크(Task)를 수행하는 독립적인 실행 단위'입니다.

정적인 프로그램이 하나의 프로세스로 변환될 때, 운영체제 내부에서는 다음과 같은 두 가지 핵심 아키텍처 작업이 수행됩니다.

### 1. 프로세스 고유의 독립된 4대 메모리 영역 할당

운영체제는 각 프로세스가 서로의 영토를 침범하지 못하도록 가상 메모리를 활용해 독립된 주소 공간을 부여하며, 이는 크게 4가지 영역으로 나뉩니다.

- **코드(Code) 영역**: 개발자가 작성한 소스 코드가 컴파일되어 기계어 형태로 저장되는 읽기 전용(Read-Only) 공간입니다.
- **데이터(Data) 영역**: 전역 변수(Global)나 정적 변수(Static)가 저장되는 공간으로, 프로그램 시작 시 할당되어 종료 시 수거됩니다.
- **힙(Heap) 영역**: 프로그래머가 런타임(Runtime)에 동적으로 메모리를 할당하는 공간입니다. (자바/스프링 환경에서 `new` 키워드로 생성된 객체들이 저장되고, 가비지 컬렉터(GC)가 관리하는 핵심 영역입니다.)
- **스택(Stack) 영역**: 함수 호출 시 필요한 지역 변수, 매개 변수, 반환 주소 등이 임시로 저장되는 공간입니다. 함수가 끝나면 자동으로 컴파일러에 의해 제거됩니다.

### 2. 커널 내부의 관리 도구, PCB (Process Control Block) 생성

운영체제 커널은 수많은 프로세스를 스케줄링하고 관리하기 위해, 프로세스 생성과 동시에 커널 메모리 영역에 PCB(프로세스 제어 블록)라는 고유한 메타데이터 구조체를 생성합니다. 여기에는 **Process ID(PID), 프로세스 상태(Ready, Running 등), 다음 실행할 코드 주소를 가리키는 프로그램 카운터(PC), CPU 레지스터 스냅샷, 그리고 메모리 관리 정보**가 담겨 있으며, CPU가 프로세스를 교체하는 '컨텍스트 스위칭'의 핵심 이정표가 됩니다.

결론적으로 프로세스는 프로그램에 **운영체제의 생명력(PCB)과 독립된 영토(4대 메모리 구조)가 부여되어 CPU와 소통하는 활성화된 실행 주체**라고 정의할 수 있습니다.

### 꼬리 질문

- 🤗 스포 방지!

  프로그램과 프로세스, 스레드의 차이에 대해 설명해 주세요.

  프로세스의 정의와 내부 구조(메모리, PCB)를 완벽히 정렬해 냈기 때문에, 면접관은 "그렇다면 이 프로세스 안에서 실행 흐름만 분리해 낸 스레드(Thread)와의 차이점"이나 "프로세스 간의 자원 공유 한계"로 무조건 꼬리 질문을 던질 것입니다.

    - [ ]  방금 프로세스는 독립된 가상 메모리 공간(힙, 스택 등)을 할당받는다고 하셨습니다. 그렇다면 하나의 프로세스 내부에서 생성되는 스레드(Thread)는 이 프로세스의 4대 메모리 영역 중 어떤 영역을 서로 '공유'하고, 어떤 영역을 '독립적으로' 할당받아 쓰는지 대조해서 설명해 주세요. (예: Heap/Data/Code 공유, Stack 및 PC 레지스터 독립 할당 원리 유도)
    - [ ]  프로세스는 서로 철저히 격리되어 있어 보안상 안전하지만, 반대로 두 개의 프로세스가 서로 데이터를 주고받으려면 일반적인 메모리 접근 방식으로는 불가능합니다. 운영체제 커널의 도움을 받아 **서로 다른 프로세스끼리 통신하기 위해 사용하는 기술 체계**를 무엇이라 하나요? (예: **IPC - Inter-Process Communication / 파이프, 소켓, 공유메모리** 등 유도)
    - [ ]  자바/스프링 부트 애플리케이션 서버를 리눅스 환경에 구동시키면, OS 레벨에서는 **하나의 프로세스**가 뜬 것일까요, 아니면 유저의 요청 수만큼 **수많은 프로세스**가 뜬 것일까요? JVM 아키텍처 관점에서 설명해 주세요.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 이 질문을 통해 확인하고 싶은 핵심은 세 가지입니다.

1. **정적 상태와 동적 상태의 구별**: 디스크에 저장된 수동적인 '프로그램'과 메모리에 적재되어 살아 움직이는 능동적인 '프로세스'의 차이를 아는가.
2. **OS 관리 인프라의 이해**: 커널이 프로세스를 스케줄링하기 위해 할당하는 핵심 메타데이터(`PCB`)의 존재를 인지하는가.
3. **메모리 격리성(Isolation)**: 프로세스가 자신만의 독립된 가상 메모리 주소 공간을 가짐으로써 얻는 안정성을 이해하고 있는가.

### 답변 분석

- **프로그램과 프로세스의 인과관계 대조**: 첫머리에서 정적(디스크)과 동적(메모리 적재)의 개념 대조를 명확히 선언하여 용어의 본질을 관통했습니다.
- **구체적인 메모리 맵(Map) 빌드업**: 코드, 데이터, 힙, 스택의 역할과 백엔드 개발자 관점의 해석(자바의 `new` 객체와 힙 영역, GC)을 매끄럽게 엮어 실무 적합성을 어필했습니다.
- **커널 오브젝트(`PCB`)의 등판**: 많은 지원자가 메모리 구조만 말하고 끝내지만, 골드 답변은 커널이 프로세스를 바라보는 실제 거울인 `PCB (Process Control Block)`의 개념과 내부 명세를 명시하여 전산학적 깊이를 완성했습니다.

### 핵심 키워드

디스크의 정적 프로그램 $\rightarrow$ 메모리의 동적 프로세스로의 전환독립된 가상 메모리 아키텍처 (Code, Data, Heap, Stack)커널이 프로세스를 제어하기 위한 메타데이터 주소록 (PCB)
