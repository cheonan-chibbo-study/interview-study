## ❓질문

```text
동시에 두 개 이상의 인터럽트가 발생하면, 어떻게 처리해야 하나요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

동시에 두 개 이상의 인터럽트가 발생하거나 인터럽트 처리 도중 또 다른 인터럽트가 가로채기를 시도할 때, 운영체제와 CPU 하드웨어는 미리 정의된 '인터럽트 우선순위(Interrupt Priority)'와 CPU 내부의 '마스킹(Masking) 메커니즘'을 사용하여 이를 정교하게 통제합니다.

처리 방식은 크개 **두 가지 시나리오**로 구동됩니다.

### 1. 순차적 처리 (인터럽트 비활성화 방식)

현재 실행 중인 인터럽트 서비스 루틴(ISR)이 끝날 때까지 뒤따라오는 모든 인터럽트를 잠시 대기시키는 방식입니다.

- CPU 내부의 상태 레지스터(EFLAGS)에는 인터럽트 수용 여부를 결정하는 **`IF (Interrupt Flag)` 비트**가 존재합니다.
- 특정 인터럽트가 터져 ISR이 시작되면, CPU는 `IF` 비트를 자동으로 `0`으로 바꿉니다. 이 상태를 **'인터럽트 불가능(Interrupt Disable)'** 상태라고 합니다.
- 이 도중에 다른 하드웨어 인터럽트 신호가 들어오면 CPU는 이를 무시하고 큐에 대기시킨 뒤, 현재 ISR이 끝난 후 `IRET` 명령어로 원래 복귀하면서 `IF` 비트가 다시 `1`로 켜지는 순간 대기하던 다음 인터럽트를 순차적으로 처리합니다.

### 2. 중첩 처리 (우선순위 기반 가로채기 방식)

마치 유저 프로그램이 인터럽트에 의해 중단되듯, **'낮은 순위의 ISR'이 실행되다가 '높은 순위의 인터럽트'가 터지면 ISR 내부에서 다시 컨텍스트 스위칭이 일어나는 중첩(Nested) 방식**입니다. 이때 하드웨어적으로 **우선순위 중제자**가 개입합니다.

- **하드웨어 제어 (APIC / PIC)**: CPU 앞단에는 인터럽트 요청을 스케줄링하는 **프로그래밍 가능한 인터럽트 컨트롤러(현대 아키텍처에서는 APIC - Advanced PIC)** 칩이 존재합니다. 이 칩이 하드웨어 번호별로 매겨진 우선순위를 비교합니다.
- **동작 원리**: 현재 실행 중인 인터럽트보다 **더 높은 레벨의 인터럽트**가 들어오면, APIC는 CPU에게 신호를 보냅니다. CPU는 현재 실행 중이던 '낮은 레벨의 ISR 컨텍스트'를 커널 스택에 한 번 더 백업하고, '높은 레벨의 ISR'로 점프하여 먼저 해결한 뒤 역순으로 복귀합니다.

### # 하드웨어 레벨의 절대적 우선순위 스펙

동시에 완전히 같은 나노초(ns) 단위에 인터럽트가 충돌할 때를 대비해, CPU 제조사와 OS는 절대적인 우선순위 계급표를 가집니다.

1. **최우선순위 - NMI (Non-Maskable Interrupt)**: 정전(Power Failure), 메모리 패리티 에러 등 하드웨어 파괴 직전의 신호입니다. 이는 앞서 말한 `IF` 플래그 비트를 무시하고 CPU가 무조건 모든 것을 멈추고 즉시 실행합니다.
2. **2순위 - 소프트웨어 예외**: 코드 실행 중 발생하는 `Divide-by-Zero`, `Page Fault` 등입니다.
3. **3순위 - 하드웨어 타이머 인터럽트**: 스케줄링의 기준이 되므로 일반 I/O보다 우선합니다.
4. **최하순위 - 일반 주변장치 I/O**: 네트워크 카드 패킷 도착, 디스크 read/write 완료, 키보드/마우스 입력 등입니다.

요약하자면, 동시에 인터럽트가 발생하면 하드웨어 인터럽트 컨트롤러(**APIC**)가 **정해진 우선순위 계급표에 따라 판정**을 내리며, 커널은 상황에 따라 **`IF` 레지스터 플래그를 마스킹하여 순차적으로 밀어내거나, 스택에 ISR을 중첩하여 쌓는 방식으로 시스템 꼬임 없이 안전하게 처리**합니다.

### 꼬리 질문

- 🤗 스포 방지!

  인터럽트 동시 처리 체계와 하드웨어 중제자 원리까지 마스터했기 때문에, 면접관은 "동시 다발적 패킷이 몰려 우선순위가 낮은 인터럽트가 영원히 실행되지 못하는 기아(Starvation) 상태나 병목"으로 깊이를 시험하려 할 것입니다.

    - [ ]  만약 네트워크 인터럽트가 최하위 순위인데, 기가비트 트래픽이 몰려 상위 인터럽트(타이머 등) 처리에 밀려 **네트워크 인터럽트가 계속 드롭되거나 처리되지 못하는 기아(Starvation) 상태**가 발생하면 OS는 이를 어떻게 방어하나요? (예: 우선순위를 유동적으로 조정하거나, 앞서 배운 하반부 처리(Bottom Half) 분할을 통해 소프트웨어 인터럽트로 격하시켜 스케줄링하는 원리 유도)
    - [ ]  방금 설명하신 중첩 처리 방식(Nested Interrupt)은 커널 스택에 ISR 상태를 계속 누적해서 쌓아 올립니다. 만약 악의적인 하드웨어 오작동으로 인해 **최상위 인터럽트들이 무한히 중첩되어 계속 터진다면 커널 메모리에는 어떤 치명적인 에러**가 발생할까요? (예: **커널 스택 오버플로우 / Kernel Stack Overflow**로 인한 커널 패닉 유도)
    - [ ]  현대의 **멀티 코어(Multi-core) CPU 환경**에서는 단일 APIC가 아닌 각 코어마다 로컬 APIC가 존재합니다. 한 코어가 인터럽트를 처리하느라 바쁠 때, 다른 놀고 있는 코어에게 인터럽트 처리를 분산 유도하기 위해 코어들끼리 서로 인터럽트를 던지는 기술을 무엇이라 하나요? (예: **IPI - Inter-Processor Interrupt / 프로세서 간 인터럽트** 메커니즘 유도)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 확인하고 싶은 골드 포인트는 두 가지입니다.

1. **우선순위(Priority) 체계의 이해**: 모든 인터럽트는 동등하지 않으며, 시스템 붕괴를 막는 치명적인 신호(전원 차단 등)와 일반 장치 신호의 계급이 하드웨어 레벨에서 나누어져 있음을 아는가.
2. **인터럽트 마스킹(Masking) 메커니즘**: 현재 중요한 인터럽트를 처리하는 중일 때, 다른 인터럽트가 방해하지 못하도록 CPU의 눈과 귀를 잠시 가리는 하드웨어 플래그 제어 능력을 이해하고 있는가.

### 답변 분석

- **하드웨어 메커니즘 레지스터 명시**: 단순히 "순서대로 한다"가 아니라, CPU 내부의 **`IF(Interrupt Flag)` 비트 마스킹 원리**를 선명하게 짚어내어 로우레벨 구현 능력을 어필했습니다.
- **핵심 물리 장치(APIC) 지목**: 인터럽트의 교통정리를 담당하는 실제 메인보드/CPU 내장 칩셋인 APIC(인터럽트 컨트롤러)를 정확히 선언해 답변의 격을 높였습니다.
- **우선순위 스펙의 구체적 나열**: 면접관이 가장 좋아하는 NMI(차단 불가능한 인터럽트)의 개념과 일반 타이머, I/O 간의 계급 구조를 체계적으로 분류해 꼼꼼함을 증명했습니다.

### 핵심 키워드

- 인터럽트 컨트롤러(APIC)를 통한 하드웨어 레벨의 우선순위 중재
- CPU `EFLAGS` 레지스터의 `IF(Interrupt Flag)` 제어를 통한 마스킹(Masking)
- 차단이 불가능한 최우선순위 신호인 NMI(Non-Maskable Interrupt)
