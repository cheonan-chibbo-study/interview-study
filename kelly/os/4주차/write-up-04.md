## 😎 답변 및 꼬리 질문

### 답변

운영체제에서 제공하는 시스템 콜은 커널이 관리하는 핵심 자원의 종류와 목적에 따라 크게 5가지 유형(프로세스 제어, 파일 조작, 장치 관리, 정보 유지, 통신 및 보안)으로 분류할 수 있습니다.

**1. 프로세스 제어 (Process Control)**
프로세스의 생성, 실행, 종료 및 동기화를 담당하는 시스템 콜입니다.

- **주요 시스템 콜**: `fork()` (자식 프로세스 복사 생성), `exec()` (새로운 프로그램 실행 파일 로드), `exit()` (프로세스 종료), `wait()` (자식 프로세스 종료 대기), `kill()` (프로세스에 시그널 전송)
- *백엔드 관점*: JVM이 OS 위에서 멀티 스레드를 생성하거나, 서버에서 스크립트를 독립된 프로세스로 실행할 때 커널의 이 영역을 호출합니다.

**2. 파일 조작 (File Manipulation)**
디스크나 스토리지에 존재하는 파일의 생성, 삭제, 읽기, 쓰기를 관장합니다. 리눅스 환경에서는 디렉터리(폴더) 역시 파일의 일종으로 취급되어 이 유형에 포함됩니다.

- **주요 시스템 콜**: `open()` (파일 열기 및 파일 디스크립터 발급), `close()` (파일 닫기), `read()` (파일 읽기), `write()` (파일 쓰기), `unlink()` (파일 삭제)
- *백엔드 관점*: 스프링 부트에서 로그 파일(Logback)을 남기거나 유저가 업로드한 이미지를 디스크에 저장할 때 매일 사용하는 가장 빈번한 시스템 콜입니다.

**3. 장치 관리 (Device Management)**
모니터, 키보드, 마우스 같은 하드웨어 주변 장치뿐만 아니라 디스크 드라이브, 가상 디바이스의 권한을 요청하고 제어합니다. 리눅스는 'Every object is a file(모든 것은 파일이다)'라는 아키텍처 철학을 가짐으로, 장치 제어도 파일 조작 시스템 콜을 확장해서 사용합니다.

- **주요 시스템 콜**: `ioctl()` (입출력 장치 제어 및 특수 설정), `read() / write()` (장치 버퍼에 데이터 입력/출력)

**4. 정보 유지 (Information Maintenance)**
운영체제 커널이 관리하는 시스템의 시간, 날짜, OS 버전, 특정 프로세스의 메타데이터 및 통계 정보를 유저 영역으로 가져오거나 설정하는 함수입니다.

- **주요 시스템 콜**: `getpid()` (현재 프로세스 ID 조회), `gettimeofday()` (현재 정밀 시스템 시간 조회), `chmod()` (파일이나 자원의 권한 정보 변경)
- *백엔드 관점*: 우리가 애플리케이션 로그에 `[PID: 12345]` 처럼 현재 서버 프로세스의 ID를 찍거나 API 수행 시간을 측정하기 위해 시스템 시간을 조회할 때 트리거됩니다.

**5. 통신 및 보안 (Communication & Protection)**
프로세스 간 데이터 통신(IPC) 및 네트워크 소켓 통신, 그리고 자원의 접근 권한 관리(보안)를 수행합니다.

- **주요 시스템 콜**: `socket()` (네트워크 통신용 엔드포인트 생성), `connect() / accept()` (네트워크 연결 요청 및 수락), `pipe()` (파이프 기반 IPC 채널 생성), `shmget()` (공유 메모리 영역 할당)
- *백엔드 관점*: 톰캣 인프라가 클라이언트와 TCP/IP 소켓을 맺고 HTTP 요청/응답 패킷을 주고받는 백엔드 통신의 근간이 되는 시스템 콜 유형입니다.

요약하자면 시스템 콜은 **프로세스, 파일, 장치, 정보, 통신**이라는 운영체제 커널의 5대 핵심 인프라 모듈을 유저 프로그램이 안전하게 빌려 쓸 수 있도록 정교하게 설계된 분류 체계입니다.

### 꼬리 질문

- 🤗 스포 방지!

  시스템 콜의 전체 지도를 완벽하게 그렸기 때문에, 면접관은 5대 유형 중 백엔드 동시성과 대규모 데이터 전송에 가장 큰 병목을 유발하는 '통신 및 파일 조작(IPC/네트워크)'의 세부 메커니즘으로 현전 기량을 테스트하려 할 것입니다.

    - [ ]  방금 5번 통신 유형에서 프로세스 간 통신을 뜻하는 IPC(Inter-Process Communication)를 언급하셨습니다. 리눅스 시스템 콜이 제공하는 여러 IPC 기법 중, 커널 모드 전환 없이 유저 영역의 메모리 공간을 공유하여 가장 압도적인 속도를 자랑하는 방식은 무엇인가요? (예: **공유 메모리 / Shared Memory (`shmget`)** 메커니즘 유도)
    - [ ]  파일 조작 유형 중 `write()` 시스템 콜을 호출하여 데이터를 저장할 때, 커널은 성능 최적화를 위해 디스크에 바로 쓰지 않고 '페이지 캐시(Page Cache / Buffer Cache)'라는 커널 메모리에 임시로 모아둡니다. 만약 이 상태에서 갑자기 정전이 나면 데이터가 날아가게 되는데, 이를 방지하고 강제로 디스크 하드웨어까지 동기화(Flush)하라고 명령하는 시스템 콜의 이름은 무엇인가요? (예: **`fsync()` 시스템 콜** 유도)
    - [ ]  전통적으로 자바에서 프로세스를 새로 띄울 때 사용하는 **`fork()` 시스템 콜**은 부모 프로세스의 메모리를 그대로 복사하므로 메모리 낭비와 부하가 심합니다. 리눅스 커널은 이를 최적화하기 위해 자식이 실제로 데이터를 '수정'하기 전까지는 부모의 메모리 페이지를 공유하게 만드는 기법을 사용하는데, 이 고도의 메모리 최적화 기술 명칭은 무엇인가요? (예: **COW - Copy-On-Write / 쓰기 시 복사** 원리)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 확인하고 싶은 골드 포인트는 "운영체제가 관리하는 핵심 서브시스템(프로세스, 파일, 네트워크, 장치, 보안)들과 시스템 콜이 어떻게 매핑되어 있는지 체계적인 분류 기준을 가지고 있는가"입니다. 각 유형별 대표적인 커널 API 명칭을 정확히 던져주면 완벽합니다.

### 답변 분석

- **POSIX 표준 가이드라인에 맞춘 분류**: OS 교과서 및 실무 표준 규격인 5대 카테고리를 정확히 선언하여 기본기가 선명함을 증명했습니다.
- **실무적 직관 연결**: 단순히 이론 명칭만 나열하지 않고 "리눅스는 장치도 파일로 보기에 `read/write`를 공유한다"라거나, **IPC와 네트워크를 통신 카테고리**로 묶는 등 아키텍처적 유연함을 보여주었습니다.
- **백엔드 매핑 코드 리마인드**: 자바/스프링에서 로그를 찍거나(`getpid`), 파일 업로드를 하거나(`write`), 소켓을 열 때(`socket`)라는 실전 개발 상황을 1:1로 결합하여 면접관에게 단순 암기자가 아님을 어필했습니다.

### 핵심 키워드

- OS 서브시스템 기준의 5대 유형 분류 (프로세스, 파일, 장치, 정보, 통신)
- 리눅스의 '모든 것은 파일이다' 철학에 따른 시스템 콜의 상호 확장성
- 백엔드 생태계(톰캣, JVM, 파일 I/O)의 실제 커널 호출 영역 매핑
