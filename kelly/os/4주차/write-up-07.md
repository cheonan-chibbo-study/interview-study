## ❓질문

```text
서로 다른 시스템 콜을 어떻게 구분할 수 있을까요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

운영체제와 CPU는 수백 개에 달하는 서로 다른 시스템 콜을 문자열 이름이 아닌, **하드웨어 연산에 최적화된 고유의 '시스템 콜 번호(System Call Number)'와 이를 커널 함수 주소와 매핑해 둔 '시스템 콜 테이블(System Call Table)'을 통해 정확하게 구분하고 실행**합니다.

구체적으로 시스템 콜을 식별하고 라우팅하는 내부 메커니즘은 다음과 같은 3가지 핵심 축으로 작동합니다.

**첫째, 고유 정수 번호(System Call Number) 할당**

운영체제는 각각의 시스템 콜마다 고유한 정수 번호를 아키텍처별로 미리 정의해 둡니다. 예를 들어, 64비트 리눅스(x86_64) 커널 기준 코드를 보면 다음과 같이 정수 번호가 매핑되어 있습니다.

- `0번`: `sys_read`
- `1번`: `sys_write`
- `2번`: `sys_open`
- `59번`: `sys_execve`

**둘째, CPU 레지스터(Register)를 통한 번호 전달**

유저 영역의 응용 프로그램이 시스템 콜 라이브러리를 호출하면, 라이브러리는 실행할 시스템 콜에 해당하는 고유 정수 번호를 **CPU의 특정 레지스터(64비트 리눅스의 경우 `RAX` 레지스터)에 세팅**한 뒤, `syscall` 특권 명령을 내려 커널 모드로 진입합니다. 즉, CPU 레지스터가 시스템 콜을 구분하는 '임시 저장소' 역할을 합니다.

**셋째, 시스템 콜 테이블(System Call Table)의 $O(1)$ 인덱싱**
커널 모드로 진입한 운영체제의 공통 핸들러는 `RAX` 레지스터에 적힌 번호를 읽어 들입니다. 그리고 커널 메모리 영역에 주소 배열 형태로 존재하는 **`시스템 콜 테이블(System Call Table)`의 인덱스로 이 번호를 그대로 사용**합니다.

만약 레지스터에 `1`이 들어있었다면, 커널은 `sys_call_table[1]` 번 자리를 조회하여 그곳에 저장된 `sys_write()` 커널 함수의 실제 메모리 주소를 찾아내어 곧바로 점프합니다. 배열의 인덱스 조회를 사용하기 때문에 수백 개의 시스템 콜 중 내가 원하는 기능을 **$O(1)$이라는 고정된 속도로 정확하게 찾아내어 분기**할 수 있습니다.

요약하자면, 운영체제는 수많은 시스템 콜을 '레지스터에 담긴 고유 정수 번호'로 식별하고, 커널 내부에서는 이를 '배열 인터페이스(System Call Table)의 인덱스'로 매핑하여 단 한 치의 오차도 없이 명확하게 구분하여 실행합니다.

### 꼬리 질문

- 🤗 스포 방지!

  시스템 콜 번호와 테이블 인덱싱이라는 내부 분기 원리를 완벽하게 묘사했기 때문에, 면접관은 "테이블에 없는 번호가 들어왔을 때의 예외 처리"나 "인자의 개수가 다를 때의 레지스터 제어 방식"으로 한 단계 더 깊숙이 파고들 것입니다.

    - [ ]  만약 악의적인 사용자나 버그로 인해, `RAX` 레지스터에 시스템 콜 테이블의 최대 범위를 넘어가는 **음수나 너무 큰 정수(예: 9999번)를 세팅하고 시스템 콜을 트리거**하면 커널은 이 예외 상황을 어떻게 구분하고 방어하나요? (예: 테이블 바운디드 체크/Range Check 메커니즘 및 `ENOSYS` 에러 반환 유도)
    - [ ]  시스템 콜은 번호 외에도 함수마다 넘겨주어야 하는 인자(Argument)의 개수와 종류가 다릅니다. 예를 들어 `read()`는 인자가 3개 필요하죠. 시스템 콜 번호는 `RAX` 레지스터에 담긴다고 하셨는데, 이러한 **추가 인자 데이터들은 CPU의 어떤 규칙으로 커널에 전달**되나요? (예: 리눅스 x86_64 Calling Convention - `RDI`, `RSI`, `RDX` 등의 레지스터 파라미터 전달 규칙)
    - [ ]  리눅스 커널을 직접 빌드하거나 커널 모듈을 개발할 때, 새로운 커널 기능을 추가하고 싶다면 방금 설명하신 **시스템 콜 테이블에 내가 만든 커널 함수를 커스텀하게 등록하는 것이 가능할까요?** 만약 가능하다면 어떤 과정을 거쳐야 할지 아키텍처 관점에서 설명해 주세요.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 확인하고 싶은 골드 포인트는 두 가지입니다.

1. **시스템 콜 번호(ID)의 존재**: 운영체제 커널은 문자열(예: "open", "write")이 아닌, 하드웨어 연산에 최적화된 '고정된 정수 번호'로 시스템 콜을 구별한다는 점을 아는가.
2. **테이블 라우팅 메커니즘**: CPU가 소프트웨어 인터럽트를 타고 커널로 넘어왔을 때, 레지스터에 담긴 번호를 기반으로 배열의 인덱스 탐색(O(1))을 수행하여 커널 함수 주소를 찾아가는 과정을 인지하고 있는가.

### 답변 분석

- **바이너리/전산학적 식별 원리 명시**: 단순 추측성 답변을 배제하고, 리눅스 실제 스펙 규격(`0번 read`, `1번 write`)을 예시로 들어 답변의 신뢰도를 극대화했습니다.
- **레지스터 명칭의 구체화**: 인터페이싱이 일어나는 물리적 장치인 **`RAX` 레지스터**를 정확히 지목하여 컴퓨터 구조 지식이 탄탄함을 증명했습니다.
- **자료구조적 관점($O(1)$)의 해석 결합**: 시스템 콜 테이블이 왜 배열 형태인지를 "수백 개 분기 처리를 속도 지연 없이 $O(1)$로 해내기 위함"이라는 자료구조 및 알고리즘적 시선과 결합하여 설명함으로써, 깊이 있고 똑똑한 답변 구조를 완성했습니다.

### 핵심 키워드

- 아키텍처별 고유 정수 번호 (System Call Number)
- CPU 특정 레지스터(`RAX`)를 통한 식별 ID 전달
- 함수 포인터 배열인 시스템 콜 테이블(System Call Table)의 $O(1)$ 인덱싱
