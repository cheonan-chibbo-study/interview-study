## 😎 답변 및 꼬리 질문

### 답변

시스템 콜(System Call)은 유저 영역의 응용 프로그램이 하드웨어 자원에 접근하거나 운영체제가 제공하는 핵심 기능을 사용하기 위해, 커널 영역의 서비스를 요청할 때 사용하는 유일한 프로그래밍 인터페이스(통로)입니다.

컴퓨터의 CPU는 시스템 보안과 안정성을 위해 실행 권한을 '유저 모드(User Mode)'와 '커널 모드(Kernel Mode)'로 엄격히 분리하여 관리합니다. 일반 응용 프로그램은 권한이 제한된 유저 모드에서 실행되므로, 디스크 파일 읽기/쓰기, 네트워크 송수신, 메모리 할당, 프로세스 생성 같은 하드웨어 밀접 연산을 직접 수행할 수 없습니다. 만약 유저 프로그램이 하드웨어를 무단으로 제어하려 하면 CPU는 보호 오류를 발생시키고 프로그램을 강제 종료합니다.

따라서 응용 프로그램이 `파일 열기(open)`나 `네트워크 전송(write)` 같은 특권 명령을 수행해야 할 때 **시스템 콜을 호출**하게 되며, 내부 동작 시퀀스는 다음과 같습니다.

1. 응용 프로그램이 API(예: Java의 파일 IO 객체나 C의 `printf`)를 호출하면, 하부의 표준 라이브러리가 해당하는 **시스템 콜 번호**를 CPU 레지스터에 세팅합니다.
2. 이후 CPU에 **트랩(Trap) 또는 소프트웨어 인터럽트(Software Interrupt)** 명령을 발생시킵니다.
3. 이 신호를 받은 CPU는 즉시 실행 중이던 유저 코드를 중단하고, 유저 모드에서 커널 모드로 모드 전환(Mode Switch)을 수행합니다.
4. 커널은 시스템 콜 테이블(System Call Table)을 참조하여 요청된 번호에 매핑된 커널 내부 함수(커널 서비스 루틴)를 대신 안전하게 실행합니다.
5. 자원 처리가 끝나면 다시 **커널 모드에서 유저 모드로 복귀**하고, 자바나 C 프로그램에 결과값을 반환하며 프로세스의 제어권을 넘겨줍니다.

요약하자면, 시스템 콜은 응용 프로그램의 무분별한 자원 접근으로부터 **시스템 전체의 무결성과 안정성을 지키기 위해 운영체제가 제공하는 최소한의 안전한 대리인 메커니즘**입니다.

    - [ ]  자바 진영에서 웹 서버의 I/O 처리 성능을 극대화하기 위해 도입한 Java NIO (New I/O)나 네티(Netty) 프레임워크는 내부적으로 리눅스의 어떤 동기식 비차단(Non-blocking) 멀티플렉싱 관련 시스템 콜을 활용하여 동시성 성능을 높이나요? (예: `select`, `poll`, 그리고 현대 표준인 **`epoll` 시스템 콜**의 원리)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 확인하고 싶은 핵심은 두 가지입니다.

1. **커널 모드와 유저 모드의 분리 이유** : 응용 프로그램이 메모리나 하드웨어를 직접 건드려 전체 시스템을 다운시키는 코너 케이스를 방어하기 위한 OS의 보안 메커니즘을 이해하는가.
2. **소프트웨어 인터럽트 원리** : 시스템 콜이 단순한 '함수 호출(Function Call)'이 아니라, CPU의 권한 레벨이 바뀌는 무거운 하드웨어 제어 과정임을 인지하는가.

### 답변 분석

- **모드 분리의 당위성 짚기**: "시스템 콜은 호출이다"로 시작하지 않고, 유저 모드와 커널 모드가 분리된 이유(시스템 안정성/보안)를 하드웨어 보안 관점에서 먼저 탄탄하게 짚었습니다.
- **추상적 개념을 아키텍처 명세로 구체화**: 트랩(Trap), 소프트웨어 인터럽트, 레지스터, **시스템 콜 테이블(System Call Table)** 등 OS의 실제 내부 컴포넌트 키워드를 순서대로 엮어 흐름을 시퀀스 단위로 증명했습니다.
- **일반 함수 호출과의 트레이드오프 암시**: 모드 전환(Mode Switch)이 일어난다는 점을 명시하여, 이것이 일반적인 메모리 주소 이동인 '함수 호출'보다 레지스터 백업 등으로 인해 **비용이 무거운 연산**임을 은근히 내비쳤습니다.

### 핵심 키워드

- 유저 모드와 커널 모드의 전이 (Privilege Mode Transition)
- 하드웨어 보호를 위한 특권 명령 대행 제도
- 트랩(Trap)/소프트웨어 인터럽트 및 시스템 콜 테이블 참조 메커니즘
