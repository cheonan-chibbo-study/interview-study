## 😎 답변 및 꼬리 질문

### 답변

시스템 콜이 운영체제에서 실행되는 과정은 단순한 메모리 주소 이동이 아니라, CPU의 권한 레벨을 안전하게 상승시키고 커널 내부의 이정표를 찾아가는 '하드웨어와 OS 커널의 정교한 5단계 협업 시퀀스'로 진행됩니다.

**1단계: 시스템 콜 번호 세팅 및 트랩 발동 (User Mode)**
응용 프로그램이 파일 쓰기(`write`) 같은 I/O 함수를 호출하면, 표준 라이브러리(glibc 등)는 해당 시스템 콜의 고유 번호(예: 리눅스 x86_64 기준 `write`는 1번)를 CPU의 특정 레지스터(예: `RAX`)에 세팅합니다. 그 후, **`syscall` 명령어나 소프트웨어 인터럽트 명령(`int 0x80`)을 실행하여 CPU에 트랩(Trap)을 발생**시킵니다.

**2단계: 커널 모드 전환 및 CPU 레지스터 백업 (Hardware)**
트랩 신호를 감지한 CPU는 즉시 유저 모드에서 커널 모드(Ring 0特権)로 권한을 상승시킵니다. 이때, 커널은 현재 실행 중이던 유저 프로그램의 CPU 레지스터 상태(현재 실행 주소인 `RIP`, 스택 주소인 `RSP` 등)가 파괴되지 않도록, 해당 프로세스의 '커널 스택(Kernel Stack)' 공간에 안전하게 복사하여 백업(Context Save)합니다. 이후 CPU의 스택 포인터는 유저 스택에서 커널 스택으로 전환됩니다.

**3단계: IDT를 통한 핸들러 점프 (Hardware -> Kernel)**
CPU는 시스템 콜을 처리하기 위해 메모리 상에 고정되어 있는 `IDT (Interrupt Descriptor Table, 인터럽트 디스크립터 테이블)`를 참조합니다. 이 테이블에는 시스템 콜 인터럽트가 들어왔을 때 백엔드에서 실행해야 하는 커널의 핵심 게이트웨이 함수인 **'시스템 콜 핸들러(System Call Handler)'의 실제 메모리 주소**가 적혀 있습니다. CPU는 이 주소를 읽어 핸들러 코드로 점프합니다.

**4단계: 시스템 콜 테이블 참조 및 비즈니스 로직 실행 (Kernel Mode)**
구동된 시스템 콜 핸들러는 앞서 1단계에서 응용 프로그램이 `RAX` 레지스터에 저장해 두었던 시스템 콜 번호(예: 1번)를 확인합니다. 그리고 커널 내부의 `시스템 콜 테이블(System Call Table)`에서 1번에 매핑된 진짜 알맹이 함수인 `sys_write()`의 주소를 찾아내어, 하드웨어 컨트롤러나 파일 시스템을 제어하는 실제 비즈니스 로직을 안전하게 대행합니다.

**5단계: 레지스터 복원 및 유저 모드 복귀 (Kernel -> User Mode)**
커널의 작업이 완료되면 결과값(성공 코드 또는 에러 넘버)을 다시 레지스터에 세팅합니다. 그리고 2단계에서 커널 스택에 백업해 두었던 유저 프로그램의 옛날 레지스터 상태들을 다시 CPU로 원상 복구(`Context Restore`)시킨 뒤, **`sysret` 또는 `iret` 명령을 통해 CPU 권한을 다시 유저 모드로 강등**시키며 자바나 C 응용 프로그램의 다음 코드로 제어권을 완전히 반환합니다.

### 꼬리 질문

    - [ ]  방금 2단계와 5단계에서 유저 레지스터를 커널 스택에 백업하고 복원하는 **'모드 스위칭(Mode Switching)'** 비용을 언급하셨습니다. 만약 자바 프로그램에서 루프문을 돌며 1바이트짜리 데이터를 디스크에 만 번 `write()` 하면, 이 시스템 콜 시퀀스가 만 번 발생하여 서버 CPU가 비명을 지르게 됩니다. 자바 백엔드 개발 시 이 시스템 콜 모드 스위칭 횟수를 줄이기 위해 파일 I/O나 네트워크 단에서 사용하는 대표적인 클래스나 디자인 패턴은 무엇인가요? (예: 데이터를 메모리에 모았다가 한 번에 시스템 콜을 찌르는 **`BufferedOutputStream` / 버퍼링 기술** 유도)
    - [ ]  과거 인텔 CPU에서 발생했던 치명적인 하드웨어 보안 결함인 **멜트다운(Meltdown) 취약점**은, 유저 프로그램이 시스템 콜을 호출하지 않고도 CPU의 예측 실행(Speculative Execution) 기능을 악용해 커널 메모리 영역을 무단으로 읽어갈 수 있는 버그였습니다. OS 진영은 이를 막기 위해 유저 모드일 때는 커널 메모리 매핑 자체를 완전히 지워버리는 **PTI(Page Table Isolation)** 기술을 도입했는데, 이 패치가 적용된 후 시스템 콜의 실행 성능(속도)에는 어떤 악영향이 생겼을까요?
    - [ ]  하드웨어 장치(마우스 클릭, 네트워크 카드에 패킷 도착 등)가 CPU에 신호를 보내는 **하드웨어 인터럽트(Asynchronous)** 처리 과정과, 방금 설명하신 시스템 콜 프로그램 코드가 의도적으로 발생시키는 **소프트웨어 인터럽트(Synchronous/Trap)** 처리 과정은 'IDT를 참조한다'는 점에서 매우 유사합니다. 두 인터럽트가 발생하는 시점과 제어 메커니즘의 가장 큰 차이점은 무엇일까요?

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 진짜 확인하고 싶은 골드 포인트는 두 가지입니다.

1. **소프트웨어 인터럽트(트랩)의 물리적 처리 방식**: 시스템 콜이 실행될 때 CPU가 커널 영역의 함수 주소를 어떻게 찾아서 점프하는지 그 이정표(`IDT`)를 아는가.
2. **컨텍스트 백업과 커널 스택의 존재**: 유저 영역에서 실행되던 CPU 레지스터 값들이 모드 전환 시 어디에 안전하게 보관(`Kernel Stack`)되는지 인지하고 있는가.

### 답변 분석

- **컴퓨터 구조학적 하드웨어 키워드 총동원**: 단순히 "커널이 처리한다"가 아니라, `RAX 레지스터`, `Trap`, `Ring 0 권한`, `커널 스택(Kernel Stack)` 등의 로우레벨 키워드를 명확한 인과관계로 배치했습니다.
- **IDT와 시스템 콜 테이블의 2단계 구조 완벽 분리**: 수많은 지원자가 인터럽트 벡터(`IDT`)와 실제 함수 매핑 테이블(`System Call Table`)을 묶어서 혼동합니다. 골드 답변은 "IDT를 통해 공통 핸들러로 먼저 가고, 그 핸들러 안에서 번호를 판별해 시스템 콜 테이블을 본다"는 운영체제의 2단계 라우팅 메커니즘을 정확하게 분리하여 깊이감을 보여주었습니다.
- **컨텍스트 백업(Save/Restore)의 시점 명시**: 유저의 원래 실행 흐름을 잃어버리지 않기 위해 **커널 스택에 레지스터를 백업하는 시점**을 명시하여 시스템 프로그래밍의 기초가 단단함을 입증했습니다.

### 핵심 키워드

- 레지스터 세팅 및 트랩(Trap) 발동을 통한 권한 상승 (Ring 0)
- 유저 컨텍스트의 커널 스택(Kernel Stack) 백업 메커니즘
- IDT(인터럽트 디스크립터 테이블)와 시스템 콜 테이블의 단계별 참조 흐름
