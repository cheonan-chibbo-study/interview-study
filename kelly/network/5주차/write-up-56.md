## ❓질문

```text
사용자의 요청이 무수히 많아지면, 소켓도 무수히 생성되나요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

**네, 이론적으로는 사용자들의 동시 접속 요청이 무수히 많아지면, 4-Tuple 식별을 위해 커널 공간에 소켓 객체 역시 1:1로 무수히 생성되는 것이 맞습니다.** TCP 핸드셰이크가 완료되어 연결이 확립(`ESTABLISHED`)될 때마다 서버 운영체제는 해당 클라이언트 전용 데이터 파이프라인인 소켓을 하나씩 새로 개설하기 때문입니다.

하지만 실제 운영 환경에서는 소켓이 무한정 생성되도록 방치하지 않으며, **운영체제 커널의 하드웨어 자원 한계와 백엔드 서버의 설정에 의해 물리적 제어**를 받게 됩니다. 소켓이 무수히 생성될 때 마주하는 3가지 현실적인 장벽은 다음과 같습니다.

### 1. 파일 디스크립터 (File Descriptor / FD)의 한계

리눅스 운영체제는 '모든 것은 파일이다'라는 철학을 가집니다. 따라서 커널은 소켓을 생성할 때도 일반 파일과 동일하게 '파일 디스크립터(FD)'라는 정수형 ID 번호를 발급하여 프로세스에게 넘겨줍니다.

- 리눅스 시스템은 하나의 프로세스가 가질 수 있는 오픈 파일 개수(**`nofile`**)에 엄격한 상한선(기본값 보통 1024개 등)을 둡니다.
- 만약 이 설정을 최적화하지 않고 유저 요청이 폭발하면, 소켓을 더 만들고 싶어도 **`Too many open files`** 에러가 발생하며 새로운 소켓 생성이 가차 없이 차단됩니다.

### 2. 커널 메모리(RAM)의 한계

소켓은 단순한 논리적 개념이 아니라, 내부에 송신 버퍼(`wmem`)와 수신 버퍼(`rmem`)를 물리적으로 할당받는 **무거운 커널 오브젝트**입니다.

- 리눅스에서 소켓 1개당 대략 수 KB에서 수십 KB의 메모리를 소모합니다.
- 동시 접속자가 10만 명, 100만 명으로 늘어나 소켓이 무수히 생성되면, 응용 프로그램(JVM)이 쓰기도 전에 **OS 커널 메모리가 먼저 고갈되어 서버 전체가 다운되는 커널 패닉**이 올 수 있습니다.

### 3. 백엔드 애플리케이션(스프링 부트/톰캣)의 스레드 풀 및 커넥션 제어

우리 스프링 부트 앞단에 있는 톰캣(Tomcat) 서버는 무분별한 소켓 생성을 막기 위해 하이퍼 파라미터로 방어벽을 칩니다.

- **`maxConnections`**: 서버가 동시에 유지할 수 있는 최대 소켓(커넥션) 수의 한계를 정합니다. 이 한계치에 도달하면 커널의 백로그 큐에서 대기시키며 더 이상 소켓을 `ESTABLISHED` 상태로 만들지 않습니다.
- 현대 톰캣은 **NIO(Non-blocking I/O)** 방식을 채택하여 하나의 스레드가 수많은 소켓을 멀티플렉싱으로 관리하므로 스레드 폭발은 막아내지만, 소켓 자체의 물리적 메모리 점유는 여전히 OS 레벨에서 제어 대상이 됩니다.

요약하자면, 사용자 요청 수만큼 소켓이 비례하여 생성되는 것이 본질이지만, 실무에서는 **리눅스의 파일 디스크립터(FD) 제한, 커널 메모리 용량, 그리고 톰캣의 커넥션 풀(MaxConnections) 설정을 통해 적정 수준으로 제어 및 격리**하여 운영합니다.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 확인하고 싶은 골드 포인트는 두 가지입니다.

1. **커넥션과 소켓의 1:1 관계 인지**: TCP 통신에서 클라이언트와의 연결 수립(Established) 상태 하나당 커널 메모리에 소켓 객체가 1개씩 할당된다는 본질을 이해하고 있는가.
2. **리눅스 자원(OS 리소스)의 한계 이해**: 소켓이 무한히 늘어날 수 없는 이유가 '파일 디스크립터(File Descriptor) 한계'와 **'커널 메모리 고갈'** 때문임을 알고, 이를 제어하는 실무 아키텍처를 인지하고 있는가.

### 답변 분석

- **이론과 실무의 완벽한 밸런스**: "그렇다"라는 원론적 답변에 멈추지 않고, 실무 환경에서 시스템을 보호하기 위해 걸어두는 **물리적 제약 조건들**을 명확히 대조하여 시니어급 시야를 보여주었습니다.
- **리눅스 아키텍처 철학 대입**: 소켓의 한계가 왜 파일 개수 제한(**파일 디스크립터 / `Too many open files`**)과 직결되는지 리눅스 시스템 구조의 본질을 찔러 높은 점수를 획득했습니다.
- **백엔드 생태계(Tomcat NIO) 매핑**: 서은 님이 다루는 스프링 부트 인프라 환경(`maxConnections`) 설정을 녹여내어, 인프라 부하를 제어하는 실전 백엔드 엔지니어의 통찰력을 입증했습니다.

### 핵심 키워드

- 연결 확립(`ESTABLISHED`) 시 커넥션과 소켓의 1:1 매핑 생성
- 리눅스 커널의 파일 디스크립터(File Descriptor) 제한 및 `Too many open files` 에러
- 소켓 버퍼 점유로 인한 커널 메모리 고갈 위험 및 톰캣 `maxConnections` 방어벽
