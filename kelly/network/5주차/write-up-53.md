## 😎 답변 및 꼬리 질문

### 답변

**소켓(Socket) 통신**과 **웹소켓(WebSocket) 통신**은 모두 양방향 실시간 통신을 지향한다는 공통점이 있지만, **네트워크 작동 계층(Layer)과 웹(Web) 생태계 및 브라우저 환경에서의 호환성**에서 결정적인 아키텍처적 차이가 있습니다.

### 1. 소켓 통신 (L4 - 전송 계층 기반)

- **개념**: 네트워크상에서 두 호스트가 통신할 수 있도록 OS 커널이 제공하는 **전송 계층(L4, 주로 TCP/IP) 기반의 네트워크 프로그래밍 인터페이스(API)** 자체를 의미합니다.
- **특징**: 특정 포트(Port)를 직접 열고 바인딩하여 바이트 스트림을 전송하는 로우레벨(Low-Level) 통신입니다.
- **웹 환경의 한계**: 보안상의 이유로 **웹 브라우저(크롬, 사파리 등)는 자바스크립트 수준에서 하드웨어 포트를 직접 제어하는 일반 TCP 소켓 통신을 표준 스펙으로 허용하지 않습니다.** 또한, 기업용 방화벽에서 80(HTTP)이나 443(HTTPS) 이외의 커스텀 포트는 대부분 차단하기 때문에 웹 서비스에는 적용하기 어렵습니다.

### 2. 웹소켓 통신 (L7 - 응용 계층 프로토콜)

- **개념**: 웹 브라우저와 웹 서버 간의 제한 없는 실시간 양방향 통신을 지원하기 위해, HTTP 표준 포트 위에서 작동하도록 설계된 응용 계층(L7) 프로토콜(RFC 6455)입니다.
- **웹 환경 최적화**: 웹 브라우저에서 100% 지원하며, 기존 웹 서버 인프라가 사용하는 **80번과 443번 포트를 그대로 공유**하기 때문에 추가적인 방화벽 설정 없이 무사히 통과합니다.

### # 두 통신의 연결 수립 및 구동 차이점

- **연결 방식 (핸드셰이크)**: 소켓 통신은 시작부터 끝까지 TCP 3-Way Handshake만으로 통로를 엽니다. 반면, 웹소켓은 최초 연결 시 기존의 익숙한 HTTP/HTTPS 요청(GET)을 먼저 보냅니다. 이때 HTTP 헤더에 `Upgrade: websocket`, `Connection: Upgrade` 스펙을 심어 보내며, 서버가 이를 수락하면 프로토콜이 HTTP에서 웹소켓 프로토콜(`ws://` 또는 `wss://`)로 전환(Protocol Upgrade)되면서 영구적인 가상 연결 파이프라인이 확립됩니다.
- **데이터 포맷**: 소켓 통신은 날것의 바이트 스트림을 개발자가 직접 파싱해야 하므로 프로토콜 규격을 일일이 설계해야 합니다. 반면 웹소켓은 프레임(Frame)이라는 정형화된 응용 레이어 단위(텍스트 데이터, 바이너리 데이터 등)로 데이터를 패킹하여 안전하게 송수신합니다.

요약하자면, 소켓은 OS가 제공하는 하부 네트워크 연결 도구(인터페이스)이고, 웹소켓은 이 소켓 기술을 웹 브라우저 환경에 맞게 커스터마이징하여 **HTTP의 포트를 빌려 쓰는 고성능 응용 프로토콜**입니다.

### 꼬리 질문

- 🤗 스포 방지!

  웹소켓과 소켓의 구조적 차이점과 포트 호환성까지 완벽히 정렬하셨기 때문에, 면접관은 서은 님이 개발하는 스프링 부트 서버단에서 실시간 알림이나 메시징 기능을 대규모 트래픽으로 받아낼 때 발생하는 자원 소모(C10K 문제)나 대안 기술(SSE)로 파고들 것입니다.

    - [ ]  스프링 부트 환경에서 수만 명의 웹소켓 유저가 동시에 접속해 유지(`ws://`) 상태를 지속하면, 전통적인 HTTP의 일회성 커넥션에 비해 서버 메모리와 스레드 자원에 어떤 부하(C10K 문제)가 걸리며, 이를 방지하기 위해 톰캣 내부 아키텍처나 웹플럭스(WebFlux) 같은 비동기 Non-blocking I/O(NIO) 엔진은 어떻게 동작하나요?
    - [ ]  만약 우리 서비스에서 채팅처럼 '양방향'으로 데이터를 주고받을 필요는 없고, 오직 **서버에서 클라이언트로 실시간 공지사항이나 주식 가격 알림을 일방향으로 밀어주기(Push)만 하면 되는 요구사항**입니다. 이때 무거운 웹소켓 대신 HTTP 프로토콜 통로를 그대로 유지하면서 가볍게 구현할 수 있는 L7 표준 기술의 이름은 무엇인가요? (예: **SSE - Server-Sent Events** 기술과의 트레이드오프 유도)
    - [ ]  웹소켓 프로토콜(`ws://`)은 HTTP의 보안막(TLS)을 타지 않기 때문에 패킷이 중간에 노출될 위험이 있습니다. 실제 스프링 부트 상용 환경에서 실시간 채팅 서비스를 안전하게 구축하려면, 어떤 웹소켓 프로토콜 접두사를 강제해야 하며 이를 위해 인프라 앞단(Nginx 등)에서 어떤 SSL/TLS 설정 처리를 연동해 주어야 하나요? (예: **`wss://` (WebSocket Secure)** 인프라 프록시 세팅 유도)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 확인하고 싶은 골드 포인트는 두 가지입니다.

1. **네트워크 레이어의 명확한 구분**: 소켓은 전송 계층(L4)의 네트워크 인터페이스 자체이고, 웹소켓은 응용 계층(L7)의 고도화된 프로토콜임을 인지하고 있는가.
2. **웹 생태계(HTTP)와의 호환성 이해**: 방화벽이나 80/443 포트 제약이 심한 웹 브라우저 환경에서, 어떻게 웹소켓이 HTTP의 한계를 극복하고 실시간 양방향 통신을 열어젖히는지 그 연결 수립 원리를 아는가.

### 답변 분석

- **OSI 7계층 기준의 선명한 경계선**: 소켓은 L4 레벨의 인터페이스 인터페이스고 웹소켓은 L7 영역의 표준 규격이라는 학술적 명확성을 명시했습니다.
- **현대 웹의 한계와 등장 배경 매핑**: 왜 웹소켓이 굳이 따로 나왔는지에 대해 "웹 브라우저의 보안 제약"과 "방화벽 포트(80/443) 우회"라는 아키텍처적 당위성을 정확하게 꼬집었습니다.
- **프로토콜 업그레이드(Upgrade) 상세 기술**: 웹소켓의 핵심 시퀀스인 HTTP 핸드셰이크를 통한 **프로토콜 스위칭 메커니즘 헤더 규격**을 언급하여 단순 암기자가 아님을 완벽히 어필했습니다.

### 핵심 키워드

- L4 TCP 인터페이스(소켓) vs L7 응용 프로토콜(웹소켓)
- 웹 브라우저 제약 및 기존 80/443 포트 방화벽 우회 호환성
- HTTP 요청을 통한 연결 수립 및 웹소켓(`ws/wss`) 프로토콜 업그레이드(Upgrade)
