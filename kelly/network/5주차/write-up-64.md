## ❓질문

```text
IP가 송신자와 수신자를 정확하게 전송되는 것을 보장해 주나요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

**아닙니다. IP 프로토콜 자체는 패킷이 수신자에게 정확하고 안전하게 전송되는 것을 보장해 주지 않습니다.** IP가 상주하는 네트워크 계층(L3)의 임무는 오직 '지정한 IP 주소를 나침반 삼아 수많은 라우터 경로를 거쳐 목적지 기기까지 패킷을 어떻게든 배달하는 것'에만 집중되어 있습니다. 이 과정에서 IP는 다음과 같은 아키텍처적 한계를 가집니다.

#### 1. IP 프로토콜의 2대 한계

- **비연결성 (Connectionless)**: 패킷을 보내기 전에 상대방이 실제로 켜져 있는지, 패킷을 받을 준비가 되어 있는지 확인하는 사전 교감 과정이 전혀 없습니다. 목적지가 오프라인 상태여도 IP는 일단 패킷을 일방적으로 던져버립니다.
- **비신뢰성 (Unreliable / Best-Effort)**: 패킷이 전송되는 도중에 백본 라우터의 메모리가 폭발하여 패킷이 증발하거나(패킷 유실), 네트워크 경로가 우회되면서 1번 패킷보다 3번 패킷이 먼저 도착하더라도(순서 역전), IP 레이어 자체는 이를 감지하거나 복구할 수 있는 능력이 전혀 없습니다. 오직 최선만 다해 배달(Best-Effort)할 뿐입니다.

#### 2. 전송 보장의 진실: L4 TCP 프로토콜과의 아름다운 협업

그렇다면 서은 님이 개발하시는 'Dogether' 서버와 클라이언트가 데이터 오염 없이 정확하게 통신할 수 있는 비결은 무엇일까요? 바로 **L3 IP의 무책임함을 L4 전송 계층의 TCP(Transmission Control Protocol)가 완벽하게 위에서 감싸주고 보완**하기 때문입니다.

- **연결 보장**: TCP는 IP 패킷을 쏘기 전에, 먼저 **3-Way Handshake**라는 가상 연결 통로를 개설하여 송수신자가 완벽히 준비되었는지 검증합니다.
- **정확성 및 순서 보장**: TCP는 송신자가 보낸 모든 패킷 데이터 헤더에 고유한 시퀀스 번호(Sequence Number)를 매겨 보냅니다. 만약 수신자가 패킷을 받다가 중간 번호가 펑크 난 것을 발견하면, 송신자에게 재전송 요청(ARQ 메커니즘)을 날려 다시 받아냅니다. 또한 뒤죽박죽 도착한 패킷들도 시퀀스 번호를 나침반 삼아 원래 순서대로 안전하게 재조립(디멀티플렉싱)하여 최종 자바 응용 프로그램단에 넘겨줍니다.

요약하자면, **IP는 주소를 기반으로 패킷을 최선 다해 목적지 기기까지 토스하는 거친 하부 인프라 역할**만 담당할 뿐 전송을 보장하지 않으며, 실제 '정확한 전송과 무결성'은 그 윗단계에서 **TCP 프로토콜이 재전송 및 순서 제어 장치를 가동하여 완벽하게 보장**해 주는 것입니다."

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 확인하고 싶은 골드 포인트는 두 가지입니다.

1. **IP 프로토콜의 본질적 스펙 인지**: IP가 가진 2대 한계인 **비연결성**과 비신뢰성(Best-Effort Delivery)의 개념을 정확하게 설명할 수 있는가.
2. **계층 간 역할 분담(L3 vs L4)의 이해**: 네트워크 계층(L3 IP)의 한계를 전송 계층(L4 TCP)이 어떻게 에러 제어 및 흐름 제어로 커버하여 종단 간 무결성을 완성하는지 아는가.

### 답변 분석

- **프로토콜 명세 용어의 정확한 구사**: 단순히 "보장 안 된다"가 아니라, 전산학 표준 명세 용어인 비연결성(Connectionless)과 **비신뢰성 / Best-Effort Delivery**의 정의를 정확히 정돈하여 신뢰도를 높였습니다.
- **계층형 아키텍처(L3-L4 Stack)의 상호작용 논증**: IP의 결함을 무조건 단점으로 몰아세우지 않고, "IP가 배송에만 집중하기에 가볍고 빠르며, 복잡한 신뢰성 처리는 윗선(TCP)이 담당한다"는 **네트워크 계층 분할 설계의 당위성**을 완벽하게 관통했습니다.

### 핵심 키워드

- IP 프로토콜의 태생적 스펙인 비연결성(Connectionless)과 비신뢰성(Unreliable)
- 유실, 순서 뒤바뀜, 목적지 오프라인 등의 상황을 해결하지 못하는 Best-Effort 특성
- 상위 전송 계층(L4) TCP의 3-Way Handshake 및 시퀀스 번호 기반 재전송 제어를 통한 무결성 확보
