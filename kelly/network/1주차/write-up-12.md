## ❓질문

```text
그런데, 브라우저는 어떤 서버가 TCP를 쓰는지 UDP를 쓰는지 어떻게 알 수 있나요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

브라우저는 처음부터 서버가 어떤 프로토콜을 쓰는지 알 수 없기 때문에, **최초 접속 시에는 가장 안전하고 대중적인 TCP(HTTP/1.1 또는 HTTP/2)로 일단 먼저 연결을 시도**합니다. 이 안전한 첫 연결 과정에서 서버로부터 힌트를 얻어 다음 요청부터 UDP(HTTP/3)로 전환하는 방식을 취하며, 크게 두 가지 메커니즘을 사용합니다.

첫째, **HTTP 응답 헤더의 `Alt-Svc` (Alternative Services) 활용**입니다.
브라우저가 처음 TCP로 요청을 보내면, HTTP/3를 지원하는 서버는 응답 헤더에 `Alt-Svc: h3=":443"; ma=86400` 같은 값을 실어 보냅니다. 이는 '우리는 443 포트로 HTTP/3(h3)도 지원하고 있으며, 이 정보는 앞으로 하루(86400초) 동안 유효하다'라는 힌트입니다. 브라우저는 이 정보를 자체 캐시에 저장해 두었다가, **다음 요청부터는 TCP를 건너뛰고 해당 서버로 곧바로 UDP(HTTP/3) 연결을 시도**하게 됩니다.

둘째, **최신 DNS 스펙인 `HTTPS Resource Record` 활용**입니다.
`Alt-Svc` 방식은 첫 번째 요청이 무조건 TCP여야 하므로 1-RTT의 지연이 무조건 발생한다는 단점이 있습니다. 이를 보완하기 위해 최신 웹 생태계에서는 브라우저가 도메인의 IP를 조회하는 DNS 단계에서 **HTTPS 레코드**를 함께 조회합니다. 서버가 DNS에 '우리는 HTTP/3를 지원한다'는 정보를 미리 등록해 두면, 브라우저는 **첫 번째 접속부터 TCP 핸드셰이크를 완전히 생략하고 바로 UDP 기반의 QUIC 핸드셰이크를 맺어** 극단적인 성능 최적화를 달성합니다.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

이 질문은 "이론적인 HTTP/3의 스펙을 넘어, 현실의 브라우저와 서버가 실제로 어떻게 '최초 연결'을 맺고 프로토콜을 협상(Negotiation)하는지 실무적 부트스트랩(Bootstrap) 과정을 이해하고 있는가?"를 검증하기 위한 것입니다.

브론즈나 실버는 "그냥 브라우저가 알아서 3.0으로 요청해 봅니다"라고 뭉뚱그리지만, 골드 답변은 최초의 TCP 연결 과정과 그 과정에서 서버가 주는 힌트(Alt-Svc 헤더), 그리고 최신 DNS 기술(HTTPS 레코드)을 명확히 엮어서 설명해야 합니다.

### 답변 분석

- **현실적인 제약 인정으로 시작**: "처음에는 알 수 없으므로 안전하게 TCP로 먼저 찌른다"는 실무적인 디폴트 동작(Fallback 관점)을 명확히 짚어 신뢰감을 줍니다.
- **표준 기술 명칭(`Alt-Svc`)과 구체적 동작 서술**: `Alt-Svc` 헤더의 실제 포맷 의미(프로토콜 타입, 포트, Max-Age 캐싱 주기)를 분해하여 설명함으로써 기술을 겉핥기가 아닌 코드로 이해하고 있음을 증명합니다.
- **최신 기술 트렌드(DNS HTTPS 레코드)로 완성**: `Alt-Svc`가 가진 '첫 요청은 무조건 TCP여야 한다'는 구조적 한계를 짚고, 이를 보완하는 최신 기술(RFC 9460 기반의 HTTPS 레코드)까지 확장하여 기술 트렌드에 매우 민감하고 깊이 있는 지원자라는 인상을 확고히 심어줍니다.

### 핵심 키워드

- 최초 접속 시 TCP 디폴트 연결 (Fallback)
- `Alt-Svc` (Alternative Services) 응답 헤더 캐싱
- DNS의 HTTPS 리소스 레코드 (0단계 디스커버리)
