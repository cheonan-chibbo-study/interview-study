HTTP의 특성인 Stateless에 대해 설명해 주세요.

## 😎 답변 및 꼬리 질문

### 답변

HTTP의 Stateless 특성은 서버가 클라이언트의 이전 상태를 기억하지 않고, 각 요청을 완전히 독립적인 것으로 처리하는 성질입니다.

- 이 방식은 서버가 클라이언트의 상태를 유지할 필요가 없기 때문에 서버 자원을 아낄 수 있고, 동일한 요청을 어느 서버로 보내도 동일한 응답을 받을 수 있어 **서버의 수평 확장(Scale-out)에 매우 유리하다는 강력한 장점**이 있습니다.
- 다만, 실제 서비스에서는 로그인 유지나 장바구니처럼 상태를 보존해야 하는 비즈니스 요구사항이 필수적입니다. 따라서 실무에서는 HTTP의 Stateless 장점을 살리면서도 상태를 관리하기 위해 **쿠키, 세션, 또는 JWT 같은 토큰 기반의 인증 메커니즘을 도입하여 적절한 트레이드오프**를 가져갑니다.

### 꼬리 질문

- 🤗 스포 방지!
    - [ ]  방금 세션이나 JWT를 사용해 상태를 보완한다고 하셨는데, 서버를 수평 확장(Scale-out)한 환경에서 '전통적인 서버 세션' 방식을 사용하면 어떤 문제가 발생할 수 있을까요?
    - [ ]  그렇다면 그 세션의 정합성 문제를 해결하기 위해 어떤 방식을 사용할 수 있나요? (예: Session Clustering, Sticky Session, Redis 등)
    - [ ]  JWT는 Stateless한 토큰 방식이라고 하는데, 이미 발급된 JWT 토큰을 서버 측에서 강제로 만료시키거나 제어하고 싶을 때는 어떻게 해결해야 할까요?

## 질문 의도 및 답변 분석

### 질문 의도

모든 웹 개발자가 아는 기본 개념이지만, 브론즈와 실버는 단순 정의(상태를 저장하지 않는다)만 말하고 끝냅니다.

- 면접관이 진짜 보고 싶은 것은 "Stateless라는 제약 조건 때문에 실무에서 어떤 불편함이 생기고,
- 이를 해결하기 위해 백엔드 아키텍처 관점에서 어떤 기술적 트레이드오프(Session, JWT 등)를 선택하는가?"입니다.
- 즉, 기본 인프라의 한계를 극복하는 확장성 있는 설계 감각이 있는지 검증하려는 질문입니다.

### 답변 분석

- **수평 확장(Scale-out) 관점의 장점 연결**: 실버 답변은 '상태를 저장하지 않아서 서버 부담이 적다' 정도로 끝납니다. 반면 골드 답변은 백엔드 아키텍처의 핵심인 '수평 확장성(Scale-out)'과 연결 지었습니다. 상태가 없기 때문에 로드밸런서가 어떤 서버로 요청을 튀겨줘도 상관없다는 서버 확장의 용이성을 짚어낸 점이 고득점 포인트입니다.
- **비즈니스 요구사항과의 충돌(불편함) 명시**: 이론적으로는 Stateless가 좋지만, 현실(로그인, 장바구니)에서는 적용하기 어렵다는 실무적 한계를 짚었습니다.
- **현실적인 해결책과 트레이드오프 언급**: HTTP 프로토콜 자체는 Stateless하게 유지하되, 애플리케이션 레벨에서 **쿠키/세션/JWT**를 활용해 이 단점을 보완한다는 흐름을 제시함으로써, 교과서적 지식을 넘어 실제 웹 서비스를 설계할 줄 아는 개발자임을 어필합니다.

### 핵심 키워드

- 수평 확장(Scale-out)
- 상태 보존의 한계 (로그인/장바구니)
- 쿠키, 세션, JWT를 통한 보완
