## 😎 답변 및 꼬리 질문

### 답변

HOL Blocking(Head-of-Line Blocking)은 네트워크나 프로토콜 구조에서 '가장 앞에 있는 첫 번째 요청(또는 패킷)의 처리가 지연되면, 뒤에 있는 나머지 모든 요청이 무조건 대기하게 되는 병목 현상'을 의미합니다. 이 현상은 웹의 역사에서 해결해야 했던 가장 큰 성능 과제 중 하나였습니다.

HOL Blocking은 크게 **애플리케이션 계층**과 **전송 계층** 두 곳에서 발생합니다.

첫째, **HTTP/1.1의 애플리케이션 레벨 HOLB**입니다. HTTP/1.1은 하나의 TCP 커넥션에서 여러 요청을 보낼 때, 반드시 '순서대로' 응답을 받아야 하는 규칙이 있습니다. 만약 무거운 이미지 요청이 첫 번째로 들어와 처리가 늦어지면, 그 뒤에 있는 가벼운 텍스트 요청들은 첫 번째 응답이 끝날 때까지 라인을 점유당한 채 대기해야만 했습니다.

이를 해결하기 위해 HTTP/2에서는 멀티플렉싱(Multiplexing)을 도입하여 하나의 커넥션에 요청들을 잘게 쪼갠 '프레임' 단위로 섞어서 보내 순서 문제를 해결했습니다.

둘째, **TCP 전송 계층 레벨의 HOLB**입니다. HTTP/2가 애플리케이션 레벨의 HOLB는 해결했지만, 근간인 TCP 프로토콜 자체의 한계는 넘지 못했습니다. TCP는 데이터의 신뢰성과 순서를 보장해야 하므로, 전송 중 단 하나의 패킷이라도 유실되면 수신 측 OS의 버퍼에서 그다음 패킷들을 재전송이 완료될 때까지 붙잡아 둡니다. 즉, HTTP/2 내의 특정 스트림과 무관한 패킷 유실 때문에 전체 커넥션이 멈추는 TCP 레벨의 HOLB가 발생한 것입니다.

최근 이 문제까지 완벽히 해결하기 위해 **HTTP/3에서는 TCP를 버리고 UDP 기반의 QUIC 프로토콜을 채택**했습니다. 각 요청 스트림을 서로 완전히 독립적인 패킷 단위로 처리하기 때문에, 하나의 패킷이 유실되더라도 다른 스트림의 데이터 전송에는 전혀 영향을 주지 않는 구조로 HOL Blocking을 종식시켰습니다.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

HOLB(Head-of-Line Blocking, 패킷 적체 직행 현상)는 "앞에 선 차량 한 대 때문에 뒤에 있는 모든 차량이 가지 못하는 고속도로 정체"와 같습니다.

면접관이 확인하고 싶은 것은 "HTTP/1.1 프로토콜의 구조적 한계로 인해 발생하는 애플리케이션 레벨의 HOLB가 무엇인지 정확히 이해하고, 이것이 HTTP/2와 HTTP/3로 진화하면서 전송 계층(TCP/UDP) 레벨의 HOLB로 어떻게 양상이 변화하고 해결되었는지 전체 맥락을 짚을 수 있는가?"입니다.

### 답변 분석

- **직관적인 개념 정의** : HOLB의 본질을 한 줄로 명확히 정의하여 면접관이 흐름을 쉽게 따라오도록 유도했습니다.
- **두 가지 레벨(HTTP vs TCP)의 철저한 이분법적 분석** : 많은 지원자가 HOLB를 이야기할 때 HTTP/1.1의 문제만 언급합니다. 하지만 골드 답변은 "HTTP/1.1의 애플리케이션 레벨 HOLB"와 "HTTP/2에서 여전히 남아있던 TCP 레벨의 HOLB"를 명확히 구분하여 설명했습니다. 이 차이를 아는 것만으로도 네트워크의 내부 동작 원리를 깊이 이해하고 있다는 인상을 줍니다.
- **프로토콜 진화 흐름과의 유기적 연결** : 단순히 현상 설명에 그치지 않고, 그 현상 때문에 HTTP/2(멀티플렉싱)와 HTTP/3(UDP/QUIC)가 탄생할 수밖에 없었던 당위성을 논리적으로 풀어내어 답변의 완성도를 극대화했습니다.

### 핵심 키워드

- Head-of-Line Blocking (선두 요청 지연으로 인한 후속 대기)
- HTTP/1.1 애플리케이션 레벨 HOLB (순서 보장 규칙 때문)
- TCP 전송 계층 레벨 HOLB (신뢰성 보장을 위한 재전송 대기)
