```text
HTTP/1.1과 HTTP/2의 차이점은 무엇인가요?
```

## 😎 답변 및 꼬리 질문

### 답변

HTTP/1.1과 HTTP/2의 가장 근본적인 차이는 '하나의 TCP 커넥션을 얼마나 효율적으로 활용하여 데이터를 동시에 전송할 수 있는가'에 있습니다. 이 차이로 인해 HTTP/2는 대역폭 낭비를 줄이고 웹 로딩 속도를 획기적으로 개선했습니다.

핵심적인 차이점은 크게 세 가지입니다.

첫째, **멀티플렉싱(Multiplexing)의 도입**입니다. HTTP/1.1은 하나의 커넥션에서 한 번에 하나의 요청만 순서대로 처리할 수 있어 앞선 요청이 지연되면 뒤의 요청들이 대기하는 애플리케이션 레벨의 HOL Blocking 문제가 있었습니다. 반면 HTTP/2는 데이터를 잘게 쪼갠 '바이너리 프레임' 단위로 섞어서 전송하기 때문에, 하나의 커넥션 위에서 순서와 상관없이 수많은 요청과 응답을 동시에 비동기적으로 처리할 수 있습니다.

둘째, **데이터 메시지의 포맷 변경**입니다. HTTP/1.1은 사람이 읽을 수 있는 일반 '텍스트 텍스트' 기반으로 메시지를 전송했지만, HTTP/2는 컴퓨터가 파싱하기 훨씬 빠르고 압축률이 높은 **'바이너리 프레임(Binary Frame)'** 구조로 변경되어 파싱 오버헤드를 대폭 줄였습니다.

셋째, 헤더 압축(HPACK)과 서버 푸시(Server Push)입니다. HTTP/1.1은 매 요청마다 중복된 수많은 헤더 정보를 그대로 전송하여 대역폭을 낭비했습니다. HTTP/2는 **HPACK** 알고리즘을 사용해 중복 헤더를 압축하고 변경된 부분만 전송합니다. 또한, 클라이언트가 요청하지 않아도 서버가 판단하여 필요한 리소스(CSS, JS 등)를 미리 보내주는 **서버 푸시** 기능이 추가되었습니다.

### 꼬리 질문

    - [ ]  HTTP/2가 하나의 TCP 커넥션을 공유하는 멀티플렉싱을 사용하기 때문에, 네트워크 패킷 유실이 심한 환경에서는 오히려 HTTP/1.1보다 전체적인 전송 속도가 느려질 수 있다고 하는데, 그 원인이 무엇인가요? (전송 계층 레벨 HOLB 유도)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

HTTP/1.1과 HTTP/2의 차이는 단순히 "버전이 바뀌어서 더 빨라졌다"가 아닙니다. 면접관이 진짜 보고 싶은 것은 두 버전 사이의 '패킷 전송 방식의 근본적인 패러다임 전환'을 이해하고 있는가입니다.

텍스트 기반의 동기식 프로토콜(1.1)이 어떻게 바이너리 프레임 기반의 비동기식 멀티플렉싱 프로토콜(2)로 진화했는지, 그리고 이 변화가 웹 백엔드 및 프론트엔드 최적화에 어떤 실무적 영향을 미쳤는지 논리적으로 비교할 수 있는지 검증하려는 질문입니다.

### 답변 분석

- **구조적 비교를 통한 가독성 극대화** : 단순히 두 버전을 나열하는 대신, 전송 방식(멀티플렉싱), 포맷(바이너리), 부가 기능(헤더 압축/서버 푸시)이라는 **3가지 명확한 축을 기준으로 대칭 비교**하여 답변이 매우 체계적이라는 인상을 줍니다.
- **이전 개념(HOLB)과의 자연스러운 링크** : 앞에서 배웠던 HTTP/1.1의 고질적인 한계인 'HOL Blocking'을 언급하고, HTTP/2가 '바이너리 프레임 기반 멀티플렉싱'으로 이를 해결했다는 인과관계를 매끄럽게 연결했습니다.
- **성능 최적화 키워드 선점** : HPACK 헤더 압축, 서버 푸시, 바이너리 프레임 파싱 오버헤드 감소 등 웹 성능을 논할 때 가장 가점 요소가 되는 실무 키워드들을 적재적소에 완벽하게 녹여냈습니다.

### 핵심 키워드

- 멀티플렉싱 (바이너리 프레임 기반의 비동기 병렬 전송)
- 텍스트 기반(1.1) vs 바이너리 프레임 기반(2.0)
- HPACK 헤더 압축 및 서버 푸시
