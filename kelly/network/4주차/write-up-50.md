## ❓질문

```text
빨리 끊어야 할 경우엔, (즉, 4-way Handshake를 할 여유가 없다면) 어떻게 종료할 수 있을까요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

4-Way Handshake를 수행할 시간적 여유가 없거나 시스템에 심각한 오류가 발생해 연결을 즉시 파괴해야 할 경우, TCP는 **헤더의 RST(Reset) 플래그 비트를 1로 설정한 패킷을 전송하여 즉각적인 '강제 종료(Abrupt Termination)'를 수행**합니다.

이 방식은 정상 종료와 구별되는 명확한 하부 레이어 메커니즘을 가집니다.

**1. 패킷 식별 및 전송량 (`RST = 1`)**

- 송신 측은 TCP 헤더의 제어 비트 중 **`RST` 플래그만 `1`로 켜서(000100)** 상대방에게 단 하나의 패킷만 던집니다. 4-Way처럼 4번의 티키타카를 할 필요가 없이 단 1번의 이동(0-RTT 급)으로 상황이 종료됩니다.

**2. 소켓 자원의 즉각적인 해제 및 `TIME_WAIT` 생략**

- **정상 종료(`FIN`)**: 소켓을 닫은 후에도 분실 패킷 방역을 위해 클라이언트가 `TIME_WAIT` 상태로 변해 몇 분간 메모리 포트를 붙잡고 있습니다.
- **강제 종료(`RST`)**: RST 패킷을 보내거나 받는 순간, 양쪽 호스트의 OS 커널은 **`TIME_WAIT` 단계를 완전히 생략(Bypass)하고 즉시 소켓 상태를 `CLOSED`로 전환하여 메모리와 포트 자원을 즉각 회수**합니다. 수신 측 애플리케이션(자바 등)에는 `Connection reset by peer` 같은 익셉션이 던져집니다.

**3. 실무 백엔드 인프라에서의 실제 활용 예시**
백엔드 엔지니어가 이 RST 강제 종료를 의도적으로 활용하거나 마주하는 대표적인 상황은 다음과 같습니다.

- **서버의 포트 고갈 방지 (SO_LINGER 옵션 튜닝)**: 대규모 트래픽 환경에서 서버가 연결을 먼저 끊다 보면 `TIME_WAIT` 소켓이 수만 개 쌓여 포트가 고갈되는 장애가 생깁니다. 이때 자바 소켓 옵션인 **`SO_LINGER` 시간 설정을 `0`초로 강제 세팅**하면, 파일 스트림을 닫을 때 `FIN` 대신 `RST`가 날아가 `TIME_WAIT` 없이 포트가 즉시 반환됩니다.
- **비정상 패킷 방어 및 예외 처리**: 서버가 8080 포트만 열어두었는데 클라이언트가 9090 포트로 잘못된 `SYN`을 찌르거나, 핸드셰이크가 맺어지지 않은 상태에서 뜬금없이 데이터 패킷이 들어오면, 서버 커널은 "난 그런 세션 모른다"라며 **`RST` 패킷을 응답으로 던져 상대방의 연결 시도를 강제로 멸절**시킵니다.

요약하자면, 4-Way Handshake를 할 여유가 없을 때는 **TCP 헤더의 RST 스위치를 켜서 단 한 발의 패킷으로 소켓과 포트 자원을 즉시 증발시키는 강제 리셋 방식을 사용**합니다.

    - [ ]  자바/스프링 백엔드 로그에 **`java.io.IOException: Connection reset by peer`** 라는 에러가 붉게 도배되는 장애 상황을 겪었습니다. 방금 배운 TCP RST 매커니즘 관점에서, 우리 서버와 통신하던 클라이언트나 외부 인프라 장비단에 어떤 구체적인 물리적 문제가 발생했을 때 이 로그가 찍히게 되는지 원인을 역추적해 주세요.
    - [ ]  보안 및 네트워크 장비 중에서, 공격자가 중간에 끼어들어 정상적인 두 호스트 간의 TCP 통신을 강제로 끊어버리기 위해 위조된 **가짜 RST 패킷을 양쪽에 마구 찔러넣는 공격** 기법을 무엇이라 하나요? (예: **TCP RST Injection / 세션 하이재킹 및 차단 공격** 유도)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 이 질문을 통해 확인하고 싶은 핵심은 "TCP의 정상 종료(Graceful)와 강제 종료(Abrupt)의 메커니즘 차이"를 알고 있는가입니다.

4-Way Handshake는 자원을 차분히 정리하는 'Graceful Close'인 반면, **RST 플래그**를 던지는 방식은 자원 청소고 대기(TIME_WAIT)고 뭐고 즉시 소켓을 날려버리는 방식입니다.

이 방식이 작동하는 물리적 원리와 실무에서 어떤 장애를 막기 위해 쓰이는지 보려는 것입니다.

### 답변 분석

- **명확한 플래그(`RST`) 제어권 명시**: 4-Way의 `FIN`과 완벽히 대조되는 **`RST (Reset)` 플래그의 명칭과 비트 토글 원리**를 정확히 타겟팅했습니다.
- **아키텍처적 트레이드오프(`TIME_WAIT` 생략) 터치**: 강제 종료 시 커널 레벨에서 **`TIME_WAIT` 방역 단계를 생략하고 포트를 즉시 회수한다**는 내부 현상을 짚어내어 기본기의 단단함을 입증했습니다.
- **실무 소켓 옵션(`SO_LINGER`) 결합**: 자바/네트워크 프로그래밍의 실무 치트키인 `SO_LINGER = 0` 설정을 예시로 들어, 포트 고갈 장애를 해결하는 백엔드 아키텍트의 실전 감각을 완벽히 어필했습니다.

### 핵심 키워드

- TCP 헤더의 RST(Reset) 플래그를 통한 1회성 강제 종료 (Abrupt Termination)
- 커널 소켓 자원의 즉시 회수 및 `TIME_WAIT` 대기 상태의 생략
- 자바/OS 소켓 인터페이스의 `SO_LINGER = 0` 타임아웃 설정을 통한 실무 적용
