## 😎 답변 및 꼬리 질문

### 답변

네, 핸드셰이크의 패킷 왕복 수를 완전히 없애고 첫 패킷부터 데이터를 전송하는 **0-RTT(Zero Round Trip Time) 기법**은 기존의 전통적인 TCP 구조에서는 불가능하며, 현대 웹 표준인 **UDP 기반의 QUIC(HTTP/3) 프로토콜과 TLS 1.3 보안 표준의 결합**을 통해 구현 가능합니다.

0-RTT가 작동하는 핵심 원리는 '첫 번째 연결 시 교환한 암호화 키를 기억해 두었다가, 재연결 시 연결 수립 패킷에 비즈니스 데이터를 처음부터 함께 실어 보내는 방식'입니다. 구체적인 동작 시퀀스는 다음과 같습니다.

**1. 최초 연결 단계 (1-RTT 핸드셰이크)**

- 클라이언트와 서버가 사상 처음으로 연결할 때는 0-RTT가 불가능하며, 1-RTT(1번의 왕복)로 연결을 수립합니다.
- QUIC은 전통적인 TCP 3-Way Handshake(L4)와 TLS 암호화 핸드셰이크(L7)를 하나의 레이어로 통합했습니다. 따라서 단 1번의 왕복 패킷(`Client Hello` <-> `Server Hello`)만으로 전송 통로 개방과 암호화 키 교환을 동시에 끝냅니다.
- 이때, 서버는 클라이언트의 신원을 인증하는 일종의 티켓인 **'정식 세션 티켓(Session Ticket)'** 또는 '사전 공유 키(PSK, Pre-Shared Key)'를 발급하여 클라이언트 브라우저에 저장시킵니다.

**2. 재연결 단계 (0-RTT 핸드셰이크)**

- 사용자가 페이지를 새로고침하거나, 앱을 껐다 켜서 동일한 서버에 재연결(Session Resumption)을 시도할 때 0-RTT가 발동합니다.
- 클라이언트는 최초 연결 때 받아두었던 사전 공유 키(PSK)를 꺼내어, 이를 기반으로 보낼 HTTP 요청 데이터(GET, POST 등의 비즈니스 Payload)를 **처음부터 암호화**합니다.
- 그리고 서버에 연결 요청을 찌르는 첫 번째 첫 패킷(`Client Hello`)을 보낼 때, **그 패킷 내부에 암호화된 HTTP 데이터 본문을 함께 실어서(Piggyback)** 날려버립니다.
- 서버는 패킷을 받자마자 자신이 과거에 발급했던 PSK 쿠키를 검증해 데이터를 즉시 복호화하고, **연결 수락 신호(`Server Hello`)와 함께 유저가 요청한 웹 데이터 결과물(HTTP Response)을 바로 응답 패킷에 실어 돌려줍니다.**

결과적으로 클라이언트 입장에서는 연결이 맺어지기를 기다리는 시간(RTT)을 단 0초도 낭비하지 않고, **첫 번째 패킷 요청과 동시에 서버로부터 데이터를 받아오는 극단의 성능 최적화**를 달성하게 됩니다.

### 꼬리 질문

- 🤗 스포 방지!

  0-RTT의 고도화된 최적화 원리를 완벽히 이해했기 때문에, 면접관은 이 0-RTT가 가진 '치명적인 보안상의 약점(Replay Attack)'을 아는지 꼬리 질문을 던져 보안 의식 수준을 테스트하려 할 것입니다.

    - [ ]  0-RTT는 첫 번째 패킷에 암호화된 데이터 본문(Request)이 그대로 담겨 날아갑니다. 만약 중간에 해커가 이 0-RTT 패킷을 통째로 가로챈 뒤, 서버를 향해 똑같은 패킷을 수만 번 그대로 다시 전송(중복 요청)하는 '재전송 공격(Replay Attack)'을 감행하면 서버는 속아 넘어가게 됩니다. 만약 그 데이터가 '1억 원 송금 요청'이었다면 끔찍한 금융 사고가 날 텐데, 이를 방지하기 위해 0-RTT 환경에서 백엔드 엔지니어는 어떤 조치를 취해야 할까요? (예: **`GET` 요청 같은 조회성(Idempotent/Idempotency) 메서드에만 0-RTT를 제한적으로 허용**하고, 상태를 바꾸는 `POST/PUT`은 금지하는 정책 유도)
    - [ ]  QUIC 프로토콜이 0-RTT/1-RTT를 구현할 수 있는 또 다른 무기는, 기존 TCP의 무거운 핸드셰이크를 버리고 **가볍고 제어권이 열려있는 UDP**를 선택했기 때문입니다. 그렇다면 UDP 자체에는 원래 신뢰성이 없는데, QUIC은 패킷 유실이나 순서 꼬임 문제를 어떻게 자체적으로 해결하나요? (예: 애플리케이션 소스 코드 레이어에 구현된 독립 스트림 및 자체 ARQ 메커니즘)
    - [ ]  스프링 부트 서버 앞단에 최신 **Nginx** 웹 서버를 프록시로 두고 HTTP/3(QUIC)를 활성화하려고 합니다. 이때 클라이언트 브라우저가 우리 서버가 HTTP/3를 지원한다는 사실을 최초에 인지하고 다음부터 QUIC 요청을 보내도록 만들기 위해, 백엔드/Nginx 서버가 HTTP 응답 헤더에 강제로 심어 보내야 하는 특수 헤더의 명칭은 무엇인가요? (예: **`Alt-Svc` (Alternative Services) 헤더** 규격 유도)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 확인하고 싶은 골드 포인트는 두 가지입니다.

1. **기술적 단절의 인지**: 0-RTT(Round Trip Time, 왕복 시간 0번)는 기존 TCP 아키텍처의 한계를 깨부수고 등장한 **UDP 기반의 QUIC 프로토콜**의 영역임을 명확히 인지하고 있는가.
2. **0-RTT의 핵심 원리**: 첫 번째 연결 때 발급받은 사전 공유 키(PSK, Pre-Shared Key)를 재사용하여, 두 번째 연결부터는 연결 수립 패킷에 암호화된 비즈니스 데이터(HTTP Request)를 **'동시에 실어서'** 던지는 메커니즘을 이해하고 있는가.

### 답변 분석

- **기존 지식과의 모순점 정면 돌파**: 앞서 설명한 3-Way Handshake(TCP)의 필수성과 모순되지 않도록, "TCP를 버리고 UDP 기반의 QUIC과 TLS 1.3을 융합했기 때문에 스펙적으로 가능해진 것"이라고 아키텍처 패러다임의 전환을 명쾌하게 짚었습니다.
- **핵심 메커니즘(PSK)의 정확한 정의**: 0-RTT의 심장인 사전 공유 키(Pre-Shared Key)와 최초 연결(1-RTT) -> 재연결(0-RTT)로 이어지는 세션 재개(Session Resumption)의 인과관계를 타임라인 단위로 완벽하게 복기했습니다.
- **비즈니스 데이터 탑재 방식의 구체화**: 첫 번째 패킷에 "암호화된 HTTP Payload를 함께 실어서 던진다"는 물리적 현상을 생생하게 묘사하여 면접관의 고개를 끄덕이게 만듭니다.
