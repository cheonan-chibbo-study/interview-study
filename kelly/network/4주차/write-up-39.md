## ❓질문

```text
본인이 새로운 통신 프로토콜을 TCP나 UDP를 사용해서 구현한다고 하면, 어떤 기준으로 프로토콜을 선택하시겠어요?
```

## 😎 답변 및 꼬리 질문

### 답변

제가 새로운 통신 프로토콜을 설계한다면, 단순히 '속도냐 신뢰성이냐'의 이분법적 논리를 넘어, '데이터 유실이 비즈니스에 치명적인가', '허용 가능한 한계 레이턴시(지연 시간)는 얼마인가', 그리고 '애플리케이션 레이어에서 신뢰성을 직접 구현할 수 있는 엔지니어링 비용이 확보되었는가'라는 3가지 축을 기준으로 선택하겠습니다.

첫째, '데이터 무결성이 비즈니스의 생명줄인가'를 보겠습니다.
만약 파일 전송, 금융 거래, 회원 인증처럼 단 1비트의 오류나 유실도 전체 시스템의 붕괴나 금전적 손실로 이어지는 도메인이라면 **고민 없이 TCP를 선택**하겠습니다. TCP가 제공하는 커널 레벨의 흐름·혼잡·오류 제어 메커니즘은 수십 년간 검증된 가장 안전한 방어벽이며, 이를 선택함으로써 개발자는 비즈니스 로직에만 온전히 집중할 수 있기 때문입니다.

둘째, '네트워크 패킷 레이턴시가 유실의 가치보다 압도적으로 중요한가'를 보겠습니다.
동영상 스트리밍, 실시간 VoIP 통화, 대규모 멀티플레이어 FPS 게임 등은 패킷이 0.1초만 늦게 와도 유저 경험이 완전히 망가집니다. 이전 프레임이 유실되었다고 해서 TCP처럼 재전송을 기다리느라 화면이 멈추는 것보다, 유실된 패킷은 과감히 버리고(Drop) 다음 최신 좌표나 프레임을 즉시 렌더링하는 것이 이득입니다. 이 경우 **주저 없이 UDP를 선택**하겠습니다.

셋째, 'TCP의 신뢰성은 필요하지만 TCP의 구조적 한계(HOLB, 핸드셰이크 지연)를 극복해야 하는 최첨단 성능 중심의 도메인인가'를 고려하겠습니다.
만약 글로벌 초고속 웹 통신이나 모바일 핸드오버(Wi-Fi와 LTE 간 이동) 환경에서 극단의 최적화가 필요하다면, **가벼운 UDP를 하부 통로로 선택하되, 유실된 패킷의 재전송이나 순서 보장 같은 신뢰성 메커니즘을 애플리케이션 레이어(혹은 QUIC 같은 커스텀 라이브러리)에서 직접 소프트웨어 코드로 구현하는 방식**을 채택하겠습니다.

다만, 이 방식은 TCP 커널이 알아서 해주던 복잡한 동기화와 흐름 제어 로직을 우리 팀의 소스 코드로 직접 짜야 하므로, **프로젝트의 타임라인과 팀원들의 네트워크 프로그래밍 숙련도(엔지니어링 비용)를 냉정하게 저울질하여 최종 의사결정**을 내리겠습니다.

    - [ ]  현실 세계의 공공기관이나 기업용 인프라(방화벽/NAT 장비) 중에는 보안상의 이유로 TCP 이외의 모든 프로토콜(UDP 등) 패킷을 무조건 유해 트래픽으로 간주하여 차단(Drop)하도록 보수적으로 설정된 방화벽이 아주 많습니다. 만약 우리가 UDP 기반의 커스텀 프로토콜을 출시했는데 이런 인프라 장비 때문에 패킷이 통과하지 못한다면, 백엔드 아키텍트로서 어떤 차선책(Fallback)을 준비하시겠습니까? (예: UDP 통신 실패 시 즉시 안정적인 TCP/TLS 통신으로 전환하는 Fallback 메커니즘 설계)
    - [ ]  사용자가 대용량 파일을 업로드하는 백엔드 서버를 설계할 때, 네트워크 레이턴시가 긴 국가(예: 한국에서 미국 서버로 업로드) 환경이라면 일반적인 단일 TCP 연결보다 **UDP 기반 전송 솔루션(Aspera 등)이 압도적으로 빠릅니다.** TCP의 어떤 제어 알고리즘 때문에 장거리 통신에서 속도 저하가 발생하는 것일까요? (예: 혼잡 제어의 윈도우 크기 위축 및 대역폭-지연곱 / BDP 문제)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 이 질문을 통해 확인하고 싶은 핵심은 세 가지입니다.

1. **트레이드오프 판단력**: 완벽한 기술은 없으며, 서비스 특성에 따라 무엇을 포기하고 무엇을 취할지 엔지니어링적 결단을 내릴 수 있는가?
2. **인프라/인력 비용 고려**: UDP를 선택했을 때 뒤따르는 '애플리케이션 레이어에서의 직접 구현 비용(시간, 인력, 버그 리스크)'을 현실적으로 인지하고 있는가?
3. **현대 기술 트렌드 이해**: 최신 프로토콜(HTTP/3, QUIC)이 가벼운 UDP 위에 신뢰성을 얹은 것처럼, 이러한 패러다임을 내 설계에 녹여낼 수 있는가?

### 답변 분석

- **입체적인 3대 설계 축 제시**: 초보적인 '속도 vs 안정성' 프레임을 깨고, 비즈니스 치명도, 한계 레이턴시, 개발 비용(엔지니어링 리소스)이라는 시니어급 관점을 명확한 축으로 세웠습니다.
- **도메인 특성과의 1:1 매핑**: 금융/인증(TCP), 스트리밍/게임(UDP)이라는 명확한 실무 예시를 들어 기준의 당위성을 입증했습니다.
- **현실적인 개발 비용(현실 감각) 터치**: 많은 지원자가 무작위로 "성능을 위해 UDP 위에 직접 구현하겠습니다!"라고 호기롭게 외치지만, 면접관은 '그게 얼마나 버그가 많이 생기고 짜기 힘든데 쉽게 말하나'라고 생각합니다. 답변의 마지막에 "엔지니어링 리소스를 냉정하게 저울질하겠다"는 현실적 트레이드오프를 언급함으로써 깊은 신뢰감을 줍니다.

### 핵심 키워드

- 비즈니스적 데이터 유실 치명도 (금융/인증 vs 스트리밍/게임)
- 한계 레이턴시 수용 범위 (TCP의 재전송 지연 감수 여부)
- 애플리케이션 레이어 직접 구현에 따른 엔지니어링 비용 계산
