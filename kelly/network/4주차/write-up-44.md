## 😎 답변 및 꼬리 질문

### 답변

TCP가 연결을 맺을 때 효율적인 2-Way Handshake를 채택하지 않고 반드시 3-Way Handshake를 고수하는 이유는 '양방향 통신 경로의 완벽한 상호 신뢰성 검증'과 '네트워크 지연 패킷으로 인한 서버의 유령 연결(Ghost Connection) 및 자원 고갈을 방지하기 위해서'입니다.

만약 악수(Handshake)를 단 2번만 주고받는 **2-Way 방식을 도입했을 때 발생하는 결정적인 두 가지 문제점**은 다음과 같습니다.

**첫째, 양방향(Full-Duplex) 신뢰성 검증의 불완전성입니다.**
TCP 통신은 클라이언트에서 서버로 가는 통로와, 서버에서 클라이언트로 오는 통로가 모두 정상인지 확인해야 합니다.

- 1단계(`SYN`): 클라이언트의 송신 능력이 정상이고, 서버의 수신 능력이 정상임을 확인합니다.
- 2단계(`SYN+ACK`): 서버의 송신 능력이 정상이고, 클라이언트의 수신 능력이 정상임을 확인합니다.
  하지만 **여기서 과정을 끝내버리면(2-Way), 서버는 '클라이언트가 내 목소리(SYN+ACK)를 제대로 들었는지' 최종 확답을 받을 길이 없습니다.** 클라이언트가 수신 불능 상태에 빠졌더라도 서버는 알 방법이 없으므로, 일방적으로 연결이 수립되었다고 신뢰하는 논리적 모순이 생깁니다.

**둘째, '지연된 유실 패킷(Delayed 패킷)'으로 인한 서버 자원 누수 장애입니다.**
2-Way 방식에서는 서버가 클라이언트의 `SYN`을 받고 `SYN+ACK`를 보내는 즉시 연결 상태를 `ESTABLISHED`로 전환하고 소켓 자원을 할당해야 합니다. 이 메커니즘은 다음과 같은 대형 장애 시나리오를 유발합니다.

1. 클라이언트가 서버에 연결 요청(`SYN_1`)을 보냈으나, 네트워크 혼잡으로 인해 이 패킷이 라우터 어딘가에 갇혀 지연(Delayed)되었습니다.
2. 클라이언트는 타임아웃이 발생하여 새롭게 연결 요청(`SYN_2`)을 다시 보냈고, 이는 정상적으로 3-Way 연결을 맺고 통신을 완료한 뒤 안전하게 종료되었습니다.
3. 그런데 한참 뒤, 과거에 길을 잃었던 첫 번째 패킷(`SYN_1`)이 뒤늦게 서버에 도착합니다.
4. **2-Way 환경이라면, 서버는 이것이 옛날에 취소된 유령 패킷인지 모른 채**, 곧바로 연결 수락 패킷(`SYN+ACK`)을 보내며 혼자서 연결 수립 상태(`ESTABLISHED`)로 들어가 소켓 메모리 자원을 할당해 버립니다.
5. 정작 클라이언트는 이미 통신을 끝내고 커넥션을 잊은 상태이므로 서버의 요청을 무시합니다.

결과적으로 서버는 **대답 없는 클라이언트를 상대로 유령 연결(Ghost Connection)을 유지하며 커널 메모리와 소켓 자원을 무한정 낭비**하게 되며, 대규모 트래픽 환경에서는 서버가 다운되는 치명적인 결함으로 이어집니다.

따라서 TCP는 클라이언트가 최종적으로 "나도 네 목소리 잘 들었어"라고 확답을 주는 **3단계 최종 ACK**를 받아야만 서버가 자원을 할당하도록 설계함으로써, 이러한 네트워크 유령 패킷 오작동을 완벽하게 원천 차단하고 있습니다.

### 꼬리 질문

- 🤗 스포 방지!

  2-Way의 치명적인 자원 누수 결함과 3-Way의 당위성을 완벽히 논증했기 때문에, 면접관은 "그렇다면 3-Way가 완벽한 건 알겠는데, 연결을 끊을 때는 왜 3번이 아니라 4번(**4-Way Handshake**)을 해야 하는가?"라며 자연스럽게 종료 프로세스로 파고들 것입니다.

    - [ ]  연결을 수립할 때 쌍방 검증을 위해 3번이면 충분하다고 하셨는데, 반대로 **TCP 연결을 해제할 때(FIN)는 왜 3-Way가 아니라 한 단계 더 많은 4-Way Handshake 과정**을 거쳐야만 하는지 아키텍처 관점의 이유를 설명해 주세요. (예: 반오픈 / Half-Close 상태의 잔여 데이터 전송 보장 유도)
    - [ ]  방금 2-Way 방식에서 '옛날 지연 패킷'이 뒤늦게 서버에 들어와 유령 소켓을 만든다고 하셨습니다. 그렇다면 현대 TCP 3-Way Handshake 환경에서는, 아주 재수가 없어서 **옛날 지연 패킷의 시퀀스 번호와 현재 새롭게 열린 패킷의 시퀀스 번호가 우연히 완벽히 일치**하여 서버가 속아 넘어가는 현상을 방지하기 위해 최초 시퀀스 번호(ISN)를 어떻게 생성하나요? (예: 난수 기반 및 시간 기반의 클럭 조합 생성 스펙 유도)
    - [ ]  만약 3-Way Handshake의 마지막 3단계인 클라이언트의 `ACK` 패킷이 네트워크 상에서 유실되어 서버에 도달하지 못했다면, 서버와 클라이언트는 각각 자신의 소켓 상태를 어떻게 처리하고 대응하나요? (예: 서버의 `SYN+ACK` 재전송 메커니즘)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 확인하고 싶은 골드 포인트는 두 가지입니다.

1. **양방향 통신(Full-Duplex)의 특성 이해**: TCP는 일방향이 아닌 '쌍방향' 프로토콜이므로, 송신과 수신 통로 모두가 안전한지 확인하려면 수학적으로 최소 3번의 티키타카가 필수적임을 알고 있는가.
2. **네트워크 유실 및 지연 패킷(Ghost Connection) 문제**: 과거에 길을 잃었던 패킷이 뒤늦게 도착했을 때, 2-Way 환경에서는 서버가 이를 정상 요청으로 오인해 **자원을 헛되이 낭비하는 치명적인 아키텍처적 결함**이 발생함을 인지하고 있는가.

### 답변 분석

- **이분법적 기준을 넘은 아키텍처적 증명**: 단순히 "쌍방향이라서 그렇다"라는 뻔한 대답을 넘어, **'지연된 패킷(Delayed Packet)'이 유발하는 실제 서버 인프라 장애 시나리오**를 완벽하게 서사 구조로 증명했습니다. 면접관이 무조건 감탄하는 포인트입니다.
- **소켓 상태 머신의 모순 지적**: 2-Way 체제에서 서버가 무작정 `ESTABLISHED` 상태로 먼저 진입할 때 발생하는 **자원 선할당의 위험성**을 논리적으로 짚어 깊이감을 입증했습니다.

### 핵심 키워드

- 클라이언트의 최종 수신 능력 검증 불가 (양방향 신뢰성 붕괴)
- 지연된 구형 패킷(Delayed Packet)의 재유입 리스크
- 서버 혼자 자원을 낭비하는 유령 연결(Ghost Connection) 방지
