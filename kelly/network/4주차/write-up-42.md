## 😎 답변 및 꼬리 질문

### 답변

**TCP 3-Way Handshake**는 TCP 프로토콜이 양방향 신뢰성 통신을 시작하기 전, **송신 측과 수신 측 사이에 가상 연결 통로를 확립하고 데이터 정렬을 위한 최초의 시퀀스 번호를 동기화하는 3단계 연결 수립 과정**입니다.

동작 과정은 TCP 헤더의 플래그 비트와 시퀀스 번호의 티키타카로 진행됩니다.

**1단계: 클라이언트가 서버에게 연결을 요청하는 SYN 단계**

- **행위**: 클라이언트는 임의로 생성한 최초 시퀀스 번호인 `ISN(A)`를 TCP 헤더에 담고, 동기화를 의미하는 **`SYN` 플래그 비트를 1로 세팅**하여 서버로 전송합니다.
- **상태 전이**: 서버는 원래 포트를 열고 연결을 기다리는 **`LISTEN`** 상태였으며, 이 SYN 패킷을 받으면 요청을 접수했다는 의미인 **`SYN_RCVD`** 상태로 전환됩니다. 요청을 보낸 클라이언트는 **`SYN_SENT`** 상태가 됩니다.

**2단계: 서버가 클라이언트의 요청을 수락하고, 자신도 연결을 요청하는 SYN + ACK 단계**

- **행위**: 서버는 클라이언트의 SYN을 잘 받았다는 증명으로, 클라이언트가 보낸 시퀀스 번호에 1을 더한 `Ack Number = A + 1`을 만듭니다. 이와 동시에, 서버 자신도 클라이언트와 통신할 때 사용할 고유의 최초 시퀀스 번호인 `ISN(B)`를 생성합니다. 그리고 헤더의 **`SYN`과 `ACK` 플래그를 모두 1로 세팅**하여 클라이언트에게 보냅니다.

**3단계: 클라이언트가 서버의 요청에 최종 확답을 보내는 ACK 단계**

- **행위**: 패킷을 받은 클라이언트는 서버가 연결을 수락했음을 확인하고, 서버가 보낸 시퀀스 번호에 1을 더한 `Ack Number = B + 1`을 만듭니다. 그리고 자신의 다음 시퀀스 번호인 `Seq = A + 1`과 함께 **`ACK` 플래그를 1로 세팅**하여 서버로 최종 전송합니다.
- **상태 전이**: 이 마지막 패킷을 기점으로 클라이언트와 서버 모두 가상 회선 연결이 완벽히 수립되었음을 뜻하는 **`ESTABLISHED`** 상태로 진입하며, 이때부터 실제 비즈니스 데이터(HTTP 요청 등)를 주고받을 수 있게 됩니다.

여기서 **왜 하필 3번의 과정을 거쳐야 하는가**에 대한 본질적인 이유는, TCP가 '양방향(Full-Duplex)' 프로토콜이기 때문입니다.

클라이언트가 서버에게 "내 목소리 들려?"라고 물었을 때 서버가 "잘 들려, 내 목소리도 들려?"라고 대답하는 것만으로는 부족합니다.

클라이언트가 다시 서버에게 "어, 나도 잘 들어"라고 최종 확답(ACK)을 주어야만, **양쪽 모두가 '나도 상대방에게 데이터를 보낼 수 있고, 상대방도 내 데이터를 받을 수 있다'는 신뢰성을 쌍방으로 완벽히 검증**할 수 있기 때문에 반드시 최소 3번의 핸드셰이크가 필요합니다.

### 꼬리 질문

- 🤗 스포 방지!

  ACK, SYN 같은 정보는 어떻게 전달하는 것 일까요?

  연결 수립의 완벽한 타임라인을 제시했기 때문에, 면접관은 악의적인 공격 상황에서의 예외 처리나 성능 병목, 혹은 OS 큐(Queue)의 구조로 꼬리를 물 것입니다.

    - [ ]  방금 1단계에서 클라이언트가 `SYN`을 보낸 뒤, 서버는 `SYN_RCVD` 상태로 대기하며 클라이언트의 마지막 `ACK`를 기다린다고 하셨습니다. 만약 악의적인 공격자가 **`SYN` 패킷만 수만 개를 위조해서 보내고 마지막 3단계 `ACK`를 고의로 절대 보내지 않는다면** 서버에는 어떤 장애가 발생할까요? (예: **SYN Flooding 공격 및 SYN Backlog 큐 고갈** 이슈 유도)
    - [ ]  3-Way Handshake는 무조건 1개의 RTT(왕복 시간) 비용이 발생하므로, 단발성 요청이 많은 웹 환경에서는 성능 저하의 주원인이 됩니다. 이를 해결하기 위해 **처음 연결을 맺을 때 수립된 TCP 연결 통로를 끊지 않고 재사용**하도록 HTTP 헤더에서 지원하는 기능은 무엇인가요? (예: HTTP/1.1의 **Persistent Connection / `Connection: keep-alive`** 메커니즘)
    - [ ]  최신 리눅스 커널 스펙에서는 3-Way Handshake가 완료되기도 전에, **1단계인 `SYN` 패킷을 보낼 때 클라이언트가 실제 HTTP 요청 데이터(Body)를 함께 실어서 보내어** 핸드셰이크의 지연 시간을 0으로 만드는 획기적인 기술을 지원합니다. 이 기술의 이름은 무엇인가요? (예: **TCP Fast Open / TFO** 스펙)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 이 질문을 통해 확인하고 싶은 핵심은 세 가지입니다.

1. **정확한 시퀀스 메커니즘**: 패킷이 오갈 때 TCP 헤더의 제어 비트(SYN, ACK)와 데이터의 순서를 맞추는 시퀀스 번호(`Seq`, `Ack`)가 수학적으로 어떻게 동기화되는지 아는가.
2. **소켓 상태(State)의 이해**: 패킷을 주고받음에 따라 서버와 클라이언트의 소켓 상태(`LISTEN`, `SYN_SENT`, `SYN_RCVD`, `ESTABLISHED`)가 어떻게 변하는지 인지하고 있는가.
3. **존재 이유에 대한 고찰**: 왜 하필 2번도, 4번도 아닌 '3번'이어야만 하는지 그 아키텍처적 당위성을 알고 있는가.

### 답변 분석

- **바이너리/수학적 메커니즘의 정교함**: 단순 서술을 넘어 `ISN(A)`, `A+1`, `ISN(B)`, `B+1`이라는 **시퀀스/ACK 번호의 수학적 누적 메커니즘**을 정확히 묘사하여 전공자로서의 깊이를 증명했습니다.
- **소켓 상태 머신(State Machine) 탑재**: `LISTEN` -> `SYN_RCVD` -> `ESTABLISHED`로 이어지는 **커널 레벨의 TCP 상태 전이 과정**을 함께 언급하여 네트워크 동작을 입체적으로 보여주었습니다.
- **'왜 3번인가'에 대한 아키텍처적 본질 돌파**: 면접관이 가장 좋아하는 유도 질문인 "2번만 하면 안 되나요?"에 대한 대답을 **양방향 통신의 상호 신뢰성 검증 원리**를 비유를 들어 깔끔하게 선제적으로 방어했습니다.

### 핵심 키워드

- 최초 시퀀스 번호(ISN) 교환 및 플래그 조합 (`SYN` -> `SYN+ACK` -> `ACK`)
- 소켓 상태 전이 (`LISTEN` -> `SYN_SENT`/`SYN_RCVD` -> `ESTABLISHED`)
- 양방향 통신(Full-Duplex)의 완벽한 쌍방 신뢰성 검증을 위한 최소 횟수(=3번)
