## ❓질문

```text
ACK, SYN 같은 정보는 어떻게 전달하는 것 일까요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

ACK나 SYN 같은 제어 정보는 애플리케이션의 데이터 본문에 담기는 것이 아니라, 전송 계층에서 생성되는 **HTTP 요청 데이터 앞에 붙는 'TCP 헤더(Header)'의 특정 비트(Bit) 영역을 0 또는 1로 토글하는 방식으로 전달**됩니다.

TCP 헤더는 기본적으로 20바이트(Byte)의 고정된 크기를 가지며, 컴퓨터가 읽기 가장 좋은 바이너리(이진수) 구조로 설계되어 있습니다. 이 헤더 내부에는 출발지·목적지 포트 번호, 시퀀스 번호 등과 함께 'TCP 플래그(Flags)' 또는 '제어 비트(Control Bits)'라고 불리는 **6비트(혹은 최신 스펙에서 9비트) 크기의 영역**이 존재합니다.

이 플래그 영역은 각각의 비트가 독립된 하나의 의미를 가지는 '스위치' 역할을 합니다. 대표적인 스위치들의 명칭과 메커니즘은 다음과 같습니다.

1. **SYN (Synchronize) 플래그**: 연결 수립 시 시퀀스 번호를 동기화하자는 스위치입니다.
2. **ACK (Acknowledgment) 플래그**: 승인 번호(Ack Number) 필드에 유효한 값이 담겨있으니 확인하라는 스위치입니다. (초초기 SYN 패킷을 제외한 모든 패킷은 이 스위치가 1로 켜져 있습니다.)
3. **FIN (Finish)**: 통신을 종료하겠다는 스위치입니다.
4. **RST (Reset)**: 연결에 심각한 오류가 생겨 강제로 연결을 초기화하겠다는 스위치입니다.

동작하는 물리적 방식을 예로 들면, 3-Way Handshake의 2단계인 **`SYN + ACK` 패킷**이 날아갈 때, 컴퓨터 내부 메모리(바이너리 패킷 데이터)에서는 이 플래그 비트 열이 `010010`과 같은 이진수로 세팅됩니다. 즉, SYN 자리에 해당하는 비트도 `1`로 켜고, ACK 자리에 해당하는 비트도 `1`로 켠 채 전송하는 것입니다.

수신 측의 운영체제(OS) 커널은 패킷이 도착하면 데이터 본문을 읽기 전, 이 20바이트짜리 TCP 헤더의 플래그 영역을 가장 먼저 마스킹(Bit Masking) 연산으로 읽어냅니다. 그리고 **'SYN 비트와 ACK 비트 위치에 1이 켜져 있구나'라는 것을 하드웨어 및 커널 레벨에서 즉시 판별**하여 3-Way Handshake 프로세스를 수행하게 됩니다.

### 꼬리 질문

- 🤗 스포 방지!

  TCP 헤더의 구조와 플래그 비트 메커니즘을 완벽하게 관통했기 때문에, 면접관은 헤더 내부의 다른 핵심 필드(시퀀스 번호의 데이터 크기 한계)나 인프라 보안 장비에서의 플래그 변조 공격으로 꼬리를 물 것입니다.

    - [ ]  방금 TCP 헤더의 구조를 완벽하게 설명해 주셨는데, TCP 헤더 내에서 패킷의 순서를 맞추는 '시퀀스 번호(Sequence Number)' 필드와 '승인 번호(Acknowledgment Number)' 필드는 각각 몇 비트(또는 몇 바이트)의 공간을 할당받아 사용하나요? (예: 각각 32비트 / 4바이트 할당 규격 유도)
    - [ ]  TCP 시퀀스 번호 필드가 32비트 크기라는 것은 표현할 수 있는 숫자의 한계($2^{32}-1$)가 있다는 뜻입니다. 만약 기가비트 초고속 네트워크 환경에서 엄청나게 큰 대용량 데이터를 주고받다가 **시퀀스 번호가 한계치를 넘어서 다시 0으로 돌아오는 '랩어라운드(Wrap-around)' 현상**이 발생하면, 패킷 순서가 꼬이는 문제를 TCP는 헤더의 어떤 옵션 필드로 해결하나요? (예: **TCP 타임스탬프 / Timestamp 옵션** 유도)
    - [ ]  보안 장비나 방화벽 중에서, TCP 헤더의 플래그 비트를 악의적으로 조작하여 정상적인 규격에 어긋나는 조합(예: **SYN 플래그와 FIN 플래그를 동시에 1로 켜서 보내는 공격**)을 감지하고 차단하는 보안 기술이나 공격 명칭에 대해 들어본 적이 있나요? (예: **Xmas Scan** 또는 비정상 플래그 스캔 공격 방어)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 이 질문을 통해 확인하고 싶은 핵심은 "TCP 헤더(Header)의 구조"입니다.

`SYN`, `ACK`는 어떤 문자열이나 파일 형태로 전송되는 것이 아니라, 데이터 본문(Payload) 앞에 붙는 **20바이트짜리 TCP 헤더 속 '제어 비트(Control Bits/Flags)' 영역의 특정 1비트 스위치를 켜고 끄는(0 또는 1) 방식**으로 전달된다는 패킷 레벨의 진실을 알고 있는지 검증하려는 것입니다.

### 답변 분석

- **바이너리 레벨의 명확한 메커니즘 제시**: 추상적인 텍스트가 아니라 "20바이트 고정 헤더 내의 '플래그 비트' 영역의 1비트 스위치를 0과 1로 토글한다"는 본질을 정확하게 짚었습니다.
- **현실적인 컴퓨터 연산(이진수) 대입**: `SYN+ACK` 단계를 이진수 비트열(`010010` 등)로 비유하여 설명함으로써 컴퓨터 구조와 네트워크 스펙을 깊게 이해하고 있음을 증명했습니다.
- **커널 레벨의 수신 처리 과정 묘사**: OS 커널이 패킷을 받아서 **비트 마스킹(Bit Masking) 연산**으로 플래그를 읽어낸다는 디테일을 언급하여 지식의 깊이를 한 층 더 끌어올렸습니다.

### 핵심 키워드

- 20바이트 크기의 TCP 고정 헤더 (TCP Header)
- 6비트/9비트 영역의 TCP 플래그 / 제어 비트 (Control Bits)
- 비트 스위치 토글 (0 또는 1) 및 OS 커널의 비트 마스킹 판별
