## 😎 답변 및 꼬리 질문

### 답변

SOP(Same-Origin Policy, 동일 출처 정책)는 '웹 브라우저가 서로 다른 출처(Origin)에서 로드된 문서나 스크립트가 다른 출처의 리소스와 상호작용하는 것을 차단하는 브라우저 레벨의 가장 기본적인 보안 정책'입니다. 넷스케이프 시절부터 이어져 온 웹 보안의 핵심 근간입니다.

SOP에서 말하는 **'출처(Origin)'가 같다고 판단하기 위해서는 URL의 3가지 구성 요소인 프로토콜(Scheme), 호스트(Host/도메인), 그리고 포트(Port)가 모두 완벽히 일치**해야 합니다. 이 중 단 하나라도 다르면 브라우저는 '교차 출처(Cross-Origin)'로 판단하여 리소스 접근을 제한합니다.

SOP가 브라우저에 반드시 필요한 이유는 **유저의 민감한 세션 정보와 데이터를 해커의 공격으로부터 보호하기 위해서**입니다.

만약 브라우저에 SOP 정책이 없다면 치명적인 보안 구멍이 생깁니다. 예를 들어, 사용자가 은행 사이트(`https://bank.com`)에 로그인하여 브라우저 쿠키에 세션 ID가 남아있는 상태라고 가정해 보겠습니다. 이 상태에서 사용자가 해커가 만든 악성 낚시 사이트(`https://hacker.com`)에 접속하면, 악성 사이트에 심겨진 자바스크립트 코드가 은행 서버의 이메일 조회 API(`https://bank.com/api/emails`)에 강제로 비동기 요청(Fetch/Axios)을 날릴 수 있습니다.

이때 브라우저의 특성상 은행 쿠키가 자동으로 실려 전송되므로, 은행 서버는 정상 요청으로 판정해 유저의 마이페이지나 금융 정보 데이터를 응답(Response)으로 돌려주게 됩니다. **SOP가 없다면 해커의 악성 스크립트가 이 응답 데이터에 그대로 접근(Read)하여 해커 서버로 탈취**해 갈 수 있습니다.

브라우저는 이 무서운 시나리오를 막기 위해, 스크립트가 실행된 출처(`hacker.com`)와 요청한 타겟 출처(`bank.com`)가 다르면 **서버가 돌려준 응답 데이터를 자바스크립트가 읽지 못하도록 브라우저 엔진 레이어에서 완전히 차단**해 버리는데, 이것이 바로 SOP 정책입니다.

### 꼬리 질문

- 🤗 스포 방지!

  SOP의 근본 철학과 브라우저 차단 메커니즘을 완벽하게 증명했기 때문에, 면접관은 이제 100% 확률로 "그렇다면 다른 출처 간에 합법적으로 데이터를 주고받아야 하는 현대 아키텍처에서는 SOP를 어떻게 우회하거나 허용하는가?"라며 CORS와 최신 기술로 파고들 것입니다.

    - [ ]  현대 웹 아키텍처에서는 프론트엔드 서버(예: `localhost:3000`)와 백엔드 API 서버(예: `api.com:8080`)의 출처가 다른 경우가 허다합니다. 이처럼 SOP 정책을 깨고 타 출처 간의 안전한 데이터 교환을 허용하기 위해 웹 표준으로 제정된 메커니즘은 무엇인가요? (예: **CORS - Cross-Origin Resource Sharing** 유도)
    - [ ]  브라우저의 SOP 정책은 자바스크립트를 통한 API 응답 데이터 접근(Fetch, XMLHttpRequest)은 엄격하게 막지만, HTML 태그인 **`<img>`, `<link>`, `<script>`** 등의 외부 리소스 로딩은 교차 출처라도 아무런 제약 없이 허용해 줍니다. 이렇게 설계된 이유와 이로 인해 발생할 수 있는 보안적 허점은 무엇이 있을까요?
    - [ ]  과거 CORS 표준이 정립되기 전, 오래된 레거시 브라우저 환경에서 SOP 정책을 우회하여 다른 도메인의 데이터를 자바스크립트로 가져오기 위해 꼼수로 사용했던 **JSONP(JSON with Padding)** 방식의 원리와 보안상 취약점에 대해 설명해 주세요.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

많은 개발자가 CORS 에러는 잘 알면서도, 그 에러가 왜 발생하는지 즉, **SOP라는 브라우저의 보호 장치**에 대해서는 깊게 생각하지 않는 경우가 많습니다.

면접관이 이 질문을 통해 확인하고 싶은 핵심은 "SOP가 정의하는 '출처(Origin)'의 3대 구성 요소를 정확히 아는가"와 "SOP가 없다면 해커들이 앞서 배운 쿠키/세션 자동 전송 메커니즘과 CSRF 공격을 이용해 어떻게 전 세계 웹사이트의 데이터를 무단으로 훔쳐 갈 수 있는지, 그 위험성을 인지하고 있는가"입니다.

### 답변 분석

- **출처(Origin)의 3대 요소 명시**: 단순 이론이 아니라 프로토콜, 호스트, 포트라는 3개 축이 완벽히 일치해야 상호작용이 가능하다는 표준 스펙 규격을 선명하게 정의했습니다.
- **SOP가 없을 때의 파괴적 시나리오 입증**: "보안을 위해서"라는 추상적인 답변 대신, **은행 쿠키 자동 전송과 해커의 비동기 요청(Fetch/Axios)을 통한 데이터 탈취 시나리오**를 구체적으로 그려내어 왜 SOP가 존재해야 하는지 당위성을 완벽히 입증했습니다. (실버 답변은 CSRF와 SOP의 데이터 접근 차이를 혼동함)
- **주체와 차단 시점의 정확성**: 서버가 요청을 안 받는 게 아니라, **서버는 응답을 돌려주었으나 '브라우저'가 자바스크립트단에서 읽지 못하게 차단하는 것**이라는 메커니즘적 선후 관계를 정확히 인지하고 있음을 보여주었습니다.

### 핵심 키워드

- 프로토콜, 호스트, 포트의 삼위일체 (Same-Origin 조건)
- 다른 출처의 스크립트가 내 응답 데이터를 읽어가는 것(Read)을 방어
- 서버의 차단이 아닌 브라우저 내부 엔진 레이어의 차단 메커니즘
