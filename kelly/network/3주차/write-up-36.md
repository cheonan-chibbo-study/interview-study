## 😎 답변 및 꼬리 질문

### 답변

체크섬(Checksum)은 네트워크를 통해 데이터를 전송할 때, '전송 과정에서 데이터가 유실되거나 변조(오염)되지 않고 무결하게 도착했는지 검증하기 위해 사용하는 정수 형태의 데이터 서명'입니다.

동작 원리는 일종의 수학적인 약속입니다.

1. **송신 측**은 전송할 데이터(메시지)를 일정 단위(예: 16비트)로 쪼갠 뒤, 이를 모두 더합니다. 그리고 그 더한 값에 **'1의 보수(1's Complement)'** 연산을 취해 반전된 값을 만듭니다. 이 값을 헤더의 '체크섬 필드'에 심어서 패킷을 보냅니다.
2. **수신 측**은 패킷을 받으면, 받은 데이터 전체와 헤더에 적힌 체크섬 값을 **그대로 다시 한 번 다 더해봅니다.** 3. 만약 전송 과정에서 단 1비트의 오류도 없었다면, 송신 측이 보수를 취해 보냈기 때문에 최종 더한 결과값의 모든 비트가 1(즉, binary로 1111...)이 나와야 합니다.

만약 최종 연산 결과가 모두 1이 아니라면, 전송 도중 노이즈나 감쇄 현상으로 인해 데이터가 변조(오염)되었다고 판단합니다.

이때 프로토콜별 처리 방식에 차이가 있습니다.
**TCP**는 신뢰성 프로토콜이므로 체크섬 오류가 나면 해당 패킷을 버리고(Drop) 송신 측에 재전송을 요청하지만, **UDP**는 비신뢰성/실시간 프로토콜이므로 체크섬 오류를 감지하면 재전송 요청 없이 그저 패킷을 즉시 파기(Drop)하고 다음 패킷을 받는 식으로 무책임하게 처리한다는 차이가 있습니다.

### 꼬리 질문

- 🤗 스포 방지!

  TCP와 UDP 중 어느 프로토콜이 Checksum을 수행할까요?

  그렇다면, Checksum을 통해 오류를 정정할 수 있나요?

  체크섬의 원리와 한계, 그리고 프로토콜 동작을 명확히 짚었기 때문에 면접관은 체크섬의 암호학적 한계나 하부 레이어(데이터 링크 계층)의 또 다른 검증 기술과 비교하려 할 것입니다.

    - [ ]  체크섬은 단순히 비트들을 더하는 방식이기 때문에, 만약 전송 도중에 두 개 이상의 비트 위치가 절묘하게 교차하여 바뀌는 오류(에러가 서로 상쇄되는 현상)가 발생하면 체크섬 검증을 통과해 버리는 치명적인 맹점이 있습니다. 이를 보완하기 위해 데이터 링크 계층(L2)의 이더넷 프레임에서 사용하는 훨씬 강력한 오류 검증 기술은 무엇인가요? (예: **CRC - Cyclic Redundancy Check / 순환 중복 검사** 유도)
    - [ ]  체크섬은 '데이터 오염'은 잡아낼 수 있지만, 해커가 의도적으로 데이터를 위조하고 체크섬 값까지 새로 계산해서 보내는 **'의도적인 데이터 변조(악의적 공격)'는 막을 수 없습니다.** 그렇다면 파일 다운로드 사이트나 보안 영역에서 데이터가 해커에 의해 위·변조되지 않았음을 완벽히 증명하기 위해 체크섬 대신 사용하는 기술은 무엇인가요? (예: MD5, SHA-256 같은 **암호학적 해시 함수 / Cryptographic Hash** 검증)
    - [ ]  IPv4 헤더에는 오직 헤더의 오염만 검증하는 'Header Checksum' 필드가 존재했습니다. 그런데 최신 **IPv6 스펙에 이르러서는 이 헤더 체크섬 필드가 완전히 삭제**되었습니다. 라우터의 처리 속도를 높이기 위해 이를 과감히 삭제할 수 있었던 레이어 아키텍처 관점의 이유는 무엇일까요? (예: 상위 Layer인 TCP/UDP와 하위 Layer인 이더넷이 이미 검증을 잘해주고 있기 때문에 중복 연산 제거)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

많은 지원자가 체크섬을 "데이터가 잘 가고 있는지 확인하는 번호" 정도로만 뭉뚱그려 대답합니다.

면접관이 진짜 보고 싶어 하는 골드 포인트는 "체크섬이 단순한 확인란이 아니라, 네트워크 전송 과정에서 발생할 수 있는 '비트 오염(Bit Error/Noise)'을 감지하기 위해 데이터 전체를 수학적으로 연산하여 만들어낸 '고정된 크기의 검증 데이터'라는 점"을 인지하고 있는가입니다.

또한 TCP와 UDP 헤더에서 이 체크섬이 어떻게 취급되는지 실무적 차이를 아는가입니다.

### 답변 분석

- **명확한 수학적 메커니즘 제시**: 단순히 "더해서 확인합니다"가 아니라, "1의 보수(1's Complement) 연산을 취해 필드에 채우고, 수신 측에서 모두 더했을 때 모든 비트가 1이 되는지 확인한다"는 전송 계층의 실제 바이너리 연산 원리를 정확하게 설명했습니다. 이 부분에서 전공 지식의 깊이가 완전히 증명됩니다.
- **프로토콜별 예외 처리(TCP vs UDP) 연결**: 앞서 배운 TCP와 UDP의 철학을 체크섬 에러 상황과 유기적으로 엮어냈습니다. 오류 감지 시 TCP의 '재전송 메커니즘 트리거'와 UDP의 '조용한 드롭(Drop)'을 대조하여 네트워크 흐름을 거시적으로 볼 줄 안다는 인상을 심어줍니다.

### 핵심 키워드

- 데이터 전송 중 비트 오염/변조를 감지하는 무결성 검증 도구
- 1의 보수(1's Complement)를 활용한 송·수신 측 덧셈 연산 매커니즘
- 에러 감지 시 프로토콜별 후속 대처 (TCP 재전송 vs UDP 즉시 파기)
