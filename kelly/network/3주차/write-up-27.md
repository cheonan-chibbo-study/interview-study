## ❓질문

```text
왜 HTTPS Handshake 과정에서는 인증서를 사용하는 것 일까요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

HTTPS 핸드셰이크 과정에서 인증서를 사용하는 근본적인 이유는 **클라이언트가 접속하려는 서버가 사칭 서버가 아닌 '진짜 서버'임을 검증하고, 서버가 제공한 공개키의 무결성을 보장하여 중간자 공격(MITM)을 원천 차단하기 위해서**입니다.

만약 인증서가 없다면, 해커가 클라이언트와 서버 중간에 끼어들어 서버의 진짜 공개키를 가로채고, 자신의 가짜 공개키를 클라이언트에게 보내는 **중간자 공격**에 무력하게 당하게 됩니다. 클라이언트는 그 키가 진짜 서버의 키인 줄 알고 대칭키를 암호화해 보내어 데이터가 탈취당하는 치명적인 보안 사고가 발생합니다.

이 신뢰 문제를 해결하기 위해 웹 생태계는 신뢰할 수 있는 제3의 기관인 공인인증기관(CA, Certificate Authority)과 **디지털 서명** 체계를 도입했습니다.

동작 메커니즘은 다음과 같습니다.

1. 서버는 자신의 도메인 정보와 **자신의 진짜 공개키**를 CA에 보내어 인증서 발급을 요청합니다.
2. CA는 이를 검증한 뒤, 해당 정보들을 CA의 개인키로 암호화(디지털 서명)하여 '디지털 인증서'를 발급합니다.
3. HTTPS 핸드셰이크가 시작되면 서버는 클라이언트에게 이 인증서를 건넵니다.
4. 브라우저(클라이언트)는 OS와 브라우저 내부에 전 세계 신뢰할 수 있는 CA들의 공개키를 이미 내장(Embedded)하고 있습니다. 이 내장된 CA의 공개키로 서버가 보낸 인증서를 복호화합니다.

만약 중간에 해커가 인증서 내부의 서버 공개키를 자신의 가짜 키로 변조했다면, CA의 개인키로 서명된 해시값과 일치하지 않아 복호화 과정에서 즉시 위·변조가 발각됩니다.

결과적으로 인증서는 **'이 도메인은 진짜가 맞으며, 여기 첨부된 서버의 공개키는 절대로 위조되지 않은 진짜 서버의 키다'라는 것을 CA의 신용을 담보로 증명해 주는 신뢰의 증서** 역할을 하기 때문에 안전한 하이브리드 암호화를 시작할 수 있게 됩니다.

### 꼬리 질문

- 🤗 스포 방지!

  인증서의 본질과 CA 체계를 완벽하게 파악했기 때문에, 면접관은 실제 대칭키를 교환하는 암호학적 프로토콜의 변화나 인증서 만료/보안 이슈로 깊게 파고들 것입니다.

    - [ ]  방금 설명해 주신 방식을 통해 서버의 진짜 공개키를 얻었습니다. 그렇다면 실제 데이터를 암호화할 '대칭키'를 클라이언트와 서버가 서로 주고받을 때 사용하는 알고리즘 방식인 **RSA 키 교환 방식**과, 최근 안전성을 이유로 표준이 된 **디디-헬만(Diffie-Hellman / ECDHE) 방식**의 차이는 무엇인가요? (예: 순방향 비밀성 / Forward Secrecy 유도)
    - [ ]  간혹 웹 서핑을 하다 보면 브라우저가 '이 사이트는 보안 연결이 승인되지 않았습니다'라며 빨간색 경고창을 띄우는 경우가 있습니다. 인증서 메커니즘 관점에서 백엔드 서버에 어떤 문제가 생겼을 때 이런 경고가 발생하는 걸까요? (예: 인증서 만료, 도메인 불일치, 사설 인증서 사용 등)
    - [ ]  최근에는 비용 부담 없이 무료로 HTTPS 인증서를 발급해 주는 **Let's Encrypt** 같은 서비스 덕분에 거의 모든 웹사이트가 HTTPS를 적용하고 있습니다. 무료 인증서와 유료 대형 CA(예: DigiCert 등) 인증서 간에 기술적인 암호화 성능 차이가 존재하는 것일까요?

## 🤔 질문 의도 및 답변 분석

### 질문 의도

공개키 방식(RSA, ECC 등)을 쓰면 해커가 데이터를 가로채도 복호화할 수 없으니 안전할 것 같습니다. 하지만 치명적인 맹점이 있습니다. "내가 지금 패킷을 주고받는 저 대상이 진짜 네이버(Naver) 서버가 맞을까? 아니면 네이버를 사칭한 해커의 서버일까?"라는 점입니다.

면접관이 확인하고 싶은 것은 "공개키 암호화 알고리즘만으로는 해결할 수 없는 '중간자 공격(MITM)'의 위험성을 인지하고 있으며, 이를 방어하기 위해 공인인증기관(CA)과 디지털 서명 기술이 어떻게 유기적으로 결합하여 '공개키의 무결성과 신뢰성'을 보장하는가"를 이해하고 있는지 검증하려는 것입니다.

### 답변 분석

- **보안적 한계(중간자 공격) 제시**: 단순히 "인증서는 신뢰를 줍니다"가 아니라, **인증서가 없을 때 발생할 수 있는 '중간자 공격(MITM)' 시나리오**를 명확히 짚어 인증서의 당위성을 완벽하게 증명했습니다.
- **공개키 암호학의 역방향 메커니즘 활용**: 앞선 꼬리 질문 예시에서 다루었던 **"CA의 개인키로 암호화하고, 클라이언트가 CA의 공개키로 복호화한다"는 전자서명(위·변조 방지)의 원리**를 HTTPS 통신 흐름에 정확히 녹여냈습니다.
- **브라우저의 구체적 구동 방식 언급**: 브라우저나 OS 내부에 '전 세계 CA의 공개키가 이미 탑재되어 있다'는 구체적인 시스템 내부 동작 스펙을 짚어 지식의 디테일을 보여주었습니다.

### 핵심 키워드

- 중간자 공격(MITM) 방어 및 서버 신원 검증
- 공인인증기관(CA)의 개인키를 활용한 디지털 서명(위·변조 차단)
- 클라이언트(브라우저)에 내장된 CA 공개키를 통한 무결성 검증
