## 😎 답변 및 꼬리 질문

### 답변

프리플라이트(Preflight)는 브라우저가 교차 출처(Cross-Origin)로 본 요청을 보내기 전, **HTTP `OPTIONS` 메서드를 사용해 타겟 서버가 이 요청을 허용하는지 안전성을 미리 검증하는 '예비 요청' 메커니즘**입니다.

프리플라이트가 반드시 필요한 근본적인 이유는 'CORS 정책을 모르는 레거시 서버를 보호하기 위해서'입니다.
만약 프리플라이트 없이 브라우저가 곧바로 대용량 데이터를 삭제하는 `DELETE` 요청을 서버에 보낸다면, CORS 설정을 모르는 옛날 서버들은 일단 데이터를 지우는 비즈니스 로직을 다 수행한 뒤 응답을 돌려줄 것입니다. 브라우저가 응답을 받고 '출처가 다르네'라며 에러를 띄워봤자, 이미 서버의 데이터는 파괴된 후입니다. 따라서 **서버의 상태를 변화시킬 수 있는 본 요청을 보내기 전, 먼저 안전하게 권한만 찔러보고 확인받는 방어벽**이 바로 프리플라이트입니다.

동작 메커니즘은 헤더를 통한 상호 협약으로 진행됩니다.

1. 브라우저는 본 요청을 보내기 전, **`OPTIONS` 메서드**로 예비 요청을 날립니다. 이때 헤더에 자신의 출처인 `Origin`과 함께, 본 요청에서 사용할 메서드를 알리는 `Access-Control-Request-Method`, 추가할 커스텀 헤더를 알리는 `Access-Control-Request-Headers`를 실어 보냅니다.
2. 백엔드 서버는 이 요청을 수신하여 자격 요건을 검증한 뒤, 허용 가능하다면 응답 헤더에 **`Access-Control-Allow-Origin`**, `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`를 담아 **200 OK** 응답을 반환합니다.
3. 브라우저는 이 응답 헤더와 자신의 요청 조건을 비교하여 매칭되면, 그제야 유저가 의도한 진짜 본 요청(`POST`, `PUT`, `DELETE` 등)을 서버로 전송합니다.

실무적으로 프리플라이트는 매번 API를 호출할 때마다 요청을 2번씩 보내게 만들므로, **네트워크 지연(Latency)이 2배로 증가하여 웹 성능에 악영향**을 미칩니다.

이를 최적화하기 위해 백엔드 엔지니어는 프리플라이트 응답 헤더에 **`Access-Control-Max-Age`** 값을 초 단위(예: 3600초)로 설정해야 합니다. 이 헤더가 적용되면 브라우저는 지정된 시간 동안 프리플라이트 결과를 자체 캐시(Preflight Cache)에 저장해 두고, 다음 요청부터는 예비 요청을 생략한 채 곧바로 본 요청만 보내 성능을 극대화할 수 있습니다.

### 꼬리 질문

- 🤗 스포 방지!

  프리플라이트의 본질과 캐싱 최적화까지 완벽하게 빌드업했기 때문에, 면접관은 인증 정보가 결합했을 때의 예외나 스프링 내부의 구체적인 필터 동작 방식으로 유도할 것입니다.

    - [ ]  방금 프리플라이트 캐싱을 위해 `Access-Control-Max-Age`를 1시간(3600초)으로 길게 설정해 두었다고 하셨습니다. 그런데 만약 브라우저가 최신 크롬(Chrome) 버전이라면, 우리가 서버에서 아무리 큰 값을 주어도 **브라우저 자체적인 최대 제한 시간(Max Cap)** 때문에 캐시가 강제로 단축됩니다. 크롬 브라우저의 프리플라이트 최대 캐시 한도 시간은 몇 분인지 아시나요? (예: 크롬 기준 최대 10분 / 600초 제한 스펙)
    - [ ]  Spring Security를 적용한 프로젝트에서 CORS 에러를 해결할 때, 일반적인 Spring MVC 설정만 해두면 Security 필터 레이어에서 프리플라이트(`OPTIONS`) 요청이 **인증되지 않은 요청(401 Unauthorized)으로 걸러져서** CORS 에러가 고쳐지지 않는 장애가 자주 발생합니다. 이 현상이 일어나는 이유와 스프링 시큐리티에서 이를 해결하는 올바른 방법은 무엇인가요? (예: 시큐리티의 `CorsFilter` 우선순위 조정 및 `cors()` 설정 활성화)
    - [ ]  만약 서비스의 보안을 강화하기 위해 모든 API 요청 헤더에 커스텀 헤더인 `X-Custom-Auth-Token`을 필수로 실어 보내도록 규칙을 정했습니다. 이때 프리플라이트 메커니즘 관점에서 백엔드 서버의 CORS 설정에 추가해 주어야 하는 명시적인 헤더 옵션은 무엇일까요? (예: `Access-Control-Allow-Headers`에 해당 커스텀 헤더 등록)

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 프리플라이트를 통해 확인하고 싶은 핵심은 세 가지입니다.

1. **존재 이유(보안적 목적)**: 왜 본 요청을 바로 보내지 않고 굳이 무겁게 네트워크 왕복(RTT)을 한 번 더 거치며 예비 요청을 보내야만 하는가? (서버 보호 목적)
2. **프로토콜 스펙**: 프리플라이트 요청 시 사용하는 HTTP 메서드(`OPTIONS`)와 오가는 핵심 헤더들의 관계를 명확히 아는가?
3. **성능 최적화**: 프리플라이트로 인해 발생하는 2배의 네트워크 비용을 실무에서 어떻게 최적화(Caching)하는가?

### 답변 분석

- **핵심 존재 이유(레거시 서버 보호) 제시**: 많은 개발자가 "CORS 허용 여부를 보려고 보낸다"로 끝냅니다. 골드 답변은 "서버가 비즈니스 로직(`DELETE` 등)을 실행해 버리는 것을 원천 방어하기 위해 존재한다"는 심오한 아키텍처적 당위성을 짚어 깊이감을 입증했습니다.
- **패킷 명세와 메커니즘의 완벽한 매핑**: `OPTIONS` 메서드, `Request-...` 계열 헤더(클라이언트)와 `Allow-...` 계열 헤더(서버)의 티키타카 과정을 시퀀스 단위로 아주 깔끔하게 정리했습니다.
- **실무적 성능 트레이드오프와 최적화 방안 제시**: 프리플라이트의 최대 단점인 '네트워크 레이턴시 증가'를 정확히 인지하고, 이를 해결하기 위한 실무 필수 헤더인 **`Access-Control-Max-Age` 캐싱 전략**을 결론으로 도출하여 당장 운영 환경을 리딩할 수 있는 실력을 보여주었습니다.

### 핵심 키워드

- CORS를 모르는 서버의 비즈니스 로직 실행을 막는 방어벽
- OPTIONS 메서드와 Request- / Allow- 헤더 쌍의 검증 매커니즘
- `Access-Control-Max-Age`를 통한 예비 요청 캐싱 및 성능 최적화
