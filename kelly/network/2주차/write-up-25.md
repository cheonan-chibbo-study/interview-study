## ❓질문

```text
규모가 커져 서버가 여러 개가 된다면, 세션을 어떻게 관리할 수 있을까요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

지적하신 대로 기술적인 엄밀함으로 보면 **세션 방식은 HTTP의 Stateless(무상태) 원칙에 정면으로 위배되는 Stateful(상태 중심) 인증 방법이 맞습니다.** 서버가 클라이언트의 이전 상태(로그인 여부)를 메모리에 저장하고 기억해야만 작동하기 때문입니다.

그럼에도 불구하고 웹 생태계가 오랜 기간 세션을 채택해 온 이유는 **'HTTP 프로토콜의 Stateless 장점'과 '비즈니스적인 보안 요구사항' 사이의 필연적인 타협(Trade-off)'** 때문이었습니다.

HTTP 자체는 무상태로 유지되어야 서버 확장성이 극대화되지만, 현실의 서비스는 유저가 페이지를 이동할 때마다 로그인이 풀리지 않아야 합니다. 이를 완벽하게 무상태로 구현하려면 매 요청마다 유저의 아이디와 비밀번호를 평문으로 실어 보내야 하는데, 이는 보안상 극도로 위험합니다. 따라서 **프로토콜 레이어는 Stateless하게 유지하되, 애플리케이션 서비스 레이어에 '세션'이라는 상태 개념을 임시로 얹어** 보안성과 유저 편의성을 확보한 것입니다.

하지만 말씀하신 무상태의 한계(서버 수평 확장 시 세션 불일치 병목, 메모리 고갈 리스크)는 현대의 대규모 트래픽 환경이나 MSA(마이크로서비스 아키텍처)에서 인프라적인 거대한 장애 요인으로 다가왔습니다.

그래서 최근 웹 생태계는 HTTP의 본질인 **Stateless 원칙을 완벽하게 계승하기 위해, 서버가 상태를 전혀 기억하지 않고 클라이언트가 스스로의 자격 증명을 증명하는 JWT(JSON Web Token) 같은 토큰 기반의 무상태 인증 방식**으로 패러다임을 전환하여 기술적 모순을 해결해 나가고 있습니다.

### 꼬리 질문

- 🤗 스포 방지!

  세션의 상태 중심적 한계를 극복하기 위해 무상태 인증인 JWT를 꺼내 들었기 때문에, 면접관은 "그렇다면 무상태인 JWT는 무조건 좋은가?"라며 JWT가 가진 치명적인 한계와 트레이드오프를 송곳처럼 찌르고 들어올 것입니다.

    - [ ]  JWT는 서버가 상태를 저장하지 않는 완벽한 무상태(Stateless) 토큰이라고 하셨습니다. 그렇다면 만약 해커가 유저의 JWT 토큰을 훔쳐 갔을 때, 서버는 이 토큰의 상태를 제어할 수 없으므로 '강제 로그아웃'이나 '토큰 만료(Blacklist) 처리'를 실무에서 어떻게 구현할 수 있을까요? (무상태 인증의 한계 유도)
    - [ ]  JWT가 무상태성을 유지하기 위해 유저의 권한 정보나 프로필 등을 토큰 바디(Payload)에 다 쑤셔 넣어 보냅니다. 이로 인해 세션 방식과 비교했을 때 **네트워크 대역폭(Bandwidth)이나 보안(Payload 평문 노출)** 측면에서 어떤 단점이 생기나요?
    - [ ]  결국 실무에서는 JWT의 단점(제어 불가능)을 해결하기 위해 Redis 같은 인프라에 리프레시 토큰(Refresh Token)의 상태를 저장하곤 합니다. 이렇게 되면 결국 다시 세션처럼 Stateful해지는 것 아닌가요? 이에 대해 엔지니어로서 어떻게 생각하시나요?

## 🤔 질문 의도 및 답변 분석

### 질문 의도

이 질문은 "HTTP의 무상태 원칙을 지키는 것과, 현실의 비즈니스 요구사항(인증 유지)을 만족시키기 위해 아키텍처를 타협하는 '트레이드오프(Trade-off)'의 본질을 이해하고 있는가?"를 검증하기 위한 것입니다.

면접관이 듣고 싶어 하는 골드 답변은 "세션 방식이 HTTP의 무상태 프로토콜 규격을 깨뜨리는 상태 중심(Stateful) 설계가 맞음을 인정"하고, 그럼에도 불구하고 **왜 웹 역사에서 세션을 쓸 수밖에 없었는지, 그리고 현대 아키텍처(MSA 등)에서 이 모순을 해결하기 위해 무상태 인증(JWT)으로 어떻게 패러다임이 진화하고 있는지** 논리적으로 풀어내는 것입니다.

### 답변 분석

- **질문의 통찰 인정 및 정면 돌파**: "세션은 무상태 원칙에 위배되는 설계가 맞다"라고 쿨하게 인정하면서 논리를 시작하여, 기술적 줏대와 메타인지를 보여주었습니다. (면접관의 꼬리 잡기 유도를 원천 차단함)
- **프로토콜 레이어와 애플리케이션 레이어의 분리**: HTTP 프로토콜 자체는 여전히 Stateless하지만, 그 위의 소프트웨어(애플리케이션) 레벨에서 상태를 관리하는 우회책을 썼다는 구조적 레이어 분리 개념을 제시했습니다.
- **기술의 발전 흐름(JWT)으로의 논리적 확장**: 세션의 구조적 모순이 결국 대규모 확장성(Scale-out)을 가로막았고, 이를 극복하기 위해 **HTTP의 진짜 무상태 철학을 이어받은 JWT**가 탄생했다는 서사 구조를 만들어 답변의 격을 높였습니다.

### 핵심 키워드

- 세션은 완벽한 Stateful(상태 중심) 설계가 맞음
- 보안/편의성을 위한 프로토콜(Stateless)과 애플리케이션(Stateful)의 타협
- 완벽한 무상태 인증(JWT)으로의 패러다임 전환 배경
