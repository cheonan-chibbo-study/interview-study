## 😎 답변 및 꼬리 질문

### 답변

HTTP 메서드의 멱등성(Idempotency)은 '동일한 요청을 한 번 보내는 것과 여러 번 연속해서 보내는 것이 서버의 리소스 상태를 항상 동일하게 유지하는 특성'을 의미합니다.

여기서 가장 중요한 핵심은 멱등성의 기준이 클라이언트가 받는 응답(Response)이 아니라, **서버의 실제 데이터 상태**라는 점입니다. 주요 메서드별 멱등성 특징은 다음과 같습니다.

첫째, **GET, PUT, DELETE는 멱등성을 보장**합니다.

- **GET**은 리소스를 단순히 조회만 하므로 백 번을 호출해도 서버 상태는 변하지 않습니다.
- **PUT**은 리소스를 완전히 '덮어쓰기(Replace)'하므로, 같은 데이터로 아무리 여러 번 덮어써도 최종 서버 데이터는 항상 같습니다.
- **DELETE**는 리소스를 삭제합니다. 첫 번째 호출 때 데이터가 삭제되고, 두 번째 호출부터는 '삭제할 데이터가 없다'는 404 응답을 받게 되지만, **'서버에 해당 리소스가 존재하지 않는다'는 최종 상태는 동일하므로 멱등합니다.**

둘째, **POST는 멱등하지 않습니다.** 호출할 때마다 서버에 새로운 리소스가 계속 추가(Insert)되므로 서버 상태가 매번 변합니다.

셋째, **PATCH는 설계에 따라 멱등할 수도 있고, 멱등하지 않을 수도 있습니다.**`{"age": 20}` 처럼 특정 필드를 고정된 값으로 변경하는 PATCH 요청은 덮어쓰기 형식이므로 멱등합니다. 하지만 `{"operation": "increment", "value": 1}` 처럼 **기존 값에 특정 수치를 더하는 방식으로 설계된 PATCH 요청은 호출할 때마다 값이 계속 증가하므로 비멱등(Non-idempotent)하게 동작**합니다.

실무에서는 네트워크 불안정으로 인해 클라이언트가 타임아웃을 겪고 동일한 **POST(결제, 주문)** 요청을 재시도할 때 중복 결제가 발생하는 치명적인 리스크가 있습니다. 이를 방지하기 위해 서버 측에서 헤더에 '멱등성 키(Idempotency Key)'를 요구하고, Redis 같은 분산 캐시에 이 키를 임시 저장하여 중복 요청을 원천 차단하는 방식으로 비멱등성의 한계를 극복하곤 합니다.

    - [ ]  방금 결제 시스템에서 '멱등성 키(Idempotency Key)'를 사용해 중복 POST 요청을 막는다고 하셨는데, 이 멱등성 키 메커니즘을 구현할 때 동일한 키를 가진 두 요청이 '동시에(구동 시간 차이 없이)' 서버로 인입되는 동시성 이슈(Race Condition)가 발생하면 어떻게 방어해야 할까요? (예: Redis Distributed Lock 분산 락 유도)
    - [ ]  네트워크 인프라 장비나 프록시 서버, 혹은 대형 오픈소스 라이브러리(예: Spring Cloud OpenFeign, WebClient 등)들은 요청이 실패했을 때 자체적으로 **재시도(Retry) 메커니즘**을 돌리곤 합니다. 이때 이 인프라들이 자동으로 재시도를 해도 안전한 메서드와, 절대 함부로 자동 재시도를 하면 안 되는 메서드는 멱등성과 어떤 관계가 있을까요?
    - [ ]  사용자가 게시글의 **'조회수 증가'** API를 호출할 때, 이 행위는 데이터를 읽어오는 것이지만 내부적으로는 조회수를 +1 시키는 가변 연산이 일어납니다. 그렇다면 이 조회수 증가 API는 HTTP 메서드를 GET으로 설계하는 것이 맞을까요, 아니면 다른 메서드를 써야 할까요? 안전성과 멱등성 관점에서 설명해 주세요.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

"동일한 요청을 여러 번 보내도 결과가 같다"는 교과서적인 정의는 브론즈와 실버도 답변합니다.

면접관이 진짜 보고 싶은 골드 포인트는 두 가지입니다.

1. **멱등성의 정확한 대상**: 멱등성은 '클라이언트가 받는 응답'이 아니라 **'서버의 리소스 상태(DB 등)'가 기준**이라는 점을 정확히 인지하는가.
2. **실무적 감각**: 실제 개발 환경이나 외부 요인에 의해 이 멱등성이 깨지거나(PATCH의 예외), 멱등하지 않은 메서드(POST)의 문제를 실무에서 어떻게 방어하는지 아는가.

### 답변 분석

- **정확한 정의와 오개념 교정**: 멱등성의 기준이 '응답'이 아닌 '서버의 리소스 상태'라는 본질을 명확히 선언했습니다. (DELETE의 200 vs 404 응답 차이 예시를 통해 이를 증명함)
- **PATCH의 반전 매커니즘 제시**: 많은 지원자가 PUT은 멱등하고 PATCH는 비멱등하다고 잘못 암기합니다. 골드 답변은 "PATCH는 어떻게 설계하느냐(고정값 대입 vs 증가 연산)에 따라 달라진다"는 완벽한 표준 스펙(RFC 5789) 기준의 디테일을 보여주어 깊이감으로 면접관을 압도합니다.
- **실무 장애 대책(멱등성 키)과의 연결**: 이론적 특성을 설명하는 데 그치지 않고, 현실 세계의 **결제 중복 장애 시나리오**와 이를 **Redis 및 멱등성 키**로 해결하는 시니어급 아키텍처 관점을 자연스럽게 녹여냈습니다.

### 핵심 키워드

- 서버의 리소스 상태가 기준 (클라이언트 응답 숫자가 아님)
- DELETE가 멱등한 이유 (최종 상태의 동일성)
- PATCH의 조건부 멱등성 (증가 연산 시 비멱등)
