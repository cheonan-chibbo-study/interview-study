## 😎 답변 및 꼬리 질문

### 답변

**쿠키**와 **세션**의 가장 본질적인 차이는 '상태 정보를 저장하는 위치'와 그로 인해 발생하는 '보안 및 서버 자원 소모의 트레이드오프'에 있습니다. 또한 두 기술은 독립적이지 않고 유기적으로 연결되어 동작합니다.

첫째, 쿠키(Cookie)는 상태 정보를 **클라이언트(브라우저)의 로컬 저장소에 텍스트 형태로 보관**하는 기술입니다.
서버가 응답 헤더의 `Set-Cookie`를 통해 데이터를 보내면, 브라우저는 이후 모든 요청 헤더에 이 쿠키를 자동으로 실어 보냅니다. 클라이언트에 데이터가 노출되므로 변조나 탈취 위험이 커서 보안이 취약하지만, 서버의 자원을 전혀 소모하지 않는다는 장점이 있습니다.

둘째, 세션(Session)은 보안 취약점을 해결하기 위해 중요 정보를 클라이언트가 아닌 **서버 측 메모리나 데이터베이스에 안전하게 보관**하는 기술입니다.

동작 원리를 보면 쿠키와 세션은 하나로 묶여 움직입니다. 유저가 로그인을 하면 서버는 세션 저장소에 유저 정보를 저장하고, 이 저장소의 열쇠 역할을 하는 고유한 무작위 문자열인 '세션 ID(Session ID)'를 발급합니다. 그리고 이 세션 ID를 **쿠키에 실어서 클라이언트에게 전달**합니다. 브라우저는 다음 요청부터 이 세션 ID가 담긴 쿠키를 서버로 보내고, 서버는 해당 ID로 세션 저장소를 조회하여 유저를 식별합니다.

결과적으로 세션은 중요 데이터가 서버에 있어 쿠키보다 보안성이 압도적으로 우수합니다. 하지만 **동시 접속자가 많아질수록 서버 메모리 부하가 급증**하며, 결정적으로 서버를 여러 대로 늘리는 **수평 확장(Scale-out) 환경에서 각 서버 간 세션 데이터가 공유되지 않는 정합성 문제**가 발생하므로 실무에서는 이를 해결하기 위한 추가적인 인프라 설계가 필수적입니다.

### 꼬리 질문

- 🤗 스포 방지!

  세션의 동작 원리와 다중 서버 환경(스케일 아웃)에서의 치명적인 한계를 언급했기 때문에, 면접관은 이를 해결하기 위한 실무 인프라 기술이나 대체 기술(JWT)로 무조건 꼬리를 물 것입니다.

    - [ ]  방금 말씀하신 '서버를 수평 확장했을 때 세션 정보가 공유되지 않는 문제'를 해결하기 위해, 실제 운영 환경이나 Spring Boot 실무에서는 어떤 아키텍처적 기법이나 인프라를 도입하나요? (예: Sticky Session, Session Clustering, Redis/Memcached 기반의 분산 세션 스토리지 유도)
    - [ ]  쿠키는 보안에 취약하다고 하셨는데, 자바스크립트를 이용한 해킹 공격(XSS)이나 네트워크 스니핑으로부터 쿠키에 담긴 세션 ID를 안전하게 보호하기 위해 웹 브라우저와 서버가 제공하는 대표적인 쿠키 보안 옵션 속성들은 무엇이 있나요? (예: `HttpOnly`, `Secure`, `SameSite` 속성)
    - [ ]  최근 현대 아키텍처(MSA 등)에서는 세션 스토리지의 병목과 관리 비용을 줄이기 위해 쿠키/세션 기반 인증 대신 JWT(JSON Web Token)를 적극적으로 도입합니다. 세션 방식과 비교했을 때 JWT 방식이 가지는 장점과, 반대로 JWT가 가지는 치명적인 단점은 무엇인가요?

## 🤔 질문 의도 및 답변 분석

### 질문 의도

면접관이 이 질문을 통해 확인하고 싶은 핵심은 두 가지입니다.

1. **역할의 유기적 관계**: 쿠키와 세션이 서로 독립된 기술이 아니라, **세션을 구현하기 위해 쿠키가 메커니즘적으로 어떻게 활용되는지** 정확히 알고 있는가.
2. **백엔드 아키텍처 관점의 문제의식**: 세션이 서버 자원을 사용하기 때문에 발생하는 **보안적 이점**과, 반대로 서버를 수평 확장(Scale-out)할 때 발생하는 **서버 가용성/정합성의 한계**를 인지하고 있는가.

### 답변 분석

- **독립된 개념이 아닌 '유기적 관계' 증명**: 쿠키와 세션을 따로 설명하지 않고, "세션 ID를 쿠키에 실어 보낸다"는 구체적인 동작 메커니즘을 엮어내어 두 기술의 인과관계를 완벽히 정리했습니다.
- **보안과 리소스의 트레이드오프 명시**: 쿠키는 '보안 취약/서버 자원 free', 세션은 '보안 우수/서버 메모리 부하'라는 엔지니어링적 장단점을 선명하게 대조했습니다.
- **백엔드 아키텍처(Scale-out)의 한계 제시**: 대규모 서비스 설계 시 반드시 마주치는 '서버 수평 확장 시 세션 불일치 문제'를 답변의 피날레로 장식했습니다. 면접관은 이 답변을 듣자마자 "스케일 아웃 환경까지 고려할 줄 아는구나"라며 깊은 인상을 받게 됩니다.

### 핵심 키워드

- 클라이언트 저장(쿠키) vs 서버 저장(세션)
- 세션 ID를 매개체로 하는 쿠키와 세션의 유기적 결합
- 수평 확장(Scale-out) 시 세션 불일치 및 서버 메모리 부하 리스크
