## ❓질문

```text
Stateless의 의미를 살펴보면, 세션은 적절하지 않은 인증 방법 아닌가요?
```

---
## 😎 답변 및 꼬리 질문

### 답변

서버가 여러 대로 확장되면 로드밸런서가 요청을 분산하기 때문에, A 서버에서 로그인한 유저의 다음 요청이 B 서버로 인입될 경우 세션 정보가 없어 로그인이 풀리는 '세션 불일치 문제'가 발생합니다. 이를 해결하기 위한 아키텍처적 접근은 크게 3가지가 있습니다.

첫째, **스sticky 세션(Sticky Session)** 방식입니다.
로드밸런서가 클라이언트의 쿠키나 IP를 확인하여, **처음 로그인 요청을 처리했던 특정 서버로 이후 모든 요청을 고정(타게팅)하여 보내는 방식**입니다. 구현이 매우 쉽고 서버 간 데이터 공유 비용이 없다는 장점이 있지만, 특정 서버에 트래픽이 몰려 **부하 분산(Load Balancing)의 균형이 깨질 수 있고, 특정 서버가 다운되면 해당 서버에 종속된 유저들의 세션 데이터가 전부 유실**된다는 치명적인 가용성 한계가 있습니다.

둘째, **세션 클러스터링(Session Clustering)** 방식입니다.
모든 애플리케이션 서버가 **자신의 세션 데이터를 다른 모든 서버와 실시간으로 동기화(Replication)하여 복제본을 가지는 방식**입니다. 유저가 어떤 서버로 접속해도 세션을 완벽히 공유하고 하나의 서버가 죽어도 가용성이 보장되지만, 서버 수가 늘어날수록 **동기화를 위한 네트워크 트래픽 오버헤드가 기하급수적으로 증가하며, 모든 서버의 메모리에 동일한 세션이 중복 적재되어 메모리 낭비가 심하다는 확장성 한계**가 있습니다.

셋째, **별도의 공용 세션 스토리지(In-Memory DB) 구축** 방식입니다.
애플리케이션 서버 내부 메모리에는 세션을 저장하지 않고, **모든 서버가 참조할 수 있는 독립적인 공용 저장소인 Redis나 Memcached를 두어 세션을 중앙 집약적으로 관리하는 방식**입니다.

서버가 아무리 수평 확장(Scale-out)되어도 무상태(Stateless)를 유지할 수 있어 트래픽 분산이 완벽히 이루어지고, 서버 한두 대가 장애로 죽어도 세션 유실이 전혀 없습니다. 비록 공용 저장소 접근을 위한 네트워크 I/O 비용이 추가되고, 세션 스토리지 자체가 단일 장애점(SPOF)이 되지 않도록 Redis Cluster 등의 고가용성(HA) 레이어를 추가해야 하는 인프라 관리 비용이 들지만, **현대 대규모 트래픽 환경에서는 확장성과 안정성이 가장 우수하여 표준(Best Practice)으로 채택하고 있는 방식**입니다.

### 꼬리 질문

- 🤗 스포 방지!

  다중 서버 세션 관리와 Redis 분산 저장소 이야기까지 완벽하게 꺼냈기 때문에, 면접관은 인-메모리 DB인 Redis의 구체적인 메커니즘이나 장애 극복(Failover) 시나리오로 한 단계 더 깊숙이 파고들 것입니다.

    - [ ]  공용 세션 스토리지로 관계형 DB(MySQL 등) 대신 **Redis나 Memcached 같은 In-Memory DB**를 사용하는 근본적인 이유는 무엇인가요? 성능과 데이터 특성 관점에서 설명해 주세요.

    - [ ]  Redis 세션 방식을 도입하면 애플리케이션 서버들은 완벽한 무상태(Stateless)가 되어 언제든 늘리고 줄일 수 있습니다. 그렇다면 트래픽 급증 시 서버를 자동으로 늘려주는 클라우드 인프라의 **'오토 스케일링(Auto Scaling)'** 기능을 활용할 때, Sticky Session 방식과 Redis 세션 방식 중 어떤 쪽이 오토 스케일링에 훨씬 유리할까요?

## 🤔 질문 의도 및 답변 분석

### 질문 의도

이 질문은 단순히 "Redis를 씁니다"라는 한 줄 답변을 원하는 게 아닙니다.

면접관이 진짜 보고 싶은 것은 "다중 서버 환경에서 발생하는 데이터 불일치 문제를 해결하기 위해 엔지니어들이 고안해 낸 아키텍처들의 진화 과정(Sticky Session -> Session Clustering -> Redis 분산 세션)을 알고 있으며, 각 기술이 자원 효율성, 가용성, 데이터 정합성 관점에서 어떤 장단점(Trade-off)을 가졌는지 정확히 비교·판단할 수 있는가"입니다.

### 답변 분석

- **서사적 구조와 발전 단계 제시**: 무작위 기술 나열이 아니라 **Sticky Session -> Clustering -> Shared Session Storage**로 이어지는 아키텍처의 진화 흐름을 논리적 단계별로 쪼개어 설명함으로써 깊이 있는 통찰을 보여주었습니다.
- **장단점(Trade-off)의 완벽한 대조**: 각 기술이 가진 이점(가용성, 구현 편의성)과 한계점(부하 불균형, 네트워크 트래픽 오버헤드, SPOF 리스크)을 명확한 인프라 키워드로 대조하여 엔지니어링 감각을 입증했습니다.
- **현대적 표준(Best Practice) 도출**: 결론부에 **Redis 기반 공용 세션 스토리지**를 제시하고, 그에 따른 잠재적 리스크(SPOF)를 해결하기 위한 고가용성(HA, Cluster) 언급까지 덧붙여 실제 대규모 시스템을 안정적으로 리딩할 수 있는 역량이 있음을 어필했습니다.

### 핵심 키워드

- 로드밸런서 종속적 고정(Sticky Session)의 부하 불균형 한계
- 모든 서버 동기화(Session Clustering)의 네트워크/메모리 오버헤드
- 외부 인-메모리 DB(Redis Shared Session)를 통한 아키텍처적 수평 확장 완성
