## 😎 답변 및 꼬리 질문

### 답변

세션 방식의 로그인 과정은 클라이언트의 자격 증명 검증, 서버의 상태 생성, 그리고 쿠키를 통한 상태 공유까지 총 4단계의 시퀀스로 진행됩니다.

**1단계: 로그인 요청 (Client -> Server)**
유저가 아이디와 비밀번호를 입력하고 로그인 버튼을 누르면, 클라이언트는 HTTPS의 POST 메서드 메시지 바디(Message Body)에 이 자격 증명 데이터를 담아 서버로 요청을 보냅니다.

**2단계: 자격 증명 검증 및 세션 생성 (Server)**
서버는 데이터베이스를 조회하여 유저가 보낸 비밀번호가 일치하는지 검증합니다. 인증이 성공하면, 서버는 웹 서버의 **세션 저장소(메모리 공간)에 해당 유저의 고유 정보(User Object)를 적재**합니다. 이와 동시에, 이 세션 데이터를 찾아갈 수 있는 유일한 열쇠인 **무작위 문자열 형태의 '세션 ID(Session ID)'를 생성**합니다.

**3단계: 세션 ID 전달 (Server -> Client)**
서버는 생성된 세션 ID를 클라이언트에게 돌려주어야 합니다. 이때 HTTP 응답 헤더의 **`Set-Cookie` 필드**를 활용하여 `Set-Cookie: JSESSIONID=무작위문자열; HttpOnly; Secure`와 같은 형태로 세션 ID를 쿠키에 실어 클라이언트에게 응답을 내려줍니다.

**4단계: 이후 요청에서의 인증 유지 (Client -> Server)**
클라이언트 브라우저는 서버가 보낸 쿠키를 로컬에 저장해 둡니다. 이후 유저가 마이페이지 조회 등 인증이 필요한 다른 API를 호출할 때, **브라우저는 요청 헤더의 `Cookie` 필드에 이 세션 ID를 자동으로 탑재하여 서버로 전송**합니다. 서버는 유저의 다른 데이터를 요구하지 않고, 오직 이 세션 ID로 서버 메모리를 조회하여 유저를 식별하고 응답을 처리합니다.

## 🤔 질문 의도 및 답변 분석

### 질문 의도

이 질문은 단순한 단어 정의를 넘어 "클라이언트와 백엔드 서버가 HTTP 프로토콜(요청/응답 헤더, 바디)을 매개체로 하여 데이터를 어떻게 주고받으며 상태를 동기화하는지 흐름을 통동적으로 이해하고 있는가?"를 검증하기 위한 것입니다.

브론즈와 실버는 "아이디 패스워드를 보내면 서버가 세션을 만들고 쿠키를 줍니다"라고 서술형으로 뭉뚱그리지만, 골드 답변은 **단계별로 서버와 클라이언트의 역할을 나누고 정확한 기술 용어(메모리 적재, Set-Cookie 헤더 등)를 사용해 시퀀스(순서) 단위로 명확히 설명** 해야 합니다.

### 답변 분석

- **시퀀스(단계별) 구조화**: 요청부터 응답, 그리고 이후의 유지 단계까지 흐름을 1~4단계로 명확히 쪼개어 가독성과 전달력을 극대화했습니다. 면접관이 머릿속으로 시퀀스 다이어그램을 그리며 들을 수 있게 만듭니다.
- **정확한 프로토콜 및 컴포넌트 명칭 사용**: 단순히 "보낸다", "받는다"가 아니라 **"POST 메시지 바디"**, **"세션 저장소 메모리 적재"**, **"응답 헤더의 `Set-Cookie`"**, **"요청 헤더의 `Cookie`"** 등 백엔드 엔지니어가 사용하는 정확한 컴포넌트와 헤더 필드명을 명시했습니다.
- **실무 보안 옵션 넌지시 언급**: 3단계에서 세션 ID를 쿠키로 보낼 때 `HttpOnly`와 `Secure` 옵션을 슬쩍 끼워 넣음으로써, 자바스크립트 해킹(XSS)이나 네트워크 탈취(Sniffing)에 대한 기본적인 방어 지식을 탑재하고 있음을 은근히 어필했습니다.

### 핵심 키워드

- POST 메시지 바디를 통한 자격 증명 전송
- 서버 세션 저장소(메모리) 적재 및 무작위 세션 ID 발급
- HTTP 헤더의 `Set-Cookie`(서버 -> 클라) 및 `Cookie`(클라 -> 서버) 메커니즘
