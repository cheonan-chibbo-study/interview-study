# 소켓 통신

## NET-053 웹소켓과 소켓 통신의 차이에 대해 설명해 주세요.

### 1. 답변

**소켓(Socket)** 은 TCP/UDP 위에서 통신하기 위한 **OS 수준의 양 끝점(endpoint)** 으로, 전송 계층 위에서 동작하는 **저수준 통신 단위**입니다. 반면 **웹소켓(WebSocket)** 은 그 위, **애플리케이션 계층 프로토콜**로, **HTTP 핸드셰이크로 연결을 맺은 뒤 같은 TCP 연결을 양방향(Full-Duplex) 통신으로 업그레이드**하는 표준 프로토콜입니다.

가장 큰 차이는 **추상화 수준과 호환성**입니다. 일반 소켓은 OS API를 직접 다루므로 브라우저에서는 쓸 수 없지만, 웹소켓은 **HTTP(80/443) 포트를 그대로 쓰고 `Upgrade` 헤더로 전환**되기 때문에 방화벽·프록시를 잘 통과하고 브라우저에서 표준으로 지원됩니다.

정리하면 웹소켓은 **"웹 환경에서 쓰기 위해 HTTP 위에서 표준화된 소켓"** 이라고 보면 됩니다. 실시간 채팅·알림·주식 시세처럼 **서버가 능동적으로 데이터를 푸시**해야 하는 경우, 매번 요청을 보내는 폴링 대신 웹소켓으로 **연결을 유지한 채 양방향 통신**을 하면 오버헤드가 훨씬 줄어듭니다.

---

### 2. 꼬리 질문

1. **웹소켓 연결은 어떻게 맺어지나요?**
   → 클라이언트가 **HTTP 요청에 `Upgrade: websocket` 헤더**를 담아 보내면, 서버가 **101 Switching Protocols** 로 응답하며 같은 TCP 연결을 웹소켓으로 전환합니다. 이후로는 HTTP가 아닌 웹소켓 프레임으로 양방향 통신합니다.

2. **웹소켓 대신 폴링/롱폴링을 쓰면 어떤 차이가 있나요?**
   → 폴링은 **주기적으로 요청을 반복**해 불필요한 트래픽과 지연이 크고, 롱폴링은 응답을 지연시켜 흉내만 냅니다. 웹소켓은 **연결을 한 번 맺고 유지**하므로 실시간성과 효율이 훨씬 좋습니다.

3. **웹소켓도 결국 소켓 위에서 동작하나요?**
   → 네. 웹소켓도 내부적으로는 **TCP 소켓 연결** 위에서 동작합니다. 다만 그 위에 **프레임 포맷과 핸드셰이크 규약**을 표준화해 얹은 것이라, 개발자는 저수준 소켓을 직접 다루지 않습니다.

## NET-054 소켓과 포트의 차이가 무엇인가요?

### 1. 답변

**포트(Port)** 는 한 호스트 안에서 **어떤 프로세스(서비스)로 데이터를 보낼지 구분하는 16비트 번호**입니다. IP가 "건물 주소"라면 포트는 "호실 번호"에 해당해, **하나의 IP에서 여러 서비스를 구분**하는 식별자입니다.

**소켓(Socket)** 은 실제 통신을 수행하는 **양 끝점(endpoint) 객체**로, **(프로토콜, 출발지 IP·포트, 목적지 IP·포트)** 의 조합으로 식별됩니다. 즉 포트는 **번호(식별자)** 하나일 뿐이고, 소켓은 그 포트를 포함해 **연결 전체를 표현하는 통신 단위**입니다.

정리하면, **포트는 소켓을 구성하는 한 요소**입니다. 같은 포트(예: 서버 80번) 위에서도 **클라이언트 IP·포트가 다르면 서로 다른 소켓**이 만들어지기 때문에, 서버 하나가 여러 클라이언트와 동시에 연결을 맺을 수 있습니다.

---

### 2. 꼬리 질문

1. **그럼 소켓은 무엇으로 유일하게 식별되나요?**
   → TCP 기준 **4-튜플(출발지 IP, 출발지 Port, 목적지 IP, 목적지 Port)** 에 프로토콜까지 더한 조합으로 식별됩니다. 이 중 하나라도 다르면 OS는 별개의 소켓으로 봅니다.

2. **포트 번호는 어떻게 나뉘나요?**
   → **0~1023 잘 알려진 포트(Well-known, HTTP 80·HTTPS 443 등)**, **1024~49151 등록 포트**, **49152~65535 동적/사설 포트**로 나뉩니다. 클라이언트는 보통 동적 포트를 임시로 할당받아 씁니다.

3. **하나의 포트에 여러 연결이 붙을 수 있나요?**
   → 네. 서버는 하나의 리스닝 포트로 여러 클라이언트를 받습니다. **목적지 포트는 같아도 출발지 IP·포트가 다르면 4-튜플이 달라져** 각각 별개의 소켓으로 처리됩니다.

---

## NET-055 여러 소켓이 있다고 할 때, 그 소켓의 포트 번호는 모두 다른가요?

### 1. 답변

아닙니다. **소켓이 다르다고 해서 포트가 반드시 다른 건 아닙니다.** 소켓은 포트 하나가 아니라 **4-튜플(출발지 IP·포트, 목적지 IP·포트) 전체로 구분**되기 때문입니다.

대표적인 예가 **서버의 리스닝 포트**입니다. 웹 서버는 80번 포트 하나로 수많은 클라이언트와 연결을 맺는데, 이때 **연결된 소켓들의 서버 측 포트는 전부 80번으로 동일**합니다. 대신 **클라이언트 IP·포트가 서로 달라서** 각 연결이 별개의 소켓으로 구분됩니다.

반대로 **클라이언트 쪽**에서는 보통 연결마다 **다른 임시 포트(ephemeral port)** 를 할당받습니다. 정리하면, 포트가 같아도 4-튜플의 나머지가 다르면 소켓은 충분히 구분되므로, **"소켓 개수 = 포트 개수"가 아닙니다.**

---

### 2. 꼬리 질문

1. **서버 포트가 같은데 어떻게 연결을 구분하나요?**
   → 커널이 들어온 패킷의 **4-튜플 전체**를 보고 매칭되는 소켓을 찾습니다. 목적지 포트(80)가 같아도 출발지 IP·포트 조합이 달라 각 클라이언트 연결을 정확히 구분합니다.

2. **그럼 한 포트로 받을 수 있는 동시 연결 수에 한계가 있나요?**
   → 이론상 출발지 IP·포트 조합의 수만큼 가능해 매우 큽니다. 실무에선 포트보다 **파일 디스크립터 수, 메모리, OS의 연결 한계(ulimit 등)** 가 먼저 병목이 됩니다.

3. **클라이언트가 같은 서버에 여러 연결을 맺으면 포트는?**
   → 연결마다 **서로 다른 임시 포트**가 할당돼 4-튜플이 달라집니다. 그래서 동일 서버·동일 포트로도 여러 연결을 동시에 유지할 수 있습니다.

---

## NET-056 사용자의 요청이 무수히 많아지면, 소켓도 무수히 생성되나요?

### 1. 답변

기본적으로는 그렇습니다. **TCP는 연결 지향**이라, 클라이언트와 연결이 맺어질 때마다 서버는 **`accept()` 로 그 연결 전용 소켓을 하나씩 생성**합니다. 따라서 동시 접속자가 늘면 그만큼 소켓(=파일 디스크립터)도 늘어납니다.

문제는 **소켓 하나당 메모리·파일 디스크립터 같은 자원을 차지**한다는 점입니다. 그래서 무한정 생성되는 게 아니라 **OS의 파일 디스크립터 한계, 메모리, 포트 자원**에 의해 제한되고, 이를 넘으면 더 이상 연결을 못 받습니다. 과거 **C10K 문제**가 바로 이 지점입니다.

실무에서는 이를 완화하기 위해 **연결 재사용(Keep-Alive), 커넥션 풀, I/O 멀티플렉싱(epoll 등), 비동기 이벤트 기반 서버**를 씁니다. 즉 "요청마다 소켓·스레드 하나"가 아니라, **소수의 스레드로 다수의 소켓을 효율적으로 다루는 구조**로 설계합니다.

---

### 2. 꼬리 질문

1. **소켓이 너무 많아지면 구체적으로 무엇이 문제인가요?**
   → 각 소켓이 **파일 디스크립터와 커널 메모리(버퍼)** 를 점유합니다. 한계를 넘으면 `Too many open files` 에러가 나거나, 컨텍스트 스위칭·메모리 압박으로 **성능이 급격히 저하**됩니다.

2. **스레드 기반 모델의 한계는 무엇인가요?**
   → "연결 하나당 스레드 하나" 방식은 연결이 많아지면 **스레드 수가 폭증**해 메모리와 컨텍스트 스위칭 비용이 커집니다. 그래서 **이벤트 루프 + I/O 멀티플렉싱(epoll, NIO)** 으로 적은 스레드가 많은 연결을 처리합니다.

3. **그럼 연결 수를 줄이는 방법은?**
   → **HTTP Keep-Alive** 로 연결을 재사용하고, **커넥션 풀**로 미리 만든 연결을 돌려쓰며, 불필요한 연결은 빠르게 닫습니다. HTTP/2의 **멀티플렉싱**도 한 연결로 여러 요청을 처리해 소켓 수를 줄입니다.

## NET-057 멀티플렉싱과 디멀티플렉싱에 대해 설명해 주세요.

### 1. 답변

전송 계층 관점에서, **멀티플렉싱(Multiplexing)** 은 **여러 소켓(애플리케이션)에서 나온 데이터를 한데 모아 하나의 네트워크로 내보내는** 과정이고, **디멀티플렉싱(Demultiplexing)** 은 **도착한 세그먼트를 헤더 정보를 보고 올바른 소켓으로 나눠주는** 과정입니다.

핵심은 **포트 번호(와 IP)** 입니다. 송신 측은 여러 프로세스의 데이터를 **포트로 구분해 캡슐화 후 전송(멀티플렉싱)** 하고, 수신 측은 세그먼트 헤더의 **포트 번호를 보고 어느 소켓으로 줄지 결정(디멀티플렉싱)** 합니다. 이 덕분에 IP 하나에서 **여러 애플리케이션이 동시에 네트워크를 공유**할 수 있습니다.

참고로 같은 "멀티플렉싱"이라도 **I/O 멀티플렉싱(select/epoll)** 은 의미가 다릅니다. 그건 **한 스레드가 여러 소켓의 이벤트를 한꺼번에 감시**하는 기법으로, 전송 계층의 다중화와는 층위가 다른 개념입니다.

---

### 2. 꼬리 질문

1. **TCP와 UDP의 디멀티플렉싱은 어떻게 다른가요?**
   → **UDP**는 **목적지 IP·포트(2-튜플)** 만으로 소켓을 찾습니다. **TCP**는 **4-튜플(출발지·목적지 IP·포트) 전체**로 구분하므로, 같은 목적지 포트라도 출발지가 다르면 다른 소켓으로 전달됩니다.

2. **전송 계층 멀티플렉싱과 I/O 멀티플렉싱(epoll)은 무엇이 다른가요?**
   → 전송 계층 멀티플렉싱은 **데이터를 포트로 모으고 나누는** 프로토콜 동작이고, I/O 멀티플렉싱은 **여러 소켓의 읽기/쓰기 가능 여부를 한 스레드가 감시**하는 프로그래밍 기법입니다. 이름만 비슷할 뿐 층위가 다릅니다.

3. **포트 외에 다른 정보가 디멀티플렉싱에 쓰이나요?**
   → TCP는 포트 외에 **IP 주소까지 포함한 4-튜플**을 씁니다. 그래서 동일 포트로 들어온 여러 연결도 출발지 정보로 정확히 구분해 각 소켓에 전달합니다.

---

## NET-058 디멀티플렉싱의 과정에 대해 설명해 주세요.

### 1. 답변

디멀티플렉싱은 **도착한 세그먼트를 올바른 소켓으로 전달하는 과정**입니다. 패킷이 도착하면 OS는 먼저 **IP 헤더에서 목적지 IP**, 그다음 **전송 계층 헤더에서 포트 번호**를 확인합니다.

**UDP** 의 경우는 단순합니다. **(목적지 IP, 목적지 포트)** 2-튜플만 보고, 해당 포트에 바인딩된 소켓을 찾아 데이터를 넣어줍니다. 그래서 출발지가 달라도 같은 포트면 동일 소켓으로 들어갑니다.

**TCP** 는 **4-튜플(출발지 IP·포트, 목적지 IP·포트) 전체**를 키로 소켓을 찾습니다. 커널은 연결 소켓들을 이 4-튜플로 관리하다가, 매칭되는 게 있으면 그 소켓 버퍼에 데이터를 넣고, 없으면 리스닝 소켓에서 새 연결로 처리합니다. 이 덕분에 **같은 서버 포트(예: 80)로 들어온 여러 연결도 각각의 소켓으로 정확히 분배**됩니다.

---

### 2. 꼬리 질문

1. **매칭되는 소켓이 없으면 어떻게 되나요?**
   → TCP는 보통 **RST 패킷**으로 응답해 연결을 거부하고, UDP는 **ICMP Port Unreachable** 을 보내거나 패킷을 폐기합니다. 즉 받을 소켓이 없으면 데이터는 버려집니다.

2. **리스닝 소켓과 연결 소켓은 어떻게 구분되나요?**
   → 리스닝 소켓은 **목적지 포트만으로 새 연결(SYN)** 을 받고, 연결이 맺어지면 `accept()` 로 **4-튜플이 완성된 연결 소켓**이 따로 생깁니다. 이후 데이터는 연결 소켓으로 디멀티플렉싱됩니다.

3. **커널은 소켓을 어떻게 빠르게 찾나요?**
   → 4-튜플(또는 2-튜플)을 키로 하는 **해시 테이블**로 소켓을 관리해, 패킷마다 빠르게 매칭합니다. 연결이 많아도 평균 O(1)에 가깝게 올바른 소켓을 찾을 수 있습니다.

---
