# IP 주소

## NET-059 IP 주소는 무엇이며, 어떤 기능을 하고 있나요?

### 1. 답변

**IP 주소(Internet Protocol Address)** 는 네트워크에 연결된 **각 장치를 식별하는 논리적 주소**입니다. 우편으로 치면 "주소" 역할을 해서, 데이터를 **어디로 보내고 어디서 왔는지** 알 수 있게 합니다.

핵심 기능은 두 가지입니다. 첫째, **식별(Identification)** — 호스트가 네트워크상에서 누구인지 구분합니다. 둘째, **라우팅(Routing)** — 목적지 IP의 네트워크 부분을 보고 라우터들이 패킷을 **올바른 경로로 전달**합니다. IP 주소는 **네트워크 부분(어느 네트워크인지)** 과 **호스트 부분(그 안의 어느 장치인지)** 으로 나뉩니다.

정리하면 IP는 **OSI 3계층(네트워크 계층)** 에서 동작하며, 패킷이 인터넷을 가로질러 목적지까지 도달하도록 하는 **주소 체계이자 길찾기의 기준**입니다. 다만 IP 자체는 **전달을 보장하지 않는(비신뢰성)** 프로토콜이라, 신뢰성은 상위의 TCP가 담당합니다.

### 2. 꼬리 질문

1. **IP 주소의 네트워크 부분과 호스트 부분은 어떻게 나뉘나요?**
   → **서브넷 마스크**로 나눕니다. 마스크에서 1인 부분이 **네트워크**, 0인 부분이 **호스트**를 가리킵니다. 예로 `/24` 면 앞 24비트가 네트워크입니다.

2. **IP 주소는 어떻게 할당되나요?**
   → 수동으로 고정 할당하거나, **DHCP** 서버가 자동으로 임대(유동 IP)합니다. 공인 IP는 ISP·IANA 체계를 통해 분배됩니다.

3. **IP는 왜 전달을 보장하지 못하나요?**
   → IP는 **비연결형·비신뢰성** 설계라 순서·도착을 보장하지 않습니다. 손실·재전송·순서 보장은 상위 계층인 **TCP**가 책임집니다.

## NET-060 IPv6는 IPv4의 주소 고갈 문제를 해결하기 위해 만들어졌지만, 아직도 수많은 기기가 IPv4를 사용하고 있습니다. 고갈 문제를 어떻게 해결할 수 있을까요?

### 1. 답변

IPv6로 완전히 넘어가지 않고도 IPv4 고갈을 늦추는 핵심 기술이 **NAT(Network Address Translation)** 와 **사설 IP** 입니다. 내부망에서는 **사설 IP(예: 192.168.x.x)** 를 자유롭게 쓰고, 외부로 나갈 때만 **공인 IP 하나로 변환**해 공유합니다. 그래서 수백 대가 공인 IP 한 개를 함께 쓸 수 있습니다.

또 다른 방법은 **CIDR(Classless Inter-Domain Routing)** 입니다. 과거 클래스(A·B·C) 단위로 IP를 통째로 나눠주던 방식은 낭비가 컸는데, CIDR은 **필요한 만큼만 가변 길이로 쪼개 할당**해 주소 활용도를 크게 높였습니다.

정리하면, **NAT + 사설 IP로 공인 IP를 아껴 쓰고**, **CIDR로 효율적으로 분배**하며, 근본적으로는 **IPv6 전환**으로 해결합니다. 실무에서는 이 세 가지가 함께 쓰여 고갈 시점을 미루고 있습니다.

### 2. 꼬리 질문

1. **NAT가 어떻게 IP를 절약하나요?**
   → 내부 사설 IP들을 **공인 IP 하나(+포트)** 로 변환(PAT)해 내보냅니다. 수많은 기기가 공인 IP 하나를 공유하므로 공인 주소 소비가 크게 줍니다.

2. **CIDR 표기 `/24` 는 무슨 뜻인가요?**
   → 앞 **24비트가 네트워크**라는 뜻으로, 호스트는 8비트(약 254개)입니다. 클래스에 얽매이지 않고 **필요한 크기로 유연하게** 나눌 수 있습니다.

3. **그래도 IPv6가 필요한 이유는?**
   → NAT는 **종단 간 직접 연결을 깨고** 추가 처리 부담을 줍니다. IPv6는 주소가 사실상 무한해 **NAT 없이** 모든 기기에 공인 주소를 줄 수 있어 근본 해결책입니다.

## NET-061 IPv4와 IPv6의 차이에 대해 설명해 주세요.

### 1. 답변

가장 큰 차이는 **주소 길이**입니다. **IPv4는 32비트(약 43억 개)** 로 고갈됐고, **IPv6는 128비트(사실상 무한)** 라 주소 부족이 없습니다. 표기도 IPv4는 점 10진수(`192.168.0.1`), IPv6는 콜론 16진수(`2001:db8::1`)로 다릅니다.

성능·구조 면에서도 IPv6가 개선됐습니다. **헤더가 단순화·고정 길이**라 라우터가 빠르게 처리할 수 있고, **체크섬을 없애** 중간 라우터 부담을 줄였습니다. 또 **IPSec을 기본 지원**하고, **NAT 없이** 종단 간 통신이 가능합니다.

정리하면, IPv6는 **주소 공간 확장**이 핵심이고, 그 위에 **헤더 효율화, 보안 내장, 자동 주소 설정(SLAAC)** 같은 개선이 더해졌습니다. 다만 IPv4와 직접 호환되지 않아 **전환 기술(듀얼 스택 등)** 이 필요합니다.

### 2. 꼬리 질문

1. **IPv6가 헤더에서 체크섬을 뺀 이유는?**
   → 상위(TCP/UDP)와 하위(이더넷)에서 이미 오류를 검출하므로 **중복**입니다. 라우터마다 체크섬을 재계산하던 부담을 없애 **전달이 빨라집니다.**

2. **IPv6의 자동 주소 설정(SLAAC)이란?**
   → 호스트가 라우터 정보를 받아 **DHCP 없이 스스로 IP를 구성**하는 방식입니다. 설정 부담을 줄여줍니다.

3. **IPv4와 IPv6는 바로 호환되나요?**
   → 아닙니다. 헤더·주소 체계가 달라 **직접 통신 불가**라, 듀얼 스택·터널링·NAT64 같은 **전환 기술**이 필요합니다.

## NET-062 수많은 사람들이 유동 IP를 사용하고 있지만, 수많은 공유기에서는 고정 주소를 제공하는 기능이 이미 존재합니다. 어떻게 가능한 걸까요?

### 1. 답변

여기서 "고정 주소"는 보통 **공유기 내부 네트워크 안에서의 고정 사설 IP** 를 말합니다. 공유기가 **DHCP 서버** 역할을 하면서, 특정 기기에는 항상 같은 사설 IP를 주도록 **DHCP 예약(MAC 주소 기반 고정 할당)** 을 하기 때문에 가능합니다.

원리는 이렇습니다. 공유기는 외부 ISP로부터는 **유동 공인 IP** 를 받지만, 내부에서는 자기가 직접 IP를 나눠줍니다. 이때 **기기의 MAC 주소를 보고 "이 MAC에는 항상 192.168.0.50을 줘라"** 라고 설정하면, 그 기기는 재접속해도 같은 내부 IP를 유지합니다.

외부에서 접속해야 하는 경우라면 **DDNS(Dynamic DNS)** 를 함께 씁니다. 공인 IP가 바뀔 때마다 **도메인 이름에 최신 IP를 갱신**해줘서, 유동 공인 IP라도 고정된 주소(도메인)처럼 접근할 수 있습니다. 정리하면 **내부는 DHCP 예약, 외부는 DDNS** 로 "고정처럼" 만듭니다.

### 2. 꼬리 질문

1. **DHCP 예약은 무엇을 기준으로 IP를 고정하나요?**
   → 기기의 **MAC 주소**를 기준으로 합니다. 특정 MAC에 특정 사설 IP를 항상 매핑해 재접속해도 같은 IP를 부여합니다.

2. **DDNS는 어떤 문제를 해결하나요?**
   → **유동 공인 IP가 바뀌면 외부 접속이 끊기는** 문제를 해결합니다. IP가 바뀔 때마다 도메인 레코드를 자동 갱신해 도메인으로 항상 접근하게 합니다.

3. **포트포워딩은 왜 필요한가요?**
   → 외부에서 들어온 요청을 **NAT 뒤의 특정 내부 기기·포트로 전달**하기 위해서입니다. 공유기가 공인 IP의 특정 포트를 내부 사설 IP로 연결해줍니다.

## NET-063 IPv4를 사용하는 장비와 IPv6를 사용하는 같은 네트워크 내에서 통신이 가능한가요? 가능하다면 어떤 방법을 사용하나요?

### 1. 답변

IPv4와 IPv6는 **헤더와 주소 체계가 달라 직접 통신은 불가능**합니다. 그래서 둘을 이어주는 **전환(transition) 기술**이 필요합니다. 대표적으로 세 가지가 있습니다.

첫째, **듀얼 스택(Dual Stack)** — 장비가 **IPv4와 IPv6를 모두 지원**해, 상대에 맞춰 골라 통신합니다. 가장 안정적이고 널리 쓰입니다. 둘째, **터널링(Tunneling)** — IPv6 패킷을 **IPv4 패킷 안에 캡슐화**해 IPv4망을 건너게 합니다(예: 6to4). 셋째, **변환(Translation)** — **NAT64/DNS64** 처럼 IPv6 패킷을 IPv4로(또는 반대로) **변환**해 서로 다른 프로토콜끼리 통신하게 합니다.

정리하면, **같은 프로토콜이면 듀얼 스택, 다른 망을 건너야 하면 터널링, 서로 다른 버전끼리 직접 통신해야 하면 변환(NAT64)** 을 씁니다. 현실에선 전환기가 길어 이 기술들을 함께 운용합니다.

### 2. 꼬리 질문

1. **듀얼 스택의 장단점은?**
   → 장점은 **호환성과 안정성**입니다. 단점은 **두 프로토콜을 모두 유지**해야 해 관리·자원 부담이 늘고, IPv4 주소도 계속 필요하다는 점입니다.

2. **터널링은 어떤 상황에 쓰나요?**
   → 중간 경로가 **IPv4만 지원**할 때, IPv6 패킷을 IPv4로 감싸 통과시킵니다. IPv6 섬(island)들을 IPv4 바다 위로 연결하는 셈입니다.

3. **NAT64와 일반 NAT는 어떻게 다른가요?**
   → 일반 NAT는 IPv4↔IPv4 변환이지만, **NAT64는 IPv6↔IPv4** 처럼 **서로 다른 프로토콜 간**을 변환합니다. 보통 DNS64와 함께 동작합니다.

## NET-064 IP가 송신자와 수신자를 정확하게 전송되는 것을 보장해 주나요?

### 1. 답변

아닙니다. **IP는 전달을 보장하지 않습니다.** IP는 **비연결형(connectionless)·비신뢰성(unreliable)** 프로토콜이라, 패킷을 목적지로 **"최선을 다해 보내기(Best-Effort)"** 만 할 뿐 도착·순서·중복을 보장하지 않습니다.

그래서 패킷이 **유실되거나, 순서가 뒤바뀌거나, 중복**될 수 있습니다. IP가 하는 일은 **주소를 보고 라우팅해 전달을 시도**하는 것까지입니다. 손상 여부를 헤더 체크섬으로 확인해 깨진 패킷을 버릴 수는 있지만, **재전송하거나 순서를 맞춰주지는 않습니다.**

이 신뢰성은 **상위 계층인 TCP**가 책임집니다. TCP가 **순서 번호, ACK, 재전송, 흐름·혼잡 제어**로 빠진 조각을 채우고 순서를 맞춰 신뢰성을 보장합니다. 반대로 **UDP**를 쓰면 그 보장이 없어, 속도가 중요한 경우(스트리밍·게임)에 적합합니다. 정리하면, **IP는 길 안내, 신뢰성은 TCP** 의 몫입니다.

### 2. 꼬리 질문

1. **그럼 신뢰성은 누가 보장하나요?**
   → **TCP**가 보장합니다. 순서 번호·ACK·재전송·흐름/혼잡 제어로 **유실·순서 뒤바뀜·중복**을 바로잡아 신뢰성 있는 전달을 만듭니다.

2. **Best-Effort 방식의 장점은 무엇인가요?**
   → 연결 설정·상태 관리가 없어 **단순하고 빠르며 확장성이 좋습니다.** 신뢰성이 필요하면 상위 계층에서 선택적으로 얹으면 됩니다.

3. **IP 체크섬은 무엇을 검사하나요?**
   → IPv4는 **헤더만** 체크섬으로 검사합니다(데이터는 검사 안 함). 헤더가 손상되면 패킷을 폐기하지만, 재전송은 하지 않습니다.

## NET-065 IPv4에서 수행하는 Checksum과 TCP에서 수행하는 Checksum은 어떤 차이가 있나요?

### 1. 답변

가장 큰 차이는 **검사 범위**입니다. **IPv4 체크섬은 IP 헤더만** 검사하고, **TCP 체크섬은 TCP 헤더 + 실제 데이터(페이로드) + 의사 헤더(pseudo header)** 까지 검사합니다.

이유가 명확합니다. IP는 **라우팅(길찾기)** 이 목적이라 **주소·TTL 등 헤더 정보가 깨지지 않았는지**만 확인하면 됩니다. 게다가 라우터를 지날 때마다 TTL이 줄어 헤더가 바뀌므로 **매 홉마다 헤더 체크섬을 재계산**합니다. 반면 TCP는 **데이터의 신뢰성**이 목적이라 페이로드까지 검사해 **내용 손상**을 잡아냅니다.

또 TCP 체크섬에 포함되는 **의사 헤더**에는 출발지·목적지 IP가 들어가, **엉뚱한 목적지로 잘못 배달된 패킷**까지 걸러낼 수 있습니다. 정리하면, **IPv4 체크섬은 헤더 무결성(전달 경로용)**, **TCP 체크섬은 데이터 무결성(내용 보장용)** 으로 목적과 범위가 다릅니다.

### 2. 꼬리 질문

1. **IP 체크섬은 왜 데이터를 검사하지 않나요?**
   → IP의 역할은 **전달(라우팅)** 이라 헤더만 맞으면 충분합니다. 데이터 무결성은 **상위 계층(TCP/UDP)** 이 책임지므로 중복 검사를 피합니다.

2. **의사 헤더(pseudo header)는 왜 포함하나요?**
   → 출발지·목적지 IP 등을 함께 검사해 **잘못 배달되거나 주소가 변조된 패킷**을 탐지하기 위해서입니다. 계층 간 일관성을 확인합니다.

3. **IPv6에는 왜 헤더 체크섬이 없나요?**
   → 상·하위 계층에서 이미 오류를 검출하므로 **중복**이고, 라우터의 재계산 부담을 없애 **성능을 높이기** 위해 제거했습니다.

## NET-066 TTL(Hop Limit)이란 무엇인가요?

### 1. 답변

**TTL(Time To Live)** 은 IP 패킷이 네트워크에서 **거칠 수 있는 최대 라우터(홉) 수**를 제한하는 값입니다. IPv6에서는 같은 개념을 **Hop Limit** 이라고 부릅니다.

동작은 단순합니다. 패킷이 **라우터를 하나 지날 때마다 TTL이 1씩 감소**하고, **0이 되면 그 패킷은 폐기**됩니다. 그리고 폐기한 라우터는 송신자에게 **ICMP Time Exceeded** 메시지를 보냅니다.

쓰는 이유는 **무한 루프 방지**입니다. 라우팅 설정 오류 등으로 패킷이 네트워크를 영원히 떠도는 것을 막아, **망 자원이 고갈되는 것을 방지**합니다. 응용으로 **`traceroute`** 가 있는데, TTL을 1부터 하나씩 늘려 보내며 각 홉의 라우터를 알아내 **경로를 추적**합니다. 정리하면, TTL은 패킷의 **수명 제한 장치**입니다.

### 2. 꼬리 질문

1. **TTL이 0이 되면 어떻게 되나요?**
   → 해당 라우터가 패킷을 **폐기**하고 송신자에게 **ICMP Time Exceeded**를 보냅니다. 그래서 패킷이 영원히 떠돌지 않습니다.

2. **traceroute는 TTL을 어떻게 활용하나요?**
   → TTL을 **1, 2, 3…** 으로 늘려가며 패킷을 보냅니다. 각 단계에서 폐기한 라우터가 보내는 ICMP 응답으로 **경로상의 라우터들을 순서대로** 알아냅니다.

3. **TTL은 시간 단위인가요?**
   → 이름은 "Time"이지만 실제로는 **홉(라우터) 수** 로 동작합니다. 그래서 IPv6는 더 정확하게 **Hop Limit** 이라 부릅니다.

## NET-067 IP 주소와 MAC 주소의 차이에 대해 설명해 주세요.

### 1. 답변

둘 다 주소지만 **계층과 역할**이 다릅니다. **MAC 주소**는 **2계층(데이터 링크)** 의 **물리적 주소**로, NIC(랜카드)에 **제조 시 새겨진 고유 식별자**라 보통 변하지 않습니다. **IP 주소**는 **3계층(네트워크)** 의 **논리적 주소**로, 네트워크 환경에 따라 **할당·변경**됩니다.

역할도 다릅니다. **IP는 전체 인터넷에서 목적지 네트워크까지 찾아가는 "최종 목적지 길찾기(라우팅)"** 에 쓰이고, **MAC은 같은 로컬 네트워크 안에서 실제 장치 간 "바로 옆 전달(프레임 전송)"** 에 쓰입니다. 비유하면 IP는 "받는 사람 집 주소", MAC은 "그 동네에서 실제 문 앞까지 배달"에 해당합니다.

둘은 함께 동작합니다. 패킷은 IP로 목적지 네트워크까지 라우팅되고, 마지막 구간에서는 **ARP(Address Resolution Protocol)** 로 **IP에 대응하는 MAC을 찾아** 실제 프레임을 전달합니다. 정리하면, **IP는 어디로 갈지(논리·전역), MAC은 바로 누구에게 줄지(물리·지역)** 를 담당합니다.

### 2. 꼬리 질문

1. **ARP는 무슨 역할을 하나요?**
   → **IP 주소에 대응하는 MAC 주소를 알아내는** 프로토콜입니다. 같은 네트워크에서 "이 IP를 가진 장치의 MAC이 뭐냐"고 물어 실제 프레임 전달을 가능하게 합니다.

2. **왜 IP만으로는 전달이 안 되나요?**
   → IP는 **목적지 네트워크까지의 길찾기**만 합니다. 실제 같은 망 안에서 **장치에 프레임을 꽂는 건 MAC** 주소로 이뤄지므로 둘 다 필요합니다.

3. **MAC 주소는 정말 안 바뀌나요?**
   → 하드웨어에 고정된 값이지만 소프트웨어로 **변경(MAC 스푸핑)** 할 수 있습니다. 기본적으로는 전 세계 고유하도록 설계돼 있습니다.
