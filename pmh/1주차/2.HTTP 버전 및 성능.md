# HTTP 버전 및 성능

## NET-007 HTTP/1.1과 HTTP/2의 차이점은 무엇인가요?

### 1. 답변

#### 1. 데이터 전송 방식 (텍스트 vs 바이너리)
HTTP/1.1은 사람이 읽을 수 있는 텍스트 기반 프로토콜입니다. 반면 HTTP/2는 메시지를 바이너리 프레임(frame) 단위로 쪼개서 전송합니다. 바이너리 방식은 파싱이 더 빠르고 효율적이며 오류가 적습니다.

#### 2. 멀티플렉싱 (Multiplexing) — 가장 중요한 차이
HTTP/1.1에서는 하나의 TCP 연결에서 요청-응답이 순차적으로 처리됩니다. 앞선 요청이 끝나야 다음 요청을 보낼 수 있어 **HOL 블로킹(Head-of-Line Blocking)** 문제가 발생합니다. 이를 우회하려고 브라우저는 보통 도메인당 여러 개(6개 정도)의 TCP 연결을 동시에 엽니다.

HTTP/2는 하나의 연결 안에서 여러 요청과 응답을 동시에(병렬로) 주고받을 수 있습니다. 각 메시지는 스트림(stream)으로 구분되어 서로 섞여 전송되더라도 올바르게 재조합됩니다. 따라서 단일 연결만으로 효율적인 통신이 가능합니다.

#### 3. 헤더 압축 (HPACK)
HTTP/1.1은 매 요청마다 헤더를 반복적으로, 압축 없이 평문으로 전송합니다(쿠키 등으로 헤더가 비대해지면 오버헤드가 큼). HTTP/2는 HPACK 알고리즘으로 헤더를 압축하고, 이전에 보낸 헤더는 인덱스 테이블로 관리해 중복 전송을 줄입니다.

#### 4. 서버 푸시 (Server Push)
HTTP/2는 클라이언트가 요청하지 않아도 서버가 필요할 것으로 예상되는 리소스(예: CSS, JS)를 미리 보낼 수 있습니다. (다만 실효성 논란으로 이후 점차 비권장·폐기되는 추세입니다.)

#### 5. 스트림 우선순위 (Prioritization)
HTTP/2는 어떤 리소스를 먼저 받을지 우선순위를 지정할 수 있어, 중요한 자원을 먼저 렌더링하도록 최적화할 수 있습니다.

#### 요약 비교

| 구분 | HTTP/1.1 | HTTP/2 |
|------|----------|--------|
| 전송 형식 | 텍스트 | 바이너리 프레임 |
| 멀티플렉싱 | 미지원 (순차 처리) | 지원 (단일 연결 병렬) |
| HOL 블로킹 | 발생 | 애플리케이션 계층에서 해결* |
| 헤더 압축 | 없음 | HPACK |
| 서버 푸시 | 없음 | 지원 |
| 연결 수 | 도메인당 여러 개 | 보통 하나면 충분 |

**참고: HTTP/2도 TCP 기반이라 TCP 계층의 HOL 블로킹 은 남**

### 2. 꼬리 질문

1. **HTTP/2의 멀티플렉싱이 HTTP/1.1의 파이프라이닝과 어떻게 다른가요?**
    - 파이프라이닝은 요청을 순서대로 보내지만 응답도 순서대로 받아야 해서 앞 요청이 지연되면 뒤 응답도 막힙니다. 멀티플렉싱은 스트림 단위로 독립적으로 처리되어 순서에 무관하게 응답을 받을 수 있습니다.

2. **HTTP/2 서버 푸시를 실무에서 사용할 때 주의할 점이 있나요?**
    - 클라이언트가 이미 캐시를 보유한 리소스를 푸시하면 오히려 낭비가 됩니다. 예측이 어렵고 캐시 무효화 처리가 복잡해 많은 서비스에서 실질적 효과가 제한적이었고, Chrome은 2022년 서버 푸시 지원을 제거했습니다.

3. **HTTP/3가 TCP 대신 UDP를 사용하는데, 신뢰성은 어떻게 보장하나요?**
    - QUIC 프로토콜이 UDP 위에서 패킷 재전송, 순서 보장, 흐름 제어를 직접 구현합니다. 스트림 단위로 독립적으로 복구하기 때문에 TCP의 HOL Blocking 없이도 신뢰성을 확보할 수 있습니다.

## NET-008 HOL Blocking에 대해 설명해 주세요.

### 1. 답변

**HOL Blocking(Head-Of-Line Blocking)** 이란 앞의 요청 또는 패킷이 지연될 때, 뒤에 있는 요청들이 모두 대기해야 하는 문제입니다.

HTTP/1.1에서는 **하나의 커넥션에서 응답을 순서대로** 받아야 하기 때문에, 첫 번째 요청이 느려지면 이후 요청들이 전부 block됩니다. 이를 우회하려고 브라우저가 **도메인당 6개의 커넥션**을 병렬로 열어서 처리했는데, 이는 서버 리소스 낭비로 이어집니다.

HTTP/2는 멀티플렉싱으로 **애플리케이션 레벨의 HOL Blocking**은 해결했지만, 결국 하나의 **TCP 커넥션**을 공유하기 때문에 TCP 패킷 손실 시 **전체 스트림이 멈추는 TCP 레벨 HOL Blocking**이 남아있습니다.

HTTP/3(QUIC)는 **스트림별로 독립적인 패킷 재전송**을 처리해 하나의 스트림에서 패킷 손실이 발생해도 다른 스트림에 영향을 주지 않아 HOL Blocking을 근본적으로 해결합니다.

---

### 2. 꼬리 질문

1. **HTTP/1.1에서 HOL Blocking을 실무적으로 어떻게 완화했나요?**
    - 도메인 샤딩(여러 서브도메인으로 커넥션 분산)이나 리소스 번들링(JS/CSS 파일 합치기)으로 요청 수를 줄이는 방식을 사용했습니다. 다만 HTTP/2 환경에서는 오히려 번들링이 캐시 효율을 낮출 수 있습니다.

2. **TCP 레벨 HOL Blocking은 네트워크 환경이 좋으면 무시해도 되나요?**
    - 안정적인 네트워크에서는 패킷 손실이 거의 없어 영향이 작지만, 모바일 환경이나 불안정한 네트워크에서는 패킷 손실률이 높아 HTTP/2가 오히려 HTTP/1.1보다 느릴 수 있습니다. HTTP/3의 실용적 가치가 모바일 환경에서 더욱 부각됩니다.

3. **QUIC이 스트림별로 독립적으로 패킷 손실을 처리하는 원리는 무엇인가요?**
    - QUIC은 각 스트림에 독립적인 **흐름 제어 컨텍스트**를 부여하고, 손실된 패킷 재전송 시 해당 스트림만 대기하도록 설계되어 있습니다. TCP처럼 단일 시퀀스 번호가 아닌 스트림별 오프셋으로 관리하기 때문에 독립적 복구가 가능합니다.

## NET-009 HTTP/3.0의 주요 특징에 대해 설명해 주세요.

### 1. 답변

HTTP/3.0의 가장 큰 특징은 기존 **TCP 기반에서 벗어나 UDP 기반의 QUIC 프로토콜**을 사용한다는 점입니다.

HTTP/1.1은 요청마다 연결을 맺어 **HOL Blocking** 문제가 심했고, HTTP/2는 멀티플렉싱으로 이를 해결했지만 **TCP 레벨의 HOL Blocking**은 여전히 존재했습니다. HTTP/3는 QUIC을 통해 이 문제를 근본적으로 해결합니다.

주요 특징은 다음과 같습니다.

- **연결 수립 속도 개선**: TLS 1.3을 QUIC에 내장해 **1-RTT 또는 0-RTT** 핸드셰이크가 가능합니다.
- **스트림 독립성**: 스트림별로 독립적으로 패킷 손실을 처리하므로 하나의 스트림 지연이 다른 스트림에 영향을 주지 않습니다.
- **Connection Migration**: IP가 바뀌어도 **Connection ID** 기반으로 연결을 유지할 수 있어 모바일 환경에서 유리합니다.
  실무 관점에서는 아직 서버 및 인프라 지원이 HTTP/2보다 성숙하지 않아 도입 시 **로드밸런서, 방화벽의 UDP 지원 여부**를 반드시 확인해야 합니다. 고트래픽 서비스나 모바일 중심 서비스에서 성능 이점이 뚜렷합니다.

> HTTP/1.1 (TCP, 순차처리) → HTTP/2 (TCP, 멀티플렉싱) → HTTP/3 (QUIC/UDP, 스트림 독립)
 
---

### 2. 꼬리 질문

1. **QUIC이 UDP를 사용하는데, 신뢰성은 어떻게 보장하나요?**
   - QUIC은 UDP 위에서 자체적으로 **패킷 재전송, 순서 보장, 혼잡 제어**를 구현합니다. 신뢰성을 애플리케이션 레벨에서 처리하므로, OS 커널을 거치지 않아 TCP보다 오버헤드가 적습니다.
2. **0-RTT 핸드셰이크는 어떤 보안 문제가 있을 수 있나요?**
   - 이전 세션 정보를 재사용하므로 **Replay Attack** 취약점이 존재합니다. 따라서 0-RTT는 멱등성이 보장되는 GET 요청 등에만 허용하는 것이 권장됩니다.
3. **HTTP/2를 이미 사용 중인데 HTTP/3로 전환할 실익이 있을까요?**
   - 패킷 손실률이 높은 **모바일, 불안정한 네트워크 환경**에서는 유의미한 성능 개선이 있습니다. 반면 안정적인 내부망 서버 간 통신은 HTTP/2로도 충분한 경우가 많습니다.
---

## NET-010 왜 HTTP는 TCP를 사용하나요?

### 1. 답변

HTTP는 웹에서 문서, 이미지, API 응답 등을 전달하는 프로토콜인데, 이런 데이터는 **순서가 맞고 유실 없이 완전하게 전달되어야** 합니다. TCP는 이를 위한 **신뢰성 있는 전송**을 기본으로 제공하기 때문입니다.

TCP의 핵심 특징은 다음과 같습니다.

- **3-way 핸드셰이크**로 연결을 확립해 양방향 통신을 보장합니다.
- **패킷 순서 보장 및 재전송**: SEQ/ACK 번호로 유실된 패킷을 감지하고 재전송합니다.
- **혼잡 제어 / 흐름 제어**: 네트워크 상태에 따라 전송 속도를 조절합니다.
  반면 UDP는 이런 보장이 없지만 **빠른 전송이 필요한 스트리밍, DNS, 게임** 등에서 사용됩니다.

실무 관점에서 보면, HTTP가 TCP를 선택한 것은 **정확성 > 속도** 트레이드오프입니다. 다만 HTTP/3는 이 한계를 극복하기 위해 UDP 기반 QUIC으로 전환했으며, QUIC 자체가 신뢰성을 자체 구현하는 형태입니다.
 
---

### 2. 꼬리 질문

1. **TCP의 3-way 핸드셰이크가 성능에 미치는 영향은 무엇인가요?**
   - 연결마다 **1.5-RTT** 지연이 발생하며, HTTPS는 TLS 핸드셰이크까지 추가됩니다. 이를 줄이기 위해 **Keep-Alive, Connection Pool, HTTP/2 멀티플렉싱** 등이 사용됩니다.
2. **TCP Keep-Alive와 HTTP Keep-Alive의 차이는 무엇인가요?**
   - TCP Keep-Alive는 **OS 레벨**에서 연결이 살아있는지 확인하는 메커니즘이고, HTTP Keep-Alive는 **애플리케이션 레벨**에서 하나의 TCP 연결로 여러 HTTP 요청을 재사용하는 개념입니다. 동작 레이어가 다릅니다.
3. **대용량 파일 전송 시 TCP에서 발생할 수 있는 성능 문제와 해결 방법은?**
   - **수신 버퍼 크기** 제한으로 처리량이 낮아질 수 있으며, 패킷 손실 시 재전송 지연이 누적됩니다. 실무에서는 **TCP 윈도우 크기 튜닝, CDN 활용, 청크 전송(Chunked Transfer Encoding)** 등으로 해결합니다.



## NET-011 HTTP/3에서는 UDP를 사용하나요? 위에서 언급한 UDP의 문제가 해결되었나요?

### 1. 답변

네, HTTP/3는 UDP 기반의 **QUIC 프로토콜** 위에서 동작합니다.

기존 UDP의 문제점은 **신뢰성 없는 전송**, **순서 보장 없음**, **혼잡 제어 없음**이었는데, QUIC은 이를 애플리케이션 레이어에서 직접 구현해 해결했습니다.

**해결된 문제들:**
- **패킷 손실 복구**: QUIC이 자체적으로 재전송 로직을 구현
- **순서 보장**: 스트림 단위로 순서를 보장하되, 스트림 간 독립성을 유지
- **혼잡 제어**: TCP와 유사한 혼잡 제어 알고리즘을 QUIC 레이어에서 구현
- **HOL Blocking 해결**: HTTP/2의 TCP 기반 **Head-of-Line Blocking** 문제를 스트림 독립 처리로 해결
  또한 **TLS 1.3을 기본 내장**해 핸드셰이크 비용을 줄이고, 기존 연결이 있으면 **0-RTT**로 즉시 데이터 전송도 가능합니다.

결과적으로 "UDP의 빠름 + TCP의 신뢰성"을 모두 챙긴 프로토콜이라고 볼 수 있습니다.
 
---

### 2. 꼬리 질문

1. **QUIC의 0-RTT가 보안상 취약할 수 있다는데, 어떤 위험이 있고 실무에서 어떻게 대응하나요?**
   - 0-RTT는 이전 세션 키를 재사용하므로 **Replay Attack** 취약점이 존재합니다. 멱등성이 보장되지 않는 POST 요청 등에는 0-RTT를 비활성화하거나, 서버에서 Replay 감지 로직을 별도로 구현해야 합니다.
2. **HTTP/2의 멀티플렉싱도 HOL Blocking을 해결했다고 알고 있는데, HTTP/3와 차이가 있나요?**
   - HTTP/2는 애플리케이션 레이어의 HOL Blocking은 해결했지만, **TCP 레이어의 HOL Blocking**은 여전히 존재합니다. 패킷 하나가 손실되면 TCP 특성상 전체 스트림이 대기합니다. HTTP/3는 UDP 기반이라 스트림이 완전히 독립적으로 처리되어 이 문제까지 해결합니다.
3. **모바일 환경에서 HTTP/3가 특히 유리하다고 하는데, 이유가 뭔가요?**
   - 모바일은 Wi-Fi ↔ LTE 전환 시 IP가 바뀌어 TCP는 연결을 끊고 재연결해야 합니다. QUIC은 **Connection ID** 기반으로 연결을 유지해 IP가 바뀌어도 끊김 없이 통신이 지속됩니다. 이를 **Connection Migration**이라 합니다.

## NET-012 브라우저는 어떤 서버가 TCP를 쓰는지 UDP(QUIC)를 쓰는지 어떻게 알 수 있나요?

### 1. 답변

브라우저는 처음에는 어떤 프로토콜을 써야 할지 모르기 때문에, 일반적으로 **첫 연결은 TCP(HTTP/1.1 또는 HTTP/2)** 로 시작합니다.

서버가 HTTP/3를 지원한다면, 응답 헤더에 **`Alt-Svc`** 헤더를 포함시켜 브라우저에게 알려줍니다.

```
Alt-Svc: h3=":443"; ma=86400
```

이를 받은 브라우저는 이후 요청부터 **UDP 443 포트로 QUIC 연결을 시도**하고, 성공하면 HTTP/3로 통신합니다. 실패하면 자동으로 TCP로 폴백합니다.

추가로 **DNS HTTPS 레코드(SVCB/HTTPS RR)** 를 통해 DNS 조회 단계에서 미리 HTTP/3 지원 여부를 알 수 있어, 첫 연결부터 QUIC을 사용하는 것도 가능합니다.

실무에서는 대부분 `Alt-Svc` 방식이 사용되며, **Cloudflare, Google, Meta** 같은 대형 서비스들이 이 방식으로 HTTP/3를 제공하고 있습니다.

### 2. 꼬리 질문

1. **방화벽이나 네트워크 장비가 UDP를 막으면 HTTP/3를 아예 사용할 수 없게 되나요?**
   - 네, UDP 443이 차단되면 QUIC 연결이 실패합니다. 다만 브라우저는 이를 감지하고 **자동으로 TCP 폴백**하도록 설계되어 있어 서비스 자체는 유지됩니다. 실제로 기업 방화벽 환경에서는 HTTP/3 사용률이 낮은 편입니다.
2. **`Alt-Svc` 헤더를 캐싱한다고 하셨는데, `ma=86400` 값이 의미하는 게 뭔가요? 이 값이 너무 크면 어떤 문제가 생길 수 있나요?**
   - `ma`는 **max-age**로 브라우저가 해당 Alt-Svc 정보를 캐싱하는 시간(초)입니다. 값이 너무 크면 서버가 HTTP/3 지원을 중단했을 때도 브라우저가 계속 QUIC을 시도하다 타임아웃이 발생할 수 있습니다. 서버 변경 계획이 있다면 배포 전에 `ma` 값을 줄여두는 것이 안전합니다.
3. **DNS HTTPS 레코드 방식이 `Alt-Svc`보다 어떤 점에서 더 유리한가요?**
   - `Alt-Svc`는 첫 TCP 연결 후 서버 응답을 받아야 HTTP/3를 알 수 있지만, **DNS HTTPS 레코드**는 DNS 조회 시점에 프로토콜 정보를 받아오므로 **첫 연결부터 QUIC을 사용**할 수 있습니다. RTT를 한 번 더 줄일 수 있어 초기 응답 속도 면에서 유리합니다.
