# 추가 CS 정리 (HTTP 기초 / HTTP 버전 및 성능)

---

## 1. Keep-Alive란 무엇이고, 어떻게 작동해서 문제를 해결하나요?

### 답변

**Keep-Alive(지속 연결)** 는 HTTP 요청/응답이 끝난 후에도 TCP 연결을 끊지 않고 유지해서, 다음 요청에 재사용하는 기능입니다.

HTTP는 기본적으로 **Connectionless** 특성을 가집니다.  
즉, 요청/응답이 끝나면 TCP 연결을 종료합니다.  
문제는 TCP 연결을 새로 맺을 때마다 **3-way Handshake** 비용(1~1.5 RTT)이 발생한다는 점입니다.  
페이지 하나에 수십 개의 리소스(JS, CSS, 이미지)가 있다면 매번 연결을 새로 맺는 오버헤드가 상당합니다.

Keep-Alive는 이 문제를 해결합니다.

```
[Keep-Alive OFF]
요청1 → 3-way HS → 응답1 → 연결종료
요청2 → 3-way HS → 응답2 → 연결종료  ← 매번 연결 비용 발생

[Keep-Alive ON]
요청1 → 3-way HS → 응답1 ─┐
요청2 →            응답2 ─┤ 연결 재사용
요청3 →            응답3 ─┘ → 연결종료
```

HTTP/1.1부터는 Keep-Alive가 **기본값(default)**입니다.  
`Connection: close` 헤더를 명시해야 연결이 종료됩니다.

**주의점:**  
- Keep-Alive는 TCP 연결을 재사용하는 것이지, 사용자 상태를 저장하는 게 아닙니다. → Stateless는 그대로 유지됩니다.
- timeout 설정이 중요합니다. 너무 길면 미사용 연결이 서버 자원을 점유합니다.
- HTTP/1.1은 Keep-Alive를 써도 **순차 처리(HOL Blocking)** 문제는 여전히 존재합니다. → HTTP/2 멀티플렉싱으로 해결했습니다.

## 2. 멀티플렉싱이란 무엇이고, 어떻게 작동해서 문제를 해결하나요?

### 답변

**멀티플렉싱(Multiplexing)** 은 하나의 TCP 연결에서 여러 요청과 응답을 동시에, 독립적으로 주고받는 기능입니다. HTTP/2에서 도입되었습니다.

**HTTP/1.1의 문제:**  
Keep-Alive로 연결은 재사용하지만, 요청/응답은 반드시 순차적으로 처리해야 합니다.  
앞 요청이 느리면 뒤 요청은 무조건 기다려야 합니다. → **HTTP 레벨 HOL Blocking**

브라우저는 이를 우회하려고 도메인당 최대 6개의 TCP 연결을 병렬로 열었지만, 이는 서버 자원 낭비입니다.

**HTTP/2 멀티플렉싱 동작:**  
HTTP/2는 메시지를 **프레임(Frame)** 단위로 쪼개고, 각 프레임에 **스트림 ID**를 붙입니다.  
여러 스트림의 프레임이 하나의 TCP 연결에서 뒤섞여 전송되더라도, 수신 측이 스트림 ID로 재조합합니다.

```
[HTTP/1.1 - 순차 처리]
연결1: 요청A → 응답A → 요청B → 응답B (순서 고정)

[HTTP/2 - 멀티플렉싱]
연결1: [A프레임][B프레임][A프레임][C프레임]... (순서 무관, 동시 전송)
       스트림ID로 재조합 → A응답 / B응답 / C응답
```

**효과:**
- 단일 연결로 병렬 요청 처리 → 연결 수 감소
- 응답 순서에 상관없이 먼저 온 것부터 처리 가능
- 리소스 우선순위 지정도 가능

**한계:**  
HTTP/2도 TCP 한 연결을 공유하기 때문에, TCP 패킷 손실 시 **전체 스트림이 멈추는 TCP 레벨 HOL Blocking**이 남아있습니다. → HTTP/3(QUIC)으로 해결합니다.

## 3. QUIC이란 무엇이고, 어떻게 작동해서 문제를 해결하나요?

### 답변

**QUIC(Quick UDP Internet Connections)** 은 구글이 개발하고 HTTP/3의 기반 프로토콜이 된 **UDP 기반의 전송 프로토콜**입니다.  
신뢰성이 없는 UDP 위에서 TCP가 제공하던 기능(재전송, 순서 보장, 흐름 제어 등)을 자체 구현한 것입니다.

**기존 TCP의 문제점:**
- 연결마다 3-way Handshake(1 RTT) + TLS Handshake(1~2 RTT) 지연
- 패킷 하나 손실 시 전체 스트림 대기(TCP 레벨 HOL Blocking)
- IP 변경 시 연결 재수립 필요(모바일 환경에서 불리)

**QUIC의 해결 방식:**

| 문제 | QUIC의 해결 |
|------|------------|
| 연결 지연 | TLS 1.3 내장 → 1-RTT 또는 0-RTT 핸드셰이크 |
| TCP HOL Blocking | 스트림별 독립 패킷 재전송 → 한 스트림 손실이 다른 스트림에 영향 없음 |
| IP 변경 시 재연결 | Connection ID 기반 연결 유지 → 와이파이↔LTE 전환에도 연결 유지 |

**신뢰성 구현 방식:**  
QUIC은 UDP 위에서 자체적으로 패킷 번호(Packet Number)와 스트림 오프셋으로 손실을 감지하고 재전송합니다.  
각 스트림이 독립적인 흐름 제어 컨텍스트를 가지므로, 특정 스트림의 재전송이 다른 스트림을 막지 않습니다.

**실무 주의점:**  
일부 방화벽이나 네트워크 장비가 UDP를 차단하는 경우가 있어 HTTP/2로 fallback이 필요할 수 있습니다.  
로드밸런서의 UDP 지원 여부도 도입 전 반드시 확인이 필요합니다.

## 4. HOL Blocking(Head-Of-Line Blocking) 문제란 무엇인가요?

### 답변

**HOL Blocking** 이란 앞선 요청 또는 패킷이 지연/손실될 때, 뒤에 있는 요청들이 모두 대기해야 하는 문제입니다. HTTP 레벨과 TCP 레벨, 두 곳에서 발생합니다.

**HTTP 레벨 HOL Blocking (HTTP/1.1)**  
Keep-Alive로 연결을 재사용하더라도, 하나의 연결에서 요청/응답은 순서대로 처리됩니다.  
첫 번째 요청이 느리면(예: 큰 이미지 로딩) 두 번째, 세 번째 요청은 무조건 대기합니다.  
→ HTTP/2 **멀티플렉싱**으로 해결 (스트림 단위 병렬 처리)

**TCP 레벨 HOL Blocking (HTTP/2)**  
HTTP/2는 멀티플렉싱으로 HTTP HOL Blocking을 해결했지만, 여전히 단일 TCP 연결을 사용합니다.  
TCP 패킷 하나가 손실되면 OS가 해당 패킷 재전송을 기다리는 동안 **모든 스트림이 멈춥니다.**  
HTTP/2가 내부적으로 아무리 스트림을 분리해도, 결국 TCP 계층에서 묶여버립니다.  
→ HTTP/3 **QUIC**으로 해결 (스트림별 독립 패킷 관리)

```
HTTP/1.1: [요청A 지연] → B, C 대기       ← HTTP HOL Blocking
HTTP/2:   TCP 패킷 손실 → A,B,C 전체 대기 ← TCP HOL Blocking  
HTTP/3:   스트림A 패킷 손실 → A만 대기, B,C는 진행 ← 해결!
```

## 5. SSL/TLS를 통한 데이터 암호화란 무엇인가요?

### 답변

**SSL/TLS** 는 HTTP 통신에서 데이터를 암호화해 도청, 변조, 위장을 방지하는 보안 프로토콜입니다.  
현재는 SSL이 deprecated되어 **TLS(Transport Layer Security)** 를 사용하며, HTTPS = HTTP + TLS입니다.

**TLS Handshake 과정 (TLS 1.2 기준):**

```
Client → Server: ClientHello (지원 암호화 방식 목록)
Client ← Server: ServerHello (선택된 방식) + 인증서(공개키 포함)
Client: 인증서 검증 (CA 신뢰 체계)
Client → Server: 세션 키 교환 (공개키로 암호화)
→ 이후 세션 키(대칭키)로 실제 데이터 암호화
```

**핵심 개념:**
- **비대칭 암호화**: 핸드셰이크 단계에서 키 교환에 사용 (RSA, ECDHE 등). 느리지만 안전합니다.
- **대칭 암호화**: 실제 데이터 전송에 사용 (AES 등). 빠릅니다.
- **CA(Certificate Authority)**: 서버 인증서의 신뢰를 보장하는 공인 기관입니다.

**TLS 1.3의 개선:**  
TLS 1.2는 2 RTT가 필요했지만, TLS 1.3은 **1 RTT**로 단축했습니다.  
QUIC은 TLS 1.3을 내장해 **0-RTT** 재연결도 가능합니다.

**실무 포인트:**  
HTTPS는 현재 사실상 필수입니다. SEO 불이익, 브라우저 경고, 보안 규제 등의 이유로 HTTP만 사용하는 서비스는 거의 없습니다.

## 6. 쿠키(Cookie), 세션(Session), JWT의 차이는 무엇인가요?

### 답변

HTTP는 Stateless라 로그인 상태를 자체적으로 유지할 수 없습니다.  
이를 해결하는 세 가지 대표적인 방법입니다.

### 쿠키(Cookie)
- 브라우저가 서버로부터 받아 저장하는 작은 데이터 조각입니다.
- 이후 요청마다 자동으로 서버에 전송됩니다.
- 클라이언트 측 저장이므로 탈취/변조 위험이 있습니다.
- `HttpOnly`, `Secure`, `SameSite` 플래그로 보완합니다.

### 세션(Session)
- 로그인 정보를 **서버 메모리(또는 Redis)** 에 저장하고, 클라이언트에는 **Session ID** 만 쿠키로 전달합니다.
- 서버가 상태를 저장하므로 Stateful 방식입니다.
- 보안에 유리하지만 서버 확장 시 **세션 공유 문제**가 발생합니다(Sticky Session 또는 Redis 세션 저장소로 해결).

### JWT(JSON Web Token)
- 사용자 정보(Payload)를 토큰에 담고 **서버가 서명(Signature)** 하여 클라이언트에 전달합니다.
- 서버는 상태를 저장하지 않고 **서명 검증만** 합니다. → 완전한 Stateless 방식입니다.
- 서버 수평 확장이 쉽고 MSA 환경에 유리합니다.
- 단점: 토큰 탈취 시 만료 전까지 무효화가 어렵습니다. (Refresh Token + Blacklist로 보완)

| 구분 | 저장 위치 | Stateless | 확장성 | 보안 |
|------|---------|-----------|--------|------|
| 쿠키 | 클라이언트 | - | 좋음 | 취약 |
| 세션 | 서버 | ✗ | 제한적 | 강함 |
| JWT | 클라이언트(토큰) | ✓ | 매우 좋음 | 토큰 관리 중요 |

## 7. MSA(마이크로서비스 아키텍처)란 무엇인가요?

### 답변

**MSA(Microservice Architecture)** 는 하나의 큰 애플리케이션을 **기능별로 독립된 소규모 서비스**로 분리해 개발/배포하는 아키텍처 스타일입니다.  
반대 개념은 하나의 거대한 애플리케이션(단일 서비스)으로 구성된 **모놀리식(Monolithic) 아키텍처**입니다.

**MSA의 핵심 특징:**
- 각 서비스는 **독립적으로 배포, 확장, 장애 격리**가 가능합니다.
- 서비스 간 통신은 **HTTP(REST) 또는 메시지 큐(Kafka, RabbitMQ)** 를 사용합니다.
- 각 서비스는 **자체 DB**를 가질 수 있습니다(DB 독립성).

**HTTP와의 관계:**  
MSA에서 서비스 간 통신이 HTTP 기반 REST API인 경우가 많습니다.  
각 서비스가 Stateless해야 서비스 인스턴스를 자유롭게 늘리거나 줄일 수 있습니다.

**모놀리식 vs MSA:**

| 구분 | 모놀리식 | MSA |
|------|---------|-----|
| 배포 | 전체 재배포 | 서비스별 독립 배포 |
| 확장 | 전체 확장 | 필요한 서비스만 확장 |
| 장애 | 전체 영향 | 서비스별 격리 가능 |
| 복잡도 | 낮음 | 높음(분산 시스템 복잡성) |

## 8. 클라우드 환경이란 무엇이고, HTTP와 어떤 관련이 있나요?

### 답변

**클라우드 환경** 은 물리 서버를 직접 운용하지 않고, AWS/GCP/Azure 같은 클라우드 제공자의 인프라를 인터넷을 통해 사용하는 환경입니다.

**HTTP와의 관계:**  
클라우드 환경에서는 서버 인스턴스가 수시로 생성/삭제됩니다.  
따라서 서비스가 **Stateless** 해야 어떤 인스턴스가 요청을 받아도 동일하게 처리할 수 있습니다.

**클라우드 환경에서 중요한 HTTP 개념:**
- **로드밸런서**: 트래픽을 여러 서버에 분산합니다. Stateless 구조여야 어느 서버로 가도 문제없습니다.
- **Auto Scaling**: 트래픽 증가 시 서버를 자동으로 추가합니다. 상태 없는 서버는 즉시 투입 가능합니다.
- **CDN**: 정적 리소스를 엣지 서버에 캐싱해 HTTP 응답 속도를 개선합니다.
- **API Gateway**: MSA 환경에서 클라이언트 요청을 각 서비스로 라우팅하는 HTTP 진입점입니다.

## 9. Kubernetes란 무엇인가요?

### 답변

**Kubernetes(K8s)** 는 컨테이너(Docker) 기반 애플리케이션의 **자동 배포, 확장, 운영을 관리하는 오케스트레이션 플랫폼**입니다.

**핵심 개념:**
- **Pod**: 컨테이너를 실행하는 가장 작은 단위입니다.
- **Deployment**: Pod의 원하는 상태(replica 수 등)를 선언하고 유지합니다.
- **Service**: Pod에 고정된 DNS/IP를 부여해 HTTP 통신을 가능하게 합니다.
- **Ingress**: 외부 HTTP 트래픽을 내부 서비스로 라우팅합니다.

**HTTP와의 관계:**  
Kubernetes는 Pod를 수시로 생성/삭제하고, IP가 바뀝니다.  
서비스가 Stateless하지 않으면 Pod가 교체될 때 세션이 유실됩니다.  
따라서 Kubernetes 환경에서는 Stateless 구조 + JWT 또는 외부 세션 저장소(Redis)가 표준입니다.

## 10. MSA, Kubernetes 환경에서 Stateless 구조가 사실상 표준인 이유는 무엇인가요?

### 답변

MSA와 Kubernetes 환경에서 **Stateless 구조가 표준인 이유는 동적 환경 특성 때문**입니다.

**이유:**

1. **Pod/컨테이너가 수시로 교체됩니다.**  
   Kubernetes는 업데이트, 장애, 스케일링 등으로 Pod를 자주 재시작합니다.  
   Pod에 세션 정보가 있으면 재시작 시 세션이 모두 유실됩니다.

2. **어떤 Pod가 요청을 받을지 알 수 없습니다.**  
   로드밸런서가 트래픽을 여러 Pod에 분산하는데, 세션이 특정 Pod에 묶여 있으면 정상 동작이 불가합니다.

3. **Auto Scaling 시 새 Pod는 기존 상태를 모릅니다.**  
   트래픽 급증 시 새 Pod가 추가되어도, Stateless라면 즉시 요청을 처리할 수 있습니다.

**결론:**  
상태가 필요한 경우, **서버에 저장하지 않고 외부(Redis, DB)에 위임**하거나 **JWT로 클라이언트가 상태를 들고 다니는** 방식을 채택합니다.  
이 구조여야 서버(Pod)는 언제든 교체 가능한 무상태 단위가 됩니다.

## 11. probe 패킷이란 무엇인가요?

### 답변

**probe 패킷** 은 TCP Keep-Alive에서 사용하는 **연결 생존 여부 확인용 특수 패킷**입니다.

**배경:**  
Keep-Alive로 연결을 유지하면, 통신이 없는 유휴 상태에서 상대방이 갑자기 죽거나 네트워크가 끊겨도 서버는 알 수가 없습니다. 이런 상태를 **Half-Open Connection(유령 연결)** 이라고 합니다.

**동작:**  
OS는 일정 시간 동안 데이터 교환이 없으면 자동으로 probe 패킷을 전송합니다.

```
서버 → probe 패킷 → 클라이언트
클라이언트가 살아있다면: ACK 응답
클라이언트가 죽었다면: 응답 없음 → 서버가 연결 종료
```

**파라미터 (Linux 기준):**
- `tcp_keepalive_time`: 유휴 후 첫 probe 전송까지 대기 시간 (기본 7200초)
- `tcp_keepalive_intvl`: probe 재전송 간격
- `tcp_keepalive_probes`: probe 최대 횟수 (초과 시 연결 종료)

**HTTP Keep-Alive와의 차이:**  
TCP probe 패킷은 OS/TCP 레벨이고, HTTP Keep-Alive는 애플리케이션 레벨입니다.  
목적이 다릅니다. TCP는 연결 생존 확인, HTTP는 연결 재사용 성능 최적화입니다.

## 12. HTTP/2의 바이너리 프레이밍이란 무엇인가요?

### 답변

**바이너리 프레이밍(Binary Framing)** 은 HTTP/2가 HTTP 메시지를 전송하는 방식입니다.  
HTTP/1.1이 사람이 읽을 수 있는 **텍스트 형식**으로 전송했다면, HTTP/2는 메시지를 **바이너리 프레임(Frame)** 단위로 쪼개서 전송합니다.

**HTTP/1.1 방식:**
```
GET /index.html HTTP/1.1\r\n
Host: example.com\r\n
...
```
→ 텍스트 그대로 전송. 파싱 비용이 상대적으로 큽니다.

**HTTP/2 방식:**  
메시지를 **HEADERS 프레임**, **DATA 프레임** 등으로 분리하고, 각 프레임에 헤더(길이, 타입, 스트림 ID)를 붙여 바이너리로 인코딩합니다.

```
[프레임 헤더 9바이트] + [프레임 payload]
 - Length: 프레임 크기
 - Type: HEADERS / DATA / SETTINGS / PING ...
 - Flags: 종료 여부 등
 - Stream ID: 어느 요청인지 식별
```

**바이너리 프레이밍의 이점:**
- 파싱 속도가 빠르고 구현 오류가 적습니다.
- 텍스트에서 불필요했던 줄바꿈/공백 구분자가 필요 없습니다.
- **스트림 ID 덕분에 멀티플렉싱이 가능**합니다. 여러 요청의 프레임이 섞여 전송되어도 재조합할 수 있습니다.

## 13. HPACK 헤더 압축으로 중복 헤더 전송을 줄인다는 게 무슨 의미인가요?

### 답변

**HPACK** 은 HTTP/2에서 사용하는 헤더 압축 알고리즘입니다.

**문제 배경:**  
HTTP/1.1은 매 요청마다 헤더를 평문으로 전부 전송합니다.  
`Cookie`, `Authorization`, `User-Agent` 같은 헤더는 요청마다 거의 동일한데도 매번 반복 전송됩니다.  
헤더만 수백~수천 바이트가 되는 경우도 많습니다.

**HPACK의 동작 원리:**

1. **정적 테이블**: 자주 쓰이는 헤더 이름/값(예: `:method: GET`, `:status: 200`)을 미리 인덱스로 정의합니다.
2. **동적 테이블**: 클라이언트/서버가 통신하면서 주고받은 헤더를 테이블에 추가하고 인덱스로 참조합니다.
3. **허프만 인코딩**: 자주 등장하는 문자에 짧은 코드를 배정해 추가 압축합니다.

**예시:**
```
첫 번째 요청: "Authorization: Bearer eyJ..." → 전체 전송 + 테이블 저장
두 번째 요청: "Authorization: Bearer eyJ..." → 인덱스 번호 하나만 전송 (수 바이트)
```

**효과:**  
헤더 크기를 최대 85~90% 가량 줄일 수 있습니다.  
특히 반복 요청이 많은 API 서버에서 유효합니다.

## 14. HTTP/2의 스트림(Stream)이란 무엇인가요?

### 답변

**스트림(Stream)** 은 HTTP/2에서 **하나의 TCP 연결 안에서 독립적으로 요청/응답을 주고받는 논리적 통로**입니다.

하나의 TCP 연결이 여러 개의 스트림을 동시에 가질 수 있으며, 각 스트림은 고유한 **스트림 ID(홀수: 클라이언트 시작, 짝수: 서버 시작)** 를 가집니다.

**동작 방식:**
```
TCP 연결 하나
├── Stream ID: 1 (요청A → 응답A)
├── Stream ID: 3 (요청B → 응답B)
└── Stream ID: 5 (요청C → 응답C)

전송 예:
[Stream1: HEADERS][Stream3: HEADERS][Stream1: DATA][Stream5: HEADERS][Stream3: DATA]...
```

수신 측은 스트림 ID를 보고 어떤 요청에 해당하는 프레임인지 판별해 재조합합니다.  
따라서 **응답 순서와 무관하게** 각 스트림을 독립적으로 처리할 수 있습니다.

**스트림 상태:**  
`idle → open → half-closed → closed` 순으로 상태가 변하며, 스트림이 닫혀도 TCP 연결은 유지됩니다.

**스트림 우선순위:**  
HTTP/2는 스트림에 우선순위를 지정할 수 있어, CSS/JS처럼 중요한 리소스를 먼저 전송할 수 있습니다.

## 15. 1-RTT 또는 0-RTT 핸드셰이크란 무엇인가요?

### 답변

**RTT(Round-Trip Time)** 는 패킷이 왕복하는 데 걸리는 시간입니다.  
1-RTT와 0-RTT는 QUIC(HTTP/3)에서 연결 수립에 걸리는 왕복 횟수를 뜻합니다.

**기존 TCP + TLS 1.2 방식:**  
TCP 3-way Handshake: 1 RTT  
TLS 1.2 Handshake: 2 RTT  
→ 총 **3 RTT** 후에야 데이터 전송 가능합니다.

**TLS 1.3 방식:**  
TCP 3-way Handshake: 1 RTT  
TLS 1.3 Handshake: 1 RTT  
→ 총 **2 RTT** (개선됨)

**QUIC의 1-RTT:**  
QUIC은 TLS 1.3을 내장하고, TCP Handshake와 TLS Handshake를 **동시에** 처리합니다.  
→ 첫 연결 시 단 **1 RTT** 만에 데이터 전송이 가능합니다.

**QUIC의 0-RTT:**  
이전에 연결한 적 있는 서버라면, 저장해 둔 세션 정보(PSK, Pre-Shared Key)를 활용해 핸드셰이크 없이 **즉시 데이터 전송**이 가능합니다.

```
[TCP + TLS 1.2]  Client → Server → Client → Server → Client → Server : 데이터
[QUIC 1-RTT]     Client → Server → Client : 데이터 (첫 연결)
[QUIC 0-RTT]     Client → Server : 데이터 (재연결, 왕복 0번)
```

**0-RTT 보안 주의점:**  
이전 세션 키를 재사용하므로 **Replay Attack** 취약점이 있습니다.  
멱등성이 보장되는 GET 요청에만 허용하고, 데이터 변경이 있는 POST/PUT 등에는 사용하지 않는 것이 권장됩니다.

## 16. QUIC이 UDP 기반임에도 신뢰성을 어떻게 구현했나요?

### 답변

**QUIC은 TCP가 OS/커널 레벨에서 제공하던 신뢰성 기능을 애플리케이션 레벨(QUIC 레이어)에서 자체 구현**했습니다.

**신뢰성 구현 메커니즘:**

### 1. 패킷 번호(Packet Number) 기반 손실 감지
QUIC은 각 패킷에 고유한 패킷 번호를 부여합니다.  
수신 측이 ACK를 보내지 않은 패킷을 추적해 손실을 감지하고 재전송합니다.  
TCP의 SEQ/ACK 번호와 유사하지만, **재전송 패킷도 새 번호**를 받아 손실 감지가 더 명확합니다.

### 2. 스트림별 독립 흐름 제어
각 스트림이 독립적인 오프셋(offset)으로 데이터 위치를 추적합니다.  
특정 스트림에서 패킷 손실이 발생해도 **해당 스트림만 재전송**을 기다리고, 다른 스트림은 계속 진행합니다.  
→ TCP HOL Blocking을 원천적으로 차단합니다.

### 3. 혼잡 제어(Congestion Control)
TCP와 유사한 혼잡 제어 알고리즘(예: CUBIC, BBR)을 QUIC 레이어에서 직접 구현합니다.  
OS 커널 업데이트 없이 QUIC 라이브러리 업데이트만으로 혼잡 제어 알고리즘을 개선할 수 있는 장점이 있습니다.

### 4. 순서 보장
스트림 내에서는 오프셋으로 순서를 보장하고, 수신 측이 올바른 순서로 재조합합니다.

**정리:**

| 기능 | TCP | QUIC |
|------|-----|------|
| 손실 감지/재전송 | OS 커널 (SEQ/ACK) | QUIC 레이어 (Packet Number) |
| HOL Blocking | 전체 연결 블로킹 | 해당 스트림만 대기 |
| 혼잡 제어 | OS 커널 | QUIC 레이어 (업데이트 용이) |
| 순서 보장 | 전체 연결 단위 | 스트림 단위 |

## 17. DNS HTTPS 레코드(SVCB/HTTPS RR)란 무엇인가요?

### 답변

**DNS HTTPS 레코드(HTTPS RR)** 는 DNS에 HTTPS 연결에 필요한 정보를 미리 게시해두는 새로운 DNS 레코드 타입입니다(RFC 9460, 2023년 표준화).  
기존 A/AAAA 레코드가 IP 주소만 알려줬다면, HTTPS RR은 **어떤 프로토콜로, 어떤 포트로, 어떤 암호화 파라미터로 연결해야 하는지**까지 DNS 단계에서 전달합니다.

**배경 - 기존 방식의 문제점:**  
클라이언트가 `http://example.com`에 처음 접속하면, 서버는 HTTPS로 리다이렉트하거나 HSTS 헤더를 응답합니다.  
이 과정에서 **최소 1번의 평문 HTTP 요청이 발생**하고, 그 사이에 공격자가 개입할 수 있습니다(**SSL Stripping 공격**).  
HTTP/3(QUIC)도 마찬가지로, 클라이언트가 서버가 QUIC을 지원하는지 모르면 처음엔 TCP로 연결한 후 `Alt-Svc` 헤더를 받아야 했습니다.

**HTTPS RR의 동작 방식:**

```
[기존 방식]
1. DNS 쿼리 → IP 주소만 획득
2. HTTP로 접속 시도 (평문 노출 위험)
3. 서버 응답: 301 → HTTPS로 리다이렉트
4. HTTPS 재연결

[HTTPS RR 방식]
1. DNS 쿼리 → IP + 지원 프로토콜(h3, h2) + 포트 + ECH 키 등 획득
2. 처음부터 HTTPS + HTTP/3으로 바로 연결
```

**레코드 구조 예시:**

```
example.com.  HTTPS  1 . alpn="h3,h2" ipv4hint=93.184.216.34 ech=...
```

- `alpn`: 지원하는 프로토콜 (`h3` = HTTP/3, `h2` = HTTP/2)
- `ipv4hint` / `ipv6hint`: 연결에 사용할 IP 힌트
- `port`: 기본 443이 아닌 포트 사용 시 명시
- `ech`: ECH(Encrypted Client Hello) 공개키 — TLS 핸드셰이크의 SNI까지 암호화

**HTTPS RR이 해결하는 문제들:**

| 문제 | 해결 방식 |
|------|---------|
| 평문 HTTP 첫 요청 노출 | DNS 단계에서 HTTPS 강제 → 처음부터 암호화 연결 |
| HTTP/3 지원 여부 미리 알 수 없음 | `alpn=h3` 정보로 바로 QUIC 연결 가능 |
| SNI 노출(어느 도메인 접속하는지 평문 노출) | ECH 파라미터 제공으로 TLS 핸드셰이크 전체 암호화 가능 |

**SVCB RR과의 관계:**  
HTTPS RR은 SVCB(Service Binding) RR의 특수화된 버전입니다.  
SVCB는 범용 서비스 바인딩 레코드이고, HTTPS RR은 HTTPS 전용으로 특화된 타입입니다.

**실무 포인트:**  
- 주요 브라우저(Chrome, Firefox, Safari)와 클라우드 CDN(Cloudflare, AWS)이 이미 지원합니다.
- HTTPS RR을 활용하면 최초 접속부터 HTTP/3를 사용할 수 있어 연결 지연을 추가로 줄일 수 있습니다.
- 단, DNS 응답 자체는 평문이므로 HTTPS RR의 완전한 효과를 위해서는 **DoH(DNS over HTTPS)** 또는 **DoT(DNS over TLS)** 와 함께 사용하는 것이 권장됩니다.
