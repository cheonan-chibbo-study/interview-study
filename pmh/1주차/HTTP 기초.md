# HTTP 기초

## NET-001 HTTP에 대해 설명해 주세요.

### 1. 답변

**HTTP(HyperText Transfer Protocol)** 는 웹에서 클라이언트와 서버가 데이터를 주고받기 위한 **애플리케이션 계층 프로토콜**입니다.  
브라우저와 서버가 **Request / Response 방식**으로 통신하며, HTML뿐 아니라 JSON, 이미지 같은 다양한 데이터를 전송할 수 있습니다.

HTTP의 핵심 특징은 **Stateless**와 **Connectionless**입니다.  
서버가 클라이언트 상태를 저장하지 않아 확장성이 좋고, 대규모 트래픽 처리에 유리합니다.  
대신 인증 상태 유지를 위해 Cookie, Session, JWT 같은 기술을 함께 사용합니다.

버전도 계속 발전해왔는데,
- **HTTP/1.1**: Keep-Alive 지원
- **HTTP/2**: 멀티플렉싱 지원
- **HTTP/3**: QUIC 기반으로 지연 감소
  처럼 성능 개선 중심으로 발전했습니다.

실무에서는 REST API, MSA, 인증, 캐시, CDN 등 거의 모든 웹 서비스의 기반 기술이라 매우 중요합니다.

---

### 2. 꼬리 질문

#### 1. HTTP와 HTTPS의 차이점은 무엇인가요?

HTTP는 데이터를 평문으로 전송하고, HTTPS는 **SSL/TLS**를 통해 데이터를 암호화해서 전송합니다.  
HTTPS는 중간자 공격이나 패킷 탈취를 방지할 수 있어서 현재 웹 서비스에서는 거의 필수입니다.

---

#### 2. HTTP가 TCP 기반인데 왜 HTTP/3에서는 UDP 기반 QUIC를 사용할까요?

TCP는 연결 지연과 HOL Blocking 문제가 있는데, QUIC은 UDP 기반으로 이를 개선했습니다.  
덕분에 연결 속도가 빠르고, 패킷 손실 상황에서도 성능 저하를 줄일 수 있습니다.

---

#### 3. Stateless 구조에서 로그인 상태는 어떻게 유지하나요?

HTTP 자체는 상태를 저장하지 않기 때문에, 로그인 정보는 Cookie, Session, JWT 같은 별도 기술로 관리합니다.  
최근에는 서버 확장성과 분산 환경 때문에 JWT 기반 인증도 많이 사용합니다.

## NET-002 HTTP의 특성인 Stateless에 대해 설명해 주세요.

### 1. 답변

**Stateless**는 서버가 클라이언트의 이전 요청 상태를 저장하지 않는 특성입니다.  
즉, 각각의 HTTP 요청은 서로 독립적으로 처리되며, 서버는 이전 요청 정보를 기억하지 않습니다.

예를 들어 사용자가 로그인 후 요청을 보내더라도, HTTP 자체만으로는 로그인 상태를 유지하지 못합니다.  
그래서 실제 서비스에서는 **Cookie, Session, JWT** 같은 별도 인증 방식을 사용해 상태를 관리합니다.

Stateless 구조의 가장 큰 장점은 **확장성**입니다.  
특정 서버가 사용자 상태를 기억하지 않아도 되기 때문에, 어떤 서버가 요청을 처리해도 동일하게 동작할 수 있어 로드밸런싱과 서버 확장에 유리합니다.

반면 단점은 매 요청마다 인증 정보 같은 필요한 데이터를 계속 보내야 해서 네트워크 비용이 증가할 수 있다는 점입니다.

실무에서는 MSA나 클라우드 환경처럼 서버가 자주 늘어나거나 변경되는 구조에서 매우 중요한 개념입니다.

---

### 2. 꼬리 질문

#### 1. Stateless 환경에서 로그인 상태는 어떻게 유지하나요?

쿠키, 세션, JWT 같은 인증 정보를 클라이언트가 저장하고 요청마다 함께 전달합니다.  
서버는 해당 정보를 검증해 사용자를 식별합니다.

#### 2. Session 방식과 JWT 방식의 차이는 무엇인가요?

Session은 서버가 상태를 저장하고, JWT는 토큰 자체에 사용자 정보를 담습니다.  
JWT는 Stateless 구조에 더 적합하지만 토큰 관리가 중요합니다.

#### 3. Stateless 구조가 MSA 환경에서 유리한 이유는 무엇인가요?

서비스 인스턴스가 자주 늘어나거나 줄어들어도 상태 공유 문제가 적기 때문입니다.  
어떤 서버가 요청을 처리해도 동일하게 동작할 수 있습니다.

---

## NET-003 Stateless와 Connectionless에 대해 설명해 주세요.

### 1. 답변

**Stateless**는 서버가 상태를 저장하지 않는 것이고,  
**Connectionless**는 요청/응답 후 연결을 끊는 특성입니다.

Stateless는 서버가 클라이언트의 이전 상태를 저장하지 않는 구조입니다.
즉, 매 요청(Request)마다 필요한 인증 정보나 데이터를 모두 포함해서 보내야 합니다.
이 구조 덕분에 서버 확장이 쉽고, 여러 서버로 요청을 분산하는 로드 밸런싱에 유리합니다.

반면 로그인 상태 유지처럼 상태가 필요한 경우에는 Session, Cookie, JWT 같은 별도 기술을 사용합니다.

Connectionless는 요청/응답이 끝나면 TCP 연결을 종료하는 방식입니다.
계속 연결을 유지하지 않기 때문에 서버 자원을 효율적으로 사용할 수 있습니다.

다만 매번 TCP 연결을 새로 맺으면 성능 오버헤드가 발생해서,
HTTP/1.1에서는 Keep-Alive, HTTP/2에서는 Multiplexing, HTTP/3에서는 QUIC 기반 연결 최적화로 성능을 개선했습니다.

실무에서는 이 두 특성 덕분에 HTTP 서버가 대규모 트래픽 환경에서도 확장성과 자원 효율성을 확보할 수 있다는 점이 중요합니다.

---

### 2. 꼬리 질문

#### 1. Keep-Alive는 어떤 문제를 해결하나요?

매 요청마다 TCP 연결을 새로 맺는 비용을 줄여줍니다.  
연결 재사용으로 응답 속도와 서버 성능이 개선됩니다.

#### 2. TCP 3-Way Handshake 비용이 왜 성능에 영향을 주나요?

연결 생성마다 네트워크 왕복(RTT)이 발생하기 때문입니다.  
요청 수가 많아질수록 지연 시간과 서버 부하가 증가합니다.

#### 3. HTTP/2의 멀티플렉싱은 무엇을 개선했나요?

하나의 연결에서 여러 요청을 동시에 처리할 수 있게 개선했습니다.  
기존 HTTP/1.1의 Head-of-Line Blocking 문제를 줄였습니다.

---

## NET-004 왜 HTTP는 Stateless 구조를 채택하고 있을까요?

### 1. 답변

HTTP가 Stateless 구조를 채택한 이유는 **확장성과 서버 자원 효율성** 때문입니다.

만약 서버가 사용자 상태를 계속 저장하면:
- 메모리 사용량 증가
- 특정 서버에 세션 종속 발생
- 서버 확장 어려움

같은 문제가 생깁니다.

반면 Stateless 구조에서는 어떤 서버가 요청을 처리해도 동일하게 동작할 수 있습니다.  
그래서 로드밸런싱과 수평 확장에 매우 유리합니다.

특히 클라우드, MSA, Kubernetes 환경에서는 Stateless 구조가 사실상 표준입니다.

상태 유지가 필요한 경우에는 Session 저장소나 JWT 같은 별도 메커니즘으로 해결합니다.

---

### 2. 꼬리 질문

#### 1. Stateful 구조에서는 어떤 확장성 문제가 발생하나요?

사용자 상태가 특정 서버에 저장되면 요청 분산이 어려워집니다.  
서버 장애 시 세션 유실 문제도 발생할 수 있습니다.

#### 2. JWT가 Stateless 인증 방식이라고 불리는 이유는 무엇인가요?

서버가 사용자 상태를 저장하지 않고 토큰만 검증하기 때문입니다.  
그래서 서버 확장 시 세션 공유가 필요 없습니다.

#### 3. Redis를 Session 저장소로 사용하는 이유는 무엇인가요?

Redis는 메모리 기반이라 조회 속도가 매우 빠릅니다.  
여러 서버가 세션을 공유할 수 있어 분산 환경에 적합합니다.

## NET-005 Connectionless의 논리대로면 성능이 되게 좋지 않을 것으로 보이는데, 해결 방법이 있을까요?
### 1. 답변

맞습니다. **HTTP의 Connectionless 특성** 때문에 요청마다 TCP 연결을 새로 맺고 끊으면,  
매번 **3-way Handshake / 4-way Handshake** 비용이 발생해서 성능이 크게 떨어질 수 있습니다.

이 문제를 해결하기 위해 나온 대표적인 방식이 **HTTP Keep-Alive(지속 연결)** 입니다.  
HTTP/1.1부터는 기본적으로 Keep-Alive를 사용해서, 하나의 TCP 연결을 재사용할 수 있습니다.

추가로 HTTP 버전이 발전하면서 성능 개선도 이루어졌습니다.

- **HTTP/1.1** : Keep-Alive 지원, 하지만 HOL Blocking 문제 존재
- **HTTP/2** : 하나의 연결에서 여러 요청을 동시에 처리하는 **Multiplexing** 지원
- **HTTP/3** : TCP 대신 **QUIC(UDP 기반)** 사용 → 연결 지연 및 재전송 문제 개선

즉, Connectionless 구조 자체는 유지하면서도,  
실제로는 연결 재사용과 멀티플렉싱으로 성능 문제를 해결한 것입니다.

실무에서도 API 서버 성능 최적화에서  
**커넥션 풀**, **Keep-Alive 설정**, **HTTP/2 사용 여부**가 매우 중요합니다.

---

### 2. 꼬리 질문

#### 1. HTTP Keep-Alive가 너무 오래 유지되면 어떤 문제가 발생할 수 있나요?

Keep-Alive를 오래 유지하면 사용하지 않는 연결이 계속 서버 자원을 점유하게 됩니다.  
동시 접속자가 많아질 경우 커넥션 부족, 메모리 사용 증가 문제가 발생할 수 있어 timeout 설정이 중요합니다.

---

#### 2. HTTP/2의 Multiplexing이 HOL Blocking 문제를 어떻게 개선했나요?

HTTP/1.1은 하나의 요청이 지연되면 뒤 요청들도 대기해야 했습니다.  
HTTP/2는 하나의 TCP 연결에서 여러 스트림을 동시에 처리해서 요청 간 대기 문제를 크게 줄였습니다.

---

#### 3. HTTP/3에서 TCP 대신 QUIC(UDP 기반)을 사용하는 이유는 무엇인가요?

TCP는 패킷 하나가 유실되면 이후 데이터까지 모두 대기하는 HOL Blocking 문제가 있습니다.  
HTTP/3는 UDP 기반 QUIC을 사용해 스트림별 독립 처리를 가능하게 하여 지연과 재전송 비용을 줄였습니다.
## NET-006 TCP의 keep-alive와 HTTP의 keep-alive의 차이는 무엇인가요?
### 1. 답변

두 개 모두 이름은 같지만, 목적과 동작 계층이 다릅니다.

먼저 **TCP Keep-Alive**는 **전송 계층(TCP)** 기능입니다.  
오랫동안 데이터 송수신이 없는 연결에 대해 주기적으로 probe 패킷을 보내서,  
상대방 연결이 살아있는지 확인하는 기능입니다.

즉, 서버가 죽었거나 네트워크가 끊긴 “유령 연결(Half-Open Connection)”을 감지하기 위해 사용합니다.

반면 **HTTP Keep-Alive**는 **응용 계층(HTTP)** 기능입니다.  
요청마다 TCP 연결을 새로 생성하지 않고,  
하나의 TCP 연결을 재사용해서 여러 HTTP 요청을 처리하는 목적입니다.

즉, HTTP Keep-Alive는 성능 최적화가 핵심이고,  
TCP Keep-Alive는 연결 상태 확인과 장애 감지가 핵심입니다.

실무에서는 HTTP Keep-Alive로 성능을 개선하고,  
TCP Keep-Alive로 비정상 연결 정리 및 커넥션 누수를 방지합니다.

---

#### 1. TCP Keep-Alive 주기를 너무 짧게 설정하면 어떤 문제가 발생할까요?

불필요한 Keep-Alive 패킷이 자주 발생해서 네트워크 트래픽과 서버 부하가 증가할 수 있습니다.  
특히 대규모 서버 환경에서는 연결 수가 많아 성능 저하 원인이 될 수 있습니다.

---

#### 2. HTTP/1.0과 HTTP/1.1의 Keep-Alive 동작 차이를 설명해 주세요.

HTTP/1.0은 기본적으로 요청마다 연결을 종료했고, Keep-Alive를 명시해야 했습니다.  
반면 HTTP/1.1은 Keep-Alive가 기본 동작이며, 명시적으로 close 해야 연결을 종료합니다.

---

#### 3. 로드밸런서 환경에서 Keep-Alive timeout 설정이 중요한 이유는 무엇인가요?

로드밸런서와 서버 간 timeout 값이 다르면 연결이 예상보다 빨리 끊길 수 있습니다.  
이 경우 Broken Pipe, Connection Reset 같은 오류가 발생할 수 있어 timeout 정합성이 중요합니다.
