# 프로세스

## OS-013 프로세스가 무엇인가요?

### 1. 답변

**프로세스(Process)** 는 **실행 중인 프로그램**입니다. 디스크에 저장된 정적인 프로그램(코드)이 **메모리에 올라와 CPU에 의해 실제로 실행되고 있는 상태**를 말합니다.

조금 더 풀면, 프로그램은 그냥 **명령어들의 집합(파일)** 일 뿐이고 아무 동작도 하지 않습니다. 이 프로그램을 실행하면 운영체제가 **메모리 공간을 할당**하고 **실행에 필요한 자원**을 부여하는데, 이렇게 **생명을 얻어 활동하는 실체**가 프로세스입니다. 그래서 같은 프로그램(예: 메모장)을 두 번 실행하면 **별개의 프로세스 두 개**가 생깁니다.

각 프로세스는 **독립된 메모리 공간**을 가지며, 보통 네 가지 영역으로 구성됩니다. **코드(Text) 영역**(실행할 명령어), **데이터(Data) 영역**(전역·정적 변수), **힙(Heap)**(실행 중 동적으로 할당하는 메모리), **스택(Stack)**(함수 호출, 지역 변수, 매개변수)입니다.

또한 운영체제는 각 프로세스를 관리하기 위해 **PCB(Process Control Block)** 라는 자료구조를 둡니다. PCB에는 **프로세스 ID(PID), 상태, 프로그램 카운터, 레지스터 값, 메모리 정보, 우선순위** 등이 담겨, 프로세스를 전환(Context Switch)하거나 스케줄링할 때 그 정보를 저장·복원하는 데 쓰입니다.

정리하면, 프로세스는 **OS가 자원을 할당하고 관리하는 작업의 기본 단위**입니다. 각자 독립된 메모리를 가져 서로 침범하지 않으므로, 한 프로세스의 오류가 다른 프로세스에 직접 영향을 주지 않는 **격리성**이 핵심 특징입니다.

---

### 2. 꼬리 질문

1. **PCB(Process Control Block)는 무슨 역할을 하나요?**
   → 프로세스에 대한 **모든 정보(PID, 상태, PC, 레지스터, 메모리 정보 등)** 를 담은 자료구조입니다. 컨텍스트 스위칭 때 현재 프로세스 정보를 PCB에 저장하고 다음 프로세스의 PCB를 불러와, 중단된 작업을 정확히 재개하게 합니다.

2. **프로세스의 메모리 영역 중 힙과 스택은 어떻게 다른가요?**
   → **힙**은 실행 중 **동적으로 할당**(malloc 등)하는 영역으로 위로 자라고, **스택**은 **함수 호출·지역 변수**를 담으며 아래로 자랍니다. 둘이 반대 방향으로 커지다 충돌하면 오버플로가 납니다.

3. **프로세스 상태에는 어떤 것들이 있나요?**
   → 대표적으로 **생성(New) → 준비(Ready) → 실행(Running) → 대기(Waiting/Blocked) → 종료(Terminated)** 가 있습니다. CPU를 기다리면 Ready, 실행 중이면 Running, 입출력 등을 기다리면 Waiting 상태가 되며 이 사이를 오갑니다.

## OS-014 프로그램과 프로세스, 스레드의 차이에 대해 설명해 주세요.

### 1. 답변

세 개념은 **"정적인 코드 → 실행 단위 → 실행 흐름"** 으로 이어지는 관계입니다.

**프로그램(Program)** 은 **디스크에 저장된 정적인 파일**, 즉 실행할 명령어와 데이터의 집합입니다. 아직 실행되지 않은 **수동적(passive)** 인 상태로, 그 자체로는 아무 일도 하지 않습니다. 비유하면 **요리 레시피**입니다.

**프로세스(Process)** 는 그 **프로그램이 메모리에 올라와 실행 중인 상태**입니다. OS로부터 **독립된 메모리 공간(코드·데이터·힙·스택)과 자원을 할당**받아 동작하는 **능동적(active)** 인 실체입니다. 레시피를 보고 **실제로 요리하는 행위**에 해당합니다. 각 프로세스는 메모리가 분리돼 있어 서로 직접 침범할 수 없습니다.

**스레드(Thread)** 는 **프로세스 안에서 실제로 실행되는 흐름의 단위**입니다. 한 프로세스는 **하나 이상의 스레드**를 가질 수 있고, 같은 프로세스 안의 스레드들은 **코드·데이터·힙 영역을 공유**합니다. 다만 **스택과 레지스터(PC 등)는 각자 따로** 가집니다. 그래서 스레드는 같은 작업을 **여러 흐름으로 나눠 병렬·동시에** 처리할 수 있습니다. 요리 비유로는, 한 주방(프로세스) 안에서 **여러 요리사(스레드)가 재료를 공유하며 동시에 일하는** 것입니다.

핵심 차이를 정리하면, **자원 할당의 단위는 프로세스**, **CPU 스케줄링(실행)의 단위는 스레드**입니다. 그리고 메모리 측면에서, **프로세스 간에는 메모리가 독립적이라 통신에 IPC가 필요**하지만, **스레드 간에는 메모리를 공유해 통신이 쉽고 빠른** 대신, 공유 자원에 동시에 접근할 때 **동기화 문제(경쟁 상태)** 를 신경 써야 합니다. 또 스레드는 자원을 공유하므로 **생성·전환 비용이 프로세스보다 훨씬 가볍습니다**.

---

### 2. 꼬리 질문

1. **스레드들은 무엇을 공유하고 무엇을 따로 갖나요?**
   → 같은 프로세스의 스레드들은 **코드·데이터·힙 영역을 공유**합니다. 반면 **스택과 레지스터(PC 등)** 는 각 스레드가 독립적으로 가집니다. 그래서 데이터 공유는 쉽지만, 함수 호출 흐름은 스레드마다 분리됩니다.

2. **멀티프로세스 대신 멀티스레드를 쓰면 어떤 장단점이 있나요?**
   → 스레드는 **메모리를 공유해 통신이 빠르고, 생성·전환 비용이 가볍습니다**. 대신 한 스레드의 오류가 **프로세스 전체를 죽일 수 있고**, 공유 자원에 대한 **동기화(경쟁 상태) 문제**를 다뤄야 합니다. 프로세스는 격리성이 좋지만 무겁습니다.

3. **경쟁 상태(Race Condition)는 왜 생기고 어떻게 막나요?**
   → 여러 스레드가 **공유 자원에 동시에 접근**해 실행 순서에 따라 결과가 달라질 때 생깁니다. **뮤텍스(Mutex)·세마포어(Semaphore)** 같은 동기화 기법으로 **임계 구역(Critical Section)** 에 한 번에 하나만 접근하게 막아 해결합니다.
