# 쿠키와 세션

## NET-022 쿠키와 세션의 차이에 대해 설명해 주세요.

### 1. 답변

쿠키와 세션은 모두 **HTTP의 무상태(Stateless) 특성을 보완**하기 위해 사용자 상태를 유지하는 수단입니다.

**쿠키(Cookie)** 는 데이터를 **클라이언트(브라우저)에 저장**합니다.
서버가 응답 시 `Set-Cookie` 헤더로 전달하면, 이후 요청마다 브라우저가 **자동으로 Cookie 헤더에 포함**해 전송합니다.
클라이언트에 저장되므로 **서버 부하가 없고**, 만료 기간 설정으로 **장기 보관이 가능**합니다.
단, 클라이언트에 데이터가 노출되므로 **민감한 정보 저장에 부적합**하고, **4KB 용량 제한**이 있습니다.

**세션(Session)** 은 데이터를 **서버에 저장**하고, 클라이언트에는 **세션 ID만 쿠키로 전달**합니다.
실제 데이터는 서버(메모리, Redis, DB)에 있으므로 **보안성이 높고 저장 용량 제한이 없습니다.**
단, 서버에 상태를 저장하므로 **서버 메모리 부하**가 생기고,
**다중 서버(Scale-out) 환경에서 세션 불일치 문제**가 발생할 수 있습니다.

| 구분 | 쿠키 | 세션 |
|------|------|------|
| **저장 위치** | 클라이언트 | 서버 |
| **보안** | 낮음 | 높음 |
| **용량** | 4KB 제한 | 제한 없음 |
| **만료** | 설정 가능 (장기) | 브라우저 종료 또는 타임아웃 |
| **서버 부하** | 없음 | 있음 |

실무에서는 **세션 불일치 문제** 때문에 Scale-out 환경에서는 세션을 **Redis 같은 중앙 저장소**로 관리하거나, 아예 **JWT 기반의 Stateless 인증**으로 전환하는 경우가 많습니다.

---

### 2. 꼬리 질문

1. **서버가 여러 대인 환경에서 세션을 어떻게 관리하시겠어요?**
    - 세 가지 방법이 있습니다. **Sticky Session**(같은 서버로만 라우팅)은 단순하지만 특정 서버 장애 시 세션이 소실됩니다. **Session Clustering**(서버 간 세션 복제)은 네트워크 비용이 큽니다. 가장 일반적인 방법은 **Redis 같은 외부 중앙 저장소에 세션을 저장**해 모든 서버가 공유하는 방식입니다.

2. **쿠키의 보안 취약점에는 어떤 것이 있고, 어떻게 방어하나요?**
    - 대표적으로 **XSS(스크립트로 쿠키 탈취)** 와 **CSRF(쿠키를 이용한 위조 요청)** 가 있습니다. XSS는 `HttpOnly` 플래그로 JavaScript 접근을 차단하고, `Secure` 플래그로 HTTPS에서만 전송하도록 방어합니다. CSRF는 `SameSite` 속성 설정 또는 **CSRF 토큰**을 함께 검증해 방어합니다.

3. **세션 방식 대신 JWT를 사용하면 어떤 장단점이 있나요?**
    - JWT는 **서버에 상태를 저장하지 않아(Stateless)** Scale-out에 유리하고 서버 부하가 없습니다. 단, 토큰이 탈취되면 **만료 전까지 강제 무효화가 어렵고**, Payload가 Base64로 인코딩되어 있어 **민감 정보를 담으면 노출 위험**이 있습니다. 이를 보완하기 위해 **짧은 Access Token + Refresh Token 조합**을 실무에서 주로 사용합니다.
## NET-023 세션 방식의 로그인 과정에 대해 설명해 주세요.

### 1. 답변

세션 방식 로그인은 **서버가 인증 상태를 직접 관리**하는 방식입니다.
전체 흐름은 다음과 같습니다.

**1. 로그인 요청**
클라이언트가 `POST /login`으로 아이디·비밀번호를 전송합니다.

**2. 인증 처리 및 세션 생성**
서버가 DB에서 사용자를 조회하고 비밀번호를 검증합니다.
인증 성공 시 **고유한 Session ID를 생성**하고,
서버 메모리 또는 Redis에 `{ sessionId → 사용자 정보 }`를 저장합니다.

**3. Session ID 전달**
서버는 응답 헤더에 `Set-Cookie: JSESSIONID=abc123`을 담아 클라이언트에 전달합니다.
브라우저는 이 쿠키를 저장합니다.

**4. 인증이 필요한 요청**
이후 요청마다 브라우저가 **자동으로 Cookie 헤더에 Session ID를 포함**해 전송합니다.
서버는 Session ID로 저장소를 조회해 **사용자 정보를 확인하고 인가(Authorization)를 처리**합니다.

**5. 로그아웃**
서버에서 해당 Session ID를 **즉시 삭제**하면 이후 요청은 인증 실패 처리됩니다.

실무에서는 Session ID를 **UUID처럼 예측 불가능한 값**으로 생성하고,
**HttpOnly + Secure 쿠키 설정**으로 탈취를 방어하며,
**세션 만료 시간(timeout)** 을 짧게 유지하는 것이 중요합니다.

---

### 2. 꼬리 질문

1. **Session ID가 탈취되면 어떻게 되나요? 어떻게 방어하나요?**
   - 공격자가 탈취한 Session ID로 **정상 사용자처럼 요청(세션 하이재킹)** 할 수 있습니다. `HttpOnly`·`Secure` 쿠키 설정, **HTTPS 강제**, 로그인 시 **Session ID 재발급(Session Fixation 방어)**, IP·User-Agent 검증을 추가로 적용해 방어합니다.

2. **세션 만료 시간은 어떻게 설계하시겠어요?**
   - 보안 민감도에 따라 다릅니다. 금융·결제 서비스는 **5~15분의 짧은 타임아웃**을 적용하고, 일반 서비스는 **30분~수 시간**으로 설정합니다. 또한 **마지막 요청 시간 기준으로 갱신되는 Sliding Expiration** 방식을 적용하면 사용 중 강제 로그아웃을 방지할 수 있습니다.

3. **로그아웃 시 클라이언트 쿠키만 삭제하면 안 되나요?**
   - 안 됩니다. 클라이언트 쿠키만 삭제하면 **서버에 세션이 남아있어** 탈취된 Session ID로 계속 접근이 가능합니다. 반드시 **서버 세션 저장소에서도 삭제**해야 완전한 로그아웃이 보장됩니다.

---

## NET-024 Stateless의 의미를 살펴보면, 세션은 적절하지 않은 인증 방법 아닌가요?

### 1. 답변

좋은 질문입니다. 맞습니다. **세션은 HTTP의 Stateless 원칙에 위배됩니다.**

HTTP Stateless란 **서버가 이전 요청의 상태를 기억하지 않는다**는 원칙입니다.
세션은 서버가 상태(`sessionId → 사용자 정보`)를 직접 저장하고 관리하므로
**Stateful한 구조**입니다.

하지만 이것이 세션이 잘못된 방식이라는 의미는 아닙니다.
**Stateless는 HTTP 프로토콜 설계 원칙**이고,
**인증은 애플리케이션 레벨의 요구사항**이기 때문입니다.
사용자 인증이라는 비즈니스 요구를 충족하기 위해 **의도적으로 Stateful한 세션을 도입한 것**입니다.

다만 세션 방식은 다음과 같은 **실무적 한계**가 있습니다.

- **Scale-out 시 세션 불일치**: 서버가 늘어날수록 세션 공유 문제 발생
- **서버 메모리 부하**: 동시 접속자가 많을수록 서버 자원 소모
- **MSA 환경 부적합**: 여러 서비스가 세션 저장소를 공유해야 하는 복잡성

이러한 한계 때문에 현대 아키텍처에서는 **JWT 기반의 Stateless 인증**이 많이 채택됩니다.
JWT는 토큰 자체에 사용자 정보를 담아 **서버가 상태를 저장하지 않아도** 인증이 가능합니다.
단, **토큰 강제 만료가 어렵다는 단점**이 있어
서비스 특성에 따라 **세션과 JWT를 적절히 선택하거나 혼용**하는 것이 중요합니다.

---

### 2. 꼬리 질문

1. **JWT가 Stateless라면 로그아웃을 어떻게 구현하나요?**
   - JWT는 서버에 상태가 없어 **토큰을 즉시 무효화하기 어렵습니다.** 실무에서는 **Access Token 만료 시간을 짧게(5~15분)** 유지하고, 로그아웃 시 **Refresh Token을 Redis 블랙리스트에 등록**하거나 삭제해 재발급을 차단하는 방식으로 처리합니다.

2. **세션과 JWT를 혼용하는 경우가 있나요? 어떤 상황에서 선택하시겠어요?**
   - 관리자 페이지처럼 **즉각적인 권한 회수가 필요한 기능은 세션**, 일반 사용자 API나 **MSA 환경의 서비스 간 인증은 JWT**를 사용하는 혼용 전략이 실무에서 사용됩니다. 강제 로그아웃·계정 정지 같은 실시간 제어가 필요하면 세션이 더 적합합니다.

3. **세션 저장소로 Redis를 사용할 때 Redis가 다운되면 어떻게 되나요?**
   - 모든 사용자가 **인증 불가 상태**가 되는 심각한 장애가 발생합니다. 이를 방지하기 위해 **Redis Sentinel이나 Cluster로 고가용성(HA)** 을 구성하고, **Redis 장애 시 fallback 정책**(일시적 세션 허용 또는 서비스 차단)을 사전에 설계해두는 것이 중요합니다.


## NET-025 규모가 커져 서버가 여러 개가 된다면, 세션을 어떻게 관리할 수 있을까요?

### 1. 답변

서버가 여러 대로 늘어나면 사용자의 세션 정보가 특정 서버 메모리에만 저장되어 있는 경우 문제가 발생합니다.

예를 들어 로그인 후 생성된 세션이 A 서버에 저장되었는데, 다음 요청이 B 서버로 전달되면 세션 정보를 찾지 못해 인증이 풀릴 수 있습니다.

이를 해결하는 대표적인 방법은 다음과 같습니다.

- **Sticky Session**: 로드밸런서가 사용자의 요청을 항상 같은 서버로 전달
- **Session Clustering**: 서버 간 세션 정보를 동기화
- **Session Storage 분리**: Redis, DB 등에 세션을 중앙 저장

실무에서는 주로 **Redis 기반 중앙 세션 저장소**를 사용합니다. 모든 서버가 동일한 Redis의 세션 정보를 조회하므로 어느 서버로 요청이 가더라도 동일한 세션을 사용할 수 있습니다.

최근에는 서버 상태를 저장하지 않는 **JWT 기반 Stateless 인증**을 사용하여 세션 자체를 제거하는 방식도 많이 사용됩니다.

성능과 확장성 측면에서는 **Redis/JWT 방식 > Session Clustering > Sticky Session** 순으로 선호되는 경우가 많습니다.

---

### 2. 꼬리 질문

#### 1. Sticky Session의 단점은 무엇인가요?

특정 사용자가 항상 같은 서버를 사용하므로 서버 간 부하 분산이 어려워집니다.

또한 해당 서버가 장애가 나면 세션 정보도 함께 사라져 사용자가 다시 로그인해야 할 수 있습니다.

---

#### 2. Session Clustering 방식의 문제점은 무엇인가요?

서버가 증가할수록 세션 동기화 트래픽이 급격히 증가합니다.

세션 변경 시 모든 서버에 복제해야 하므로 네트워크 비용과 관리 복잡도가 커집니다.

---

#### 3. Redis를 세션 저장소로 많이 사용하는 이유는 무엇인가요?

Redis는 **메모리 기반**이라 조회 속도가 매우 빠르고 TTL을 통해 세션 만료 관리도 쉽습니다.

또한 여러 서버가 동일한 Redis를 사용하므로 수평 확장 환경에서도 안정적으로 세션을 공유할 수 있습니다.
