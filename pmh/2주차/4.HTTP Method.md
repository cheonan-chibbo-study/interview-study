# HTTP Method

## NET-017 HTTP Method에 대해 설명해 주세요.

### 1. 답변

HTTP Method는 **클라이언트가 서버에 어떤 동작을 요청하는지 의미를 명시**하는 수단입니다.
RESTful API 설계에서 리소스에 대한 행위를 표현하는 핵심 요소입니다.

| 메서드 | 용도 | 멱등성 | 안전성 |
|--------|------|--------|--------|
| **GET** | 리소스 조회 | ✅ | ✅ |
| **POST** | 리소스 생성 | ❌ | ❌ |
| **PUT** | 리소스 전체 교체 | ✅ | ❌ |
| **PATCH** | 리소스 부분 수정 | ❌ | ❌ |
| **DELETE** | 리소스 삭제 | ✅ | ❌ |
| **HEAD** | GET과 동일하나 Body 제외 | ✅ | ✅ |
| **OPTIONS** | 서버가 허용하는 메서드 조회 | ✅ | ✅ |

실무에서는 **GET/POST/PUT/PATCH/DELETE** 5가지를 주로 사용하며,
**PUT vs PATCH** 구분이 자주 혼용되는데, PUT은 리소스 **전체를 교체**, PATCH는 **일부 필드만 수정**한다는 차이가 있습니다.
또한 **GET 요청에는 Body를 담지 않는 것**이 표준이며, 일부 인프라에서 무시되거나 오동작할 수 있습니다.

---

### 2. 꼬리 질문

1. **PUT과 PATCH의 차이를 실무 예시로 설명해 주세요.**
    - PUT으로 회원 정보를 수정할 때 일부 필드만 보내면 **나머지 필드가 null로 덮어써질 수 있습니다.** PATCH는 보낸 필드만 수정하므로 부분 업데이트에 적합합니다. 실무에서는 PATCH를 더 안전하게 사용하는 경우가 많습니다.

2. **POST 요청이 멱등하지 않다면, 결제나 주문 같은 중복 요청은 어떻게 방어하나요?**
    - 클라이언트가 `Idempotency-Key`를 헤더에 포함해 보내고, 서버에서 **해당 키를 기준으로 중복 요청을 감지**해 동일한 응답을 반환합니다. DB unique 제약이나 Redis를 활용한 분산 락으로 처리하기도 합니다.

3. **GET 요청도 서버 상태를 변경하도록 구현할 수 있는데, 이게 왜 문제인가요?**
    - GET은 **안전한(Safe) 메서드**로 간주되어 브라우저, 프록시, CDN이 **캐싱하거나 prefetch**할 수 있습니다. 상태를 변경하는 로직이 있으면 의도치 않게 **중복 실행되거나 캐시된 응답으로 변경이 누락**될 수 있습니다.

## NET-018 HTTP Method의 멱등성에 대해 설명해 주세요.

### 1. 답변

**멱등성(Idempotency)** 이란 **동일한 요청을 여러 번 보내도 서버의 상태가 항상 같은 결과를 유지하는 성질**입니다.
응답값이 같다는 의미가 아니라, **서버 상태(Side Effect)가 동일하게 유지됨**을 의미합니다.

- **멱등 O**: `GET`, `PUT`, `DELETE`, `HEAD`, `OPTIONS`
- **멱등 X**: `POST`, `PATCH`

예를 들어 `DELETE /users/1`을 두 번 호출하면 첫 번째는 200, 두 번째는 404가 반환될 수 있지만, **서버에 user/1이 존재하지 않는다는 상태는 동일**하므로 멱등합니다.

반면 `POST /orders`를 두 번 호출하면 **주문이 2개 생성**되므로 멱등하지 않습니다.

멱등성은 **네트워크 장애나 타임아웃 발생 시 클라이언트가 안전하게 재시도할 수 있는지** 판단하는 기준이 되기 때문에 실무에서 매우 중요합니다. 멱등하지 않은 API를 재시도하면 **중복 결제, 중복 주문** 같은 심각한 장애로 이어질 수 있습니다.

---

### 2. 꼬리 질문

1. **DELETE가 멱등하다고 하셨는데, 두 번째 호출 시 404를 반환하면 멱등성이 깨진 게 아닌가요?**
    - 멱등성은 **응답 코드가 아닌 서버 상태 기준**입니다. 두 번 호출해도 해당 리소스가 삭제된 상태는 동일하므로 멱등합니다. 다만 404를 반환할지 200을 반환할지는 API 설계 정책의 문제입니다.

2. **PATCH가 멱등하지 않다고 하셨는데, 경우에 따라 멱등하게 구현할 수도 있지 않나요?**
    - 맞습니다. `PATCH /users/1 { "name": "홍길동" }` 처럼 **절대값으로 수정하면 멱등**하지만, `{ "balance": "+1000" }` 처럼 **상대값으로 수정하면 멱등하지 않습니다.** HTTP 명세상 PATCH는 멱등을 보장하지 않는다고 명시하고 있습니다.

3. **네트워크 타임아웃이 발생했을 때 멱등하지 않은 API를 재시도해야 한다면 어떻게 처리하시겠어요?**
    - 클라이언트가 **고유한 `Idempotency-Key`를 생성해 헤더에 포함**하고, 서버는 해당 키로 처리 결과를 **캐싱(Redis 등)** 해 중복 요청 시 동일한 응답을 반환합니다. Stripe, 토스페이먼츠 같은 결제 시스템에서 실제로 사용하는 패턴입니다.
## NET-019 GET과 POST의 차이는 무엇인가요?

### 1. 답변

GET과 POST는 가장 많이 사용되는 HTTP 메서드로, **데이터 전달 방식과 목적**에서 핵심적인 차이가 있습니다.

**GET**은 **리소스를 조회**하는 메서드로, 데이터를 **URL 쿼리스트링**에 포함해 전송합니다.
**서버 상태를 변경하지 않는 안전한(Safe) 메서드**이며, 브라우저와 CDN, 프록시에서 **캐싱**이 가능합니다.
단, URL에 데이터가 노출되므로 **민감한 정보 전송에 부적합**하고, URL 길이 제한으로 **대용량 데이터 전송이 불가**합니다.

**POST**는 **리소스를 생성하거나 서버에 데이터를 제출**하는 메서드로, 데이터를 **요청 Body**에 담아 전송합니다.
URL에 노출되지 않아 상대적으로 안전하며, **대용량 데이터나 파일 전송**에 적합합니다.
**멱등하지 않으므로** 동일 요청을 반복하면 리소스가 중복 생성될 수 있습니다.

실무에서는 **조회는 GET, 생성·처리는 POST**를 원칙으로 하되, GET은 캐싱 전략과 함께 활용하면 **성능 최적화**에 효과적입니다.

---

### 2. 꼬리 질문

1. **GET 요청도 Body에 데이터를 담을 수 있나요?**
   - 기술적으로는 가능하지만 **HTTP 명세에서 권장하지 않으며**, 브라우저·프록시·일부 서버가 GET Body를 **무시하거나 거부**할 수 있습니다. 복잡한 조회 조건이 필요하다면 POST를 사용하거나 쿼리스트링을 설계하는 것이 안전합니다.

2. **GET이 POST보다 빠르다고 할 수 있나요? 그 이유는 무엇인가요?**
   - 단순 네트워크 속도는 비슷하지만, GET은 **브라우저·CDN·프록시 캐싱**이 가능해 반복 요청 시 서버까지 도달하지 않아 실질적으로 빠릅니다. POST는 매 요청마다 서버가 처리해야 하므로 캐싱 이점이 없습니다.

3. **로그인 요청을 GET으로 구현하면 어떤 문제가 발생하나요?**
   - 아이디·비밀번호가 **URL에 노출**되어 브라우저 히스토리, 서버 액세스 로그, 프록시 로그에 평문으로 기록됩니다. 또한 캐싱될 경우 인증 정보가 캐시에 저장되는 **심각한 보안 취약점**이 발생합니다.

## NET-020 POST와 PUT, PATCH의 차이는 무엇인가요?

### 1. 답변

세 메서드 모두 서버 상태를 변경하지만, **목적과 멱등성, 수정 범위**에서 차이가 있습니다.

**POST**는 **새로운 리소스를 생성**할 때 사용합니다.
리소스의 URI를 **서버가 결정**하며(`/users` → `/users/1` 생성), **멱등하지 않아** 동일 요청을 반복하면 리소스가 중복 생성됩니다.

**PUT**은 **리소스를 전체 교체**할 때 사용합니다.
리소스의 URI를 **클라이언트가 지정**하며(`/users/1`), 보내지 않은 필드는 **null 또는 기본값으로 덮어써집니다.**
동일 요청을 반복해도 결과가 같으므로 **멱등합니다.**

**PATCH**는 **리소스의 일부 필드만 수정**할 때 사용합니다.
보낸 필드만 업데이트하고 나머지는 유지하므로 **부분 업데이트에 적합**합니다.
단, 상대값(`+1000` 증가 등)으로 구현하면 **멱등하지 않을 수 있습니다.**

실무에서는 **전체 교체가 필요한 경우 PUT, 특정 필드만 수정할 때 PATCH**를 사용하며, PUT 사용 시 **누락 필드 덮어쓰기 버그**를 가장 주의해야 합니다.

---

### 2. 꼬리 질문

1. **PUT으로 존재하지 않는 리소스를 요청하면 어떻게 처리해야 하나요?**
   - HTTP 명세상 PUT은 **리소스가 없으면 생성(201), 있으면 교체(200)** 하는 것이 표준입니다. 다만 실무에서는 보안·정책상 이유로 **없으면 404를 반환**하도록 제한하는 경우도 많습니다.

2. **PATCH가 PUT보다 항상 더 좋은 선택인가요?**
   - 그렇지 않습니다. PATCH는 **부분 업데이트 명세가 복잡**하고(JSON Patch, JSON Merge Patch 등 표준이 여럿), 구현 방식이 팀마다 달라질 수 있습니다. 단순한 리소스라면 **PUT으로 전체를 보내는 게 오히려 명확하고 안전**할 수 있습니다.

3. **대용량 트래픽 환경에서 PUT과 PATCH 중 어떤 것이 DB 부하 측면에서 유리한가요?**
   - PATCH가 유리합니다. PUT은 **모든 컬럼을 UPDATE**하므로 불필요한 쓰기가 발생하고, 인덱스 재구성 비용이 생길 수 있습니다. PATCH는 **변경된 컬럼만 UPDATE**하므로 DB I/O와 인덱스 부하를 줄일 수 있습니다.
