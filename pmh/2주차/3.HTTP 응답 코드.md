# HTTP 응답 코드

## NET-013 HTTP 응답코드에 대해 설명해 주세요.

### 1. 답변

HTTP 응답 코드는 서버가 클라이언트의 요청을 처리한 결과를 **3자리 숫자**로 나타내는 표준 규약입니다.

크게 5가지 클래스로 분류됩니다.

- **1xx (Informational)**: 요청을 수신했고 처리 중임을 알리는 코드. `101 Switching Protocols`는 WebSocket 핸드셰이크 시 사용됩니다.
- **2xx (Success)**: 요청이 성공적으로 처리됨. `200 OK`, `201 Created`(리소스 생성 성공), `204 No Content`(삭제 등 응답 바디 불필요 시) 구분이 중요합니다.
- **3xx (Redirection)**: 추가 액션이 필요함. `301`은 **영구 이동** (SEO에 영향), `302`는 **임시 이동**, `304 Not Modified`는 **캐시 활용** 시 사용됩니다.
- **4xx (Client Error)**: 클라이언트 측 문제. `400` 잘못된 요청, `401` 인증 실패, `403` 권한 없음, `404` 리소스 없음, `429` 요청 한도 초과.
- **5xx (Server Error)**: 서버 측 문제. `500` 서버 내부 오류, `502/503/504`는 **게이트웨이·프록시 계층** 장애 시 발생합니다.

실무에서는 응답 코드를 단순히 반환하는 것에서 그치지 않고, **API 설계 시 의미에 맞는 코드를 일관성 있게 사용**하는 것이 중요합니다. 특히 `200`으로 모든 오류를 처리하면 클라이언트가 에러를 감지하기 어렵고, 모니터링 및 알람 시스템에도 악영향을 줍니다.

---

### 2. 꼬리 질문

1. **`301`과 `302`의 차이가 실제 서비스에서 왜 중요한가요?**
   `301`은 브라우저와 검색엔진이 **영구적으로** 새 URL을 기억하므로, 잘못 설정하면 캐시가 남아 원복이 어렵습니다. `302`는 임시이므로 원본 URL을 유지합니다. SEO와 캐시 전략 모두에 영향을 줍니다.

2. **`400 Bad Request`와 `422 Unprocessable Entity`는 어떻게 구분해서 사용하나요?**
   `400`은 요청 자체가 문법적으로 잘못된 경우(파싱 불가), `422`는 문법은 올바르나 **비즈니스 유효성 검증 실패**(예: 이메일 형식 오류)에 사용합니다. REST API 설계 시 명확히 구분하면 클라이언트가 오류를 처리하기 쉬워집니다.

3. **`503 Service Unavailable`과 `504 Gateway Timeout`은 어떤 상황에서 발생하며, 장애 대응 관점에서 어떻게 다르게 처리해야 하나요?**
   `503`은 서버가 **과부하 또는 점검 중**일 때(Retry-After 헤더로 재시도 시간 안내 가능), `504`는 **업스트림 서버가 응답 시간 초과**일 때 발생합니다. `504`는 타임아웃 임계값과 의존 서비스 장애를 우선 점검해야 합니다.

## NET-014 401 (Unauthorized)와 403 (Forbidden)은 의미적으로 어떤 차이가 있나요?

### 1. 답변

이름만 보면 혼동하기 쉽지만, 두 코드는 보안의 다른 계층을 나타냅니다.

- **401 Unauthorized**: 이름과 달리 실제 의미는 **"인증(Authentication) 실패"** 입니다. "당신이 누구인지 모르겠다"는 뜻으로, 로그인하지 않았거나 토큰이 만료·유효하지 않을 때 반환합니다. 클라이언트에게 **다시 인증하라는 신호**이며, `WWW-Authenticate` 헤더와 함께 반환하는 것이 표준입니다.

- **403 Forbidden**: **"인가(Authorization) 실패"** 입니다. "당신이 누구인지는 알지만, 이 리소스에 접근할 권한이 없다"는 뜻입니다. 예를 들어 일반 유저가 관리자 API를 호출할 때 반환합니다. 재인증해도 해결되지 않으므로 클라이언트가 **다른 행동을 취해야 한다**는 신호입니다.

실무에서는 보안상의 이유로 **존재하지 않는 리소스에 대한 권한 없는 접근도 의도적으로 `404`로 반환**하는 경우도 있습니다. 리소스의 존재 자체를 숨기기 위해서입니다.

---

### 2. 꼬리 질문

1. **JWT 토큰이 만료된 경우와 권한이 없는 경우, 각각 어떤 응답 코드를 반환하고 클라이언트는 어떻게 처리해야 하나요?**
   만료 시 `401`을 반환하고 클라이언트는 **Refresh Token으로 재발급** 후 재시도합니다. 권한 없음은 `403`이며 재시도가 무의미하므로 UI에서 접근 불가 메시지를 표시합니다.

2. **보안 관점에서 403 대신 404를 반환하는 전략은 어떤 상황에 유효한가요?**
   어드민 엔드포인트나 다른 사용자의 리소스에 접근 시, `403`은 리소스가 존재함을 노출합니다. `404`로 반환하면 **리소스 존재 여부 자체를 은폐**할 수 있어 정보 노출 공격(Enumeration Attack)을 방어할 수 있습니다.

3. **API Gateway나 Spring Security 같은 프레임워크에서 인증/인가 필터 체인을 설계할 때, 401과 403을 어느 레이어에서 처리하는 것이 좋을까요?**
   `401`은 **인증 필터(AuthenticationFilter)** 에서, `403`은 **인가 필터(AccessDecisionManager/AuthorizationFilter)** 에서 처리하는 것이 표준입니다. 필터 순서가 잘못되면 `403`이 와야 할 자리에 `401`이 반환되는 버그가 생기므로, 체인 순서 관리가 중요합니다.
