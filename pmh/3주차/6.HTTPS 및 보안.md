# HTTPS 및 보안
## NET-026 공개키와 대칭키에 대해 설명해 주세요.

### 1. 답변

암호화 방식은 크게 **대칭키**와 **공개키(비대칭키)** 로 나뉩니다.

**대칭키**는 암호화와 복호화에 **동일한 키**를 사용합니다. 연산이 단순해서 **속도가 빠르고** 대용량 데이터 암호화에 유리하지만, 키를 상대방에게 안전하게 전달하는 **키 교환 문제**가 있습니다. 대표적으로 **AES**가 있습니다.

**공개키**는 **공개키와 개인키 한 쌍**을 사용합니다. 공개키로 암호화하면 개인키로만 복호화할 수 있어서 키를 공개해도 안전하고, 키 교환 문제를 해결합니다. 다만 연산이 무거워 **속도가 느립니다**. 대표적으로 **RSA, ECC**가 있습니다.

그래서 실무에서는 둘을 **혼용**합니다. **TLS**가 대표적인데, 느리지만 안전한 공개키로 **데이터 암호화에 쓸 세션키(대칭키)를 안전하게 마련**하고, 실제 데이터 통신은 빠른 대칭키로 처리합니다. 즉 **"키 교환은 공개키, 데이터 전송은 대칭키"** 구조로 보안과 성능을 모두 챙기는 게 핵심입니다.

이때 세션키를 마련하는 방식은 두 가지가 있습니다. 옛날 **RSA 방식**은 클라이언트가 세션키를 만들어 **서버의 공개키로 암호화해 전송**하고, **서버만 개인키로 복호화**해 같은 키를 공유했습니다. 다만 이 방식은 **서버 개인키가 유출되면 과거 통신까지 모두 복호화**되는 약점이 있습니다. 그래서 현재 표준인 **TLS 1.3**은 **ECDHE 방식**을 사용하는데, 세션키 자체는 네트워크로 보내지 않고 **세션키를 만들기 위한 공개값(재료)만 교환**한 뒤 **양쪽이 각자 계산해서 같은 키를 도출**합니다. 완성된 키가 네트워크에 아예 흐르지 않아 더 안전하고, 세션마다 키를 새로 만들어 **순방향 비밀성(PFS)** 까지 확보하는 게 핵심입니다.

---

### 2. 꼬리 질문

1. **대칭키가 빠른데도 공개키를 함께 쓰는 이유가 뭘까요?**
   → 대칭키는 빠르지만 키를 안전하게 전달할 방법이 없습니다. 공개키로 **서버 신원을 검증**하고 이 **키 교환 문제**를 해결한 뒤, 마련된 대칭키로 빠르게 통신하기 위해서입니다.

2. **RSA 키 교환과 ECDHE 방식의 차이는 무엇인가요?**
   → RSA는 세션키를 공개키로 암호화해 **직접 전송**하지만, ECDHE는 세션키 자체는 보내지 않고 **공개값만 교환해 양쪽이 각자 계산**합니다. ECDHE는 개인키가 유출돼도 과거 통신이 안전한 **순방향 비밀성(PFS)**을 보장합니다.

3. **ECDHE는 키를 안 보낸다는데, 그럼 양쪽이 어떻게 같은 세션키를 갖나요?**
   → 각자 **비밀값**을 두고, 거기서 만든 **공개값만 주고받습니다**. 받은 공개값에 자기 비밀값을 다시 연산하면 양쪽이 수학적으로 **같은 값**에 도달합니다. 비밀값과 완성된 세션키는 네트워크에 흐르지 않고, 공개값만으로는 **역산이 불가능**해 도청자는 세션키를 알 수 없습니다.

4. **개인키가 유출되면 어떤 일이 발생하나요?**
   → RSA 방식이라면 과거 통신까지 전부 복호화될 수 있습니다. 그래서 **인증서 폐기(CRL/OCSP)**와 키 교체가 필요하고, 애초에 **ECDHE로 세션마다 키를 새로 만드는 것**이 중요합니다.

## NET-027 왜 HTTPS Handshake 과정에서는 인증서를 사용하는 것일까요?

### 1. 답변

핵심은 **서버가 진짜 그 서버가 맞는지 신원을 검증**하기 위해서입니다.

공개키 암호화만으로는 통신 내용을 숨길 순 있어도, 내가 받은 공개키가 **진짜 서버의 것인지** 알 수 없습니다. 공격자가 중간에서 자기 공개키를 끼워 넣으면 **중간자 공격(MITM)** 에 그대로 노출됩니다.

이를 막기 위해 **CA(인증기관)** 가 서명한 **인증서**를 사용합니다. 인증서 안에는 서버의 **공개키와 도메인 정보**가 들어 있고, 여기에 CA가 자신의 개인키로 **전자서명**을 합니다. 클라이언트는 이미 신뢰하는 **CA의 공개키(루트 인증서)** 로 이 서명을 검증해서, 공개키가 위변조되지 않았고 신뢰할 수 있는 서버라는 걸 확인합니다.    

정리하면 인증서는 **"이 공개키가 진짜 이 도메인의 것"임을 보증**하는 역할이고, 이를 통해 HTTPS는 **기밀성(암호화) + 무결성 + 인증**을 모두 보장합니다. 실무에서 인증서 만료나 체인 오류는 곧바로 접속 차단으로 이어지므로 **만료 모니터링과 자동 갱신(Let's Encrypt 등)** 이 중요합니다.

---

### 2. 꼬리 질문

1. **인증서가 없으면 구체적으로 어떤 공격이 가능한가요?**
   → 공격자가 중간에서 자기 공개키를 서버 것처럼 전달하는 **MITM 공격**이 가능합니다. 클라이언트는 위장 서버와 정상 통신하는 줄 알고 모든 데이터를 노출하게 됩니다.

2. **CA를 신뢰한다는 건 기술적으로 어떻게 보장되나요?**
   → OS나 브라우저에 **루트 CA 인증서**가 미리 내장돼 있습니다. 서버 인증서부터 루트까지 **신뢰 체인(Chain of Trust)**이 서명으로 연결돼 검증됩니다.

3. **인증서 검증은 통과했는데 폐기된 인증서라면 어떻게 거르나요?**
   → **CRL**(폐기 목록)이나 **OCSP**로 실시간 폐기 여부를 확인합니다. 성능 때문에 요즘은 **OCSP Stapling**으로 서버가 폐기 응답을 미리 첨부해 검증 지연을 줄입니다.


## NET-028 SSL과 TLS의 차이는 무엇인가요?

### 1. 답변

둘 다 데이터를 **암호화해 안전하게 주고받기 위한 프로토콜**이고, 사실상 **같은 계보**입니다.

**SSL(Secure Sockets Layer)**이 먼저 나왔고, 이후 표준화되면서 이름이 **TLS(Transport Layer Security)**로 바뀌었습니다. 즉 **TLS는 SSL의 후속 버전**입니다. 흐름으로 보면 **SSL 2.0/3.0 → TLS 1.0 → 1.1 → 1.2 → 1.3** 순으로 발전했습니다.

지금은 SSL과 TLS 1.0/1.1까지 모두 **보안 취약점으로 폐기(deprecated)**됐고, 현재 실무 표준은 **TLS 1.2와 TLS 1.3**입니다. 특히 **TLS 1.3**은 핸드셰이크 단계를 줄여 **1-RTT(재접속 시 0-RTT)**로 연결 지연을 크게 낮췄고, 취약한 **RSA 키 교환을 제거**하고 **ECDHE만 강제**해 **순방향 비밀성(PFS)**을 기본 보장합니다.

실무에서 흔히 "SSL 인증서", "SSL 적용"이라고 부르지만 **실제 동작하는 건 TLS**입니다. 이름만 관습적으로 SSL이 남아 있는 것이고, 서버 설정 시 **TLS 1.2 이상만 허용**하고 구버전을 막는 게 보안의 기본입니다.

---

### 2. 꼬리 질문

1. **TLS 1.3이 1.2보다 빠른 이유는 무엇인가요?**
   → 핸드셰이크 왕복(RTT)을 줄였기 때문입니다. 1.2는 보통 **2-RTT**가 필요했지만, 1.3은 **1-RTT**, 재접속 시엔 **0-RTT**까지 가능해 초기 연결 지연이 줄어듭니다.

2. **왜 SSL과 TLS 1.0/1.1은 폐기됐나요?**
   → POODLE, BEAST 같은 **알려진 취약점**과 약한 암호화 알고리즘 때문입니다. 현재는 PCI-DSS 등 보안 규정에서도 **TLS 1.2 이상**을 요구합니다.

3. **TLS 1.3에서 RSA 키 교환을 없앤 이유는 무엇인가요?**
   → RSA 방식은 **서버 개인키 유출 시 과거 통신까지 복호화**되는 약점이 있습니다. ECDHE만 강제해 세션마다 키를 새로 만드는 **순방향 비밀성(PFS)**을 기본으로 확보하기 위해서입니다.

---

## NET-029 XSS에 대해서 설명해 주세요.

### 1. 답변

**XSS(Cross-Site Scripting)**는 공격자가 **악성 스크립트를 웹 페이지에 삽입**해, 그 페이지를 보는 **다른 사용자의 브라우저에서 실행**시키는 공격입니다.

동작 원리는, 사용자가 입력한 값(댓글, 검색어 등)을 서버나 화면이 **검증·이스케이프 없이 그대로 출력**할 때 발생합니다. 예를 들어 댓글에 `<script>` 코드를 넣으면, 그걸 보는 사람의 브라우저에서 스크립트가 실행되어 **쿠키·세션 토큰 탈취, 키 입력 가로채기, 화면 위변조** 등이 가능해집니다.

유형은 크게 세 가지입니다. **Stored(저장형)**는 DB에 악성 스크립트가 저장돼 지속적으로 노출되고, **Reflected(반사형)**는 URL 파라미터 등으로 즉시 반사되며, **DOM-based**는 서버를 거치지 않고 **프론트엔드 JS가 DOM을 조작하는 과정**에서 발생합니다.

방어의 핵심은 **출력 시 이스케이프(인코딩)**입니다. 요즘은 **React, Vue** 같은 프레임워크가 기본적으로 출력값을 **자동 이스케이프**해줘서 상당 부분 방어됩니다. 다만 `dangerouslySetInnerHTML` 같은 우회 지점은 여전히 위험합니다. 여기에 **CSP(Content Security Policy) 헤더**로 허용된 스크립트 출처만 실행하게 막고, 쿠키에 **HttpOnly** 플래그를 걸어 스크립트의 쿠키 접근을 차단하는 게 실무 표준입니다.

---

### 2. 꼬리 질문

1. **Stored XSS와 Reflected XSS의 차이는 무엇인가요?**
   → Stored는 DB에 저장돼 **모든 조회자에게 지속적으로** 영향을 주고, Reflected는 악성 링크를 **클릭한 사용자에게 일회성**으로 실행됩니다. 영향 범위 면에서 Stored가 더 위험합니다.

2. **React를 쓰면 XSS가 완전히 막히나요?**
   → 기본 출력은 자동 이스케이프되지만, **완전하지 않습니다**. `dangerouslySetInnerHTML`, `href`에 들어가는 `javascript:` URL 등은 그대로 뚫리므로 별도 검증이 필요합니다.

3. **HttpOnly 쿠키가 XSS 방어에 어떻게 도움이 되나요?**
   → HttpOnly가 걸린 쿠키는 **JS의 `document.cookie`로 접근할 수 없습니다**. XSS로 스크립트가 실행돼도 **세션 토큰 탈취를 막을 수 있어** 피해를 줄입니다.

---

## NET-030 CSRF랑 XSS는 어떤 차이가 있나요?

### 1. 답변

둘 다 웹 보안 취약점이지만, **공격 대상과 신뢰 방향이 정반대**입니다.

**XSS**는 **사용자의 브라우저를 공격**합니다. 악성 스크립트를 페이지에 삽입해 **사용자 측에서 실행**시키고, 쿠키 탈취나 화면 조작을 노립니다. 즉 **사이트가 사용자를 믿는 점**을 악용합니다.

**CSRF(Cross-Site Request Forgery)**는 **서버를 속이는 공격**입니다. 사용자가 **이미 로그인된 상태**를 이용해, 공격자가 만든 위조 요청(예: 송금, 비밀번호 변경)을 **사용자도 모르게 서버로 보냅니다**. 브라우저가 쿠키를 자동 첨부하는 점을 노린 것으로, **서버가 사용자의 요청을 믿는 점**을 악용합니다.

방어 방식도 다릅니다. XSS는 **입력 검증·출력 이스케이프·CSP**로 막고, CSRF는 **요청의 출처가 정상인지 검증**하는 데 집중합니다. 대표적으로 **CSRF 토큰**(서버가 발급한 비밀값을 요청마다 확인), 그리고 요즘 표준인 쿠키의 **SameSite 속성**(Lax/Strict로 설정해 외부 사이트발 요청에는 쿠키를 안 보냄)이 있습니다. 최신 브라우저는 **SameSite=Lax가 기본값**이라 기본 방어 수준이 많이 올라갔습니다.

---

### 2. 꼬리 질문

1. **CSRF 토큰은 어떤 원리로 공격을 막나요?**
   → 서버가 발급한 **예측 불가능한 토큰**을 정상 폼/요청에만 심어두고 검증합니다. 공격자는 이 토큰 값을 알 수 없어 **위조 요청이 검증에서 걸러집니다**.

2. **SameSite 쿠키 속성이 CSRF를 어떻게 방어하나요?**
   → `SameSite=Lax/Strict`면 **다른 사이트에서 발생한 요청에는 쿠키를 자동 첨부하지 않습니다**. CSRF가 노리는 "자동 첨부되는 인증 쿠키"를 차단하는 셈입니다.

3. **JWT를 헤더로 보내면 CSRF에 안전한가요?**
   → 토큰을 **쿠키가 아니라 Authorization 헤더**로 직접 실어 보내면, 브라우저가 자동 첨부하지 않으므로 CSRF에 강합니다. 다만 토큰을 LocalStorage에 두면 **XSS에 취약**해지는 트레이드오프가 있습니다.

---

## NET-031 XSS는 프론트엔드에서만 막을 수 있나요?

### 1. 답변

아닙니다. **프론트엔드와 백엔드 양쪽에서 막아야 하며, 오히려 백엔드 방어가 더 근본적**입니다.

프론트엔드 방어만 믿으면 안 되는 이유는, **클라이언트 검증은 우회가 너무 쉽기 때문**입니다. 공격자는 브라우저를 거치지 않고 **Postman이나 직접 HTTP 요청**으로 악성 값을 서버에 그대로 보낼 수 있습니다. 즉 프론트 검증은 **사용자 편의용**이지 보안 경계가 아닙니다.

그래서 방어는 **계층별로 나눠서** 적용합니다.
- **백엔드(입력)**: 들어오는 값을 **검증·필터링**하고, 저장 시 위험 태그를 정제합니다.
- **백엔드(출력)**: 가장 중요합니다. 데이터를 화면에 내보낼 때 **반드시 이스케이프(인코딩)**합니다. Stored XSS는 결국 서버가 저장한 값을 다시 뿌릴 때 터지므로, 출력 처리가 핵심입니다.
- **프론트엔드**: **React/Vue의 자동 이스케이프**가 기본 방어선이 되고, 위험 API(`dangerouslySetInnerHTML` 등) 사용을 최소화합니다.
- **공통(브라우저 레벨)**: **CSP 헤더**로 실행 가능한 스크립트 출처를 제한하고, **HttpOnly 쿠키**로 토큰 탈취를 막습니다.

핵심은 **"입력은 신뢰하지 말고(백엔드 검증), 출력할 때 안전하게(이스케이프), 브라우저에서 한 번 더(CSP)"** 라는 다층 방어입니다. 실무에서는 어느 한쪽만으로는 절대 안전하지 않다는 게 결론입니다.

### 2. 꼬리 질문

1. **출력 이스케이프가 입력 필터링보다 중요하다고 보는 이유는 무엇인가요?**
   → 입력 필터링은 **저장 맥락**만 보지만, 같은 데이터가 HTML·JS·URL 등 **다양한 위치에 출력**될 수 있습니다. 결국 위험이 터지는 건 출력 시점이라, **출력 맥락에 맞는 인코딩**이 더 근본적입니다.

2. **CSP는 구체적으로 XSS를 어떻게 막나요?**
   → 허용된 스크립트 출처만 실행하도록 브라우저에 강제합니다. 인라인 스크립트나 외부 악성 스크립트 실행을 차단해, XSS 코드가 삽입돼도 **실행 자체를 막는 마지막 방어선**이 됩니다.

3. **DOM-based XSS는 백엔드에서 막기 어려운데 어떻게 대응하나요?**
   → DOM-based는 서버를 거치지 않고 **프론트 JS에서 발생**하므로 백엔드 검증이 닿지 않습니다. 그래서 `innerHTML` 대신 `textContent` 사용, 안전한 DOM API 활용, **CSP**로 프론트 단에서 직접 방어해야 합니다.
