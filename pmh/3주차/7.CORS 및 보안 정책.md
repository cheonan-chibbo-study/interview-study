# CORS 및 보안 정책
## NET-032 SOP 정책에 대해 설명해 주세요.

### 1. 답변

**SOP(Same-Origin Policy, 동일 출처 정책)** 는 브라우저의 가장 기본적인 **보안 정책**으로, **한 출처(Origin)에서 로드된 스크립트가 다른 출처의 리소스에 함부로 접근하지 못하도록 제한**하는 규칙입니다.

여기서 **출처(Origin)** 는 **프로토콜 + 도메인 + 포트** 세 가지로 결정됩니다. 셋 중 **하나라도 다르면 다른 출처**로 간주됩니다. 예를 들어 `https://app.com`과 `http://app.com`(프로토콜 다름), `https://app.com`과 `https://api.app.com`(도메인 다름)은 서로 다른 출처입니다.

왜 필요하냐면, 이게 없으면 사용자가 악성 사이트에 접속했을 때 그 사이트의 스크립트가 **사용자가 로그인해둔 다른 사이트(예: 은행)의 데이터를 마음대로 읽어갈 수 있기** 때문입니다. SOP는 이런 **출처 간 무단 접근을 막아** 쿠키·세션·DOM 등 민감한 정보를 보호합니다.

다만 SOP가 모든 통신을 막는 건 아닙니다. `<img>`, `<script>`, `<link>` 같은 태그를 통한 단순 로드는 허용되고, **JavaScript로 다른 출처의 응답을 읽는 것**(주로 `fetch`/`XHR`)을 제한합니다. 그런데 실무에서는 프론트엔드와 API 서버의 출처가 다른 경우가 많아서(예: `app.com` ↔ `api.app.com`), 이 제약을 안전하게 풀어주는 표준이 바로 **CORS**입니다.

---

### 2. 꼬리 질문

1. **출처(Origin)가 같다는 건 정확히 무엇을 기준으로 판단하나요?**
   → **프로토콜·도메인·포트** 세 가지가 모두 같아야 동일 출처입니다. 하나라도 다르면 다른 출처로 보고 SOP 제약을 받습니다.

2. **SOP가 있는데도 `<script>`로 외부 JS를 불러오는 건 왜 허용되나요?**
   → SOP는 **응답을 JS로 읽는 것**을 막는 것이지, 리소스 로드 자체를 막지는 않습니다. 그래서 CDN의 스크립트 로드는 되지만, 그 응답 내용을 코드로 가로채는 건 제한됩니다.

3. **SOP는 서버가 아니라 브라우저가 강제하는 정책인데, 그럼 우회가 가능한가요?**
   → SOP는 **브라우저 차원의 보호**라, Postman이나 서버 간 통신엔 적용되지 않습니다. 그래서 SOP만으로 충분하지 않고, 서버 측에서도 **인증·CSRF 토큰 검증** 등 별도 방어가 필요합니다.

## NET-033 CORS 정책이 무엇인가요?

### 1. 답변

**CORS(Cross-Origin Resource Sharing, 교차 출처 리소스 공유)** 는 SOP 때문에 막히는 **다른 출처 간의 요청을, 서버가 명시적으로 허용해주는 표준 메커니즘**입니다.

앞서 SOP가 출처가 다르면 JS의 응답 접근을 막는다고 했는데, 실무에선 프론트(`app.com`)와 API 서버(`api.app.com`)의 출처가 다른 경우가 대부분입니다. 이때 **서버가 응답 헤더로 "이 출처는 허용한다"고 알려주면** 브라우저가 통신을 허용하는 방식이 CORS입니다.

핵심은 **응답 헤더**입니다. 서버가 `Access-Control-Allow-Origin`에 허용할 출처를 명시하고, 필요하면 `Allow-Methods`, `Allow-Headers`, `Allow-Credentials` 등을 함께 내려줍니다. 브라우저는 이 헤더를 보고 접근 허용 여부를 판단합니다.

또 하나 중요한 게 **Preflight(예비 요청)**입니다. `PUT`/`DELETE`처럼 변경을 일으키거나 커스텀 헤더가 있는 요청은, 브라우저가 본 요청 전에 **`OPTIONS` 메서드로 "이 요청 보내도 되냐"고 먼저 물어봅니다**. 서버가 허용 응답을 주면 그제야 실제 요청을 보냅니다. 이를 통해 서버에 부담을 주기 전에 **안전성을 미리 확인**합니다.

실무에서 가장 흔한 함정은 **`Access-Control-Allow-Origin: *`와 `Allow-Credentials: true`를 함께 쓸 수 없다**는 점입니다. 쿠키·인증정보를 실어 보낼 때는 **와일드카드(*) 대신 출처를 명시**해야 하고, 보안상으로도 허용 출처를 구체적으로 지정하는 게 원칙입니다. 핵심은 **CORS가 보안을 "강화"하는 게 아니라, SOP 제약을 "안전하게 완화"하는 장치**라는 점입니다.

---

### 2. 꼬리 질문

1. **CORS 에러는 프론트엔드에서 해결하는 건가요, 백엔드에서 해결하는 건가요?**
   → **백엔드에서 해결**합니다. CORS 허용 여부는 **서버 응답 헤더**로 결정되기 때문입니다. 프론트는 요청만 보낼 뿐이고, 허용 출처·메서드·헤더 설정은 서버(또는 API Gateway)에서 합니다.

2. **Preflight 요청은 모든 요청에서 발생하나요?**
   → 아닙니다. `GET`이나 단순 `POST`(기본 헤더) 같은 **Simple Request**는 Preflight 없이 바로 갑니다. `PUT`/`DELETE`, 커스텀 헤더, `application/json` 등의 조건이면 **`OPTIONS` Preflight가 먼저** 발생합니다.

3. **CORS를 풀어주면 보안이 약해지는 것 아닌가요?**
   → CORS는 SOP를 **무분별하게 푸는 게 아니라 지정한 출처에만 제한적으로 허용**하는 것입니다. 다만 `Allow-Origin: *`를 인증 요청에 남발하면 위험하므로, **허용 출처를 명시하고 credentials 설정을 신중히** 하는 게 중요합니다.

## NET-034 Preflight에 대해 설명해 주세요.

### 1. 답변

**Preflight(예비 요청)** 는 CORS에서, 브라우저가 **실제 요청을 보내기 전에 "이 요청을 보내도 되는지" 서버에 먼저 물어보는 사전 확인 요청**입니다. 이때 **`OPTIONS` 메서드**를 사용합니다.

동작 흐름은 이렇습니다. 브라우저가 본 요청 전에 `OPTIONS` 요청을 보내면서, 헤더에 **`Access-Control-Request-Method`(어떤 메서드를 쓸지)**와 **`Access-Control-Request-Headers`(어떤 커스텀 헤더를 쓸지)**를 담아 묻습니다. 서버가 **`Access-Control-Allow-Origin`, `Allow-Methods`, `Allow-Headers`** 등으로 허용 응답을 주면, 그제야 브라우저가 **실제 요청을 전송**합니다. 허용되지 않으면 본 요청은 아예 보내지 않고 **CORS 에러**로 막힙니다.

모든 요청에 Preflight가 붙는 건 아닙니다. **Simple Request**(단순 요청) 조건을 만족하면 생략됩니다. `GET`, `HEAD`, 또는 `Content-Type`이 `application/x-www-form-urlencoded`·`multipart/form-data`·`text/plain`인 단순 `POST` 등이 여기 해당합니다. 반대로 **`PUT`/`DELETE`, `application/json` 본문, 커스텀 헤더(예: `Authorization`, 토큰 헤더)** 가 있으면 Preflight가 발생합니다. 실무에서 REST API는 대부분 `application/json`을 쓰므로 **Preflight가 흔하게 일어납니다**.

왜 필요하냐면, **서버 상태를 바꾸는 위험한 요청이 실제로 실행되기 전에 미리 안전성을 검증**하기 위해서입니다. CSRF처럼 의도치 않은 교차 출처 요청이 서버에 곧장 도달하는 걸 막는 역할도 합니다. 다만 단점은 **요청마다 왕복이 한 번 더 늘어나 지연이 생긴다는 것**입니다. 그래서 서버가 **`Access-Control-Max-Age`** 헤더로 Preflight 결과를 일정 시간 **캐싱**하게 해서, 같은 요청에 대한 반복적인 OPTIONS 호출을 줄이는 게 실무 최적화 포인트입니다.

---

### 2. 꼬리 질문

1. **어떤 경우에 Preflight가 발생하고, 어떤 경우엔 생략되나요?**
   → `GET`·단순 `POST`처럼 **Simple Request 조건**을 만족하면 생략됩니다. `PUT`/`DELETE`, `application/json` 본문, **커스텀 헤더**가 붙으면 Preflight가 발생합니다. 요즘 JSON 기반 API는 대부분 발생한다고 보면 됩니다.

2. **Preflight 때문에 생기는 성능 부담은 어떻게 줄이나요?**
   → 서버가 **`Access-Control-Max-Age`**로 Preflight 응답을 브라우저에 캐싱시키면, 유효 시간 동안 반복 OPTIONS 요청을 생략할 수 있습니다. 또 출처·헤더 설정을 단순화해 불필요한 Preflight를 줄입니다.

3. **Preflight 요청에는 왜 인증 쿠키나 본문을 담지 않나요?**
   → Preflight는 **"보내도 되는지 확인만"** 하는 단계라 실제 데이터·인증 정보를 싣지 않습니다. 본 요청이 허가됐을 때 비로소 쿠키·본문이 함께 전송되며, 자격증명을 쓰려면 서버가 **`Allow-Credentials: true`**를 응답해야 합니다.
